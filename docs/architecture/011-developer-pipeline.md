## 1. Главный принцип

Архитектура НЕ копирует человеческую команду (Аналитик → Ресерчер → Планировщик → Разработчик → Ревьюер). Люди делят работу так, потому что работают параллельно и не могут передать друг другу мозг. У LLM-агента ограничение обратное: **каждый handoff между ролями — потеря контекста** (сжатие живого понимания в артефакт, который следующий агент перечитает и пойдёт перепроверять теми же инструментами, оплачивая те же токены второй раз).

Единственное разделение с настоящим архитектурным обоснованием — **адверсариальное**: проверяющий не должен разделять состояние и убеждения делающего.

Отсюда правильный вопрос архитектуры: не «как организовать команду агентов», а **«как дать одному сильному агенту проверяемую среду и накопленную память проекта»**. Контекстная машина проекта (Fact Store, глоссарий, Observer-граф, code graph, семантический индекс) — конкурентное преимущество продукта — подключается к работающему агенту **инъекцией в цикл** (механизмы `knownFactsHint`/`observerHint`/`semanticSearch`/`findReferences` уже существуют и проверены), а не через промежуточного агента-переводчика.

## 2. Что отвергнуто и почему

Зафиксировано, чтобы будущие сессии не переизобретали отвергнутое:

| Отвергнуто | Причина |
|---|---|
| Отдельный Researcher-этап перед разработкой (Change Brief как межролевой контракт) | Developer имеет те же read-инструменты; brief либо избыточен, либо опасен (устаревший brief, которому поверили). Память инъецируется в цикл напрямую. |
| Отдельный Planner-агент и Execution Plan как версионируемый DAG с rollback-планом на шаг | Микро-декомпозиция — часть мышления пишущего. Роль чекпоинтов и отката выполняет git (commit на milestone, reset). Спеки planner.md/execution-engine.md — YAGNI. |
| Утверждение плана человеком как обязательный гейт | Планы всегда выглядят правдоподобно, проблемы видны в diff. Человек проверяет **результат**, а не намерение (исключение — необратимые операции, раздел 5). |
| Кросс-вендорность Reviewer как жёсткий инвариант | Для ревью кода ограничивающий фактор — способность модели к коду. Правило: **Reviewer ≥ Developer по code-способности**; кросс-вендорность — желательный бонус. Слабый чужой ревьюер генерирует шумные findings → fix-циклы → деградация кода. |
| Pre-code Impact-гейт по графу | Impact по *кандидатам* до существования кода — спекуляция поверх догадки; на легаси-графах даёт ложную уверенность. Настоящая ценность impact — **после** diff (изменённые символы → вызыватели → какие тесты гонять). |
| Детерминированная scope-сверка «diff не вышел за файлы плана» как автостоп | Честная разработка открывает файлы по ходу. Выход за ожидания — сигнал для Reviewer («объясни, зачем тронут этот файл»), а не правило. |
| Пошаговый гейт build/test после каждого шага плана | Предполагает независимую компилируемость шагов — часто ложно. Developer с `run_command` сам решает, когда верифицировать, как живой разработчик. |
| Decision-запись (ADR) на каждую задачу | Кладбище, которое никто не читает. Записывать только реально спорные решения. |

## 3. Три LLM-роли

1. **Developer** — агентный цикл (расширение проверенного `runAgenticLoop`): сам исследует (те же read-инструменты + инъекция памяти), сам решает микро-декомпозицию, сам пишет (мутационные инструменты), сам верифицирует (`run_command`). Уточняющий вопрос при неясном ТЗ — его **режим**, не отдельная роль; порог для change-задач жёстче, чем для вопросов (цена ошибки в коде выше).
2. **Reviewer** — независимая проверка: получает **задачу + diff + журнал верификации**. НЕ получает план/заметки Developer'а — иначе проверяет «соответствие плану» и слепнет к сценарию «план сам неверный» (коррелированная слепота). Тот же архетип, что Critic в Q&A-пути — один механизм, разные промпты.
3. **Observer/Curator** — существующий фоновый Observer + обязанности куратора памяти: после принятой задачи — промоушен новых фактов, пометка противоречащих (`contradicted`), пере-обход затронутых директорий. Anti-oscillation: contradicted-факт не пере-промоутится без нового свидетельства.

Модели: роли Developer/Reviewer — **отдельные поля Team** (`developer_model`/`reviewer_model` в Postgres, редактируются с фронта наравне с остальными ролями; реализовано 2026-07-17). Изначальный v1-план «переиспользовать researcher/critic» отменён в тот же день: телеметрия первых же живых ранов показала, что critic-модель (частые дешёвые вызовы, латентность) и reviewer-модель (редкие вызовы, цена ошибки — пропущенный баг) — противоположные требования, и flash-lite-критик в роли ревьюера дважды выдал фактически неверные findings. Fallback для команд без явных значений: Developer = researcher-модель (тот же архетип), Reviewer = `moonshotai/kimi-k2.7-code` (code-специализация среднего ценового диапазона, 009 принцип 5). Живой прогон с Kimi-ревьюером: APPROVED с первого раунда без шумных замечаний на задаче, эквивалентной тем, где flash-lite ошибалась.

## 4. Pipeline

```
Задача (change-задача)
 → режим ясности ТЗ: неясно → ОДИН уточняющий вопрос (существующий UI-механизм)
 → изоляция: git worktree на задачу (чекаут пользователя не трогается)
 → Developer-цикл: исследует → правит → run_command (build/lint/tests, когда доступны)
 → детерминированная верификация фиксируется в журнал (что запускалось, exit codes)
 → Reviewer: задача + diff + журнал верификации → APPROVED | NEEDS_CHANGES(findings)
 → NEEDS_CHANGES → ОДИН fix-раунд Developer'а (findings как вход; вправе аргументированно
   оспорить finding — разногласие не «дочинивается», а поднимается человеку)
 → повторный вердикт финален; двойной провал → человек
 → результат человеку: diff + вердикт + журнал верификации + (если есть) разногласие
 → merge выполняет человек (v1) → Knowledge Update (Curator)
```

**Остановки ошибок:** одинаковый провал дважды → стоп, человеку (никаких «ещё раз»); верификация невозможна (нет тестов/окружения) → честное понижение заявляемых гарантий в результате, а не молчаливый пропуск гейта; конфликт Developer/Reviewer → человеку. Бюджетные потолки цикла (turns/tokens/stuck-детектор) наследуются от research-цикла.

**Человек обязателен ровно в трёх местах:** разрешение неоднозначности ТЗ; авторизация merge и необратимых операций (миграции данных, внешние эффекты); разбор тупика. Всё остальное — без человека.

## 4.1 Вход через чат (реализовано 2026-07-17)

Пользователь пишет вопросы, задачи разработки и корректировки в **один и тот же чат** — различает их система, не пользователь. `/api/pipeline/run` перед Q&A-путём вызывает `classifyChatIntent` (`packages/ai`, LLM-классификация critic-моделью Team, не keyword-regex: «как добавить X?» — вопрос, «добавь X» — задача). Fail-closed в «question»: неверно понятый вопрос стоит одного research-рана, неверно понятая задача мутировала бы worktree без спроса.

Корректировка («он сделал, я проверил, сказал переделать») продолжает **последний develop-run того же conversationId**: новый ран входит в тот же worktree с уже доставленными изменениями (`continueFrom`), diff остаётся кумулятивным против исходного baseline. Ответ на уточняющий вопрос Developer'а продолжает разработку без классификации. Параллельное сообщение при живом develop-ране того же диалога отклоняется (409) до завершения.

Живой E2E (2026-07-17) подтвердил все три маршрута, включая кейс спора: Reviewer (слабая flash-lite модель) выдал фактически неверный finding, Developer аргументированно оспорил, вердикт ушёл человеку — механика эскалации отработала, но это живое подтверждение правила «Reviewer ≥ Developer по code-способности» из раздела 2: reviewer-модель дефолтной Team стоит усилить.

## 4.2 Бенчмарк на реальных PM-тикетах и временное ужесточение git (2026-07-17)

Прогон трёх реальных тикетов magendamd (PM-сформулированные ТЗ, не синтетика) против эталонных веток дал 0 из 3 задач на уровне эталона, с двумя серьёзными находками:

- **MD-1332**: Developer исследовал только фронтенд-репозитории (там фича уже была видна в UI) и ни разу не заглянул в собственный целевой backend-репозиторий — сделал ложный вывод «уже реализовано» с пустым diff, хотя нужный backend-эндпоинт на `master` не существовал.
- **MD-1474**: summary заявил правку `CardStatus.php`, которую Developer фактически не делал (`git diff` пустой) — расхождение между отчётом и реальным diff, а не просто неполнота.
- **MD-1498**: рабочий, но архитектурно иной код — Developer не нашёл существующую в кодовой базе конвенцию (`cloneFromMainClinic()`), использованную в эталонах и MD-1474, и MD-1498, несмотря на явную инструкцию «изучи похожие места перед тем как писать».

Вывод: система ещё не в той стадии зрелости, чтобы результат мержился без построчного человеческого ревью — это не претензия к архитектуре, а фиксация текущего уровня, который надо поднимать с помощью телеметрии `developer_runs`.

По итогам этого прогона, как временная страховка (владелец продукта, явный запрос): `runShellCommand` (`packages/agentic-research/src/tools.ts`) блокирует **все git-подкоманды, изменяющие состояние** (`commit`, `push`, `merge`, `rebase`, `reset`, `checkout`/`switch`, мутирующие `branch`/`tag`, `update-ref`, `symbolic-ref`, `remote`, `worktree`, `filter-branch`, `reflog`), а не только `push`. Обоснование строже, чем «push выглядел рискованным»: worktree разделяет object/ref-базу с реальным чекаутом пользователя, поэтому команда, выглядящая «внутри worktree», в реальности может подвинуть общий `refs/heads/master` (`git branch -f master <sha>`). Это не убавляет возможностей Developer'а: сбор diff (`collectWorktreeChanges`) и cleanup идут отдельным путём — прямыми вызовами `runGit` в `packages/repository-git`, не через `run_command`, — и никогда не зависели от того, что модель сама коммитит. Смягчение этого ограничения (например, разрешить `git commit` внутри worktree) — осознанное решение на будущее, когда накопится статистика точности по `developer_runs`, а не технический долг, который надо снять при первой возможности.

## 5. Артефакты

Необходимые: **Task Spec** (задача + результат уточнений + acceptance-критерии — единственный носитель намерения), **Diff + журнал верификации** (что прогнали, что прошло/упало), **вердикт Reviewer** (findings со статусом: принят/исправлен/оспорен).

Не заводятся: Change Brief, Execution Plan как персистентный артефакт, pre-code Impact Report, ADR-на-задачу.

## 4.3 Reuse-инструменты + надёжность edit_file (2026-07-18)

По явному запросу владельца продукта («реализация должна больше переиспользовать существующий функционал» + «не сломать существующий функционал») сделаны три связанных улучшения:

- **`semantic_search`/`find_references` подключены в Developer-цикл.** Были объявлены в протоколе с самого начала (develop-loop.ts), но `develop-runner.ts` их физически не передавал — весь пройденный ранее бенчмарк MD-1332/1474/1498 шёл только на `grep_content`/`read_file`. Переиспользует те же `buildSemanticSearchTool`/`buildGraphNavigationTool`, что и Q&A-путь (экспортированы из `pipeline-runner.ts`); `findReferences` — best-effort загрузка персистентного графа по тому же `stableId(["project", ...])`, что и Q&A, деградирует без графа честно (инструмент просто не объявляется). Живой повтор MD-1498 подтвердил: модель сама, без единого совпадения по буквальным словам задачи, вызвала `semantic_search` и нашла, что `VisitType::cloneFromMainClinic($clinic->id)` уже вызывается в `CreateClinicAction.php` на `master` — то есть корректно обнаружила существующую конвенцию, которую в оригинальном бенчмарке пропустила.
- **Промпт Developer'а** — новое правило (2): перед изменением тела/сигнатуры существующей функции — `find_references`/`grep_content` на предмет вызывающего кода, и явное требование указать в summary, что было проверено на breaking changes.
- **Обнаруженная и исправленная проблема надёжности:** на реальном 391-строчном файле (не встречалась на маленьких синтетических тестах) `edit_file` дважды подряд свалился в слепой повтор — 13+ и 15+ неудачных попыток без единого успешного изменения, до аварийной остановки по токен-лимиту (~560K токенов впустую, но diff пуст — упало безопасно, ничего не сломано). Добавлено: (1) детектор застревания на уровне файла — после 3 подряд неудач `edit_file` файл принудительно исключается из «прочитанных», что заставляет модель перечитать его заново вместо угадывания вслепую; (2) `editFile` при провале теперь показывает **реальный ближайший фрагмент файла** (по word-overlap первой существенной строки SEARCH-блока), а не голое «not found» — модель видит, чем её догадка отличается от истины, вместо повторения той же ошибки. Оба узких тест-кейса (совпадение среди 9 похожих строк вида `X::cloneFromMainClinic(...)`, полностью несвязанный SEARCH-блок) проверены изолированно.

## 4.4 Тестовая методология: Slay вместо Magenda (2026-07-18)

Magenda — реальный клиентский проект с задачами от PM, использовался как первичный бенчмарк (раздел 4.2), но по решению владельца продукта **выведен из активных тестовых прогонов**: тяжело ориентироваться при разборе результатов, и это чужой прод с риском для реального релизного цикла (см. пятничный инцидент осторожности — временный запрет git write-операций для LLM в разделе выше). Slay — собственный проект владельца, без реального PM-беклога — становится основным полигоном для дальнейшей разработки и тестирования пайплайна; задачи для него формулируются самим владельцем или придумываются в рамках сессии, эталон — не git-ветка, а «как сделал бы я» от владельца.

**Эталонная тестовая задача: цветовая схема GUI.** Формулировка (как обычный тикет, без деталей реализации) — дать пользователю возможность выбрать акцентный цвет интерфейса, сохранить как user setting на бэке с дефолтом для новых и существующих пользователей, восстанавливать при старте GUI даже до загрузки профиля с сервера. Владелец продукта заранее расписал, как сделал бы сам (`InfoDropdown.vue` → `UserSettingEnum`/`Colors` enum → `UserController`/`SettingController`/`UpdateSettingRequest` → `user_settings.ts` → `appStore`/`settings-store.ts` → CSS → `UserSettingsSeeder`).

Пять живых прогонов подряд дали ценную, но неровную картину:
- **Exploration — подтверждён дважды на разных реальных проектах.** Без единого упоминания конкретных файлов в тексте задачи модель самостоятельно, через `semantic_search`, нашла **практически весь** список владельца: `settings-store.ts`, `InfoDropdown.vue`, `SettingController.php`, `UserSettingEnum.php`, `UpdateSettingsDTO.php`, `user_settings.ts`, `app-store.ts` (оба — `src/stores` и `src-electron/store`), плюс сама обнаружила `Service`/`Repository`-слой бэка (владелец его не упоминал намеренно — «и так знал, что переиспользуется») и **не тронула его**, что и требовалось.
- **`edit_file` — два РАЗНЫХ, оба реальных failure mode.** На магендовском 391-строчном файле — застревание из-за плотности похожих строк (закрыто nearest-match hint'ом). На слеевских маленьких файлах (`user_settings.ts` — 1 строка, `app-store.ts` — 21 строка) — та же слепая петля, но текстовые подсказки («используй write_file», «вот ближайшее совпадение») модель **игнорировала** 2 прогона подряд. Решение — не третья подсказка текстом, а изменение семантики самого инструмента: **whitespace-tolerant fallback** в `editFile` (построчное сравнение с нормализацией пробелов/отступов, применяется только при уникальном совпадении — двусмысленность явно репортится, а не угадывается). 14 изолированных тестов, включая точное воспроизведение сценария с 9 похожими строками и однострочным файлом без завершающего перевода строки.
- **Отдельный, более серьёзный баг найден по пути:** при выходе рана по лимиту токенов/ходов **без** вызова `task_complete`, diff никогда не собирался — реально применённые правки молча терялись, а `develop-runner.ts` удалял worktree как «пустой». Живой прогон поймал это: `onProgress` показывал `filesChanged: 1`, а финальный результат — `changedFiles: []`. Исправлено: `finalize()` теперь на любом non-task_complete выходе, если есть успешно отредактированные файлы, сначала пытается собрать реальный diff, и только потом возвращает результат — частичный прогресс больше не исчезает молча.
- **Gotcha-память.** По ходу тестирования владелец продукта заметил вслух, что даже сам, как автор фичи, не сразу понял, что GUI может стартовать раньше, чем успеют подгрузиться данные пользователя с сервера — это вскрылось только на этапе доработки бэка. Отсюда расширение уже работающего механизма памяти паттернов (раздел 4.3): `extractCodePatternFacts` теперь фиксирует не только конвенции («как здесь принято»), но и **gotchas** («на чём здесь легко обжечься») — тем же путём, через `project_facts`, с тем же гейтом (только из approved-ранов).

**Прогон 5 (после обоих `edit_file`-фиксов) подтвердил главное — но вскрыл более приоритетную проблему.** `finalize()`-фикс сработал вживую: ран снова упёрся в `max_turns` без `task_complete`, но `changedFiles` в результате оказался непустым (реальная правка `user_settings.ts` сохранена, а не потеряна). Whitespace-tolerant matching, судя по всему, тоже сыграл роль — правка маленького файла наконец прошла. Но задача **опять не дошла до Reviewer**.

**Задача A (Server tags, тест DB-нормализации) — намеренно спровоцировала антипаттерн, но так и не показала его Reviewer'у.** Тикет явно требовал many-to-many («один тег — много серверов, один сервер — много тегов»). Модель добавила `tags` в `fillable` модели `Server` и в repository-инсерт **без единой миграции** — колонки `tags` не существует в БД вообще, этот код упал бы на реальном запуске SQL-ошибкой, и даже если бы миграция была — денормализованное поле прямо противоречило явному требованию many-to-many. Это ровно тот случай, для которого построен DB-normalization чек-лист Reviewer'а (раздел 4.3) — но ран снова упёрся в `max_turns` (27 ходов, 5 изменённых файлов, ~620K токенов) до вызова `task_complete`, и Reviewer это не увидел.

**Итог: 6 из 6 живых прогонов на Slay сегодня уперлись в `max_turns`, ни один не дошёл до Reviewer.** Это не совпадение и не тот же баг, что раньше — это системный паттерн, который важнее любого точечного фикса `edit_file`. Multi-root Slay (4 физических репозитория) требует заметно больше ходов на исследование, чем single-root Magenda, и текущий бюджет (`DEFAULT_DEVELOP_CEILING_TURNS`/`DEVELOP_TOKEN_SAFETY_LIMIT` в `develop-loop.ts`) явно не рассчитан на это. Вторая заготовленная тестовая задача (audit-событие при revoke `PasswordShare`, реиспользование конвенции логирования из `VaultCryptoService.php`) **осознанно не запущена** — с высокой вероятностью упёрлась бы в ту же стену, а туда же тратить бюджет вслепую шестой раз подряд нецелесообразно. Следующий шаг — не очередной точечный фикс, а пересмотр бюджета ходов/токенов для multi-root develop-ранов (либо поднять потолок, либо разбить крупную задачу на реально независимые под-шаги с промежуточным `task_complete`/review на каждом), прежде чем расширять тестовый набор дальше.

## 4.5 Efficiency-паритет с research + DB safety + доведение edit_file (2026-07-18, продолжение)

По итогам раздела 4.4 («6 из 6 упёрлись в max_turns») и явного требования владельца продукта («мы не бюджетировали ресерчера, мы оптимизировали его — сделай так же для Developer, плюс осторожность при работе с БД как у любого вменяемого агента: тестовые запросы через транзакцию с откатом, чувствительные действия — через апрув человека») сделан следующий, самый крупный блок работы за сессию.

**Efficiency-паритет.** `develop-loop.ts` унаследовал от research-цикла seed-grep, но не `semanticSeedFiles` (auto-read топ-файлов по смыслу до первого хода — «единственный измеренный рычаг задержки» по словам самого research-кода) и не общий stuck-детектор (только точечный, под `edit_file`). Оба перенесены: `semanticSeedFiles` подключён через тот же `buildSemanticSeedLookup`, что и Q&A-путь (экспортирован из `pipeline-runner.ts`); общий детектор (`STUCK_TURNS_THRESHOLD = 10`, hard-abort на +5) считает прогресс по `touchedFiles + seenDirs + grepTermsSeen + editedFiles` — правки теперь тоже засчитываются как прогресс, чего у research нет по конструкции (там нет записи).

**DB safety.** До этой правки `run_command` пропускал вообще любую команду, включая `php artisan migrate` — реальный, ранее не замеченный риск: если бы Developer решил накатить миграцию, ничего бы его не остановило. Теперь: детектор чувствительных БД-команд (`isSensitiveDbCommand` — generic по эко-системам: Laravel/Rails/Django/Alembic/Flyway/Prisma/Knex/TypeORM + деструктивный DDL) перехватывает такую команду ДО выполнения и останавливает весь ран (`stopped: "needs-approval"`, `pendingApproval` с точной командой и причиной от модели) — worktree сохраняется, не удаляется. Подтверждение идёт тем же чат-каналом, что и уточняющие вопросы: `classifyApprovalResponse` (fail-closed на «непонятно» — без однозначного да ничего не выполняется), одобренная команда выполняется оркестратором НАПРЯМУЮ в сохранённом worktree (не пересылается модели на повторный ввод — что видел человек, то и выполняется), результат (включая реальный вывод, не только exit code — это было отдельным найденным и исправленным багом) уходит в `sensitiveActions` и прокидывается в контекст продолжения. Промпт учит модель: `--pretend`/`--dry-run` не блокируются (это уже безопасный предпросмотр), транзакция+rollback через `DB::beginTransaction()`/`START TRANSACTION` — нормальный, неблокируемый путь для тестовых запросов.

Живой прогон подтвердил всю цепочку целиком, трижды подряд, в одном непрерывном диалоге: `php artisan migrate:status` → апрув → выполнение (Laravel сам отказался — "APPLICATION IN PRODUCTION" — по недостающему `APP_ENV=local`) → модель **увидела реальный вывод и сама адаптировалась**, добавив `APP_ENV=local` и `--pretend` → апрув → успешный dry-run с полным SQL-превью всех миграций. По пути найдены и закрыты две реальные неточности классификатора: `migrate:status` (read-only, ложно спамился как sensitive из-за substring-совпадения) и `--pretend`/`--dry-run` (безопасные флаги, не должны требовать апрува вообще) — обе пойманы ИМЕННО живыми прогонами, не придуманы заранее.

**Доведение edit_file (продолжение раздела 4.3).** Оригинальный дизайн детектора застревания (эвикция файла из `touchedFiles` + инструкция «перечитай») предполагал, что модель послушается инструкции. Живое свидетельство (второй реальный застревание на Slay, `ServersRepository.php`) показало: не послушалась — 13+ попыток `edit_file` подряд без единого промежуточного `read_file`, каждая проваливалась одинаково (эвикция без последующего чтения = постоянная стена «you have not read this file», которую модель просто игнорировала и продолжала биться). Исправлено на детерминированный путь: после порога неудач актуальное содержимое файла **впрыскивается напрямую** в наблюдение (без пропуска через общий лимит усечения), а не только упоминается в тексте инструкции — не полагается на послушание модели вообще. После фикса тот же класс застревания (Server.php, Profile.php) на повторном прогоне разрешился успешно; на новом узком месте (`ServersRepository.php`) снадобье сработало медленнее, но тоже без вечного тупика.

**Итог по эффективности:** повторные прогоны задачи A показывают устойчивый рост (2 → 6 файлов изменено за прогон, включая корректно спроектированную two-table many-to-many миграцию дважды подряд — DB-normalization осведомлённость подтверждена не один раз), но полный `task_complete`/Reviewer на самой сложной, многофайловой задаче по-прежнему не достигнут в рамках текущего бюджета ходов/токенов — открытый вопрос о пересмотре бюджета для multi-root задач (раздел 4.4) остаётся актуальным и не решён этой правкой, только облегчён.

## 4.8 Цикл «фикс → тест → оценка» ×5 на раскрытом бюджете (2026-07-18)

Владелец явно потребовал не останавливаться после раздела 4.7 (использовано 473К из 51М дневных токенов) и прогнать структурированный цикл: 5 итераций, каждая — фикс, тест на модели, оценка что чинить дальше, — затем один финальный отчёт.

**Итерация 1.** В `REVIEWER_SYSTEM_PROMPT` добавлены пункты (7) отсутствие миграции под новую персистентную колонку и (8) проверка acceptance-evidence (журнал только со статическими проверками не доказывает, что описанное в задаче поведение реально работает). Тест на V3.2 подтвердил: Reviewer стал ловить оба класса находок вживую.

**Итерация 2.** Сообщения `formatError` в `parseDevelopActions` расширены: теперь называют конкретный отсутствующий блок и прикладывают `segmentSnippet` (первые 200 символов реально присланного текста) — раньше и модель, и `actionsLog` видели только факт ошибки, не её форму.

**Итерация 3.** В системный промпт Developer'а (`buildDevelopSystemPrompt`) добавлен полный конкретный пример корректного вызова `edit_file` на реальном PHP-коде с явным указанием, что маркеры должны стоять на отдельной строке. A/B-тест на GLM-4.7: `FORMAT_ERROR` ушёл с ~10 случаев до 0 на PHP-файлах.

**Итерация 4.** Гипотеза «не хватает бюджета ходов» — `DEFAULT_DEVELOP_CEILING_TURNS` поднят с 60 до 90, `DEVELOP_TOKEN_SAFETY_LIMIT` с 600К до 950К. Решающий тест (задача с тегами серверов, V3.2, `.vue`-файлы GUI) остановился на **29-м ходу из 90 возможных** — упёрся не в потолок ходов, а в токен-лимит: ~35К токенов/ход из-за 14 `FORMAT_ERROR`, почти все на `SshTab.vue`/`AddEditServerDialog.vue` (500–600 строк). Диагностика по логам: `SEARCH`-блок парсился нормально, а закрывающий `REPLACE>>>`/`CONTENT>>>` не приходил вообще. **Настоящая причина — не поведение модели**, а инфраструктурный лимит: `packages/agentic-research/src/provider.ts`'s `MAX_COMPLETION_TOKENS = 4000` — общий для research- и develop-луп, `chat/completions`-вызовов. Полная перезапись/большой REPLACE-блок для файла в 500+ строк физически не помещается в 4000 токенов ответа — провайдер молча обрезает completion до закрывающего маркера, и парсер (корректно, но вводяще в заблуждение) репортит «маркер отсутствует». Вывод: подъём бюджета ходов раунда 4 был не лишним, но не бил в причину — причина инфраструктурная, не поведенческая.

**Итерация 5.** Фикс: `callModel`/`performCall` в `provider.ts` получили необязательный параметр `maxCompletionTokens` (по умолчанию сохраняет старые 4000 для research-луп и Reviewer'а) — эта часть осталась в коде как инфраструктура. Первая попытка — вызов Developer'а поднят до 16 000 токенов на ответ. Тест **дважды подряд** (runId `1f7b3e781...`, затем `26cfbe901...`) детерминированно ломался уже на 2-м ходу: DeepSeek V3.2 через этот провайдер вместо нашего ACTION-протокола начинал эмитить свои нативные служебные токены tool-calling (`<｜｜DSML｜｜tool_calls>...`) — 0 файлов изменено, полный останов прогресса. Контрольный прогон с возвратом на прежние 4000 токенов (runId `f0ef3314f...`) прошёл штатно: 32 хода, **13 изменённых файлов**, `FORMAT_ERROR` — всего 3 из 32 (было 14 у iteration 4). **Вывод: подъём лимита ответа до 16К не безопасен с этой связкой модель+провайдер — откачено**, `DEVELOP_MAX_COMPLETION_TOKENS`-константа убрана, вызов вернулся на общий дефолт 4000. Причина обрезания больших файлов (раздел итерации 4) реальна, но упирание в токен-бюджет на 4000 в первую очередь идёт не от обрезания ответа, а от **неограниченного роста промпт-токенов внутри одного прогона** (974К prompt-токенов уже к 32-му ходу — это и есть настоящий триггер `DEVELOP_TOKEN_SAFETY_LIMIT`). Обрезание истории разговора / суммаризация старых ходов — реальный следующий шаг, но за пределами этого пятииерационного цикла.

## 4.9 Debugger-флоу: баг простыми словами → причина → варианты → фикс по команде (2026-07-18)

Продолжение раздела 4.8. Владелец продукта: разработчик должен уметь не только писать фичи, но и находить причину бага, который пользователь описывает "максимально непонятно, простыми словами" — объяснить причину по-человечески (не только код), затем по запросу дать варианты фикса по пунктам, и **чинить только по явной команде**, никогда самостоятельно.

**Ключевое архитектурное решение: отдельный "debug-loop" НЕ понадобился.** Исследование существующего кода показало, что Q&A-пайплайн уже содержит почти всю нужную инфраструктуру:
- `looksLikeDiagnosticTask`/`classifyQuestionShape` уже распознаёт баг-репорты ("не работает", "ошибка", "почему") как diagnostic-questions;
- `buildAnswerSystemPrompt` уже пишет ответы в стиле "живой сеньор в чате", без секций-шаблонов — именно то, что нужно для "простыми словами";
- `conversationTranscript` (последние 3 реплики диалога) уже прокидывается в синтезатор ответа — многоходовый разговор ("какие варианты фикса?" после диагноза) уже работает без доп. кода;
- `loadLatestConversationTurn` уже хранит последнюю реплику диалога в Postgres (`knowledge_catalog`).

Значит баг-репорт **это уже обычный вопрос** для системы — отдельная классификация intent'а не нужна. Понадобилось три точечных изменения:

1. **Расширен `looksLikeChangeTask`** (`packages/ai/src/index.ts`) — добавлены "вариант", "почин", "как решить", "план фикс", чтобы "какие варианты фикса?" резолвился в `plan-summary-answer` (numbered-план в конце), а не в обычный лоцирующий ответ.

2. **Найден и исправлен реальный баг в существующем Q&A-пайплайне** (не новый функционал, обнаружено живым тестом): `shouldForceEvidenceLockedMode` использовала ОДИН флаг `evidenceLocked` для двух разных целей — (а) пропустить уточняющий вопрос при неоднозначности (верно для диагностики: баг легитимно затрагивает несколько модулей сразу) и (б) блокировать LLM-синтез, откатываясь на детерминированный шаблон. Diagnostic-режим корректно требовал (а), но как побочный эффект получал и (б) — **любой баг-репорт получал робо-ответ с секциями "Что это заденет"/"Что может пойти не так" вместо живой прозы**, даже с точной, хорошо обоснованной причиной. Живой тест: диагноз бага с дублирующимися portforwarding-записями получил идеальную по содержанию, но шаблонную по форме причину. Исправлено — `shouldForceEvidenceLockedMode(input, includeDiagnosticSignal)`: ambiguity-skip держит старое поведение, LLM-gate больше не блокируется одним фактом diagnostic-режима. Ретест — тот же баг, `synthesis: "llm"`, естественная проза с точным указанием строки и причины.

3. **Прокидывание контекста в команду фикса** (`apps/api/src/app.ts`): "делай пункт 2" само по себе ничего не значит без диалога, который к нему привёл. При старте СВЕЖЕГО `develop`-намерения (не `develop-correction`) роутинг теперь подтягивает `loadLatestConversationTurn` для того же `conversationId` и вставляет task+explanation предыдущей реплики как контекст перед составлением задачи разработчику. Классификатор `classifyChatIntent` не менялся — "делай X" и так соответствует его определению "develop" (императивная команда), это именно то, что не даёт чинить без явной команды: обычный вопрос про баг или про варианты фикса остаётся `question` и никогда не запускает Developer-пайплайн.

**Живой тест на 2 синтетических багах** (внедрены как uncommitted-правки в одноразовых git worktree от slay-api/slay-gui на выброшенных ветках `debug-test/*`, ни одна правка не попала в реальные ветки — все worktree и ветки удалены по завершении):
- **Баг 1 (backend, `ServersRepository::updateServer`)**: удаление старых `forwarding_ports` использовало отсутствующий ключ `$server['id']` вместо `$id` — тихо превращалось в `where('server_id', null)`, порты копились при каждом редактировании.
- **Баг 2 (frontend, `ssh-store.ts::syncServers`)**: сверка `connection_id` при обновлении списка серверов матчилась по `host` вместо `id` — при нескольких серверах на одном хосте активная SSH-сессия могла "прилипнуть" не к тому серверу.

Результат по обоим багам одинаковый: диагноз попал в точную причину и точную строку кода с первого раза, "какие варианты фикса" дал 4–5 пронумерованных вариантов с оценкой риска (в обоих случаях сам назвал верный "минимальный" вариант фаворитом), "делай пункт N" корректно классифицировался как `develop`, задача корректно собиралась с полным контекстом диалога, итоговый фикс **в обоих случаях был кодово идентичен правильному оригиналу**.

## 4.20 Дефолтные локальные порты синхронизированы: frontend 3030, backend 3031 (2026-07-20)

По прямому запросу владельца продукта локальные дефолты запуска разведены с типовыми портами других проектов: dev-сервер веб-интерфейса переведён с `3000` на `3030`, а API — с `3001` на `3031`. Вместе с этим синхронизированы обе фронтовые точки доступа к API: Vite proxy (`/api`) и fallback `API_BASE_URL` в `App.tsx` теперь смотрят на `http://127.0.0.1:3031` / `http://localhost:3031`. Preview-конфиг не менялся. Практический смысл — убрать постоянные локальные конфликты портов при параллельной работе с другими проектами.

**Побочная находка при первом фиксе бага 1 (ценная сама по себе):** Reviewer справедливо потребовал acceptance-evidence (не только моки), Developer в ответ на это переписал 3 НЕ относящихся к задаче файла моделей и начал строить новые test-фабрики/миграции с нуля, чтобы дотянуть до полноценного DB-теста — потратил 45 ходов и 953K токенов, не дошёл до `task_complete`. Проблема: reviewer-finding про качество теста не масштабировал ответ пропорционально размеру исходного фикса (1 строка). Добавлена явная инструкция в fix-round-сообщение Developer'а (`develop-loop.ts`): находка про тесты не даёт права расширять scope на неизменённые файлы/новую инфраструктуру — предпочесть усиление теста существующими средствами или оспорить несоразмерный запрос. Ретест того же бага с той же формулировкой причины: **26 ходов (было 45), 475K токенов (было 953K)** — Developer сделал точечный mock-based тест, сам эмпирически доказал что тест ловит баг (временно вернул старый код, показал что тест падает), и когда Reviewer всё равно не согласился с моками — **корректно оспорил** вместо того чтобы продолжать копать. Два раунда ревью, обе стороны имеют разумную позицию (Mockery-ожидания технически являются assertion'ами, но не эквивалент реального DB-поведения) — система верно остановилась и передала решение человеку, а не зациклилась.

## 4.10 Подрезка истории разговора Developer'а (2026-07-18)

Прямое продолжение находки из 4.8 (итерация 5): реальный узкое место при упоре в `DEVELOP_TOKEN_SAFETY_LIMIT` — не обрезание completion'а, а неограниченный рост prompt-токенов внутри одного прогона (turn 32 → 974К). `actionsLog` уже И ТАК ведёт полную компактную запись каждого действия по одной строке — реализована компакция истории (`develop-loop.ts`): после `HISTORY_COMPACT_TRIGGER_MESSAGES` (50) сырых сообщений старые ходы, кроме последних `HISTORY_KEEP_RECENT_MESSAGES` (24, ≈12 ходов), схлопываются в одно синтетическое сообщение на основе `actionsLog`; при следующем срабатывании то же сообщение расширяется, а не плодятся новые. `touchedFiles`/`editedFiles`-гейтинг живёт в JS-состоянии, а не в истории сообщений — удаление старых сообщений не портит логику допуска к `edit_file`, модель просто перечитывает файл при необходимости (тот же путь восстановления, что уже используют stuck-детекторы). Отдельного нагрузочного теста на 90 ходов не проводилось в рамках этой сессии — эффект подтверждён косвенно (recalibration-ретест бага 1 уложился в 475K вместо 953K на сопоставимой задаче), полноценная проверка на длинном прогоне — предмет следующей сессии.

## 4.11 Декомпозиция крупных задач — цикл «фикс → тест → оценка» ×3, фокус на завершении (2026-07-18)

Владелец продукта: по факту раздела 4.8-4.10 крупные фичи (десяток+ файлов) НЕ доезжали до конца ни разу за сессию, несмотря на бюджет/эффективность/компакцию. Прямая директива — заняться причиной, а не следствием, и прогнать новый цикл фикс→тест→оценка на смеси простых задач, сложных задач и багов, с фокусом на завершение.

**Архитектурное решение.** Причина неготовности крупных фич — не способность модели (диффы, которые доезжали, показывали верные архитектурные решения), а масштаб: вся фича делается одним непрерывным agentic-разговором, как если бы живой инженер писал всю фичу без единого коммита и код-ревью по пути. Реализована декомпозиция: `planDevelopSubtasks` (`packages/ai/src/index.ts`) — LLM-классификатор (на `criticModel`, тот же, что уже используется для `classifyChatIntent`), который для СВЕЖЕГО `develop`-намерения (не для `develop-correction` — правка уже скоуплена) разбивает крупную задачу на упорядоченную последовательность из 2–5 маленьких, самостоятельно завершаемых и ревьюируемых шагов; для задачи, уже ограниченной одним файлом/слоем/багфиксом, возвращает её БЕЗ изменений как единственный шаг (fail-closed к отсутствию декомпозиции при любой ошибке классификации).

**Механизм цепочки** (`develop-runner.ts`): `StartDevelopRunInput` получил `chainRemaining`/`chainInfo`; после КАЖДОГО звена, если `stopped === "task_complete" && reviewVerdict === "approved"` (чистое прохождение, без остатка споров) — `maybeAdvanceChain` автоматически запускает следующий шаг тем же `startDevelopRun`, переиспользуя ворктри текущего звена (`continueFrom`, тот же механизм, что уже несёт review-correction раунды) — цепочка работает в ОДНОМ непрерывном ворктри, а не N одноразовых. Любой другой исход звена (спорное ревью после 2 раундов, пауза на уточнение/апрув, ошибка) — цепочка ОСТАНАВЛИВАЕТСЯ, автопродолжения не будет, пока человек не решит, тот же порог, что уже держит нецепочечный прогон. `conversationId` не меняется по всей цепочке — весь существующий механизм продолжения диалога (уточнения, апрувы БД, дебаггер-флоу) работает без единой правки.

**Живой тест на реальной фиче тегов** (та самая, что не доезжала весь предыдущий цикл 4.8): декомпозировано на 4 шага, первый — "модель данных для тегов (Tag) и отношение многие-ко-многим между серверами и тегами в БД". Первое звено (миграции + модели + `belongsToMany`) дошло до ревью за **10 ходов** вместо прежних 29-45+ на весь монолит — прямое подтверждение гипотезы: маленький, чётко ограниченный шаг реально укладывается в бюджет. Содержание шагов 2-4 (API + фронт) не увидено вживую — цепочка оба раза (до и после фикса деградации ревьюера, ниже) остановилась на звене 1 по реальным/спорным находкам ревью, не по нехватке бюджета.

**Побочная находка (крупная, отдельный фикс в рамках того же цикла): деградация Reviewer'а в повторяющийся мусор.** На корректном 4-файловом диффе (миграции+модели) Reviewer выдал **100+ почти дублирующихся "находок"** вида "отсутствует проверка X — используется корректно", обрубленных на середине предложения упором в лимит токенов ответа; второй раунд повторил тот же паттерн (123+ пункта). Developer справедливо распознал единственную реальную находку (id в $fillable) и корректно оспорил остальное — но цепочка всё равно остановилась (2 отклонённых раунда = человеку), хотя код был по сути верным. Похоже на repetition-loop деградацию декодирования, не на реальные находки модели. Фикс — двойной (не полагаться на послушание модели): (1) `filterSelfContradictingFindings` в `develop-loop.ts` — детерминированный фильтр, вырезающий находки, чей ЖЕ текст сам признаёт отсутствие проблемы ("это корректно", "не требуется", "это нормально" и т.п.), плюс жёсткий потолок в 10 находок как страховка; (2) явный запрет в `REVIEWER_SYSTEM_PROMPT` на построчное подтверждение корректности вместо перечисления реальных проблем. Ретест того же диффа: 8 находок в раунде 1, 6 в раунде 2 (было 100+/123+) — деградация исчезла.

**Остаточная находка (не зачищена, зафиксирована как открытая): точность Reviewer'а после фикса деградации всё ещё ограничивает автозавершение.** На повторном прогоне того же звена Reviewer больше не спамит, но во втором раунде выдал confidently-неверную находку ("не импортирован класс Tag — фатальная ошибка", хотя `Server`/`Tag` в одном namespace `App\Models` и `use` не требуется по правилам PHP) и одну устаревшую (повторно потребовал `$table = 'tags'`, хотя diff уже содержал этот фикс из раунда 1). На простой однофайловой задаче (сообщение об ошибке валидации на русском) — та же картина: 2 находки в раунде 1 (одна реальная, одна про acceptance-evidence), но раунд 2 добавил слабые/spurious пункты (переводы на ДРУГИЕ языки, не упомянутые в задаче; namespace-подобная ошибка). **Вывод: Developer стабильно выдаёт качественный код и адекватно спорит с ревью, но Reviewer — сейчас главный тормоз полного автозавершения**, даже маленьких задач, не только крупных. Не чинилось в рамках этого цикла (риск постоянной подгонки промпта под конкретные наблюдённые ошибки вместо структурного решения) — кандидаты на следующую итерацию: 3-й раунд ревью специально для по-настоящему НОВЫХ находок (не всплывших в раунде 1), либо снижение строгости планки блокирования для второстепенных находок.

**Регрессионная проверка (важная, поймана до продакшена).** Первая версия `planDevelopSubtasks` расщепляла "исправь код" и "напиши тест на это" на ДВА отдельных шага — плохо: тест первого шага без ещё-не-существующего теста почти гарантированно ловит "нет acceptance evidence" от Reviewer и блокирует шаг 1, не давая цепочке вообще добраться до шага 2. Пойман изолированным тестом на симуляции дебаггер-флоу ("делай пункт 1, только сразу добавь тест") ДО того как это попало в реальный прогон. Промпт ужесточён: "реализация и её тест — один шаг, никогда не разделять". Ретест — багфикс-подобная задача и тривиальная опечатка обе корректно остались нерасщеплёнными.

## 4.12 Тесты — отдельный опциональный шаг + телеметрия + Reviewer-модель (2026-07-18)

Прямое продолжение раздела 4.11 по явной директиве владельца: "покрыть тестами?" — отдельный вопрос пользователю ПОСЛЕ approve, да/нет/свой вариант, а не требование, которое Reviewer навязывает по умолчанию. Причина: реальные проекты (в т.ч. Slay, основная тестовая цель) в большинстве своём не покрыты тестами по конвенции — Reviewer, требовавший acceptance-evidence, боролся с этой реальностью вместо того чтобы её уважать, и это был реальный источник потерь этой сессии (один прогон потратил 45 ходов на постройку несвязанной тестовой инфраструктуры именно из-за такой находки).

**Реализовано:**
- `classifyTestsOffer` (`packages/ai`) — классификатор ответа пользователя на вопрос "покрыть тестами?" (да/нет/свой вариант охвата), fail-closed к "не ответ на этот вопрос" (никогда не пишет тесты без явного согласия, никогда не глотает не относящееся к делу сообщение).
- `REVIEWER_SYSTEM_PROMPT`: находка (8) про acceptance-evidence/тестовое покрытие убрана как блокирующая — явный запрет писать в NEEDS_CHANGES отсутствие тестов; блокирует только РЕАЛЬНО упавшая команда верификации, которую автор сам запустил.
- `buildDevelopSystemPrompt` (инструкция 8): убрано требование добавлять персистентные тестовые файлы «на всякий случай» — вместо этого ad hoc воспроизведение (одноразовый скрипт/tinker) до/после фикса, без новых файлов в репозитории, если задача явно не просит тесты.
- `develop-runner.ts`: после ЧИСТОГО approve и завершения ВСЕЙ цепочки (не промежуточного звена) — вопрос "Покрыть это тестами? (да / нет / свой вариант)" дописывается в `summary`; следующее сообщение того же диалога проверяется через `classifyTestsOffer` ДО обычной классификации намерения (тот же паттерн, что уже держат needs-clarification/needs-approval) — на "да" запускается отдельный develop-прогон "напиши тесты для уже одобренного изменения" в том же ворктри.

## 4.18.b Глобальный менеджер worktree: группировка `проект -> чат -> деревья` и массовое удаление (2026-07-19)

По прямому запросу владельца продукта правый менеджер worktree доведён до реального операционного UX, а не списка «только текущего проекта / только текущего запуска».

- Источник данных: `/api/develop/worktrees` больше используется как **глобальный реестр** всех известных worktree, а не как срез выбранного в UI проекта.
- Фронтенд-группировка в `apps/web/src/App.tsx`: записи сворачиваются в иерархию `projectPath -> conversationId -> physical worktree`. Это важно, потому что один чат может породить несколько develop-run'ов, и менеджер обязан показывать их как **один чат** с несколькими деревьями/доработками, а не как набор несвязанных run-карточек.
- На каждом уровне доступно удаление:
  - проект: удаляет все деревья всех чатов этого проекта;
  - чат: удаляет все деревья всех develop-run'ов данного conversationId;
  - дерево: удаляет только конкретный physical worktree (`runId + label`).
- Мультиселект работает тоже на всех трёх уровнях; bulk-delete собирает реальные `(runId, label)` цели и вызывает существующий `cleanup-worktree`, не вводя отдельный bulk-backend раньше времени.
- Токены developer-run'а снова видны в UI: и в самой dev-реплике, и на уровне карточки чата в менеджере.

Это изменение не меняет правило merge: занесение в основной checkout **не удаляет** worktree автоматически, потому что в дереве могут оставаться незакоммиченные доправки и следующий develop-run того же чата может осознанно продолжить работу в нём.

## 4.18.e Единая модель выделения: source of truth = только деревья (2026-07-19)

Следующий живой прогон выявил более глубокую проблему: выделение проекта/чата хранилось как отдельное состояние рядом с выделением дерева. Это давало рассинхрон между UI-галкой уровня `project/chat` и тем, что реально удаляется.

- чекбокс чата просто выделяет/снимает **все деревья чата**;
- bulk-delete всегда собирает массив удаления только из реально выделенных деревьев;
- ключ раскрытия чата (`openChats`) переведён на составной `projectPath::conversationId`, чтобы одинаковый `conversationId` в разных проектах не конфликтовал по accordion-state.

## 4.18.f Cleanup для telemetry-visible worktree (2026-07-19)

После живой проверки кейса `выделить все -> удалить все` найден backend-разрыв: менеджер worktree показывает объединение **live in-memory run'ов** и **persisted telemetry run'ов** из `developer_runs`, а endpoint cleanup изначально умел удалять только live run через `getDevelopRunStatus(runId)`.

Из-за этого пользователь мог видеть дерево в UI, выделять его, но удаление тихо ломалось на 404 для run'ов, переживших перезапуск API.

Исправление:

- `cleanup-worktree` теперь умеет fallback в telemetry;
- если run отсутствует в live-memory, API читает `developer_runs.worktrees`, удаляет физические worktree/branches по тем же данным и затем обновляет `developer_runs.worktrees` в Postgres;
- cleanup live-run'а теперь тоже синхронно обновляет telemetry-строку, чтобы UI после refresh не реанимировал уже удалённые деревья из базы.

## 4.18.g Убрана ложная “регенерация” деревьев после удаления (2026-07-19)

После живого прогона найден ещё один backend-дефект: telemetry-выборка worktree была ограничена `limit 50`. Из-за этого сценарий выглядел так:

1. UI показывает первые 50 записей из `developer_runs`;
2. пользователь удаляет их;
3. следующий poll получает уже **следующие** 50 старых записей;
4. визуально кажется, что деревья “снова появились”.

Фикс: искусственный `limit 50` убран. Менеджер теперь видит весь актуальный набор telemetry-visible worktree, поэтому массовое удаление доезжает до реально пустого состояния, а не открывает следующую скрытую страницу истории.

## 4.14 Path-scoped директивы для Developer'а + первый живой APPROVED (2026-07-18)

Явное требование владельца: "делай только фронт"/"только бек"/"только gui" не должно оставлять Developer'у физический доступ к остальным репозиториям — раньше это работало ТОЛЬКО для Q&A (`classifyProjectScopeDirective`, `pipeline-runner.ts`), для Developer-пайплайна такого фильтра не было вообще — multi-root задача всегда создавала ворктри под ВСЕ зарегистрированные пути, независимо от того, что написал пользователь, полагаясь чисто на то, что модель сама не полезет куда не просили.

**Подключено в `app.ts`:** та же `classifyProjectScopeDirective`, что уже проверена на Q&A-пути, теперь фильтрует `projectPaths` ДО того как `createTaskWorktree` вообще запускается — при ограничении Developer физически не видит и не может отредактировать исключённый репозиторий, это не "попросили не трогать", а "ворктри для него просто не существует".

**Живой тест подтвердил и заодно поймал реальный баг.** Задача "делай только в gui: добавь placeholder..." создала ворктри ТОЛЬКО для gui (api/web/cli не тронуты вообще) — и это оказался **первый прогон за всю сессию, дошедший до чистого `APPROVED` без вмешательства человека**. Сработал и вопрос про тесты (§4.12) — "покрыть тестами? да/нет/свой вариант".

При ответе "да" вскрылся баг: scope-директива классифицировалась ЗАНОВО на каждое сообщение, включая голое "да" — которое, естественно, не матчит паттерн ограничения области и сбрасывает её на "все репозитории". Это ломало сразу два механизма: (1) `executeDevelopRun`'s проверку на переиспользование ворктри (`continueFrom.worktrees.length === originalRoots.length` — 1 против 4, не совпадает, ворктри создаются заново с нуля, а согласованная правка теряется из контекста); (2) саму заявленную область ("только gui") — молча снималась на продолжении разговора. Фикс: для ЛЮБОГО продолжения диалога (ответ на уточнение/апрув/вопрос про тесты, develop-correction) область берётся НЕ из текста реплики, а напрямую из ворктри уже идущего прогона (`priorDevelop.worktrees` labels) — переклассификация происходит только для по-настоящему нового `develop`-намерения. Ретест с фиксом: follow-up на "да" корректно переиспользовал ТОТ ЖЕ gui-ворктри, ни один другой репозиторий не тронут.

Прогон написания тестов (по "да") упёрся в `max_turns` (28 ходов, не завершён) — но честно, не как баг: в gui-проекте не было НИКАКОЙ тестовой инфраструктуры (нет vitest.config.ts, нет test-wrapper'ов), и Developer с нуля поднимал framework, конфиг, зависимости прежде чем написать сам тест — работа объективно крупнее исходной 1-строчной правки. Механика (спросить → сработать на "да" → тот же ворктри → область не расширилась) отработала верно; сам объём работы под запрошенные тесты — отдельный вопрос, не входящий в рамки этого фикса.

- `client`, задача про README (`8b8d3c3dabd8184f60bf295e66db4174835924bc`) — Reviewer поймал реальную функциональную ошибку: автор поменял `/chat` на `/чат` не в том месте. Это как раз корректный блокер: diff не выполнял задачу.
- `client`, задача про страницу проектов (`a5782a5dd1c520789c88857df243223bd3cf33c0`) — Developer вообще не сделал edit, закончил с пустым diff и без ревью. Это уже не проблема Reviewer'а, а отдельная слабость Developer'а на микрокопи.
- `slay-gui`, scope-only прогон (`76e82d9be7b02e939a01abc03c798b461ede2210`) — пустой diff после честной проверки, ревью не запускалось; это полезный контрольный кейс на область, но не кейс на точность Reviewer'а.

Из этого зафиксирован новый принцип для кода пайплайна, а не только для промпта: **BLOCKER должен означать "нельзя мержить, потому что это реально ломает/не выполняет задачу"**. Всё, что относится к wording/copy/стилистике/предпочтительной формулировке/терминологии без явного функционального дефекта, должно ехать в `NOTE`, даже если модель назвала это `BLOCKER`.

Реализован детерминированный пост-процессинг в `develop-loop.ts`:

- мягкие находки по словарю copy/style/translation автоматически демотируются из blocker в note, если в тексте самой находки нет сильного сигнала реальной поломки (`runtime`, `validation`, `migration`, `wrong place`, `не выполняет задачу`, и т.д.);
- во втором раунде повтор уже оспоренной находки без нового содержания перестаёт блокировать и уходит в notes как повтор после спора;
- UI уже показывает `noteFindings` отдельно, поэтому замечания не теряются, но перестают стопорить завершение задачи.

Открытый честный вывод после этой итерации: это должно резко снизить ложные `needs-changes` на микрокопи и спорных wording-кейсах, но НЕ решает вторую часть проблемы — Developer всё ещё может закончить микрозадачу без edit вообще. Следующий обязательный живой ретест после этого фикса: один copy-кейс на `client`, один заведомо-правильный functional/correctness-кейс и один scoped-кейс на `slay`, чтобы отдельно измерить reviewer-noise и developer-no-op.

Отдельно по cleanup: временные worktree от ранних живых прогонов были убраны не полностью — `git worktree list` показывал prunable-метаданные, а прямой `git worktree remove --force`/`git worktree prune` упёрлись в `Operation not permitted` на `.git/worktrees/root*`. Это не ломает код пайплайна, но это честно зафиксированное ограничение локальной среды, которое надо ещё дочистить отдельной безопасной попыткой в конце сессии.

### 4.15.b Micro-copy fast-path в Developer (2026-07-18)

После ретеста стало ясно, что одной починки Reviewer уже мало: на задачах вида "замени одно слово и больше ничего не трогай" сам Developer всё ещё ведёт себя слишком как общий кодовый агент — читает широко, иногда пытается завершиться без edit, а потом легко задевает соседнюю строку с тем же словом. Это не баг модели как таковой, а отсутствие отдельного режима под очень узкий класс задач.

Добавлен детерминированный `isMicroCopyTask(task)` в `develop-loop.ts`. Если задача одновременно:

- явно про замену/исправление/перевод текста;
- содержит текстовые маркеры (`слово`, `строка`, `placeholder`, `provider`, `backend`, `frontend`, `chat`, и т.п.);
- жёстко ограничена формулировкой вроде `больше ничего не меняй` / `только пользовательский текст`;
- и при этом НЕ похожа на кодовую/схемную/bugfix-задачу,

то в первый user-message Developer получает специальный **MICRO-COPY MODE**:

- сначала доказать точную строку через `grep`, потом читать только целевой файл;
- менять минимально возможную поверхность, ideally одну строку/литерал;
- считать любое соседнее изменение подозрительным по умолчанию, если оно не вытекает прямо из слов задачи;
- верифицировать точечными `grep`/`sed`, а не общими обходами.

Это не новый пайплайн и не новая роль, а узкий режим того же Developer-цикла. Цель — убрать именно observed failure mode: когда микро-задача тратит 8-12 ходов и всё равно трогает лишнюю строку.

## 4.16 Цикл фикс→прогон→анализ ×5 на Reviewer-точности (2026-07-18, продолжение)

Прямое продолжение §4.15 (сделано параллельно другой сессией/инструментом, код проверен и принят после ревью diff'а и собственного typecheck) — владелец утвердил план из 3 пунктов (round-2 судит только новые находки, blocker/note-разделение, стабилизация гранулярности декомпозиции) и попросил ещё 5 итераций фикс→прогон→анализ.

**Итерация 1 — валидация уже внесённых §4.15 фиксов на 3 свежих живых прогонах:**
- copy-задача на `client` (изменить одно слово в title) — **чистый APPROVED**, 0 blockers, 0 notes.
- scoped-задача на `slay-gui` (добавить title-tooltip к кнопке, которая уже имела `q-tooltip`) — корректно заблокирована оба раунда: раунд 1 поймал дублирование тултипов, раунд 2 — что фикс сломал i18n (убрал `q-tooltip` вместо решения дублирования). Обе находки — РЕАЛЬНЫЕ, разные, не поводов для demote — сам факт что современный Reviewer всё ещё блокирует на настоящей проблеме, а не слепо всё апрувит, отдельно ценен.
- functional-задача на `slay-api` — неубедительный тест: сама задача была двусмысленно сформулирована ("проверь и почини, если баг есть" на коде, который, вероятно, уже был корректен) — сигнал получился неоднозначный, не в счёт.

**Итерация 2 — точка 3 (гранулярность декомпозиции).** `planDevelopSubtasks` получил явное разбиение по архитектурным слоям (schema/migrations → models → repository/service → API/controller/routes → frontend UI, никогда не смешивать два слоя в одном шаге) вместо свободного суждения "сколько это один шаг". Ретест фичи тегов: **5 подзадач вместо 3-4 раньше**, шаг 1 сузился до ЧИСТО миграции (не миграция+модели, как раньше). Побочно найден новый баг: модель написала `ACTION: api: php -l ...` вместо `ACTION: run_command(api: php -l ...)` — префикс мультирута спутан с отдельным типом действия, 3 хода подряд `NO ACTION PARSED`, ран закончился `max_turns`/0 файлов (при этом ни одна из миграций не была реально создана — вероятная жертва 41 действия, сброшенных за один ход per-turn кэпом в 1-м ходу).

**Итерация 3 — фикс run_command-префикса + ретест.** Явное предупреждение против точно наблюдённой ошибки ("это НЕ отдельный тип действия, конструкция `run_command(...)` обязательна"). Ретест: миграция реально создана (1 файл, 9 ходов, без format-error луп) — баг закрыт. Но всплыл НОВЫЙ класс проблемы: раунд 1 заблокировал `unique()` на `name` ("в задаче не требуется глобальная уникальность"), Developer убрал `unique()` точно как просили, раунд 2 заблокировал ОТСУТСТВИЕ `unique()` на том же поле — прямой разворот собственной позиции ревьюера между раундами без признания разворота.

**Итерация 4 — фикс на разворот позиции между раундами.** Добавлена явная инструкция: если фикс сделал ТОЧНО то, что просил предыдущий раунд, не флагать получившееся состояние как новую проблему без явного признания "я передумал, вот почему".

**Итерация 5 — ретест, находка глубже прежней.** Разворот позиции исчез (11 ходов, 1 файл). Но обнаружилось: раунд 2 заново поднял НЕ NULL-претензию на `name`, которую раунд 1 уже поднимал и Developer уже оспорил (технически корректно — `$table->string()` в Laravel NOT NULL по умолчанию) — БЕЗ demote в note. Причина найдена: `suppressRepeatedRoundTwoBlockers` матчит находки по нормализованной СТРОКЕ (`normalizeFindingKey`) — раунд 1 и раунд 2 сформулировали одну и ту же претензию РАЗНЫМИ словами ("не имеет ограничения" vs "не имеет ограничения NOT NULL" — разный порядок и состав фразы), нормализованные ключи не совпали, repeat-detection не сработал. **Открытый, не зачищенный вывод: подавление повторов сейчас ловит только буквально-похожие переформулировки, не смысловые повторы** — для полного решения нужна семантическая, а не строковая проверка (эмбеддинги или отдельный LLM-вызов "это то же самое утверждение?"), не сделано в рамках этого цикла — кандидат на следующую итерацию.

**Итоговая оценка цикла:** Reviewer стал заметно надёжнее (чистый APPROVED на маленьких задачах, корректно держит реальные блокеры, самопротиворечия внутри раунда и разворот между раундами устранены) — но точность на многораундовом споре всё ещё ограничена буквальным, не смысловым сопоставлением находок. Все живые тесты сегодня либо доехали до approve, либо остановились на содержательном, а не шумовом споре — это качественно другой уровень, чем в начале дня (100+ находок-самоопровержений, confidently-неверные фактические claims).

## 4.17 Семантическое сопоставление находок между раундами — цикл ×5 (2026-07-18, продолжение)

Прямое продолжение открытого вывода §4.16: подавление повторов раньше матчило находки только по нормализованной строке — переформулировка одной и той же претензии другими словами (что LLM делает постоянно) проскакивала мимо. Владелец дал добро на 5 итераций фикс→прогон→анализ конкретно на этой проблеме.

**Инфраструктура.** `DevelopRunOptions.computeFindingSimilarity` — новый необязательный хук `(candidateFinding, comparisonTexts) => Promise<number>`, реализован в `develop-runner.ts` через уже существующий `embedTexts` (`@client/ai`, та же модель `qwen/qwen3-embedding-8b`, что и семантический поиск по коду) + локальный `cosineSimilarity`. Best-effort по всей цепочке: любая ошибка эмбеддинга — тихий откат к прежней строковой проверке, ревью не блокируется.

**Калибровка ДО прогона, не вслепую.** Первая прикидка порога — 0.87. Прогнал реальные эмбеддинги на живой паре из §4.16 (round 1: «не имеет ограничения», round 2: «не имеет ограничения NOT NULL» — та самая уже разрешённая претензия) ДО того, как тратить полный пайплайн-прогон: похожесть оказалась всего **0.83** — начальная прикидка 0.87 пропустила бы именно тот случай, ради которого всё делалось. Контрольная пара (другая претензия, та же таблица) дала 0.62, полностью не связанная — 0.45. Порог выставлен на **0.75** — с запасом выше false positive (0.62) и ниже настоящего совпадения (0.83).

**Итерация 1 — живой ретест, находка на уровень выше.** Полная реализация подключена, прогнан тот же сценарий (фича тегов). Round 2 повторил уже оспоренную (собственно, оспоренную ПРАВИЛЬНО) claim про `$table` — semantic-check не сработал вообще, потому что `authorDisputeSummary` (то, с чем сравниваем) вообще не был извлечён: строка оспаривания на этот раз была оформлена как пункт нумерованного списка без буквального тега «Оспорено:» — `extractDisputedSummary`'s строгий regex-по-префиксу её не увидел. Фикс не добрался до применения — сломался на шаг раньше, при извлечении входных данных для сравнения.

**Итерация 2 — расширил regex распознавания диспута ключевыми фразами, ретест выявил ГОРАЗДО более серьёзную проблему.** На этот раз diff УЖЕ содержал оба запрошенных фикса (`->unique()` на `name`, `$table = 'tags'` в модели) — проверено напрямую в сыром diff'е. Round 2 ВСЁ РАВНО заблокировал оба пункта как отсутствующие, хотя ничего не оспаривалось (Developer их просто исправил). Гипотеза: передача `priorRound.findings` в промпт ревьюера ("вот что ты сказал в прошлый раз") создаёт эффект якорения — модель повторяет собственный прошлый вердикт вместо свежей проверки текущего диффа. Это возможный побочный эффект механизма, добавленного специально для §4.15/4.16 (чтобы не блокировать повторно уже оспоренное) — обнажил обратную сторону той же медали.

**Итерация 3 — переформулировка того, как раунд 1 показывается ревьюеру.** Вместо "вот твои прошлые блокеры" — "это находки ДРУГОГО прохода по УСТАРЕВШЕЙ версии диффа, проверь сам по текущему коду, не считай их автоматически верными", плюс явное требование указать конкретный текст в ТЕКУЩЕМ диффе, подтверждающий находку, иначе не повторять. Ретест на том же сценарии: разворот позиции не наблюдался, но всплыла ТРЕТЬЯ форма той же проблемы извлечения диспута — "Оспорено:" в этот раз был секционным заголовком с булет-пунктами ниже, а не префиксом строки, да ещё с формулировкой ("было бы неконсистентным и избыточным"), не попадающей ни в один keyword из regex.

**Итерация 4 — отказ от попытки распознавать «диспут» по ключевым словам вообще.** Три разных формата провала за три итерации подряд — погоня за естественно-языковыми формулировками несогласия через regex принципиально проигрышна. `extractDisputedSummary` заменён на `extractSummaryChunks`: делит ВЕСЬ summary на параграфы/буллет-пункты по структурным маркерам (пустые строки, `-`/`•`/цифра+точка), без фильтрации по «похоже на диспут». Смысловую релевантность теперь целиком определяет порог косинусного сходства, а не regex на входе. Осознанный компромисс: теряется различение «автор оспорил» от «автор описал фикс» — находка, совпадающая с описанием ФИКСА (не оспаривания), тоже может демоутнуться, если фикс на самом деле не сработал. Живые данные сегодня — 3 из 3 случаев были пропущенные повторы, ноль случаев ложного подавления; если это когда-то развернётся — пересмотреть. Ретест: **сработало** — лог явно показывает `"Повторно заявленная находка после оспаривания (перефразировано): ...отсутствует импорт класса Server..."`, находка про уже оспоренный (и оспоренный обоснованно — Laravel autoloading, тот же паттерн что в §4.14) импорт корректно ушла в NOTE вместо повторного блока.

**Итерация 5 — повторный прогон на воспроизводимость.** Другой набор находок в раунде 1 (естественный разброс модели) не дал точь-в-точь такого же случая для проверки, но всплыла уже задокументированная (не новая) находка — confidently-неверная claim про отсутствующий импорт в том же namespace (§4.14). Не регрессия сегодняшней работы, старый открытый пункт.

## 4.18 Полноценный worktree-manager в чате + возврат токенов Developer (2026-07-19)

Новый прямой запрос владельца: текущий блок "worktree этой задачи" был слишком узким и неудобным для реальной работы. Проблемы были три:

- в multi-root задаче кнопки `занести` / `удалить` визуально воспринимались как общие на весь набор путей, хотя пользователь ожидал действия ПО КАЖДОМУ пути отдельно;
- справа от чата пустовало место, а управлять всеми деревьями по проекту было негде;
- в developer-репликах пропало отображение реально потраченных токенов, хотя это важная часть калибровки поведения моделей.

### Что изменено

**1. Один чат старается продолжать то же дерево, а не плодить новое.**  
В `app.ts` ужесточён старт develop-задачи: если у текущего `conversationId` уже есть `priorDevelop` с живыми `worktrees`, новый imperative/develop follow-up теперь тоже идёт через `continueFrom`, а не только формальный `develop-correction`. То есть продолжение разговора по той же задаче жёстче привязывается к уже существующему дереву. Это не новый архитектурный режим, а устранение практического разрыва между "пользователь продолжает ту же задачу" и "классификатор назвал это просто новым develop".

**2. Backend отдаёт реестр worktree по проекту/диалогу.**  
В `develop-runner.ts` добавлен `listDevelopWorktreeEntries(...)`, а в `app.ts` — новый read-endpoint `GET /api/develop/worktrees?projectPath=...&conversationId=...`. Он возвращает:

- run id / conversation id;
- задачу;
- статус;
- набор физических `worktrees`;
- per-run merge outcomes;
- usage по developer-run (`prompt/completion/total tokens`).

Это не замена старому `GET /api/develop/status`, а отдельный источник данных именно для менеджера деревьев.

**3. Справа от чата появился отдельный worktree-manager.**  
Во фронте (`App.tsx`) добавлена правая sticky-колонка для chat-view: `WorktreeManagerSidebar`. Она группирует worktree по run/чату в виде аккордеона и показывает:

- какой это чат;
- сколько путей внутри;
- статус;
- исходную задачу;
- реальные токены этого develop-run;
- встроенный `DevelopWorktreePanel` с per-path действиями.

То есть справа теперь не пустое место, а полноценный менеджер деревьев по проекту, без необходимости искать старое сообщение в транскрипте.

**4. Кнопки `занести` / `удалить` уже реально per-path.**  
Server-side это уже было реализовано ранее (`label` в `POST /api/develop/merge-to-checkout` и `POST /api/develop/cleanup-worktree`), но теперь это закреплено и как основной UX: менеджер и карточка используют путь-специфичный вызов, а не общее действие на весь набор roots.

**5. Merge не удаляет дерево автоматически.**  
Это поведение сохранено и теперь явно поддержано новым менеджером: после `merge-to-checkout` исходный worktree остаётся жить, потому что туда может быть дописан код или merge может быть частичным. Cleanup остаётся отдельным, явным действием пользователя.

**6. Токены Developer снова видны прямо в сообщении.**  
В `DevelopRunMessage` возвращён блок `Токены`:

- prompt;
- completion;
- total.

Раньше developer-реплика показывала summary/review/diff, но реальные токены именно develop-run были потеряны из UI, хотя backend их продолжал собирать в `developer_runs`.

### Честный остаток

Эта итерация закрывает именно UX и lifecycle worktree-manager'а. Она не решает ограничение локальной среды с `git worktree prune`/`Operation not permitted` на старых prunable metadata внутри `.git/worktrees/*` — это отдельная operational-проблема хоста, не логика нового менеджера.

### 4.18.a Починка видимости уже существующих worktree в менеджере (2026-07-19)

Сразу после 4.18 всплыл реальный баг по живому UX: пользователь видел 4 worktree в старом блоке сообщения задачи, но новый правый manager показывал пусто. Причина оказалась архитектурной, не CSS:

- sidebar сначала читала только `developRunStatuses` в памяти текущего процесса;
- старые/уже завершённые run'ы с сохранёнными worktree физически оставались и в tmp, и в Postgres `developer_runs.worktrees`;
- после рестарта API или просто вне текущей in-memory сессии эти записи исчезали из `Map`, но не из реальности.

Из-за этого менеджер показывал только "живые в памяти" деревья, а не "все реально существующие деревья проекта".

**Фикс:**

- в `develop-runner.ts` добавлен `listDevelopWorktreeEntriesFromTelemetry(...)`, читающий `developer_runs` из Postgres (`jsonb_array_length(worktrees) > 0`);
- `GET /api/develop/worktrees` теперь объединяет два источника:
  - live in-memory `developRunStatuses`;
  - persisted telemetry `developer_runs`;
- merged-список дедуплицируется по `runId`, live-запись приоритетнее telemetry-записи;
- даты из Postgres нормализуются в ISO перед сортировкой (пойман и сразу исправлен реальный баг `right.startedAt.localeCompare is not a function`).

**Живая проверка:** после фикса `GET /api/develop/worktrees?projectPath=/Users/evgenii/Desktop/client` уже вернул сохранённые worktree прошлых run'ов (`6181...`, `8b8d...`, `22c3...`, `0226...`, `6f3f...`, и т.д.) вместе с токенами. То есть новый manager теперь видит не только текущий ран, но и уже созданные ранее деревья, которые пользователь и хотел удалить/занести.

**Итог цикла:** заявленная цель (семантическое сопоставление вместо строкового) — реализована и подтверждена живым логом. По пути найдено и починено больше, чем планировалось: тройная хрупкость regex-извлечения диспута (два формата пропущены, третий пойман только на 4-й итерации) и вероятный anchoring-эффект от передачи прошлого раунда в промпт — оба явления не были бы найдены без прогона на реальных, а не синтетических кейсах. Осталось честно открытым: полное разделение «диспут vs описание фикса» для семантического сравнения не восстановлено — принятый риск, эмпирически обоснованный, но не строго доказанный.

## 4.18 Ретест anchoring-эффекта + «занеси в текущую ветку» / «почисти ворктри» (2026-07-18)

**Ретест anchoring-эффекта (§4.17, п.2 плана владельца).** Тот же сценарий (фича тегов) прогнан ещё раз с уже переформулированным промптом ("это находки ДРУГОГО прохода по УСТАРЕВШЕЙ версии, проверь сам"). Результат — качественно другой уровень: **звено 1 дошло до APPROVED (13 ходов), автоцепочка перешла на звено 2 — оно ТОЖЕ дошло до APPROVED (15 ходов)**, впервые за всю сессию два звена подряд с чистым одобрением без участия человека. Звено 3 (15 файлов — контроллеры, requests, репозитории, сервис, роуты) корректно остановилось на `needs-changes`: раунд 1 дал 3 находки, раунд 2 — **2 из 3 корректно демоутнуты в NOTE** семантическим сопоставлением ("Повторно заявленная находка после оспаривания (перефразировано)"), остался ровно 1 новый, содержательный блокер (несогласованность автопроверки владения между `AddTagRequest`/`RemoveTagRequest`). Anchoring-фикс подтверждён живьём: ни одного случая повторной блокировки уже исправленного за весь трёхзвенный прогон.

**«Занеси в текущую ветку» / «почисти ворктри» — новый флоу по прямому запросу владельца.** Явная причина: "за 5 лет коммерческого опыта ни разу не понадобился git worktree, не умею им пользоваться" — нужен флоу задача → «готово» → «занеси в текущую ветку» → изменения появляются как обычный незакоммiченный diff в реальном чекауте, ревью — в IDE, без commit/push.

Реализовано:
- `applyWorktreeDiffToRoot` (`packages/repository-git`) — берёт уже вычисленный diff ворктри (`collectWorktreeChanges`) и применяет его `git apply` напрямую на `worktree.rootPath` (РЕАЛЬНЫЙ чекаут пользователя, не ворктри). Сначала `git apply --check` (dry-run), и только при чистом результате — реальное применение. Никогда не коммитит, никогда не пушит, не трогает индекс/указатель ветки.
- `classifyPostCompletionCommand` (`packages/ai`) — распознаёт "занеси в текущую ветку" / "почисти ворктри" как отдельные, explicit-only команды (fail-closed к «ничего не делать» при неоднозначности — обе мутируют реальное состояние, угадывать нельзя). В отличие от вопроса про тесты (одноразовое окно сразу после completion), эта проверка активна ВСЁ время, пока у завершённого рана есть живые ворктри — пользователь может попросить в любой момент, не только сразу.
- `mergeDevelopRunToRealCheckout` / `cleanupDevelopRunWorktrees` (`develop-runner.ts`) — тонкие обвязки поверх вышеперечисленного, подключены в `app.ts`'s chat routing.

**Живой тест на изолированном одноразовом репо (не Slay, не сам `client` — нулевой риск на время проверки):**
- Happy path: правка сделана, `git status` показывает `M app.js` (uncommitted), `git log` — БЕЗ нового коммита, содержимое файла реально изменилось. Ровно то, что просили.
- Cleanup: `git worktree list` до/после подтверждает удаление, ветка `client/task-*` удалена, а занесённое изменение в реальном чекауте осталось нетронутым (cleanup чистит только ворктри/ветку задачи, не трогает состояние реального чекаута).
- **Safety net на конфликт** — специально разошёл реальный файл с тем, что ожидал diff (изменил ту же строку вручную по-другому). `git apply --check` корректно отказал, ничего не применилось и не сломалось, пользователю — чёткая ошибка (`patch failed: file.txt:1`, `patch does not apply`). Ни партиального применения, ни порчи файла.

## 4.19 Read-only доступ к БД — `db_query` (2026-07-18)

Явный запрос владельца, из его собственной реальной практики: "когда исследую фичу, легче увидеть реальную строку в таблице, чем абстрактно читать модель"; "когда баг-репорт — сначала воспроизвожу, потом смотрю что реально записалось в БД, и от этого отталкиваюсь"; "для сложной выборки сначала довожу чистый SQL до результата на реальных данных, потом перевожу на ORM". До этого ни у Researcher'а (Q&A/дебаггер), ни у Developer'а не было НИКАКОГО прямого доступа к данным проекта — только чтение кода.

**Двойная защита, явное требование владельца ("нужно перестраховаться и на уровне кода и на уровне модели"):**
1. **Код**: `isReadOnlyQuery` (`apps/api/src/db-query-tool.ts`) — детерминированный, регулярка допускает ТОЛЬКО одиночный `SELECT`/`WITH`/`EXPLAIN`/`SHOW`/`DESCRIBE`, блокирует множественные statement'ы (защита от `SELECT 1; DROP TABLE`) и функции с файловым I/O (`pg_read_file`, `COPY ... TO`, `INTO OUTFILE`, `LOAD_FILE`) — работает независимо от того, что попросит модель.
2. **Промпт**: инструкция инструмента прямо говорит "READ-ONLY ONLY, enforced in code regardless of what you send".

**Определение connection — по прямому решению владельца, три слоя, разрешено делать по ОРИГИНАЛЬНОМУ пути проекта, не по воркtree:**
1. Читаем `.env` проекта (`DB_CONNECTION`/`DB_HOST`/`DB_PORT`/`DB_DATABASE`/`DB_USERNAME`/`DB_PASSWORD`, generic Laravel-конвенция, плюс fallback на `DATABASE_URL`).
2. Если `DB_HOST` — реальный адрес (localhost/IP/домен с точками) — коннект напрямую.
3. Если `DB_HOST` — похоже на имя docker-сервиса (без точек, не IP) — сначала best-effort построчный (не полноценный YAML-парсер, осознанно — не тащить лишнюю зависимость ради одной узкой задачи) поиск `ports:` в `docker-compose.yml`/`compose.yml` рядом с проектом; если порт опубликован на хост — коннект напрямую на `127.0.0.1:<порт>`; если нет — находим реальный ID контейнера через `docker compose ps -q <service>` (с fallback на `docker-compose` v1), запущенный **из оригинальной директории проекта** (не воркtree — у воркtree другое имя папки, значит другой compose-project-context, контейнеры не найдутся), и обращаемся напрямую через `docker exec <id> psql/mysql ...` (это уже не зависит от директории, раз ID известен).

Не строит собственное TCP-соединение (без npm-зависимостей на `pg`/`mysql2`) — всегда шеллится в `psql`/`mysql` CLI, что даёт ОДИН код-путь что для прямого, что для docker-exec режима (разница только в том, ЧЕМ оборачивается команда).

**Живые тесты, все прошли:**
- 14/14 юнит-проверок `isReadOnlyQuery` (все read-only формы допущены, все мутации/множественные statement'ы/файловые функции отклонены).
- Резолюция connection против реального `.env` Slay — корректно распознала `pgsql`/`localhost:5448`, БД сейчас не поднята (дев-окружение не запущено) — получена чистая, не падающая ошибка `Connection refused`, без краша.
- **docker-exec режим против РЕАЛЬНОГО работающего контейнера** (`client-postgres-1`, собственная метоБД этого же проекта) — реальный `SELECT` вернул реальные имена таблиц, мутация (`DELETE`) корректно отклонена кодовым гейтом до выполнения.
- **Полный сквозной прогон через настоящий чат-пайплайн** (не изолированный вызов функции): вопрос "какие таблицы в базе и сколько строк" на одноразовом тестовом репо с `.env`, указывающим на реальный Postgres — Researcher САМ выбрал `db_query`, получил реальные данные (`code_embeddings` 10338 строк, `developer_runs` 73 и т.д.) и построил корректный ответ. Вся цепочка (чат → intent-роутинг → Q&A pipeline → agentic-loop → `db_query` доступен → модель решает им воспользоваться → connection resolved → реальный результат → синтез ответа) подтверждена вживую.

Подключено в оба цикла — Developer (`develop-loop.ts`, где ещё используется для "довести сложный SELECT до результата на реальных данных, потом перевести на ORM" сценария) и Researcher (`loop.ts`, куда уже маршрутизируется дебаггер-флоу §4.9 — напрямую покрывает "баг-репорт → воспроизвести → посмотреть что реально в БД" сценарий).

**Не сделано в этом заходе, честно открыто:** паттерн "если записи нет — дефолт, если есть — из БД" (ленивый backfill вместо сидера/деструктивного UPDATE, реальный прецедент — Magenda patient restrictions) обсуждён, но не закодирован как переиспользуемая CONVENTION-запись в Fact Store — идея зафиксирована, реализация отложена.

## 6. Известные разрывы (осознанно v2+, зафиксированы из критического ревью)

- **Верификация приёмки** («сделано то, что просили», а не только «ничего не сломали»): воспроизведение бага до/после, новый тест на фичу. v1 несёт только журнал верификации; приёмка — на человеке при merge.
- **Bootstrap-факт проекта** («как собирается/тестируется») — выясняется и сохраняется в Fact Store один раз; v1 полагается на `run_command` Developer'а per-run.
- **Синхронизация с живым репозиторием** (rebase/конфликты при долгих задачах) — v1: worktree от HEAD на момент старта, конфликт при merge виден человеку.
- ~~**Post-diff impact**~~ — закрыто §4.13: `buildImpactPreviewTool` (детерминированный, через `getFileDependents`) подключён к Reviewer-контексту. Не закрыто до конца: селекция ЗАПУСКАЕМЫХ тестов под изменённые зависимости не реализована (тесты теперь опциональны и по запросу, см. §4.12) — граф даёт только "кто зависит", не "какой тест это покрывает".
- Кураторство памяти (дедуп, decay) — по мере роста числа задач.

## 7. Телеметрия — с первого дня

Каждый development-run пишет в Postgres (`developer_runs`): токены по ролям, ходы, команды верификации и их exit codes, вердикты Reviewer, число fix-раундов, исход (delivered/failed/aborted), размер diff. Без этого невозможно ответить ни на один будущий вопрос тюнинга (нужен ли отдельный Reviewer-vendor, где сгорают токены, какой гейт что ловит) — это урок, купленный ценой слепоты в Q&A-пути.
