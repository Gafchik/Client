## 1. Vision

### Что такое Client

Client — это автономная среда разработки программного обеспечения, управляемая искусственным интеллектом. Она представляет собой платформу, которая способна принимать задачу на естественном языке, проводить полноценное исследование предметной области, проектировать архитектуру, писать, тестировать и поставлять код, а также накапливать и переиспользовать знания между проектами.

Client не является очередным "AI-кодером" или тонкой обёрткой над LLM. Это полноценная экосистема, в которой несколько специализированных модулей взаимодействуют друг с другом, управляя полным жизненным циклом разработки: от размытой идеи до развёрнутого и задокументированного решения.

### Какую проблему решает

Современная разработка программного обеспечения сталкивается с рядом фундаментальных проблем:

- **Фрагментация знаний.** Знания о проекте распределены между головами разработчиков, тикет-трекерами, wiki, комментариями в коде и чатами в мессенджерах. При уходе ключевого разработчика значительная часть контекста безвозвратно теряется.
- **Высокий порог входа.** Даже опытному разработчику требуется от нескольких дней до нескольких недель, чтобы продуктивно начать работать в незнакомой кодовой базе среднего размера.
- **Повторяемость ошибок.** Команды раз за разом наступают на одни и те же архитектурные и инженерные грабли, потому что уроки прошлых проектов не формализованы и не доступны в момент принятия решений.
- **Ручная рутина.** Значительная часть времени разработчика уходит на задачи, которые алгоритмизуемы, но не автоматизированы: поиск нужного файла, понимание зависимостей, написание boilerplate-кода, конфигурирование окружений.
- **Разрыв между исследованием и реализацией.** Разработчик сначала изучает документацию, API, best practices, а затем пишет код. Эти два этапа редко связаны автоматически — знание, добытое на этапе исследования, не формализуется и не передаётся этапу реализации.

Client решает эти проблемы через глубокую интеграцию AI-компонентов в каждый этап жизненного цикла разработки и через создание непрерывно пополняемой базы знаний, которая переживает отдельные проекты и отдельных разработчиков.

### Для кого предназначен

- **Независимые разработчики и малые команды**, которым нужна скорость и качество без найма множества узких специалистов.
- **Архитекторы и Tech Lead-ы**, которые хотят проверить гипотезы, быстро прототипировать архитектурные решения и получить документированное обоснование выбора.
- **Крупные инженерные организации**, стремящиеся накапливать и переиспользовать знания между проектами, снижая bus factor и ускоряя onboarding новых членов команды.
- **Open-source проекты**, желающие снизить порог входа для контрибьюторов за счёт автоматически поддерживаемой базы знаний и автоматизированного code review.

## 2. Goals

### Главные цели проекта

1. **Автономное выполнение задач разработки.** Система должна принимать задачу на естественном языке (например, "добавить OAuth-аутентификацию через Google в веб-приложение") и доводить её до полностью готового, протестированного, задокументированного и задеплоенного решения.

2. **Полная прослеживаемость решений (Traceability).** Каждое принятое архитектурное, дизайнерское или инженерное решение должно быть связано с контекстом, в котором оно было принято: какая задача решалась, какие альтернативы рассматривались, какие источники были изучены, какие trade-off были сделаны.

3. **Непрерывное накопление и переиспользование знаний.** Знания, добытые в ходе одного проекта, должны быть доступны в последующих проектах. Это касается как технических знаний (спецификации API, паттерны, best practices), так и проектных знаний (почему было выбрано решение X, а не Y).

4. **Модульность и заменяемость компонентов.** Каждый модуль системы должен иметь чётко определённые границы и интерфейсы. LLM-провайдеры, парсеры, базы данных, индексы — всё это должно быть заменяемо без переписывания остальной системы.

5. **Человеко-читаемая документация как код.** Вся документация (архитектурная, API, пользовательская) должна генерироваться и поддерживаться автоматически как часть процесса разработки, а не быть отдельной активностью.

6. **Работа в реальной файловой системе.** Client работает не в песочнице, а напрямую с файлами проекта. Разработчик-человек может в любой момент вмешаться, изменить код руками, и система должна адаптироваться.

### Что входит в проект

- Система индексации и анализа кодовой базы (Indexer).
- Графовая база знаний для представления сущностей и отношений (Graph).
- Модуль управления знаниями: хранение, версионирование, поиск (Knowledge).
- Модуль автоматического исследования: оркестратор сбора информации из кода, документации, Git, AST, внешних источников (Research Engine).
- Модуль сборки контекста: компиляция релевантного окружения для конкретной задачи (Context Builder).
- Модуль планирования: декомпозиция задачи на атомарные шаги (Planner).
- Модуль выполнения: реализация шагов с обратной связью (Execution).
- Модуль рабочего пространства: единая среда выполнения — файловая система, Git, Docker, терминал, sandbox (Workspace).
- Система абстракции провайдеров: унифицированный доступ к LLM, embeddings, vector stores, search (Provider System).
- Модуль анализа влияния: определение последствий изменений, анализ зависимостей, прогноз рисков (Impact Analysis Engine).
- API-слой для интеграции с IDE (на первом этапе — PhpStorm), CLI и web-интерфейсом.
- Docker-упаковка для воспроизводимости окружений.
- Real-time подсистема для стриминга прогресса выполнения.
- Парсеры для основных языков программирования (начиная с PHP, TypeScript/JavaScript).
- Событийная шина (Event Bus) для асинхронного взаимодействия между модулями.
- Система версионирования всех артефактов (Graph, Knowledge, Project State).
- Система внутренних артефактов выполнения задачи (intent, research report, impact report, context package, execution plan, execution report).

### Что НЕ входит в проект

- Собственная разработка LLM-моделей. Client использует существующие модели через унифицированный интерфейс.
- Замена production CI/CD систем. Client может генерировать конфигурации и интегрироваться с ними, но не заменяет Jenkins, GitHub Actions или GitLab CI.
- Полная автоматизация деплоя в production. Client готовит артефакты и конфигурации, но решение о деплое и сам деплой остаётся за командой.
- Управление проектной командой, замена менеджера проекта. Client — инструмент разработчика, а не замена организационным процессам.
- Встроенный хостинг или облачная платформа. Client работает локально или на собственной инфраструктуре пользователя.
- Замена систем мониторинга и observability production-систем.

## 3. Core Principles

### Принцип 1: AI не принимает архитектурные решения самостоятельно

Любое архитектурное решение, которое может иметь долгосрочные последствия для проекта, должно быть явно сформулировано, обосновано и представлено разработчику-человеку. Система предлагает варианты с анализом trade-off, но окончательный выбор остаётся за человеком. Это не ограничение — это гарантия того, что архитектура проекта остаётся осмысленной и контролируемой.

### Принцип 2: Любое изменение начинается с исследования

Система никогда не изменяет код без предварительного понимания контекста. Перед любым изменением модуль Research собирает релевантную информацию: затронутые файлы, связанные сущности, историю изменений, внешнюю документацию, best practices. Без результатов исследования планировщик не получает задачу.

### Принцип 3: Документация является частью исходного кода

Документация не пишется отдельно. Она генерируется из кода, из графа знаний, из истории принятых решений. Документация хранится в репозитории рядом с кодом (директория `docs/`), обновляется автоматически при изменениях и проходит тот же процесс review. Устаревшая документация — это баг.

### Принцип 4: LLM являются заменяемыми компонентами

Система не должна зависеть от конкретного провайдера LLM. Архитектура Provider System обеспечивает абстракцию, позволяющую переключаться между OpenAI, Anthropic, локальными моделями и любыми будущими провайдерами без изменения бизнес-логики. Это защищает от vendor lock-in и позволяет выбирать оптимальную модель под конкретную задачу (цена, качество, latency, privacy).

### Принцип 5: Архитектура модульная

Каждый модуль системы имеет:
- Чётко определённый **контракт** (интерфейс).
- **Независимый жизненный цикл** (может разрабатываться, тестироваться и развёртываться отдельно).
- **Явные зависимости** (никаких циклических зависимостей между модулями).
- **Собственное хранилище** (если модулю нужно состояние, он владеет им; другие модули получают доступ только через API модуля).

### Принцип 6: Контекст первичен, модель вторична

Качество выходного результата определяется не столько мощностью LLM, сколько качеством и полнотой контекста, который был предоставлен модели. Система инвестирует значительные ресурсы в сборку максимально релевантного и полного контекста, прежде чем отправить запрос к LLM.

### Принцип 7: Наблюдаемость (Observability) на всех уровнях

Каждое действие системы должно оставлять след, доступный для анализа. Почему было принято то или иное решение, какой контекст был предоставлен модели, какие файлы были изменены и почему — всё это должно быть доступно для аудита. Без наблюдаемости нет доверия к автономной системе.

### Принцип 8: Graceful Degradation

Система должна корректно работать при недоступности отдельных компонентов. Если LLM-провайдер недоступен, система должна переключиться на резервный. Если Knowledge Base временно недоступна, система должна работать с тем контекстом, который есть, и явно предупредить об ограничениях. Отказ одного модуля не должен приводить к полному отказу системы.

## 4. High Level Architecture

Система состоит из десяти основных модулей. Каждый модуль представляет собой логически завершённую единицу с чётко определёнными границами ответственности.

Модули взаимодействуют друг с другом **исключительно через события**. Прямые вызовы между модулями запрещены. Каждый модуль подписывается на события, которые ему необходимы, и публикует события о результатах своей работы. Это обеспечивает слабую связанность (loose coupling), возможность независимого развития модулей и наблюдаемость всех взаимодействий в системе.

---

### 4.1 Indexer

#### Назначение

Indexer — это подсистема статического анализа и индексации исходного кода проекта. Он сканирует файловую систему проекта, строит абстрактное синтаксическое дерево (AST) для каждого файла, извлекает структурные элементы (классы, функции, методы, интерфейсы, типы, константы, импорты) и регистрирует их в графовой базе знаний.

Indexer работает непрерывно: при запуске он выполняет полное сканирование, затем подписывается на изменения файловой системы и инкрементально обновляет индекс по мере изменения файлов.

#### Ответственность

- Сканирование файловой системы проекта с учётом `.gitignore` и дополнительных правил исключения.
- Парсинг исходного кода в AST для каждого поддерживаемого языка.
- Извлечение символов (symbols): классы, трейты, интерфейсы, функции, методы, свойства, константы, типы, enum-ы.
- Определение отношений между символами: наследование, реализация интерфейса, вызов метода, использование типа, импорт, композиция.
- Вычисление метрик: размер файла, цикломатическая сложность, глубина наследования, количество зависимостей.
- Инкрементальное обновление: при изменении файла переиндексируется только затронутая часть графа.
- Регистрация извлечённой информации в модуле Graph.

#### Вход

- Путь к корневой директории проекта.
- Конфигурация: список поддерживаемых расширений, правила исключения, правила разрешения путей (path aliases).
- События файловой системы (file change events) для инкрементального обновления.

#### Выход

- События добавления/обновления/удаления узлов и рёбер, направляемые в модуль Graph.
- Статистика индексации: количество проиндексированных файлов, символов, отношений, Errors/Warnings.
- AST-представления файлов, доступные для запроса другими модулями.

### 4.2 Graph

#### Назначение

Graph — это графовая база данных сущностей и отношений проекта. Это центральная структура знаний о коде, которая отвечает на вопросы типа: "Какие классы используют этот интерфейс?", "Каков полный граф зависимостей модуля X?", "Какие файлы будут затронуты при изменении метода Y?".

В отличие от реляционной базы, графовая модель естественно представляет структуру кода, где сущности (классы, функции, модули) являются узлами, а отношения (наследование, вызов, зависимость) — рёбрами.

#### Ответственность

- Хранение всех сущностей проекта как узлов графа.
- Хранение всех отношений между сущностями как рёбер графа с типизацией и атрибутами.
- Выполнение графовых запросов: поиск путей, соседей, подграфов.
- Определение зоны влияния (impact analysis): какие сущности зависят от данной.
- Поиск циклов и архитектурных нарушений.
- Версионирование графа: состояние графа привязано к состоянию кодовой базы (коммиту).
- Предоставление API для запросов от других модулей.

#### Вход

- События индексации от Indexer: создание, обновление, удаление узлов и рёбер.
- Запросы от других модулей: поиск зависимостей, анализ влияния, поиск циклов.

#### Выход

- Результаты графовых запросов: множества узлов, пути, подграфы.
- Отчёты об анализе влияния (impact reports).
- Предупреждения об архитектурных нарушениях (циклические зависимости, violation of layer boundaries).

#### 4.2.1 Graph Data Model

Модель данных графа определяет типы узлов (Node) и рёбер (Edge), из которых строится граф проекта. Каждый узел и ребро имеют тип, набор атрибутов и уникальный идентификатор, что позволяет выполнять типизированные графовые запросы.

##### Узлы (Nodes)

Узел представляет единицу знания или структурный элемент проекта. Каждый узел имеет:

- **ID** — уникальный идентификатор в рамках графа проекта.
- **Type** — тип узла (один из перечисленных ниже).
- **Label** — человеко-читаемое имя (например, полное квалифицированное имя класса).
- **Properties** — набор атрибутов, специфичных для типа узла (например, filePath, lineNumber для класса).
- **Version** — версия узла, привязанная к версии графа (см. раздел [Versioning](#8-versioning)).
- **Source** — источник, из которого узел был извлечён (Indexer, Knowledge, пользовательский ввод).

**Типы узлов:**

| Тип узла | Назначение | Пример |
|----------|-----------|--------|
| `Project` | Корневой узел, представляющий весь проект | `client-api` |
| `Module` | Логический модуль внутри проекта (директория, package, namespace) | `apps/api`, `packages/graph` |
| `Feature` | Функциональная возможность, объединяющая несколько компонентов | `OAuth Authentication` |
| `Class` | Класс (включая абстрактные) | `UserController` |
| `Interface` | Интерфейс / контракт | `UserRepositoryInterface` |
| `Trait` | Трейт / миксин | `Timestampable` |
| `Enum` | Перечисление | `UserRole` |
| `Method` | Метод класса или интерфейса | `createUser()` |
| `Function` | Функция (не метод) | `generateToken()` |
| `Property` | Свойство класса | `private string $email` |
| `Constant` | Константа класса или глобальная константа | `MAX_RETRY_COUNT` |
| `Type` | Пользовательский тип / type alias | `UserDTO` |
| `Route` | HTTP-маршрут | `POST /api/users` |
| `Component` | UI-компонент (для frontend) | `UserProfileCard` |
| `APIEndpoint` | Внешний API-эндпоинт, с которым взаимодействует проект | `https://api.stripe.com/v1/charges` |
| `DatabaseTable` | Таблица базы данных | `users` |
| `DatabaseColumn` | Колонка таблицы базы данных | `users.email` |
| `Migration` | Миграция базы данных | `20260708_create_users_table` |
| `Test` | Тестовый файл или тестовый класс | `UserControllerTest` |
| `Documentation` | Документ документации | `docs/api/authentication.md` |
| `ADR` | Архитектурное решение (Architecture Decision Record) | `ADR-003: выбрана стратегия кеширования` |
| `Task` | Задача, поставленная пользователем | `task: добавить OAuth` |
| `ResearchReport` | Отчёт об исследовании | `research: OAuth integration patterns` |
| `ImpactReport` | Отчёт об анализе влияния | `impact: изменение UserRepository` |
| `ExecutionPlan` | План выполнения задачи | `plan: OAuth implementation` |
| `ExecutionStep` | Отдельный шаг выполнения | `step: создать таблицу oauth_tokens` |
| `KnowledgeEntry` | Запись в базе знаний | `knowledge: best-practices/oauth/authorization-code-flow` |
| `ExternalResource` | Внешний ресурс (статья, документация, репозиторий) | `https://datatracker.ietf.org/doc/html/rfc6749` |

##### Рёбра (Edges)

Ребро представляет направленное отношение между двумя узлами. Каждое ребро имеет:

- **ID** — уникальный идентификатор.
- **Type** — тип отношения (один из перечисленных ниже).
- **Source Node** — узел-источник (откуда исходит отношение).
- **Target Node** — узел-цель (куда направлено отношение).
- **Properties** — атрибуты отношения (например, lineNumber для CALLS, cardinality для HAS_MANY).
- **Version** — версия ребра, привязанная к версии графа.

**Типы рёбер:**

| Тип ребра | Семантика | Пример |
|-----------|----------|--------|
| `CALLS` | Метод A вызывает метод B | `UserController::store()` → `UserRepository::create()` |
| `USES` | Класс A использует класс B (не через наследование) | `UserController` → `Request` |
| `IMPLEMENTS` | Класс A реализует интерфейс I | `MysqlUserRepository` → `UserRepositoryInterface` |
| `EXTENDS` | Класс A наследует класс B | `AdminController` → `BaseController` |
| `DEPENDS_ON` | Модуль A зависит от модуля B | `apps/api` → `packages/graph` |
| `BELONGS_TO` | Сущность A принадлежит сущности B | `Method::createUser()` → `Class::UserController` |
| `READS` | Метод A читает свойство B | `UserController::show()` → `Property::$email` |
| `WRITES` | Метод A записывает в свойство B | `UserController::update()` → `Property::$email` |
| `CREATES` | Метод A создаёт экземпляр класса B | `UserFactory::make()` → `Class::User` |
| `EMITS` | Компонент A генерирует событие B | `UserForm` → `Event::onSubmit` |
| `LISTENS` | Компонент A слушает событие B | `UserList` → `Event::UserCreated` |
| `IMPORTS` | Файл A импортирует модуль/класс B | `UserController.ts` → `UserService.ts` |
| `REFERENCES` | Документ A ссылается на документ B | `ADR-003` → `ADR-001` |
| `RESOLVES` | Задача/план A приводит к выполнению шага B | `Task::add-oauth` → `ExecutionStep::create-table` |
| `PRODUCES` | Шаг выполнения A производит артефакт B | `ExecutionStep::create-table` → `Migration::create_oauth_tokens` |
| `AFFECTS` | Изменение в A потенциально влияет на B | `Method::rename()` → `Test::UserControllerTest` |
| `RELATES_TO` | Слабая семантическая связь (ассоциация) | `ADR-003` → `Class::CacheManager` |
| `PRECEDES` | Шаг A должен быть выполнен перед шагом B | `ExecutionStep::create-table` → `ExecutionStep::create-model` |
| `CONFLICTS_WITH` | Изменение A конфликтует с изменением B | `Task::refactor-auth` → `Task::add-oauth` |

##### Зачем нужна типизированная модель данных графа

1. **Точность запросов.** Вместо размытого "найди всё, что связано с UserController", система может задать точный вопрос: "Найди все классы, которые IMPLEMENTS интерфейс, который CALLS метод из UserController". Типизация рёбер делает запросы детерминированными.

2. **Разделение ответственности между Graph и Knowledge.** Graph хранит структурные отношения кода (кто кого вызывает, кто от кого зависит). Knowledge хранит семантические знания (почему именно так, какие альтернативы рассматривались). Типизация узлов и рёбер делает эту границу чёткой: `KnowledgeEntry RELATES_TO Class` — это мост между Knowledge и Graph, но само знание хранится в Knowledge.

3. **Impact Analysis.** Тип ребра `AFFECTS` позволяет Impact Analysis Engine (см. раздел [4.10](#410-impact-analysis-engine)) находить затронутые тесты, миграции и документацию при изменении класса. Тип `DEPENDS_ON` позволяет находить затронутые модули.

4. **Версионирование.** Поскольку каждый узел и ребро имеет версию, граф может быть "отмотан" к любому состоянию проекта (коммиту), что необходимо для сравнения состояний (см. раздел [8. Versioning](#8-versioning)).

5. **Инкрементальное обновление.** При изменении одного файла Indexer обновляет только затронутые узлы и рёбра, не перестраивая весь граф. Типизация позволяет точно определить, какие типы рёбер нужно пересчитать.

### 4.3 Knowledge

#### Назначение

Knowledge — это долговременная память системы. Если Graph отвечает на вопрос "как код выглядит сейчас", то Knowledge отвечает на вопросы "почему код выглядит именно так", "какие решения были приняты", "какие альтернативы рассматривались", "какие best practices применимы в данном контексте", "какие ошибки были допущены и исправлены".

Knowledge агрегирует несколько видов знаний:
- **Проектные знания (Project Knowledge):** архитектурные решения (ADR), результаты исследований, протоколы code review.
- **Технические знания (Technical Knowledge):** документация API, спецификации протоколов, best practices, паттерны.
- **Исторические знания (Historical Knowledge):** история изменений, принятых решений, совершённых ошибок и их исправлений.
- **Внешние знания (External Knowledge):** документация фреймворков, библиотек, внешних API.

#### Ответственность

- Хранение всех типов знаний в структурированном виде.
- Эмбеддинг-индексирование для семантического поиска.
- Версионирование знаний: каждое знание имеет версию, привязанную к моменту времени и состоянию проекта.
- Связывание знаний с сущностями в Graph (например, архитектурное решение связано с затронутыми классами).
- Предоставление API для поиска и извлечения знаний.
- Приоритезация знаний: более актуальные и часто используемые знания имеют больший вес при поиске.

#### Вход

- Архитектурные решения (ADR) в структурированном формате.
- Результаты исследований от модуля Research.
- Результаты code review от модуля Execution.
- Внешняя документация (импортированная или извлечённая).
- Запросы на поиск знаний: по ключевым словам, по семантической близости, по связанным сущностям.

#### Выход

- Ранжированный список релевантных знаний для данного контекста.
- Конкретные документы (ADR, research reports, best practices) в полном тексте.
- Связанные сущности из Graph, к которым привязаны найденные знания.

### 4.4 Research Engine

#### Назначение

Research Engine — это **оркестратор исследования проекта**. Его задача — не хранить данные и не выполнять поиск самостоятельно, а координировать сбор информации из множества источников для получения ответов на инженерные вопросы.

Research Engine получает пользовательскую задачу (например, "добавить OAuth-аутентификацию через Google") и трансформирует её в серию конкретных исследовательских вопросов: "Где в проекте уже есть аутентификация?", "Какие таблицы базы данных затронуты?", "Какие внешние API будут использоваться?", "Какие миграции уже созданы для пользователей?", "Какие best practices применимы для OAuth?", "Какие есть риски безопасности?".

Каждый вопрос направляется к соответствующему источнику данных. Research Engine агрегирует ответы и формирует целостную картину, готовую для передачи следующим модулям.

#### Источники, используемые Research Engine

Research Engine не владеет данными. Он является потребителем API следующих модулей и систем:

| Источник | Что предоставляет | Пример исследовательского вопроса |
|----------|-------------------|-----------------------------------|
| **Graph** | Структурные отношения между сущностями проекта | "Какие классы вызывают `UserRepository::create()`?" |
| **Knowledge** | Накопленные знания: ADR, best practices, история решений | "Какие архитектурные решения уже приняты по аутентификации?" |
| **Git** | История изменений, blame, diff между версиями | "Кто и когда последним менял модуль аутентификации?" |
| **Indexer / AST** | Синтаксическая структура кода, сигнатуры, типы | "Какова сигнатура метода `authenticate()`?" |
| **Документация** | Project docs, внешняя документация фреймворков и библиотек | "Как фреймворк рекомендует реализовывать OAuth middleware?" |
| **External Search** (через Provider System) | Внешние статьи, документация API, спецификации | "Как выглядит актуальный OAuth 2.0 flow для Google?" |
| **File System** (через Workspace) | Содержимое файлов проекта | "Какие конфигурационные файлы содержат ключи API?" |

#### Процесс исследования

1. **Приём задачи и формулировка исследовательских вопросов.**
   Research Engine анализирует пользовательскую задачу и разбивает её на атомарные исследовательские вопросы. Каждый вопрос имеет тип (структурный, исторический, best-practice, внешний) и приоритет.

2. **Параллельный сбор информации.**
   Исследовательские вопросы, не зависящие друг от друга, отправляются к источникам параллельно:
   - Структурные вопросы → Graph (поиск классов, методов, зависимостей).
   - Исторические вопросы → Git (log, blame, diff).
   - Синтаксические вопросы → Indexer/AST (сигнатуры методов, типы параметров).
   - Knowledge-вопросы → Knowledge (семантический поиск по ADR и best practices).
   - Внешние вопросы → Provider System (поиск в интернете, обращение к документации API).

3. **Анализ перекрёстных связей.**
   Ответы из разных источников связываются друг с другом. Например, найденный через Graph класс `GoogleOAuthProvider` связывается с историей его изменений из Git, с ADR о выборе OAuth-стратегии из Knowledge, и с конфигурационным файлом из файловой системы.

4. **Выявление пробелов.**
   Research Engine определяет, на какие вопросы не удалось найти ответы, и формулирует дополнительные исследовательские вопросы. Процесс повторяется итеративно до достижения достаточной полноты исследования или исчерпания источников.

5. **Формирование Research Report.**
   Итоговый структурированный отчёт (см. раздел [7. Build Artifacts](#7-build-artifacts)) агрегирует всю собранную информацию и передаётся одновременно в Knowledge (для сохранения) и в Impact Analysis Engine (для анализа последствий).

#### Ответственность

- Формулировка исследовательских вопросов на основе пользовательской задачи.
- Маршрутизация вопросов к релевантным источникам (Graph, Knowledge, Git, Indexer/AST, документация, внешние источники).
- Оркестрация параллельного и последовательного сбора информации.
- Анализ перекрёстных связей между ответами из разных источников.
- Выявление информационных пробелов и итеративное углубление исследования.
- Формирование структурированного Research Report.
- Публикация Research Report в Knowledge для долговременного хранения.

#### Что Research Engine НЕ делает

- **Не хранит данные.** Все данные хранятся в Knowledge, Graph, Git и файловой системе. Research Engine — stateless-оркестратор.
- **Не выполняет поиск самостоятельно.** Он делегирует поиск модулям, которые владеют соответствующими данными.
- **Не принимает решений о реализации.** Research Engine собирает информацию, но не говорит, *как именно* нужно реализовать задачу. Это ответственность Planner.
- **Не анализирует влияние изменений.** Это ответственность Impact Analysis Engine.

#### Вход

- Пользовательская задача на естественном языке.
- Текущий контекст проекта (Graph version, Knowledge version, Project state).
- Ограничения по scope исследования.
- Доступ к API всех источников (Graph, Knowledge, Git, Indexer, Workspace, Provider System).

#### Выход

- **Research Report** — структурированный документ, содержащий:
  - Сформулированные исследовательские вопросы и ответы на них.
  - Найденные релевантные сущности в коде (классы, методы, файлы).
  - Релевантные архитектурные решения и документы.
  - Историю изменений затронутых компонентов.
  - Рекомендации по реализации (best practices, паттерны, примеры).
  - Предупреждения о потенциальных проблемах и конфликтах.
  - Ссылки на все источники информации.
  - Оценку полноты исследования (процент вопросов, получивших ответ).
  - Список оставшихся без ответа вопросов (для ручного уточнения пользователем).
  - Связи между найденными артефактами (перекрёстные ссылки).
- Событие `ResearchCompleted` в Event Bus (см. раздел [5. Event Driven Architecture](#5-event-driven-architecture)).

### 4.5 Context Builder

#### Назначение

Context Builder — это модуль, который принимает результаты Research и формирует итоговый контекст для LLM. В условиях ограниченного контекстного окна модели (даже при больших окнах в 200K токенов) задача эффективной упаковки релевантной информации становится критически важной.

Context Builder реализует Принцип 6: "Контекст первичен, модель вторична". Он применяет стратегии приоритезации, сжатия и структурирования информации, чтобы максимизировать полезность контекста в рамках доступного окна.

#### Ответственность

- Приём отчёта об исследовании от модуля Research.
- Определение лимитов контекстного окна целевой LLM-модели.
- Приоритезация информации: что должно быть в контексте обязательно, что желательно, что можно опустить.
- Извлечение полного содержимого приоритезированных файлов (через Workspace).
- Форматирование контекста в структурированный промпт.
- Разделение контекста на системный (system prompt), проектный (project context) и операционный (task-specific).
- Кеширование контекста: части контекста, которые не меняются между задачами (например, архитектурные принципы проекта), не должны передаваться заново каждый раз.
- Оценка заполненности контекстного окна и предупреждение при переполнении.

#### Вход

- Отчёт об исследовании от модуля Research.
- Конфигурация целевой LLM-модели (размер контекстного окна).
- Описание задачи.
- Системные инструкции и проектные принципы.

#### Выход

- Структурированный контекст, готовый для передачи LLM, включающий:
  - Системный блок: инструкции, принципы, правила.
  - Проектный блок: архитектура, структура, ключевые абстракции.
  - Операционный блок: содержимое затронутых файлов, сигнатуры, зависимости.
  - Блок знаний: релевантные ADR, best practices, примеры.
  - Блок задачи: описание того, что требуется сделать.
  - Ограничения и предупреждения.
- Метаинформация: использованный объём токенов, процент заполнения окна.

### 4.6 Planner

#### Назначение

Planner — это **модуль планирования, не LLM**. Его задача — преобразовать результаты предшествующих этапов (Research, Impact Analysis, Context) в детальный, структурированный, проверяемый план выполнения. Planner не генерирует код. Он генерирует план, который затем будет передан Execution.

Planner работает итеративно: после каждого шага Execution сообщает о результате, и Planner может скорректировать оставшуюся часть плана с учётом новой информации.

Planner является критическим звеном, обеспечивающим качество результата. Плохой план гарантирует плохой результат, каким бы мощным ни был LLM на этапе Execution.

#### Входные данные Planner

Planner не начинает работу с нуля. Он получает на вход все результаты предшествующих этапов:

| Входной артефакт | Источник | Что содержит |
|------------------|----------|-------------|
| **User Intent** | API Gateway | Исходная задача пользователя, ограничения, приоритеты |
| **Research Report** | Research Engine | Найденные сущности, архитектурные решения, best practices, внешняя информация |
| **Impact Report** | Impact Analysis Engine | Список затронутых модулей/классов/файлов, оценка рисков, конфликтующие изменения |
| **Context Package** | Context Builder | Структурированный контекст (system, project, task-specific) с приоритезированной информацией |
| **Knowledge History** | Knowledge | Предыдущие планы по схожим задачам, извлечённые уроки |

Только получив **все** эти артефакты, Planner приступает к формированию плана.

#### Процесс планирования

1. **Анализ входных артефактов.**
   Planner изучает Research Report (что нужно сделать?), Impact Report (на что это повлияет?), Context Package (каков текущий контекст проекта?). Без любого из этих трёх артефактов Planner не начинает работу.

2. **Стратегическая декомпозиция.**
   Задача разбивается на логические этапы (phases), каждый этап — на атомарные шаги. Этапы определяют порядок: сначала миграции БД, потом модели, потом сервисы, потом контроллеры.

3. **Построение DAG шагов (Directed Acyclic Graph).**
   Определяются зависимости между шагами. Какие шаги могут выполняться параллельно? Какие — только последовательно? Результат — ациклический направленный граф шагов.

4. **Определение критериев приёмки для каждого шага.**
   Не "написать код", а "метод `authenticate()` возвращает JWT-токен при валидных credentials и выбрасывает `AuthenticationException` при невалидных, покрыт тестами с coverage > 80%".

5. **Оценка рисков и альтернативных путей.**
   Используя Impact Report, Planner определяет: какие шаги наиболее рискованны? Какие имеют альтернативные пути? Например: "Если провайдер OAuth недоступен, создать mock-провайдер для тестов".

6. **Передача плана Execution.**
   Execution Plan передаётся модулю Execution. Planner остаётся активным и ожидает обратной связи.

7. **Итеративная корректировка.**
   При получении Execution Report по каждому шагу, Planner анализирует результат и при необходимости корректирует оставшиеся шаги.

#### Роль LLM в Planner

Planner использует LLM как инструмент для генерации плана, но сам процесс планирования не сводится к одному вызову LLM. Planner:
- Проверяет план на структурную целостность (все ли зависимости учтены, нет ли циклов).
- Валидирует план относительно Impact Report (все ли затронутые сущности покрыты шагами).
- Верифицирует критерии приёмки (измеримы ли они, проверяемы ли).
- Сравнивает с историческими планами из Knowledge (не повторяет ли план известные ошибки).

#### Что Planner НЕ делает

- **Не пишет код.** Planner генерирует план, а не реализацию. Код генерирует Execution.
- **Не выполняет исследование.** Planner не ищет информацию — он получает уже готовый Research Report.
- **Не анализирует влияние изменений.** Planner использует готовый Impact Report, сгенерированный Impact Analysis Engine.
- **Не принимает архитектурные решения.** Если в ходе планирования выявляется необходимость архитектурного выбора, Planner фиксирует это как точку принятия решения и требует участия человека.

#### Ответственность

- Приём и анализ всех входных артефактов (Research Report, Impact Report, Context Package, User Intent).
- Стратегическая и тактическая декомпозиция задачи.
- Построение Directed Acyclic Graph шагов с зависимостями.
- Определение измеримых критериев приёмки для каждого шага.
- Оценка рисков на основе Impact Report.
- Итеративная корректировка плана при получении Execution Report.
- Фиксация Execution Plan и всех его изменений в Knowledge для будущего анализа.
- Валидация плана на структурную целостность перед передачей Execution.

#### Вход

- User Intent (исходная задача от пользователя).
- Research Report (от Research Engine).
- Impact Report (от Impact Analysis Engine).
- Context Package (от Context Builder).
- История предыдущих планов (из Knowledge).

#### Выход

- **Execution Plan** (см. раздел [7. Build Artifacts](#7-build-artifacts)):
  - Directed Acyclic Graph шагов с зависимостями и типами рёбер `PRECEDES`.
  - Для каждого шага: описание, предусловия, ожидаемый результат, критерии приёмки, требуемые инструменты.
  - Оценка рисков и альтернативные пути для каждого шага.
  - Связи с затронутыми сущностями (через Graph: `ExecutionStep AFFECTS Class`).
- Событие `ExecutionPlanned` в Event Bus.

### 4.7 Execution

#### Назначение

Execution — это модуль, который выполняет шаги плана. Он работает в цикле: получает очередной шаг от Planner, выполняет его (взаимодействуя с Workspace для операций с файловой системой, с LLM для генерации кода, с инструментами для тестирования и линтинга), проверяет результат и отправляет обратную связь Planner-у.

Execution также выполняет роль "Developer" и "Reviewer". Каждый шаг проходит через стадию реализации и стадию ревью перед тем, как считаться выполненным.

Execution реализует циклический процесс:
- **Plan** → получить шаг от Planner.
- **Develop** → реализовать изменение (сгенерировать код, создать файлы, изменить конфигурацию).
- **Verify** → проверить результат (линтеры, тесты, статический анализ).
- **Review** → self-review сгенерированного кода на соответствие best practices и архитектурным принципам.
- **Fix** → при обнаружении проблем вернуться к Develop.
- **Report** → сообщить Planner-у о результате шага.

#### Ответственность

- Получение очередного шага плана.
- Выполнение шага: генерация изменений кода через LLM.
- Применение изменений в Workspace.
- Запуск проверок: линтеры, статический анализ, тесты.
- Self-review сгенерированного кода.
- Итеративное исправление проблем до достижения критериев приёмки.
- Отправка отчёта о выполнении шага Planner-у.
- Сохранение результатов в Knowledge: что было сделано, какие проблемы возникли, как были решены.

#### Вход

- Шаг плана от Planner.
- Контекст выполнения от Context Builder.
- Доступ к Workspace для операций с файлами.
- Конфигурация инструментов проверки (линтеры, тестовые фреймворки).

#### Выход

- Изменения в файловой системе (применённые через Workspace).
- Отчёт о выполнении шага:
  - Статус: успех / частичный успех / неудача.
  - Список изменённых файлов.
  - Результаты проверок (линтеры, тесты).
  - Обнаруженные проблемы и как они были решены.
  - Извлечённые уроки для сохранения в Knowledge.

### 4.8 Workspace

#### Назначение

Workspace — это **единая среда выполнения**, а не просто абстракция над файловой системой. Это изолированное, контролируемое и воспроизводимое окружение, которое объединяет все аспекты работы с проектом, необходимые для выполнения задачи: файлы, систему контроля версий, среду исполнения кода, терминал, артефакты сборки и sandbox-окружения.

Workspace — это то *место*, где Execution выполняет свою работу, а не просто набор утилит. Каждая задача получает собственный экземпляр Workspace, изолированный от других задач.

#### Компоненты Workspace

##### 1. Файловая система

- Чтение, запись, удаление, перемещение файлов проекта.
- Staging-область: изменения не применяются напрямую к рабочей копии. Они буферизуются, могут быть просмотрены, отменены или закоммичены.
- Snapshot-ы: возможность сохранить полное состояние файловой системы до и после операции.
- Разрешение конфликтов: если разработчик-человек изменил файл, который трогает система, конфликт детектируется, и система либо разрешает его автоматически (merge), либо запрашивает вмешательство пользователя.
- Виртуализация: Workspace не обязательно равен реальной файловой системе. Он может быть overlay над ней с копированием-при-записи (copy-on-write).

##### 2. Git

- Управление ветками: создание feature-веток для каждой задачи, переключение между ветками.
- Коммиты: атомарные коммиты для каждого успешного шага выполнения.
- Diff: просмотр изменений между любыми двумя состояниями (ветками, коммитами, staging-областью).
- Blame: определение автора и времени последнего изменения строки (используется Research Engine).
- Log: история изменений с фильтрацией по файлам, авторам, временным диапазонам.
- Интеграция с удалённым репозиторием: push, pull, fetch, создание pull request.
- Автоматическое разрешение merge-конфликтов на основе контекста задачи.

##### 3. Docker

- Управление Docker-образами: сборка, кеширование, версионирование образов для разных типов проектов (PHP, Node.js, Python).
- Управление контейнерами: создание, запуск, остановка, удаление sandbox-контейнеров.
- Монтирование томов: проброс файлов проекта в контейнер для выполнения кода.
- Сетевая изоляция: контейнеры изолированы от сети по умолчанию; доступ к сети предоставляется явно.
- Resource limits: ограничение CPU, памяти и дискового пространства для каждого sandbox-контейнера.
- Docker Compose: управление многоконтейнерными окружениями (например, приложение + база данных + Redis).

##### 4. Терминал

- Выполнение shell-команд внутри sandbox-контейнера.
- Захват stdout, stderr и кода возврата.
- Таймауты: принудительное завершение команд, превысивших лимит времени.
- Интерактивные команды: поддержка команд, требующих ввода (например, подтверждение в CLI-инструментах).
- Переменные окружения: инжекция секретов, конфигурационных переменных в окружение выполнения.
- Пайплайны команд: последовательное выполнение нескольких команд с передачей контекста.

##### 5. Запуск тестов

- Запуск тестовых фреймворков (PHPUnit, Vitest, Jest, Pytest) внутри sandbox-контейнера.
- Сбор и парсинг результатов тестирования: passed, failed, skipped, coverage.
- Изоляция тестов: каждый тестовый запуск выполняется в чистом окружении.
- Параллельный запуск: независимые тесты запускаются параллельно в разных контейнерах.
- Регрессионное тестирование: автоматический запуск тестов, затронутых изменениями (определяется через Impact Analysis Engine).

##### 6. Артефакты

- Хранение артефактов сборки (build artifacts): скомпилированные файлы, бандлы, отчёты.
- Артефакты задачи (task artifacts): Research Report, Impact Report, Execution Plan, Execution Report.
- Контроль целостности: хеширование артефактов для проверки неизменности.
- Доступ к артефактам: API для запроса артефактов другими модулями.

##### 7. Sandbox

- Полная изоляция выполнения кода от хостовой системы.
- Запрет network egress по умолчанию (кроме явно разрешённых адресов).
- Запрет доступа к чувствительным директориям хоста.
- Ограничение по времени выполнения (hard timeout).
- Ограничение по использованию ресурсов (CPU, RAM, disk I/O).
- Аудит всех выполненных команд.

##### 8. Diff

- Вычисление разницы между любыми двумя состояниями файловой системы.
- Поддержка unified diff, side-by-side diff.
- Структурированный diff: не только текстовый diff, но и diff на уровне AST (какие методы были изменены, какие сигнатуры поменялись).
- Контекстный diff: diff с привязкой к сущностям Graph (diff метода `authenticate()` связан с узлом `Method::authenticate`).

##### 9. Команды

- Унифицированный интерфейс для выполнения команд в окружении проекта.
- Поддержка различных рантаймов: shell, PHP, Node.js, Python.
- Pre- и post-command hooks: линтинг перед коммитом, форматирование после генерации кода.
- Логирование всех выполненных команд с контекстом: кто инициировал, в рамках какой задачи, с каким результатом.

#### Ответственность

- Предоставление унифицированного API для всех компонентов Workspace (файлы, Git, Docker, терминал, тесты, артефакты, sandbox).
- Изоляция окружений: каждая задача получает собственное рабочее пространство.
- Управление жизненным циклом sandbox-контейнеров.
- Сохранение и восстановление snapshot-ов.
- Обеспечение безопасности: все потенциально опасные операции выполняются в изолированном окружении.
- Разрешение конфликтов между изменениями системы и изменениями разработчика-человека.
- Предоставление информации для других модулей: содержимое файлов (для Context Builder), diff (для Execution), статус Git (для Research Engine).

#### Вход

- Команды файловой системы: чтение, запись, удаление, перемещение.
- Команды Git: создание веток, коммиты, diff, log, blame.
- Команды терминала: выполнение shell-команд с параметрами окружения.
- Команды тестирования: запуск тестов с указанием фреймворка и scope.
- Команды sandbox: создание/уничтожение окружений.
- Команды артефактов: сохранение/извлечение артефактов.
- Команды snapshot: создание/восстановление состояния.

#### Выход

- Результаты файловых операций: содержимое файлов, подтверждение записи.
- Git-информация: diff, log, blame, статус репозитория.
- Результаты выполнения команд: stdout, stderr, exit code.
- Результаты тестов: passed/failed/skipped, coverage report.
- Статус sandbox-окружения: запущен, остановлен, ошибка.
- Артефакты: файлы, отчёты, метаданные.
- Событие `FileChanged` при любом изменении файловой системы.
- Событие `TestsCompleted` при завершении тестового прогона.

#### Принципы работы Workspace

1. **Изоляция задач.** Каждая задача получает собственное рабочее пространство, основанное на snapshot-е текущего состояния проекта. Изменения не влияют на другие задачи до явного merge.

2. **Воспроизводимость.** Любое окружение может быть воссоздано из Dockerfile и snapshot-а файловой системы. Никаких "работает только на моей машине".

3. **Безопасность.** Весь пользовательский код выполняется в sandbox. Ни одна строка кода, сгенерированная AI, не выполняется на хостовой системе.

4. **Наблюдаемость.** Каждая операция в Workspace логируется с полным контекстом: какая задача, какой шаг, какая команда, какой результат.

5. **Атомарность.** Каждое изменение либо применяется полностью, либо откатывается. Нет промежуточных "полу-применённых" состояний.


### 4.10 Impact Analysis Engine

#### Назначение

Impact Analysis Engine — это модуль, который определяет последствия предлагаемого изменения до того, как это изменение будет реализовано. Это принципиально отличается от поиска файлов: поиск отвечает на вопрос "где находится X?", анализ влияния отвечает на вопрос "что сломается, если изменить X?".

Impact Analysis Engine является обязательным этапом между Research и Planning. Без Impact Report Planner не начинает работу (см. раздел [4.6](#46-planner)).

#### Почему Impact Analysis — это отдельный модуль

Поиск файлов и анализ влияния — это разные задачи с разной сложностью:

| Критерий | Поиск файлов (Research) | Анализ влияния (Impact Analysis) |
|----------|------------------------|----------------------------------|
| **Вопрос** | "Где находится?" | "Что сломается?" |
| **Направление** | От задачи к коду | От кода к коду |
| **Глубина** | Прямые ссылки | Транзитивное замыкание зависимостей |
| **Результат** | Список файлов | Граф affected-узлов с оценкой риска |
| **Инструмент** | Поиск по Graph, grep, семантический поиск | Траверс графа, статический анализ, анализ истории изменений |

Выделение Impact Analysis в отдельный модуль позволяет:
- Специализировать алгоритмы обхода графа (BFS/DFS по рёбрам `DEPENDS_ON`, `CALLS`, `AFFECTS`).
- Кешировать результаты анализа влияния для часто изменяемых сущностей.
- Строить предиктивные модели на основе истории изменений: "в прошлый раз при изменении этого метода сломались эти тесты, вероятно, сломаются и сейчас".
- Независимо масштабировать модуль (анализ влияния может быть вычислительно дорогим для больших проектов).

#### Ответственность

- **Определение зоны влияния (Impact Zone).**
  Для заданного изменения (например, "изменить сигнатуру метода `UserRepository::findByEmail()`") Impact Analysis Engine определяет:
  - Какие методы вызывают `findByEmail()`? (через ребро `CALLS` в Graph)
  - Какие классы используют `UserRepository`? (через ребро `USES` и `IMPLEMENTS`)
  - Какие тесты покрывают затронутые методы? (через ребро `AFFECTS`)
  - Какие миграции базы данных затронуты? (через ребро `AFFECTS` на `DatabaseTable`)
  - Какая документация ссылается на затронутые классы? (через ребро `REFERENCES`)
  - Какие внешние API затронуты? (через цепочку зависимостей)

- **Анализ зависимостей.**
  Транзитивное замыкание: изменение в классе A влияет на класс B, который влияет на класс C. Impact Analysis Engine определяет полную цепочку влияния, включая косвенные зависимости.

- **Поиск затронутых модулей.**
  Какие модули (директории, пакеты, namespaces) будут затронуты изменением? Это важно для планирования: если изменение затрагивает несколько модулей, риск выше.

- **Анализ рисков.**
  Для каждого затронутого узла определяется уровень риска:
  - **Критический:** изменение затронет production-код без тестового покрытия.
  - **Высокий:** изменение затронет код с тестовым покрытием < 80%.
  - **Средний:** изменение затронет код с хорошим тестовым покрытием.
  - **Низкий:** изменение затрагивает только тесты или документацию.
  - **Неопределённый:** невозможно определить уровень риска (недостаточно данных).

- **Прогноз влияния изменения на проект.**
  На основе истории изменений (из Git и Knowledge) Impact Analysis Engine прогнозирует:
  - Сколько шагов потребуется для реализации (сложность задачи).
  - Какие части изменения наиболее рискованны.
  - Какие альтернативные подходы могут снизить риск (передаётся Planner для рассмотрения).

- **Обнаружение конфликтов.**
  Если в системе есть несколько активных задач, Impact Analysis Engine проверяет, не пересекаются ли их зоны влияния. Например, две параллельные задачи изменяют один и тот же файл — это конфликт, который должен быть разрешён до начала выполнения.

- **Инкрементальный анализ.**
  При изменении плана (после получения Execution Report) Impact Analysis Engine пересчитывает зону влияния только для изменённых шагов, не перестраивая весь анализ.

#### Вход

- **Research Report** от Research Engine: список сущностей, которые планируется изменить.
- Текущее состояние **Graph**: узлы и рёбра, представляющие код проекта.
- История изменений из **Git**: какие файлы менялись вместе в прошлом.
- История из **Knowledge**: предыдущие Impact Reports и их корреляция с реальными последствиями.
- Активные задачи: список параллельно выполняющихся задач для обнаружения конфликтов.

#### Выход

- **Impact Report** (см. раздел [7. Build Artifacts](#7-build-artifacts)):
  - Зона влияния: граф затронутых узлов (классы, методы, тесты, миграции, документация).
  - Уровни риска для каждого затронутого узла.
  - Оценка сложности задачи: количество затронутых узлов, глубина транзитивного замыкания.
  - Конфликтующие изменения: список активных задач, чьи зоны влияния пересекаются с текущей.
  - Рекомендации по снижению риска: предложение альтернативных подходов.
  - Транзитивное замыкание: полная цепочка зависимостей от точки изменения до стабилизации.
- Событие `ImpactAnalysisCompleted` в Event Bus.
- Обновлённые рёбра `AFFECTS` в Graph для затронутых узлов.

### 4.9 Provider System

#### Назначение

Provider System — это слой абстракции над внешними сервисами, который реализует Принцип 4: "LLM являются заменяемыми компонентами". Он предоставляет унифицированный интерфейс для взаимодействия с:

- LLM-провайдерами (OpenAI, Anthropic, Google, локальные модели через Ollama/vLLM).
- Embedding-провайдерами (для семантического поиска).
- Поисковыми системами (для Research).
- Vector store-ами (для Knowledge).

Каждый тип провайдера имеет стандартизированный интерфейс. Добавление нового провайдера не требует изменения кода, использующего провайдер.

#### Ответственность

- Определение и поддержка стандартизированных интерфейсов для каждого типа провайдера.
- Реализация адаптеров для конкретных провайдеров.
- Маршрутизация запросов: выбор оптимального провайдера в зависимости от задачи (цена, качество, скорость, privacy).
- Fallback и retry логика: при недоступности провайдера автоматическое переключение на резервный.
- Rate limiting и управление квотами.
- Мониторинг использования: токены, стоимость, latency.
- Безопасное хранение и передача API-ключей.

#### Вход

- Запросы от других модулей на выполнение операций (completion, embedding, search, vector store query).
- Конфигурация провайдеров: API-ключи, эндпоинты, лимиты, приоритеты.

#### Выход

- Результаты выполнения запросов через выбранного провайдера.
- Метаданные: использованный провайдер, latency, стоимость, количество токенов.
- Событыя мониторинга и алерты (превышение квот, недоступность провайдера).

## 5. Event Driven Architecture

Модули Client **не вызывают друг друга напрямую**. Взаимодействие между модулями происходит исключительно через асинхронную событийную шину (Event Bus). Это архитектурное решение обеспечивает слабую связанность (loose coupling), позволяет независимо развивать и тестировать каждый модуль, а также обеспечивает полную наблюдаемость всех взаимодействий в системе.

### Принципы событийного взаимодействия

1. **Публикация-подписка (Pub/Sub).** Модуль публикует событие в шину, не зная, кто и как на него отреагирует. Заинтересованные модули подписываются на события, которые им необходимы. Отправитель и получатель не знают друг о друге.

2. **Идемпотентность.** Каждое событие содержит уникальный идентификатор и может быть обработано повторно без побочных эффектов. Это обеспечивает надёжность при сбоях.

3. **Asynchronous-first.** Все события обрабатываются асинхронно. Это не исключает возможности синхронного запроса-ответа для некоторых операций (например, запрос к Graph API), но взаимодействие между модулями всегда асинхронное.

4. **События как источник истины.** Состояние системы выводится из цепочки событий, а не хранится как текущее значение. Это позволяет восстановить состояние на любой момент времени и воспроизвести любую ситуацию (event sourcing).

5. **Версионирование событий.** Схема каждого события версионируется. Модули должны уметь обрабатывать как текущую, так и предыдущие версии событий для обеспечения обратной совместимости.

### Категории событий

События разделены на три категории по их источнику и семантике:

#### Системные события (System Events)

Порождаются инфраструктурой и модулями при изменении состояния системы. Эти события отражают факт: "что-то произошло".

| Событие | Источник | Описание |
|---------|----------|----------|
| `ProjectIndexed` | Indexer | Полная индексация проекта завершена |
| `FileChanged` | Workspace | Файл был создан, изменён или удалён |
| `GraphUpdated` | Graph | Состояние графа изменилось (добавлены/удалены/обновлены узлы или рёбра) |
| `KnowledgeUpdated` | Knowledge | База знаний обновлена (добавлен/изменён документ) |
| `SnapshotCreated` | Workspace | Создан snapshot состояния проекта |
| `ProjectStateChanged` | Система | Изменилось общее состояние проекта |

#### События жизненного цикла задачи (Task Lifecycle Events)

Порождаются модулями в процессе обработки пользовательской задачи. Эти события отражают прогресс задачи.

| Событие | Источник | Описание |
|---------|----------|----------|
| `TaskReceived` | API Gateway | Новая задача получена от пользователя |
| `ResearchStarted` | Research Engine | Исследование по задаче началось |
| `ResearchCompleted` | Research Engine | Исследование завершено, опубликован Research Report |
| `ImpactAnalysisStarted` | Impact Analysis Engine | Анализ влияния начался |
| `ImpactAnalysisCompleted` | Impact Analysis Engine | Анализ влияния завершён, опубликован Impact Report |
| `ContextBuilt` | Context Builder | Контекст для задачи собран |
| `ExecutionPlanned` | Planner | План выполнения сформирован |
| `ExecutionStarted` | Execution | Выполнение плана началось |
| `StepStarted` | Execution | Началось выполнение конкретного шага |
| `StepCompleted` | Execution | Шаг выполнен (успешно или с ошибкой) |
| `StepFailed` | Execution | Шаг не удался после исчерпания попыток |
| `TestsCompleted` | Workspace | Тестовый прогон завершён |
| `ExecutionFinished` | Execution | Все шаги плана выполнены |
| `TaskCompleted` | Система | Задача полностью обработана |

#### События мониторинга (Monitoring Events)

Порождаются системой для целей наблюдаемости и алертинга.

| Событие | Источник | Описание |
|---------|----------|----------|
| `ProviderRateLimited` | Provider System | Достигнут лимит запросов к провайдеру |
| `ProviderUnavailable` | Provider System | Провайдер недоступен, активирован fallback |
| `ContextWindowOverflow` | Context Builder | Контекст превышает лимиты окна модели |
| `ConflictDetected` | Impact Analysis Engine | Обнаружен конфликт между активными задачами |
| `ArchitectureViolation` | Graph | Обнаружено нарушение архитектурных границ |
| `SecurityAlert` | Workspace | Потенциально опасная операция заблокирована |

### Реализация Event Bus

На первом этапе Event Bus реализуется через Redis Pub/Sub, что обеспечивает:
- Надёжную доставку сообщений.
- Горизонтальное масштабирование (несколько экземпляров одного модуля могут подписываться на одну очередь).
- Персистентность событий через Redis Streams для возможности повторного чтения истории.

В перспективе — миграция на RabbitMQ или Apache Kafka для обеспечения гарантированной доставки (at-least-once/exactly-once semantics) и долговременного хранения событий.

### Детальное описание этапов

### Этап 1: Поступление задачи

1. Пользователь формулирует задачу на естественном языке.
2. Задача поступает через API Gateway.
3. API Gateway валидирует запрос, проверяет права доступа, определяет целевой проект.
4. Публикуется событие `TaskReceived`.

### Этап 2: Research Engine — Оркестрация исследования

1. Research Engine получает задачу через событие `TaskReceived`.
2. Анализирует задачу и формулирует исследовательские вопросы.
3. Параллельно направляет вопросы к источникам:
   - Graph: структурные отношения, затронутые сущности.
   - Knowledge: ADR, best practices, история решений.
   - Git: история изменений, blame, diff.
   - Indexer/AST: сигнатуры методов, типы параметров.
   - Документация: внутренняя и внешняя.
   - External Search (через Provider System): статьи, API references.
4. Анализирует перекрёстные связи между ответами.
5. Выявляет информационные пробелы и проводит дополнительные итерации.
6. Формирует структурированный Research Report.
7. Сохраняет Research Report в Knowledge.
8. Публикует событие `ResearchCompleted`.

### Этап 3: Impact Analysis Engine — Анализ влияния

1. Impact Analysis Engine получает Research Report через событие `ResearchCompleted`.
2. Определяет зону влияния: какие классы, методы, модули, тесты, миграции будут затронуты.
3. Выполняет транзитивное замыкание зависимостей.
4. Оценивает риски для каждого затронутого узла.
5. Проверяет конфликты с другими активными задачами.
6. Формирует Impact Report.
7. Публикует событие `ImpactAnalysisCompleted`.

### Этап 4: Context Builder — Сборка контекста

1. Context Builder получает Research Report и Impact Report.
2. Приоритезирует информацию: критически важное, полезное, опциональное.
3. Извлекает полное содержимое приоритетных файлов через Workspace.
4. Формирует структурированный Context Package:
   - Системный блок: инструкции и принципы.
   - Проектный блок: архитектура, ключевые абстракции.
   - Операционный блок: затронутые файлы, сигнатуры, зависимости.
   - Блок знаний: релевантные ADR, best practices.
   - Блок задачи: что требуется сделать.
   - Блок рисков: предупреждения из Impact Report.
5. Проверяет лимиты контекстного окна целевой модели.
6. Публикует событие `ContextBuilt`.

### Этап 5: Planner — Планирование

1. Planner получает три обязательных артефакта: Research Report, Impact Report, Context Package.
2. Без любого из этих трёх артефактов Planner не начинает работу.
3. Выполняет стратегическую декомпозицию задачи.
4. Строит Directed Acyclic Graph шагов с зависимостями.
5. Определяет измеримые критерии приёмки для каждого шага.
6. Оценивает риски на основе Impact Report.
7. Формирует Execution Plan.
8. Публикует событие `ExecutionPlanned`.

### Этап 6: Execution — Developer Agent

1. Execution получает Execution Plan.
2. Для каждого шага выполняет цикл:
   - **Plan:** определить конкретные действия.
   - **Develop:** сгенерировать изменения кода через LLM.
   - **Verify:** запустить линтеры, статический анализ, тесты.
   - **Review:** self-review сгенерированного кода.
   - **Fix:** исправить обнаруженные проблемы.
   - **Report:** зафиксировать результат шага.
3. Все операции выполняются через Workspace (файлы, Git, Docker, терминал).
4. Публикует события `StepStarted`, `StepCompleted`/`StepFailed`.

### Этап 7: Execution — Reviewer

1. После выполнения всех шагов Reviewer проводит финальное ревью.
2. Проверяет соответствие best practices и архитектурным принципам.
3. Валидирует тестовое покрытие.
4. Формирует Execution Report.
5. Публикует событие `ExecutionFinished`.

### Этап 8: Обновление знаний, графа и документации

1. **Knowledge Update:** Execution Report и извлечённые уроки сохраняются в Knowledge.
2. **Graph Update:** Indexer реагирует на изменённые файлы, обновляет узлы и рёбра в Graph.
3. **Documentation Update:** Затронутая документация обновляется; при необходимости генерируются новые ADR.
4. Публикуются события `GraphUpdated`, `KnowledgeUpdated`, `TaskCompleted`.
5. Пользователь получает итоговый отчёт.

### Сквозные аспекты потока

На всех этапах:
- **Наблюдаемость:** каждое действие логируется с полным контекстом для последующего аудита.
- **Traceability:** каждое решение связывается с породившим его событием, задачей и контекстом.
- **Graceful Degradation:** при сбое любого модуля система корректно обрабатывает ситуацию и информирует пользователя.
- **Контроль пользователя:** на любом этапе пользователь может вмешаться, скорректировать направление или отменить задачу.
- **Event Sourcing:** полная история событий позволяет воспроизвести любой момент выполнения задачи.

## 7. Build Artifacts

В процессе обработки каждой задачи система создаёт внутренние артефакты — структурированные документы, которые передаются между модулями. Артефакты обеспечивают прослеживаемость (traceability), возможность аудита и повторного использования знаний.

### Список внутренних артефактов

| Артефакт | Создаётся | Потребляется | Назначение |
|----------|----------|-------------|------------|
| **User Intent** | API Gateway | Research Engine | Формализованное описание задачи: что нужно сделать, ограничения, приоритеты, контекст от пользователя |
| **Research Report** | Research Engine | Impact Analysis Engine, Context Builder, Planner, Knowledge | Результаты исследования: найденные сущности, релевантные ADR, best practices, история изменений, внешняя информация |
| **Impact Report** | Impact Analysis Engine | Planner, Context Builder | Зона влияния изменения: затронутые сущности, оценка рисков, конфликты, транзитивное замыкание зависимостей |
| **Context Package** | Context Builder | Planner, Execution | Структурированный контекст для LLM: system prompt, project context, task-specific files, knowledge snippets |
| **Execution Plan** | Planner | Execution | Directed Acyclic Graph шагов с критериями приёмки, зависимостями, оценкой рисков |
| **Execution Report** | Execution (Developer + Reviewer) | Planner, Knowledge | Отчёт о выполнении: изменённые файлы, результаты проверок, обнаруженные проблемы, извлечённые уроки |
| **Knowledge Update** | Knowledge | Knowledge, будущие задачи | Сохранённые знания: Execution Report, новые ADR, извлечённые уроки, обновлённые best practices |
| **Graph Update** | Graph (через Indexer) | Graph, будущие задачи | Обновлённые узлы и рёбра графа, отражающие изменения в коде |

### Зачем нужны артефакты

1. **Traceability.** Каждый артефакт ссылается на породивший его артефакт. Execution Report ссылается на Execution Plan, который ссылается на Impact Report, который ссылается на Research Report, который ссылается на User Intent. В любой момент можно проследить: почему было сделано именно это изменение? Какие альтернативы рассматривались? Какие риски были оценены?

2. **Повторное использование знаний.** Research Report и Impact Report сохраняются в Knowledge. Когда через месяц понадобится сделать похожую задачу, система найдёт предыдущие отчёты и использует их как отправную точку, сокращая время исследования.

3. **Аудит и compliance.** Артефакты обеспечивают полный audit trail. Для regulated industries это критически важно: каждая строка кода имеет задокументированное обоснование.

4. **Восстановление контекста.** При сбое системы артефакты позволяют восстановить состояние задачи и продолжить выполнение с места сбоя.

5. **Обучение системы.** Анализ цепочек артефактов (Intent → Research → Impact → Plan → Execution) позволяет системе учиться на своих ошибках: где планирование было неточным? Где анализ влияния недооценил риски? Где исследование пропустило важный источник?

### Жизненный цикл артефактов

```
User Intent
    │
    ▼
Research Report ───────────► Knowledge (сохранение)
    │
    ▼
Impact Report ─────────────► Knowledge (сохранение)
    │
    ▼
Context Package ───────────► Execution (использование)
    │
    ▼
Execution Plan ────────────► Knowledge (сохранение)
    │
    ▼
Execution Report ──────────► Knowledge (сохранение)
    │
    ▼
Knowledge Update ──────────► Knowledge (persist)
    │
    ▼
Graph Update ──────────────► Graph (persist)
```

Все артефакты хранятся в Knowledge с версионированием. Каждый артефакт получает уникальный идентификатор (UUID) и связывается с породившей его задачей.

## 8. Versioning

Система Client непрерывно изменяет код проекта. Для обеспечения согласованности, возможности отката и анализа изменений между состояниями проекта необходима система версионирования всех ключевых артефактов.

### Версионируемые сущности

#### Graph Version

Граф проекта (модуль Graph) хранит состояние кода в виде узлов и рёбер. Это состояние изменяется при каждом изменении файлов. Graph Version — это снимок (snapshot) графа, привязанный к конкретному состоянию кодовой базы.

**Характеристики:**
- Каждая версия графа привязана к Git-коммиту (`commit hash`).
- Версия графа включает все узлы и рёбра на момент коммита.
- При инкрементальном обновлении Graph создаёт новую версию, но хранит diff относительно предыдущей.
- Graph Version позволяет "отмотать" граф к любому коммиту и выполнить запросы к состоянию кода на тот момент.

**Зачем нужно:**
- Сравнение графов между версиями: какие узлы добавились/удалились, какие рёбра изменились.
- Resync: при рассинхронизации графа с реальным состоянием кода система может перестроить граф от последней консистентной версии.
- Impact Analysis: сравнение зоны влияния до и после изменения.

#### Knowledge Version

База знаний (модуль Knowledge) накапливает документы (ADR, Research Reports, Impact Reports, Execution Reports, best practices). Knowledge Version — это снимок состояния базы знаний на определённый момент.

**Характеристики:**
- Каждая версия Knowledge привязана к временной метке и, опционально, к Git-коммиту.
- Включает все документы и их версии на момент снимка.
- Поддерживает семантическое версионирование документов: Major (несовместимые изменения), Minor (добавления), Patch (исправления).

**Зачем нужно:**
- Понимание, какие знания были доступны на момент принятия того или иного решения.
- Сравнение баз знаний между проектами: какие ADR и best practices были перенесены.
- Resync: при восстановлении после сбоя Knowledge может быть восстановлен до последней консистентной версии.

#### Project Snapshot

Project Snapshot — это полный снимок состояния проекта: файловая система + Graph + Knowledge + активные задачи. Это "точка сохранения", к которой можно вернуться.

**Характеристики:**
- Делается автоматически перед началом каждой задачи.
- Делается автоматически после завершения каждой задачи.
- Может быть сделан вручную пользователем.
- Включает: состояние файлов (через Workspace), Graph Version, Knowledge Version, список активных задач с их артефактами.

**Зачем нужно:**
- Полный откат: если задача пошла не по плану, можно откатить проект к состоянию до начала задачи.
- Анализ "до/после": сравнение двух Project Snapshots показывает полную картину изменений, внесённых задачей.
- Восстановление после сбоев: при крахе системы проект восстанавливается из последнего успешного Project Snapshot.

#### Project State

Project State — это не снимок, а динамическое описание текущего состояния проекта: какие задачи активны, какие модули находятся в процессе изменения, какие конфликты обнаружены.

**Характеристики:**
- Обновляется в реальном времени при изменении состояния.
- Включает: список активных задач, их статус, текущий шаг, зоны влияния.
- Включает: список конфликтов между активными задачами.
- Включает: метрики здоровья проекта (code coverage, technical debt index, architecture violations).

**Зачем нужно:**
- Координация параллельных задач: Impact Analysis Engine использует Project State для обнаружения конфликтов.
- Пользовательский интерфейс: дашборд состояния проекта показывает, что происходит прямо сейчас.
- Принятие решений: Planner использует Project State для учёта контекста активных задач при планировании новой.

### Стратегия версионирования

1. **Автоматические снимки.** Project Snapshot создаётся автоматически перед каждой задачей (`TaskReceived`), после каждого успешного шага (`StepCompleted`) и после завершения задачи (`TaskCompleted`).

2. **Инкрементальное хранение.** Каждая новая версия Graph и Knowledge хранит не полную копию, а diff относительно предыдущей версии. Полные снимки создаются периодически (например, каждый N-й коммит) для ускорения восстановления.

3. **Связь с Git.** Каждая версия Graph и Knowledge привязана к Git commit hash. Это позволяет однозначно сопоставить состояние системы с состоянием кода в репозитории.

4. **TTL для старых версий.** Старые версии Graph (старше, например, 30 дней или 100 версий) могут быть удалены для экономии места, если они не являются частью Project Snapshot.

5. **Resync-процедура.** При обнаружении рассинхронизации (например, граф не соответствует состоянию файлов) запускается процедура Resync:
   - Определяется последняя консистентная Graph Version.
   - Indexer переиндексирует изменения с момента этой версии до текущего состояния.
   - Graph обновляется инкрементально, без полной перестройки.

| Слой | Технология | Обоснование |
|------|-----------|-------------|
| **Frontend** | React + TypeScript + Vite | Де-факто стандарт для современных веб-приложений. TypeScript обеспечивает типобезопасность, что критически важно для сложного UI с множеством состояний. Vite обеспечивает мгновенный HMR и быструю сборку. React выбран из-за обширной экосистемы компонентов и поддержки со стороны сообщества. |
| **Backend** | Node.js + TypeScript + Fastify | Единый язык (TypeScript) на фронтенде и бекенде снижает когнитивную нагрузку и позволяет переиспользовать типы. Fastify выбран вместо Express за лучшую производительность, встроенную валидацию схем, поддержку плагинов и нативную работу с async/await. |
| **База данных (основная)** | PostgreSQL | Зрелая, надёжная реляционная база данных с отличной поддержкой JSON, полнотекстового поиска, транзакций. Используется для хранения пользовательских данных, конфигураций проектов, метаданных. Хорошо интегрируется с Prisma ORM. |
| **База данных (графовая)** | Neo4j | Наиболее зрелая и производительная графовая база данных с expressive языком запросов Cypher. Необходима для модуля Graph, где основная нагрузка — это графовые запросы (поиск путей, соседей, анализ влияния). Альтернатива: ArangoDB (мультимодельная), но Neo4j предпочтительнее как специализированное решение. |
| **Векторная база данных** | pgvector (расширение PostgreSQL) или Qdrant | pgvector позволяет хранить эмбеддинги прямо в PostgreSQL, упрощая инфраструктуру и снижая operational complexity. Для более высоких нагрузок или требований к скорости рекомендуется Qdrant — специализированная векторная база данных с продвинутыми алгоритмами приближённого поиска. |
| **Кеширование** | Redis | Быстрое in-memory хранилище для кеширования контекстов, результатов запросов, сессионных данных и real-time сообщений. Поддержка pub/sub для real-time функциональности. |
| **Docker** | Docker + Docker Compose | Стандарт контейнеризации. Каждый модуль упаковывается в отдельный контейнер. Docker Compose для локальной разработки и тестирования. В перспективе — Kubernetes для production-развёртывания. Sandbox-выполнение кода также использует Docker-контейнеры для изоляции. |
| **Realtime** | WebSocket (через Fastify WebSocket plugin) | Для стриминга прогресса выполнения задачи пользователю, включая логи, текущий шаг, промежуточные результаты. WebSocket предпочтительнее Server-Sent Events из-за двусторонней связи. При масштабировании — Redis pub/sub для координации сообщений между инстансами. |
| **Парсер (AST)** | tree-sitter | Универсальный инкрементальный парсер с поддержкой множества языков. Не требует компиляции под каждый язык отдельно. Используется в Indexer для построения AST. Альтернативы: php-parser для PHP, @babel/parser для TypeScript/JavaScript, но tree-sitter обеспечивает единый подход и лучшую производительность. |
| **LLM Providers** | OpenAI API, Anthropic API, Google Generative AI API, Ollama (локальные модели) | Множественные провайдеры через унифицированный интерфейс Provider System. OpenAI и Anthropic для задач, требующих наивысшего качества. Google — как альтернатива с большим контекстным окном. Ollama — для локального выполнения без передачи данных вовне (privacy-sensitive задачи). |
| **ORM** | Prisma | Типобезопасный ORM с отличной поддержкой TypeScript, автоматической генерацией миграций и интуитивно понятным API. Лучше чем TypeORM по удобству работы с relations и миграциями. |
| **Тестирование** | Vitest | Быстрый тестовый фреймворк, нативно совместимый с Vite и TypeScript. Используется для модульных и интеграционных тестов. |
| **Мониторинг** | OpenTelemetry + Grafana + Loki | Открытый стандарт для сбора трейсов, метрик и логов. OpenTelemetry обеспечивает инструментацию, Grafana — визуализацию, Loki — агрегацию логов. |

### Почему TypeScript является основным языком

1. **Единый язык на всём стеке.** Frontend, backend, tooling — всё на TypeScript. Это снижает когнитивную нагрузку, упрощает переиспользование кода (типы, утилиты, конфигурации) и делает проект более доступным для контрибьюторов.

2. **Типобезопасность.** Для системы, которая сама генерирует и изменяет код, типобезопасность критически важна. Ошибки, пойманные компилятором, не дойдут до рантайма. Это особенно важно, когда код генерируется AI.

3. **Экосистема.** Обширная экосистема npm-пакетов покрывает практически все потребности: от парсинга AST до интеграции с LLM API.

4. **Поддержка AI-инструментами.** TypeScript хорошо понимается современными LLM благодаря большому количеству публичного кода на GitHub.

## 10. Future Vision

### Каким должен стать Client через несколько лет развития

#### Фаза 1: Основа (год 1)

Client функционирует как персональный AI-ассистент разработчика. Он способен:
- Принимать отдельные задачи и выполнять их от начала до конца.
- Индексировать и понимать кодовую базу проекта.
- Проводить исследования перед внесением изменений.
- Накопить базу знаний в рамках одного проекта.
- Работать с ограниченным набором языков (PHP, TypeScript/JavaScript).

Основной пользователь — индивидуальный разработчик или малая команда (2-5 человек). Взаимодействие происходит через IDE-плагин (PhpStorm), CLI и web-интерфейс.

#### Фаза 2: Командная работа (год 2-3)

Client становится частью командного процесса разработки:
- **Мульти-пользовательский режим:** несколько разработчиков работают с одной системой, знания накапливаются коллективно.
- **Team Knowledge Base:** база знаний команды, которая растёт с каждым проектом. Новый член команды получает мгновенный доступ ко всем архитектурным решениям, best practices и истории проекта.
- **Автоматическое ревью:** Client выполняет предварительное ревью всех изменений, проверяя их на соответствие архитектурным принципам, стилю кода и best practices команды.
- **Интеграция с CI/CD:** Client интегрируется в пайплайны, автоматически выполняя рутинные проверки и предлагая улучшения до того, как код дойдёт до человеческого ревью.
- **Поддержка большего числа языков:** Python, Go, Rust, Java.

#### Фаза 3: Организационная память (год 3-5)

Client становится системой организационной памяти в масштабе компании:
- **Cross-project Knowledge Transfer:** знания, добытые в одном проекте, автоматически применяются в других проектах компании. Например, решение проблемы с производительностью базы данных в проекте A немедленно становится доступным для проекта B при схожих обстоятельствах.
- **Архитектурный надзор:** Client постоянно анализирует кодовую базу на предмет отклонений от утверждённой архитектуры, накопления технического долга, деградации производительности — и предлагает конкретные улучшения.
- **Автономное прототипирование:** Client способен самостоятельно исследовать новые технологии, создавать proof-of-concept и представлять результаты команде с анализом trade-off и рекомендациями.
- **Onboarding нового разработчика за часы, а не недели:** благодаря накопленной Knowledge Base и способности объяснять архитектуру и принятые решения, новый разработчик получает контекст мгновенно.

#### Фаза 4: Экосистема (год 5+)

Client становится платформой, а не просто инструментом:
- **Marketplace модулей и провайдеров:** сообщество разрабатывает и публикует адаптеры для новых LLM-провайдеров, парсеры для новых языков, коннекторы к новым инструментам.
- **Shared Knowledge Bases:** открытые базы знаний по популярным фреймворкам, библиотекам и паттернам, поддерживаемые сообществом и используемые Client по умолчанию.
- **Федеративное обучение знаний:** компании могут обмениваться обезличенными паттернами и best practices без раскрытия своего кода, улучшая общую экосистему.
- **Специализированные AI-агенты:** экосистема позволяет создавать узкоспециализированных агентов (агент по безопасности, агент по производительности, агент по доступности), которые работают параллельно и предоставляют экспертизу каждый в своей области.
- **Полная автономия для стандартных задач:** для хорошо изученных доменов (например, создание типового CRUD API, интеграция с типовым внешним сервисом) Client выполняет задачу полностью автономно, без участия человека, с гарантированным качеством.

#### Стратегические принципы развития

1. **Никакой магии.** Каждое новое поколение системы должно быть не менее объяснимым и прозрачным, чем предыдущее. Увеличение автономности не должно приводить к снижению понятности.

2. **Человек остаётся ответственным.** Независимо от уровня автономности, окончательную ответственность за код несёт человек. Система — это инструмент, а не замена инженерному суждению.

3. **Открытость.** Ядро системы должно оставаться открытым. Проприетарные компоненты (если появятся) должны быть изолированы и заменяемы.

4. **Качество важнее скорости.** Лучше получить правильное решение через час, чем сломанное через минуту.

5. **Эволюция, а не революция.** Каждая фаза строится на фундаменте предыдущей. Мы не переписываем систему под новое видение, мы расширяем существующую архитектуру.

6. **Инклюзивность.** Система должна быть полезна как соло-разработчику, так и корпорации из 1000 инженеров. Сложность должна быть опциональной — простые сценарии должны работать просто.
