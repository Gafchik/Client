## Implementation Status (актуализировано 2026-07-11)

Документ ниже описывает **целевую** полиглотную архитектуру хранения (PostgreSQL + Redis + Neo4j + Vector Store + File Storage). Она остаётся верной как долгосрочная vision и не отменяется.

Фактическое состояние реализации на сегодня существенно проще и явно зафиксировано, чтобы устранить расхождение между документацией и кодом (см. `docs/state/project-state.md`, раздел "Что уже реализовано"; открытый вопрос был явно поднят при архитектурном анализе проекта):

- **PostgreSQL исключён из проекта.** Ранее использовался только для `Project`/`Provider` CRUD; полностью выведен.
- **Единственная база данных — Neo4j.** `docker-compose.yml` поднимает только Neo4j. `Project`, `ProjectPath`, `Provider` хранятся в Neo4j как узлы (`apps/api/src/project-store.ts`, `apps/api/src/provider-store.ts`) — то есть Neo4j временно используется не только как graph storage, но и как единственное персистентное хранилище конфигурационных сущностей, которые по целевой архитектуре (раздел 4) должны жить в PostgreSQL.
- **Graph (раздел 6) частично реализован**: после стадии `graph` в pipeline снапшот узлов/рёбер (`GraphNode`/`RELATES`) сохраняется в Neo4j и переживает перезапуск API-процесса (`apps/api/src/graph-store.ts`) — это закрывает разрыв "in-memory graph без персистентности", отмеченный ранее в `project-state.md`. Однако это единый актуальный срез на проект (перезаписывается целиком на каждый run), а не version-per-commit модель из раздела 10 этого документа и не snapshot+overlay модель из `008-next-generation-architecture.md`, раздел 6.2.
- **Redis, Vector Store, File Storage (S3-совместимое) не реализованы.** Кэш контекста, эмбеддинги, артефакты и снапшоты по-прежнему хранятся как JSON на файловой системе внутри `.client/` (см. `knowledge.md`, `packages/knowledge`).
- **Runtime graph queries (`packages/graph`) продолжают работать над in-memory структурой текущего run**, а не читают напрямую из Neo4j — персистентность на сегодня служит только для переживания рестарта процесса и будущего переиспользования (Slice 4 из `008`), не для замены runtime query engine.

Следующие шаги закрытия разрыва между этим документом и кодом описаны как Slice 5+ в `008-next-generation-architecture.md` (Fact-Based Memory, Git Delta & Typed Invalidation) — миграция Knowledge/Context с плоских JSON-файлов на модель фактов, а не восстановление PostgreSQL.

## 1. Цели системы хранения данных

Система хранения данных Client решает следующие фундаментальные задачи:

### 1.1 Целостность и согласованность

Каждая единица информации в Client имеет **ровно одно авторитетное хранилище** (single source of truth). Никакая сущность не дублируется между хранилищами без явной причины. Если данные присутствуют в двух хранилищах, документация точно определяет, какое из них является авторитетным источником, а какое — производной копией (projection).

### 1.2 Производительность под рабочую нагрузку

Разные типы данных имеют принципиально разную природу доступа:

| Тип доступа | Пример | Требование |
|-------------|--------|------------|
| **Транзакционный (OLTP)** | Создание задачи, обновление статуса, запись сообщения | Атомарность, согласованность, изоляция (ACID). Задержка: миллисекунды. |
| **Графовые запросы** | Поиск кратчайшего пути зависимости, анализ влияния изменений | Траверс рёбер за миллисекунды. Глубина обхода: до 10 уровней. |
| **Семантический поиск** | Поиск Knowledge, релевантного текущей задаче | Поиск ближайших соседей в векторном пространстве. Топ-K за миллисекунды. |
| **Потоковая запись** | Логирование сообщений диалога с LLM | Высокая пропускная способность записи. Задержка чтения: некритична. |
| **Кэширование** | Горячие данные графа, результаты частых запросов | Сублинейная задержка. Автоматическая инвалидация. |
| **Файловые операции** | Чтение/запись артефактов, снапшотов | Потоковое чтение. Размеры: от байт до сотен мегабайт. |

Одно хранилище не может эффективно обслуживать все типы доступа. Поэтому Client использует **полиглотное хранение (polyglot persistence)** — комбинацию специализированных хранилищ, каждое из которых оптимально для своего класса задач.

### 1.3 Аудируемость и воспроизводимость

Каждое изменение состояния системы должно быть отслеживаемо:
- **Кто** инициировал изменение (пользователь, Agent, система).
- **Когда** изменение произошло.
- **Почему** изменение произошло (в контексте какой Task, какого Run).
- **Что именно** изменилось (предыдущее и новое состояние).

Это требование диктует:
- Неизменяемость (immutability) критических записей: Artifact, Decision, Message.
- Версионирование мутабельных сущностей: Knowledge, Graph.
- Полный журнал событий (event log) для воспроизведения состояния на любой момент времени.

### 1.4 Масштабируемость

Система хранения должна поддерживать множественные проекты с миллионами файлов каждый. Горизонтальное масштабирование достигается через:
- **Партицирование по project_id** в PostgreSQL.
- **Отдельные базы данных Neo4j** на проект.
- **Шардирование Redis** по ключам.
- **Разделение файлового хранилища** по проектам.

### 1.5 Отказоустойчивость и восстанавливаемость

- RPO (Recovery Point Objective): не более 5 минут для транзакционных данных, не более 24 часов для производных данных (которые могут быть перестроены).
- RTO (Recovery Time Objective): не более 15 минут для критических компонентов, не более 4 часов для полного восстановления.

## 2. Классификация данных по жизненному циклу

### 2.1 Постоянные данные (Persistent)

Данные, которые должны пережить перезагрузку системы, обновление версии, сбой оборудования. Хранятся в долговременных хранилищах с гарантиями сохранности.

| Категория | Сущности | Почему постоянные |
|-----------|----------|-------------------|
| **Конфигурация** | Project, Provider, настройки системы | Определяют поведение системы. Без них система не может функционировать. |
| **Задачи и выполнение** | Task, Run, их статусы и метаданные | История работы. Аудит. Восстановление после сбоя. |
| **Знания** | Knowledge, Decision (ADR) | Накопленная интеллектуальная ценность. Невосстановима из других источников. |
| **Артефакты** | Artifact (все типы) | Результаты работы. Неизменяемые документы. Аудит. |
| **Диалоги** | Conversation, Message | История взаимодействия с пользователем и LLM. Аудит. Трассировка решений. |
| **События** | Event Log | Журнал всех значимых событий. Основа для аудита и воспроизведения. |
| **Снапшоты** | Project Snapshot (Stable) | Точки восстановления проекта. |
| **Структурные метаданные** | Feature | Функциональная карта проекта. |

**Хранилище:** PostgreSQL (первичное), File Storage (для крупных Artifact-ов).

### 2.2 Временные данные (Transient)

Данные, которые существуют только в процессе выполнения операции. Их потеря приемлема, так как они могут быть восстановлены повторным выполнением операции.

| Категория | Сущности | Почему временные |
|-----------|----------|------------------|
| **Контекст LLM** | Context | Собирается для каждого запроса к LLM заново. После использования — не нужен. |
| **Кэш графа** | Горячие узлы и рёбра Graph | Ускоряет доступ к часто запрашиваемым частям графа. Восстанавливается из Neo4j. |
| **Кэш контекста** | Предсобранные фрагменты контекста (Context Package) | Ускоряет построение контекста. Восстанавливается пересборкой. |
| **Состояние очередей** | Очередь задач, очередь событий | Транзитные данные. После обработки — не нужны. |
| **Состояние Workspace** | Файловая система Workspace, Docker-контейнеры | Существуют только на время Run. После — сливаются или удаляются. |
| **Сессии пользователя** | Аутентификационные токены, состояние UI | Временные по определению. |
| **Прогресс операций** | Текущий шаг Run, процент выполнения | Актуален только во время выполнения. |
| **Метрики реального времени** | Текущая нагрузка, latency | Агрегируются в постоянное хранилище периодически. |
| **Состояние Rate Limiter** | Счётчики запросов к провайдерам | Сбрасываются с окном лимита. |

**Хранилище:** Redis (первичное), Workspace File System.

### 2.3 Вычисляемые данные (Computed)

Данные, которые могут быть полностью восстановлены из других (авторитетных) источников. Они не являются самостоятельным источником истины, но хранятся для производительности.

| Категория | Сущности | Почему вычисляемые | Восстанавливаются из |
|-----------|----------|--------------------|-----------------------|
| **Графовая модель кода** | Graph, Node, Edge | Представление (projection) структуры кода. Может быть полностью перестроен парсингом исходных файлов. | Исходный код (File в Workspace/Project) |
| **Эмбеддинги** | Векторные представления Knowledge, кода, ADR | Вычисляются из текста. Могут быть пересозданы повторным прогоном embedding-модели. | Текст Knowledge/кода + Embedding Provider |
| **Индексы и кэши** | Полнотекстовые индексы, кэши поиска | Производные от первичных данных. | PostgreSQL, Neo4j, Vector Store |
| **Метрики и аналитика** | Агрегированные метрики | Вычисляются из сырых событий и логов. | Event Log |
| **Impact Report (частично)** | Результаты анализа влияния | Содержат как вычисляемые данные (результаты графовых запросов), так и постоянные (выводы AI). | Graph (структура) + AI (выводы) |

**Хранилище:** Neo4j (Graph), Vector Store (эмбеддинги), Redis (кэши). Важно: эти хранилища всегда могут быть восстановлены из авторитетных источников. Ни одно из них не является единственным владельцем данных.

## 4. PostgreSQL: реляционная основа

### 4.1 Назначение

PostgreSQL — **главное авторитетное хранилище** системы Client. Все критические данные, требующие ACID-гарантий, хранятся в PostgreSQL. Это единственное хранилище, потеря данных в котором недопустима и невосстановима.

### 4.2 Почему PostgreSQL

| Критерий | Обоснование |
|----------|-------------|
| **ACID** | Задачи, Run-ы, Knowledge требуют строгой согласованности. Частичная запись недопустима. Транзакции гарантируют, что создание Task и связанного Run атомарно. |
| **Ссылочная целостность** | Внешние ключи (FK) гарантируют, что Run всегда ссылается на существующий Task, а Message — на существующий Conversation. |
| **JSONB** | Метаданные Knowledge, конфигурации Provider-ов, payload событий — semi-structured данные, которые удобно хранить в JSONB с возможностью индексации и запросов по вложенным полям. |
| **Полнотекстовый поиск** | Встроенный `tsvector` для поиска по текстам задач, Knowledge, Message без необходимости во внешнем поисковом движке на начальном этапе. |
| **Зрелость и экосистема** | Десятилетия эксплуатации, предсказуемое поведение, огромное сообщество, инструменты миграции, бэкапа, мониторинга. |
| **Расширяемость** | PostGIS для возможных гео-данных, pgvector как легковесная альтернатива внешнему Vector Store, TimescaleDB для временных рядов метрик. |

### 4.3 Какие данные хранятся

| Группа сущностей | Конкретные сущности | Формат хранения |
|-------------------|---------------------|-----------------|
| **Проект** | Project (метаданные, конфигурация, статус) | Строго реляционно |
| **Структурные метаданные** | Feature, связь Feature → Node | Реляционно |
| **Задачи** | Task (статус, приоритет, метаданные), связи Task → Run | Реляционно |
| **Выполнение** | Run (статус, тип, метаданные), связь Run → Workspace | Реляционно |
| **Знания (метаданные)** | Knowledge (статус, версия, тип), Decision/ADR (статус, связи supersedes) | Реляционно + JSONB для тела |
| **Диалоги** | Conversation (метаданные), Message (отправитель, timestamp, тело) | Реляционно + JSONB |
| **Конфигурация** | Provider (конфигурация, API-ключи), настройки системы | Реляционно + JSONB (зашифровано) |
| **События** | Event Log (все критические события) | Реляционно + JSONB для payload |
| **Снапшоты (метаданные)** | Snapshot (тип, версия, Git commit, ссылки на файлы) | Реляционно |
| **Artifact (метаданные)** | Artifact (тип, размер, ссылка на тело в File Storage) | Реляционно |
| **Аудит** | Audit Trail (журнал всех изменений критических сущностей) | Реляционно |

### 4.4 Принципы использования

**Партицирование.** Таблицы с высоким объёмом данных (Event Log, Message) партицируются по `project_id` и/или временному диапазону.

**Мягкое удаление.** Сущности не удаляются физически. Используется флаг `deleted_at` (soft delete). Исключение: Event Log (хранится с retention policy).

**История изменений.** Критические сущности (Task, Knowledge, Decision) имеют audit-таблицы, в которые записывается каждое изменение (кто, когда, что изменил).

**JSONB для гибкости.** Тело Knowledge, конфигурация Provider-ов, payload Artifact-ов хранятся как JSONB. Это позволяет менять схему данных без миграций для слабоструктурированных данных.

**Индексация.** Все внешние ключи индексированы. Поля, по которым выполняется поиск (статус Task, тип Knowledge, project_id), индексированы. JSONB-поля имеют GIN-индексы для запросов по вложенным ключам.

## 5. Redis: оперативная память системы

### 5.1 Назначение

Redis — **оперативное хранилище** системы Client. Все данные в Redis являются либо временными, либо кэшем (восстановимыми). Redis не является авторитетным источником ни для каких данных. Потеря данных Redis не приводит к потере информации — только к временной деградации производительности.

### 5.2 Почему Redis

| Критерий | Обоснование |
|----------|-------------|
| **Сублинейная задержка** | Операции за микросекунды/миллисекунды. Критично для кэша графа (тысячи запросов в секунду при сборке контекста) и очередей событий. |
| **TTL** | Встроенная поддержка Time-To-Live. Кэш автоматически инвалидируется без ручного управления. |
| **Pub/Sub** | Встроенная публикация/подписка для событийной системы. События доставляются подписчикам мгновенно. |
| **Структуры данных** | Sorted Sets для очередей с приоритетом, Hashes для объектов, Streams для event sourcing, HyperLogLog для приблизительных счётчиков. |
| **Атомарные операции** | INCR, DECR для rate limiting, счётчиков токенов, метрик. |
| **Персистентность (RDB/AOF)** | Опционально: для критических очередей можно включить сохранение на диск для восстановления после рестарта. |
| **Кластеризация** | Redis Cluster для горизонтального масштабирования. |

### 5.3 Какие данные хранятся

| Категория | Конкретные данные | Структура Redis | TTL |
|-----------|-------------------|-----------------|-----|
| **Кэш графа** | Горячие узлы (Node) и рёбра (Edge), часто запрашиваемые подграфы | Hash (JSON-сериализованный объект) + ключ `graph:{project_id}:node:{node_id}` | 5–15 минут |
| **Кэш контекста** | Предсобранные фрагменты контекста (Context Package) | String (JSON) | До инвалидации (событие ContextInvalidated) |
| **Кэш Knowledge** | Результаты частых запросов к Knowledge | String (JSON) | 5–15 минут |
| **Кэш эмбеддингов** | Часто запрашиваемые эмбеддинги (горячие Knowledge) | String (binary blob) | 1 час |
| **Сессии** | Пользовательские сессии, JWT-токены | Hash | Время жизни сессии |
| **Прогресс** | Текущий шаг Run, процент выполнения, статус | Hash | До завершения Run |
| **Очередь задач** | Очередь задач с приоритетами | Sorted Set (score = priority) | До обработки |
| **Очередь событий** | События для асинхронной обработки | Stream | До обработки |
| **Блокировки** | Distributed locks для координации Agent-ов | String (SET NX) | До 30 секунд (lease) |
| **Rate Limiting** | Счётчики запросов к провайдерам | String (INCR + TTL) | Окно лимита |
| **Pub/Sub** | Каналы для событийной системы (FileChanged, GraphUpdated, etc.) | Channel | N/A (мгновенная доставка) |
| **Метрики (real-time)** | Текущая нагрузка, latency, количество активных Run-ов | Hash, String | Периодически сбрасываются в PostgreSQL |

### 5.4 Принципы использования

**Кэш, а не источник.** Любые данные в Redis должны быть восстановимы из авторитетного источника (PostgreSQL, Neo4j). При старте системы Redis пуст и наполняется по мере использования (cache-aside pattern).

**TTL на всё.** Каждый ключ в Redis имеет явный TTL. Нет "вечных" ключей. Это предотвращает утечки памяти и гарантирует, что устаревшие данные будут автоматически удалены.

**Инвалидация по событиям.** При изменении данных в авторитетном источнике соответствующие ключи в Redis инвалидируются (удаляются) через событийную систему. Например, событие `GraphUpdated` вызывает удаление всех ключей `graph:{project_id}:*`.

**Ограничение памяти.** Для Redis настраивается `maxmemory` и политика вытеснения (`allkeys-lru`). При нехватке памяти наименее используемые ключи вытесняются автоматически.

## 6. Graph Storage (Neo4j): графовая модель кода

### 6.1 Назначение

Neo4j — **вычисляемое хранилище**, содержащее графовую модель кодовой базы. Это единственное хранилище, оптимизированное для графовых запросов: поиск зависимостей, анализ влияния, поиск путей, обнаружение циклов.

### 6.2 Почему Neo4j

| Критерий | Обоснование |
|----------|-------------|
| **Нативная графовая модель** | Данные хранятся как узлы и рёбра, а не эмулируются через JOIN-ы. Траверс рёбер — операция O(1), а не O(log N) как в реляционных БД. |
| **Cypher** | Декларативный язык графовых запросов. `MATCH (a:Class)-[:CALLS*1..5]->(b:Class)` — запрос, который на SQL потребовал бы рекурсивного CTE на десятки строк. |
| **Производительность** | Траверс отношений между миллионами узлов за миллисекунды. Критично для Impact Analysis Engine и Context Builder. |
| **Гибкая схема** | Узлы и рёбра могут иметь произвольные свойства. Добавление нового типа узла не требует миграции схемы. |
| **APOC и GDS** | Библиотеки для сложных графовых алгоритмов: PageRank, community detection, shortest path, centrality. Полезны для анализа архитектуры. |

### 6.3 Какие данные хранятся

| Данные | Тип Neo4j | Описание |
|--------|-----------|----------|
| **Node** | Узел (Node) с лейблом типа | Class, Interface, Method, Function, Route, File, Module, Folder, Component, DatabaseTable, Migration, APIEndpoint, Feature, Knowledge, ADR, Task, Provider |
| **Edge** | Отношение (Relationship) с типом | CALLS, EXTENDS, IMPLEMENTS, BELONGS_TO, CONTAINS, DEPENDS_ON, IMPORTS, HANDLED_BY, AFFECTS, RELATES_TO, REFERENCES, SUPERSEDES, READS, WRITES, USES, SNAPSHOT_OF, VERSION_OF |
| **Свойства узлов** | Properties | Имя, полный путь (FQN), местоположение (файл:строка), видимость, сигнатура, метрики (LOC, сложность) |
| **Свойства рёбер** | Properties | Номер строки вызова, количество вызовов, conditional (boolean) |
| **Версия графа** | Метка на узлах/свойство | Каждый узел содержит `graph_version` — версию графа, к которой он принадлежит |
| **Метрики графа** | Вычисляемые метрики | Количество узлов/рёбер по типам, связность, цикломатическая сложность |

### 6.4 Принципы использования

**Вычисляемое хранилище.** Graph не является авторитетным источником. Авторитетный источник структуры кода — исходные файлы в проекте. Graph может быть полностью перестроен из исходного кода.

**Версионирование.** Каждая версия Graph привязана к Git-коммиту. При инкрементальном обновлении узлы и рёбра помечаются актуальной версией. При Resync старые версии удаляются.

**Связь с Knowledge.** Узлы Knowledge и ADR в графе связываются с узлами кода через отношение `RELATES_TO`. Это позволяет выполнять запросы вроде: "Найти все Knowledge, связанные с этим методом, и все методы, на которые этот метод влияет".

**Изоляция проектов.** Каждый проект имеет отдельную базу данных Neo4j (или отдельный граф в мульти-графовой инсталляции). Это обеспечивает изоляцию и позволяет независимое масштабирование.

**Кэширование в Redis.** Наиболее часто запрашиваемые узлы и подграфы кэшируются в Redis для снижения нагрузки на Neo4j.

## 7. Vector Storage: семантическая память Knowledge

### 7.1 Назначение

Vector Storage — **вычисляемое хранилище**, содержащее векторные представления (embedding-и) Knowledge, кода и ADR. Предназначено для семантического поиска: найти Knowledge, релевантное текущей задаче, найти похожий код, найти связанные архитектурные решения.

### 7.2 Почему отдельное Vector Storage

| Критерий | Обоснование |
|----------|-------------|
| **ANN (Approximate Nearest Neighbors)** | Поиск ближайших соседей в пространстве высокой размерности (768–4096 измерений). Специализированные индексы (HNSW, IVF) обеспечивают поиск за миллисекунды. |
| **Независимость от embedding-модели** | При смене embedding-провайдера или модели все эмбеддинги могут быть пересозданы без изменения остальных данных. |
| **Гибкость выбора** | На старте может использоваться pgvector (встроенный в PostgreSQL) для простоты. При росте объёмов — миграция на специализированное решение (Qdrant, Milvus, Pinecone) без изменения архитектуры. |

### 7.3 Технологический выбор

**Начальный этап: pgvector (расширение PostgreSQL).**

| Критерий | Обоснование |
|----------|-------------|
| Нулевые операционные расходы | Не требует отдельного сервера, кластера, мониторинга. |
| Транзакционность | Эмбеддинг создаётся в той же транзакции, что и Knowledge. |
| Достаточная производительность | До 1M векторов — поиск за миллисекунды с HNSW-индексом. |
| Простота миграции | При переходе на внешнее Vector Storage структура данных не меняется. |

**При росте: Qdrant или аналог (специализированное Vector Storage).**

| Критерий | Обоснование |
|----------|-------------|
| Горизонтальное масштабирование | Распределённые индексы, шардирование. |
| Фильтрация | Комбинированный поиск: семантическая близость + фильтр по project_id, типу, статусу. |
| Payload | Хранение метаданных вместе с вектором, без отдельного запроса в PostgreSQL. |
| Количественная оценка | Точные метрики расстояния (cosine, euclidean, dot product). |

### 7.4 Какие данные хранятся

| Данные | Что векторизуется | Размерность | Частота обновления |
|--------|-------------------|-------------|---------------------|
| **Knowledge embedding** | Заголовок + тело + ключевые выводы | 768–1536 | При создании/обновлении Knowledge |
| **ADR embedding** | Заголовок + контекст + решение + trade-off | 768–1536 | При создании ADR |
| **Code chunk embedding** | Фрагменты кода (метод, класс, файл) | 768–1536 | При индексации (по мере изменения кода) |
| **Message embedding** | История диалога (опционально, для поиска по прошлым решениям) | 768–1536 | Периодически, батчами |

### 7.5 Принципы использования

**Вычисляемое хранилище.** Embedding-и всегда могут быть пересозданы из исходного текста (Knowledge, код) путём повторного вызова embedding-модели. Потеря Vector Storage восстановима.

**Связь с PostgreSQL.** Каждый вектор в Vector Storage имеет внешнюю ссылку на сущность в PostgreSQL (`knowledge_id`, `adr_id`). PostgreSQL — авторитетный источник. Vector Storage — индекс для поиска.

**Версионирование embedding-ей.** При обновлении Knowledge создаётся новый embedding. Старый embedding сохраняется до тех пор, пока старая версия Knowledge доступна для истории. При смене embedding-модели все embedding-и пересоздаются в фоновом режиме.

**Гибридный поиск.** Поиск Knowledge комбинирует семантическую близость (через Vector Storage) и структурные фильтры (через PostgreSQL: project_id, тип Knowledge, статус, теги).

## 8. File Storage: артефакты и снапшоты

### 8.1 Назначение

File Storage — хранилище для крупных неизменяемых файлов: тел Artifact-ов (Research Report, Execution Plan, сгенерированный код), снапшотов состояния проекта, истории Git-репозиториев, логов выполнения.

### 8.2 Почему отдельное File Storage

| Критерий | Обоснование |
|----------|-------------|
| **Размер данных** | Artifact может достигать десятков мегабайт. Снапшот проекта — сотен мегабайт. Хранение в PostgreSQL (BLOB) неэффективно и дорого. |
| **Потоковый доступ** | Artifact-ы читаются и записываются последовательно, не требуя транзакционности. |
| **Неизменяемость** | После создания Artifact/Snapshot не изменяется. Это идеально соответствует объектному хранилищу (S3-like). |
| **Дедупликация** | Снапшоты одного проекта имеют общие неизменённые файлы. |

### 8.3 Технологический выбор

**S3-совместимое хранилище** (MinIO для self-hosted, AWS S3/Cloudflare R2 для cloud).

| Критерий | Обоснование |
|----------|-------------|
| Стандартный API | S3 API поддерживается всеми облачными провайдерами и self-hosted решениями. |
| Неограниченный объём | Автоматическое масштабирование без операционного вмешательства. |
| Дешёвое хранение | Классы хранения (Standard, Infrequent Access, Glacier) для оптимизации стоимости. |
| Версионирование на уровне бакета | Встроенное версионирование объектов — дополнительная защита от случайного удаления. |
| Pre-signed URLs | Безопасный доступ к файлам без проксирования через backend. |

### 8.4 Какие данные хранятся

| Данные | Путь в бакете | Размер | Жизненный цикл |
|--------|---------------|--------|----------------|
| **Artifact body** | `artifacts/{project_id}/{artifact_id}/body.json` (или `.md`, `.diff`) | 1 KB – 50 MB | Постоянно (Retention Policy) |
| **Artifact attachments** | `artifacts/{project_id}/{artifact_id}/attachments/{file}` | Variable | Постоянно |
| **Stable Snapshot** | `snapshots/{project_id}/stable/{snapshot_version}/` | 10 MB – 1 GB | Постоянно (последние N + periodic) |
| **Pre-Run Snapshot** | `snapshots/{project_id}/pre-run/{run_id}/` | 10 MB – 1 GB | До завершения Run + grace period |
| **Incremental Snapshot** | `snapshots/{project_id}/incremental/{snapshot_version}/` | Diff от предыдущего | До создания Stable Snapshot |
| **Workspace state** | `workspaces/{run_id}/` | 10 MB – 1 GB | На время Run + 24 часа после |
| **Git working copy** | `git/{project_id}/working/` | Variable | Постоянно (рабочая копия) |
| **Git bare repo** | `git/{project_id}/bare/` | Variable | Постоянно |
| **Execution logs** | `logs/{run_id}/execution.log` | 1 KB – 100 MB | Retention Policy |
| **AI call logs** | `logs/{run_id}/ai_calls/{call_id}.json` | 1 KB – 1 MB | Retention Policy |

### 8.5 Принципы использования

**Метаданные в PostgreSQL, тело — в File Storage.** PostgreSQL хранит только метаданные Artifact (тип, размер, content-type, checksum) и ссылку на объект в File Storage. Это позволяет быстро искать Artifact-ы по метаданным без загрузки тела.

**Content-Addressable хранение (опционально).** Artifact-ы могут адресоваться по SHA-256 содержимого. Одинаковые Artifact-ы не дублируются. Это особенно полезно для снапшотов (неизменённые файлы ссылаются на тот же объект).

**Pre-signed URLs для доступа.** Фронтенд получает временную pre-signed URL для загрузки/скачивания файла напрямую из File Storage, минуя backend (экономия пропускной способности).

**Классы хранения.** Активные снапшоты — Standard. Старые снапшоты — Infrequent Access. Архивные данные — Glacier (с задержкой восстановления).

## 9. Сущность → Хранилище: полная карта

Для каждой сущности из Domain Model указано: где хранится, почему, жизненный цикл хранения, требования к консистентности.

### 9.1 Project

| Аспект | Значение |
|--------|----------|
| **Где хранится** | PostgreSQL (метаданные, конфигурация, статус) |
| **Почему** | Корневая сущность. Требует ACID: создание проекта может включать несколько таблиц (project, project_config, project_providers). Транзакция гарантирует атомарность. |
| **Жизненный цикл в хранилище** | Создаётся → активно → archived (soft delete) → deleted (hard delete после retention). |
| **Требования к консистентности** | Strong consistency. Изменения проекта (конфигурация, провайдеры) видны всем модулям немедленно. |

### 9.2 Workspace

| Аспект | Значение |
|--------|----------|
| **Где хранится** | File Storage (файловая система Workspace), PostgreSQL (метаданные: ID, связь с Run, статус) |
| **Почему** | Workspace — это файловая система, а не запись в БД. Метаданные — в PostgreSQL для связи с Run и Task. |
| **Жизненный цикл в хранилище** | Создаётся при старте Run → активно → merged/discarded. Файлы: merged — влияют на основной проект; discarded — удаляются через grace period (24 часа). |
| **Требования к консистентности** | Изоляция (каждый Run — собственный Workspace). Strong consistency метаданных в PostgreSQL. |

### 9.3 Module

| Аспект | Значение |
|--------|----------|
| **Где хранится** | Neo4j (Node с лейблом Module), Redis (кэш для горячих модулей) |
| **Почему** | Module — структурный элемент кода. Его ценность — в связях с другими Module и File. Хранение в графе обеспечивает эффективный траверс зависимостей. |
| **Жизненный цикл в хранилище** | Создаётся Indexer-ом → обновляется при изменении содержимого → удаляется при удалении модуля. |
| **Требования к консистентности** | Eventual consistency. Индексация происходит с задержкой относительно изменений в коде (секунды). |

### 9.4 File

| Аспект | Значение |
|--------|----------|
| **Где хранится** | File Storage (содержимое), Neo4j (Node с лейблом File, связи с Module, Symbol), Redis (кэш содержимого для частых запросов) |
| **Почему** | Как и Module — ценность в связях. Граф содержит связи File → Symbol. Содержимое — в File Storage (Git). |
| **Жизненный цикл в хранилище** | Обнаруживается Indexer-ом → Node создаётся в Neo4j → содержимое изменяется → Node обновляется → удаляется при удалении файла. |
| **Требования к консистентности** | Eventual consistency для Neo4j (секунды после изменения файла). Strong consistency для содержимого (Git). |

### 9.5 Folder

| Аспект | Значение |
|--------|----------|
| **Где хранится** | Neo4j (Node с лейблом Folder, связь CONTAINS с File и Folder) |
| **Почему** | Чисто структурная сущность. Хранится в графе для иерархической навигации. |
| **Жизненный цикл в хранилище** | Аналогичен Module. |
| **Требования к консистентности** | Eventual consistency. |

### 9.6 Symbol (и подтипы: Class, Interface, Method, Function, Route)

| Аспект | Значение |
|--------|----------|
| **Где хранится** | Neo4j (Node с лейблом типа: Class, Interface, Method, Function, Route), Redis (кэш сигнатур) |
| **Почему** | Symbol-ы — атомарные единицы графа кода. Их главная ценность — в отношениях: Class EXTENDS Class, Method CALLS Method. Графовая БД — единственное хранилище, где эти запросы эффективны. |
| **Жизненный цикл в хранилище** | Извлекается Indexer-ом из AST → Node создаётся в Neo4j → при изменении файла Node обновляется (удаляется и создаётся заново — проще, чем обновлять свойства) → удаляется при удалении Symbol. |
| **Требования к консистентности** | Eventual consistency. Допустима задержка между изменением кода и обновлением графа (секунды). |

### 9.7 API Endpoint

| Аспект | Значение |
|--------|----------|
| **Где хранится** | PostgreSQL (метаданные: URL, метод, документация), Neo4j (Node с лейблом APIEndpoint, связи CALLED_BY с Method) |
| **Почему** | В отличие от внутренних Symbol-ов, API Endpoint может быть создан пользователем или Research Engine. Метаданные — в PostgreSQL, структурные связи — в Neo4j. |
| **Жизненный цикл в хранилище** | Создаётся Indexer-ом, Research Engine или пользователем → обновляется → может быть deprecated. |
| **Требования к консистентности** | Eventual consistency для Neo4j, strong для PostgreSQL. |

### 9.8 Database Table

| Аспект | Значение |
|--------|----------|
| **Где хранится** | Neo4j (Node с лейблом DatabaseTable, связи с Migration и Method), PostgreSQL (метаданные, если создано вручную) |
| **Почему** | Аналогично API Endpoint: структурные связи — в Neo4j, метаданные — в PostgreSQL. |
| **Жизненный цикл в хранилище** | Аналогичен API Endpoint. |
| **Требования к консистентности** | Eventual consistency для Neo4j. |

### 9.9 Migration

| Аспект | Значение |
|--------|----------|
| **Где хранится** | Neo4j (Node с лейблом Migration, связи AFFECTS с DatabaseTable), PostgreSQL (метаданные: статус Applied/Pending, порядковый номер) |
| **Почему** | Статус миграции — критический атрибут, требующий ACID. Структурные связи — в графе. |
| **Жизненный цикл в хранилище** | Обнаруживается/создаётся → Applied/Rolled Back. |
| **Требования к консистентности** | Strong consistency для статуса. Eventual для графа. |

### 9.10 Component

| Аспект | Значение |
|--------|----------|
| **Где хранится** | Neo4j (Node с лейблом Component, связи USES с Component, BELONGS_TO с Module) |
| **Почему** | Аналогичен Symbol: ценность в графовых связях. |
| **Жизненный цикл в хранилище** | Аналогичен Symbol. |
| **Требования к консистентности** | Eventual consistency. |

### 9.11 Knowledge

| Аспект | Значение |
|--------|----------|
| **Где хранится** | PostgreSQL (метаданные + версии + тело), Vector Storage (embedding), Neo4j (Node с лейблом Knowledge, связь RELATES_TO), Redis (кэш) |
| **Почему** | Knowledge — самая сложная сущность с точки зрения хранения. Метаданные и версионирование — ACID (PostgreSQL). Семантический поиск — Vector Storage. Структурные связи — Neo4j. Горячие данные — Redis. |
| **Жизненный цикл в хранилище** | Создаётся → Active → Updated (новая версия, старая сохраняется) → Deprecated → Archived. |
| **Требования к консистентности** | Strong consistency для метаданных (PostgreSQL). Eventual для Vector Storage и Neo4j. При создании Knowledge: сначала запись в PostgreSQL, затем (асинхронно) embedding и связи в графе. |

### 9.12 Research

| Аспект | Значение |
|--------|----------|
| **Где хранится** | PostgreSQL (состояние, список вопросов, прогресс), Redis (текущий прогресс) |
| **Почему** | Research — процесс, а не результат. Состояние процесса — в PostgreSQL для восстановления после сбоя. Текущий прогресс — в Redis для скорости. |
| **Жизненный цикл в хранилище** | Создаётся → выполняется (обновляется прогресс) → завершается (создаётся Knowledge). |
| **Требования к консистентности** | Strong consistency для состояния (чтобы после сбоя Research не потерялся и не дублировался). |

### 9.13 Context

| Аспект | Значение |
|--------|----------|
| **Где хранится** | Redis (на время использования), PostgreSQL (для аудита — опционально, только метаданные: что вошло в контекст) |
| **Почему** | Context — одноразовый. Собирается для конкретного запроса к LLM. Хранение в Redis с коротким TTL. |
| **Жизненный цикл в хранилище** | Создаётся → используется (передаётся LLM) → удаляется из Redis (TTL). Метаданные аудита — в PostgreSQL (Event Log). |
| **Требования к консистентности** | Неприменимо (одноразовая сущность). |

### 9.14 Task

| Аспект | Значение |
|--------|----------|
| **Где хранится** | PostgreSQL (статус, метаданные, связи с Run и Feature) |
| **Почему** | Центральная оркестрирующая сущность. Требует ACID: изменение статуса Task может каскадно влиять на Run-ы, Workspace, Artifact-ы. |
| **Жизненный цикл в хранилище** | Created → Queued → Researching → Impact Analyzing → Planning → Executing → Reviewing → Completed → Archived. Каждый переход — транзакция. |
| **Требования к консистентности** | Strong consistency. Статус Task должен быть всегда согласован со статусами Run-ов. |

### 9.15 Run

| Аспект | Значение |
|--------|----------|
| **Где хранится** | PostgreSQL (статус, тип, метаданные, связь с Task и Workspace) |
| **Почему** | Требует ACID: создание Run атомарно резервирует Workspace. Завершение Run атомарно обновляет статус Task. |
| **Жизненный цикл в хранилище** | Created → Running → Completed / Failed / Retrying / Cancelled. |
| **Требования к консистентности** | Strong consistency. Run не может быть Completed, если Task — Cancelled. |

### 9.16 Artifact

| Аспект | Значение |
|--------|----------|
| **Где хранится** | File Storage (тело), PostgreSQL (метаданные: тип, размер, checksum, ссылка на тело, связь с Run и Task) |
| **Почему** | Тело Artifact может быть большим (Research Report — до 10 MB, сгенерированный код — до 50 MB). Метаданные — в PostgreSQL для поиска и связи. |
| **Жизненный цикл в хранилище** | Created (immutable) → Active → Superseded. Никогда не удаляется (аудит). |
| **Требования к консистентности** | Strong consistency для метаданных. Тело в File Storage: запись должна быть подтверждена перед фиксацией метаданных в PostgreSQL. |

### 9.17 Decision (ADR)

| Аспект | Значение |
|--------|----------|
| **Где хранится** | PostgreSQL (статус, версия, тело), Vector Storage (embedding), Neo4j (Node с лейблом ADR, связь SUPERSEDES) |
| **Почему** | Аналогичен Knowledge, но с дополнительными инвариантами: неизменяемость после Accepted, связь SUPERSEDES между версиями. |
| **Жизненный цикл в хранилище** | Proposed → Accepted → Implemented → Superseded / Deprecated. |
| **Требования к консистентности** | Strong consistency для статуса (PostgreSQL). Eventual для Vector Storage и Neo4j. |

### 9.18 Graph

| Аспект | Значение |
|--------|----------|
| **Где хранится** | Neo4j (все узлы и рёбра) |
| **Почему** | Graph = Neo4j. Это не отдельная таблица, а вся база данных Neo4j. |
| **Жизненный цикл в хранилище** | Создаётся Indexer-ом → инкрементально обновляется → версионируется → может быть полностью перестроен (Resync). |
| **Требования к консистентности** | Eventual consistency относительно исходного кода. |

### 9.19 Node

| Аспект | Значение |
|--------|----------|
| **Где хранится** | Neo4j (узел с лейблом типа) |
| **Почему** | Node — это узел в Neo4j. |
| **Жизненный цикл в хранилище** | Создаётся → обновляется → удаляется. |
| **Требования к консистентности** | Eventual consistency. |

### 9.20 Edge

| Аспект | Значение |
|--------|----------|
| **Где хранится** | Neo4j (отношение с типом) |
| **Почему** | Edge — это отношение в Neo4j. |
| **Жизненный цикл в хранилище** | Создаётся → обновляется → удаляется. |
| **Требования к консистентности** | Eventual consistency. |

### 9.21 Provider

| Аспект | Значение |
|--------|----------|
| **Где хранится** | PostgreSQL (конфигурация, API-ключи, статус) |
| **Почему** | Конфигурационная сущность. Требует ACID и безопасности (API-ключи шифруются). |
| **Жизненный цикл в хранилище** | Добавляется → активен → деактивирован → удалён. |
| **Требования к консистентности** | Strong consistency. Изменение Provider-а должно немедленно влиять на все новые AI-вызовы. |

### 9.22 Agent

| Аспект | Значение |
|--------|----------|
| **Где хранится** | PostgreSQL (метаданные: роль, состояние, привязка к Run), Redis (текущий статус) |
| **Почему** | Agent существует в рамках Run. Метаданные — в PostgreSQL для аудита. Текущий статус — в Redis (временный). |
| **Жизненный цикл в хранилище** | Создаётся при старте Run → активен → уничтожается при завершении Run. |
| **Требования к консистентности** | Strong consistency для метаданных в PostgreSQL. |

### 9.23 Conversation

| Аспект | Значение |
|--------|----------|
| **Где хранится** | PostgreSQL (метаданные + сообщения в JSONB), Redis (активные диалоги) |
| **Почему** | Диалоги требуют ACID: каждое Message добавляется атомарно. История — в PostgreSQL. Активный диалог (последние N сообщений) — в Redis для быстрого доступа при построении контекста. |
| **Жизненный цикл в хранилище** | Создаётся → пополняется сообщениями → завершается → архивируется. |
| **Требования к консистентности** | Strong consistency. Сообщение должно быть записано до того, как Agent получит ответ. |

### 9.24 Message

| Аспект | Значение |
|--------|----------|
| **Где хранится** | PostgreSQL (в составе Conversation, immutable) |
| **Почему** | Сообщение неизменяемо — идеально для PostgreSQL. Хранится как JSONB или в реляционной структуре. |
| **Жизненный цикл в хранилище** | Создаётся (immutable) → не изменяется никогда. |
| **Требования к консистентности** | Strong consistency (сообщение должно быть сохранено, даже если ответ LLM потерян). |

### 9.25 Feature

| Аспект | Значение |
|--------|----------|
| **Где хранится** | PostgreSQL (метаданные: название, описание, статус), Neo4j (Node с лейблом Feature, связи с Module, Class, Component, Route) |
| **Почему** | Сквозная группирующая сущность. Метаданные — в PostgreSQL. Связи с элементами кода — в Neo4j. |
| **Жизненный цикл в хранилище** | Создаётся → активно → completed / deprecated → удалено. |
| **Требования к консистентности** | Strong consistency для метаданных, eventual для связей в графе. |

### 9.26 Event

| Аспект | Значение |
|--------|----------|
| **Где хранится** | PostgreSQL (Event Log — персистентное хранение), Redis (Stream — оперативная доставка) |
| **Почему** | События требуют персистентности для аудита (PostgreSQL) и мгновенной доставки подписчикам (Redis Pub/Sub или Stream). |
| **Жизненный цикл в хранилище** | Публикуется → доставляется подписчикам (Redis) → сохраняется в Event Log (PostgreSQL) → удаляется по Retention Policy. |
| **Требования к консистентности** | At-least-once доставка. Событие должно быть гарантированно сохранено в PostgreSQL, но может быть доставлено подписчикам с задержкой или повторно. |

## 10. Data Versioning

### 10.1 Зачем нужно версионирование

Client оперирует данными, которые изменяются со временем: кодовая база проекта меняется с каждым коммитом, Knowledge обновляется при переисследовании, ADR пересматриваются. Для корректной работы системы необходимо знать, какая версия данных использовалась при принятии решений и выполнении задач.

### 10.2 Что версионируется

| Сущность | Как версионируется | Частота версий |
|----------|--------------------|----------------|
| **Graph** | Каждая версия графа привязана к Git commit hash. `graph_version = {git_commit_hash}` | Каждый коммит в отслеживаемых ветках |
| **Knowledge** | Явное версионирование. Каждое обновление создаёт новую версию: `knowledge_version = {id} + {version_number}` | При обновлении Knowledge (ручном или автоматическом) |
| **ADR** | Явное версионирование. Аналогично Knowledge: `adr_version = {id} + {version_number}` | При пересмотре решения |
| **Project Snapshot** | Каждый снапшот имеет уникальную версию: `snapshot_version = {timestamp} + {git_commit_hash}` | При создании снапшота |
| **Artifact** | Не версионируется. Вместо изменения создаётся новый Artifact, который ссылается на предыдущий (`supersedes`). | При необходимости обновления |
| **Event** | Каждое событие имеет `event_version` (версия схемы события). | При изменении схемы события |

### 10.3 Механизм версионирования Graph

Graph — самый часто обновляемый объект. Его версионирование реализуется через:

**Метод: версионирование на уровне узла (Node-Level Versioning).**

Каждый узел Neo4j содержит свойство `graph_version`:

```
(Node:Class {name: "UserService", graph_version: "abc123"})
(Node:Class {name: "UserService", graph_version: "def456"})  // после обновления
```

- При инкрементальном обновлении: старый узел удаляется (или помечается `graph_version_until`), создаётся новый узел с актуальным `graph_version`.
- При запросе Graph всегда фильтруется по актуальному `graph_version`.
- Исторические версии узлов могут храниться для возможности "путешествия во времени" (time-travel queries).

**Почему не снапшоты всей базы Neo4j:**
- Полный снапшот Neo4j для каждого коммита — дорого (миллионы узлов × тысячи коммитов).
- Node-level versioning хранит только изменённые узлы, экономя место.
- Большинство запросов требуют только актуальную версию графа.

### 10.4 Механизм версионирования Knowledge

Knowledge версионируется в PostgreSQL:

```
Knowledge (id: K-001)
  ├── Version 1 (body: "...", graph_version: "abc123", status: Active -> Superseded)
  ├── Version 2 (body: "...", graph_version: "def456", status: Active)
  └── Version 3 (body: "...", graph_version: "ghi789", status: Active)
```

- Каждая версия Knowledge сохраняется как отдельная запись с уникальным `knowledge_id + version`.
- Старые версии не удаляются — они нужны для понимания, какое знание использовалось при принятии прошлых решений.
- При обновлении Knowledge создаётся новый embedding в Vector Storage (старый сохраняется для поиска по историческим версиям, если включено).

### 10.5 Связь версий

Критическая цепочка версий, обеспечивающая воспроизводимость:

```
Git Commit (git_commit_hash)
    ↓ определяет
Graph Version (graph_version = git_commit_hash)
    ↓ определяет актуальность
Knowledge Version (привязана к graph_version)
    ↓ использовалась при
Task Execution (Run)
    ↓ породил
Artifact (ссылается на graph_version, knowledge_version)
```

Эта цепочка гарантирует, что для любого Artifact можно определить: на основе какой версии кода, какой версии графа и какой версии знаний он был создан.

## 11. Snapshot Strategy

### 11.1 Зачем нужны снапшоты

Снапшот — полная или инкрементальная копия состояния проекта на определённый момент времени. Снапшоты обеспечивают:

1. **Восстановление после сбоя.** Если Run испортил файлы, Workspace можно откатить к стабильному снапшоту.
2. **Изоляция Run-ов.** Каждый Run стартует от стабильного снапшота, не видя незавершённых изменений других Run-ов.
3. **Воспроизводимость.** Для любого Artifact можно восстановить точное состояние проекта на момент его создания.

### 11.2 Типы снапшотов

| Тип | Когда создаётся | Содержит | Хранится | Размер |
|-----|-----------------|----------|----------|--------|
| **Stable Snapshot** | После успешного завершения Run (изменения слиты в основной проект) | Полное состояние файловой системы проекта + Graph version + Knowledge version | Постоянно (последние N + периодические) | Полный |
| **Pre-Run Snapshot** | Перед стартом каждого Run | Копия состояния проекта до начала Run | До завершения Run + 24 часа grace period | Полный |
| **Incremental Snapshot** | При инкрементальном обновлении графа (после CommitPushed) | Diff файлов относительно предыдущего снапшота | До создания следующего Stable Snapshot | Diff |
| **Manual Snapshot** | Вручную пользователем | Полное состояние проекта | Постоянно (до ручного удаления) | Полный |

### 11.3 Частота и retention

| Тип | Частота создания | Retention | Обоснование |
|-----|------------------|-----------|-------------|
| **Stable Snapshot** | После каждого успешного Run | Последние 10 снапшотов + 1 в неделю за последние 3 месяца + 1 в месяц за последний год | Баланс между возможностью отката и объёмом хранилища. |
| **Pre-Run Snapshot** | Перед каждым Run | На время Run + 24 часа | Для отката при сбое Run. После успешного завершения не нужен (создаётся Stable Snapshot). |
| **Incremental Snapshot** | При каждом CommitPushed | До создания следующего Stable Snapshot | Промежуточный снапшот для быстрого восстановления без полного Resync. |
| **Manual Snapshot** | По требованию | До ручного удаления | Пользователь сам определяет ценность. |

### 11.4 Формат снапшота

Stable Snapshot включает:

```
snapshot/{project_id}/stable/{snapshot_version}/
├── manifest.json           # Метаданные снапшота:
│                           #   - snapshot_version
│                           #   - git_commit_hash
│                           #   - graph_version
│                           #   - knowledge_versions: {K-001: 3, K-002: 1, ...}
│                           #   - timestamp
│                           #   - created_by_run_id
│                           #   - parent_snapshot_version
├── files.tar.gz            # Архив файловой системы проекта
├── graph.json              # Экспорт актуальной версии графа (опционально)
└── knowledge.json          # Экспорт актуальных версий Knowledge (опционально)
```

Pre-Run Snapshot имеет аналогичный формат, но содержит состояние до начала Run.

Incremental Snapshot содержит только diff: изменённые/добавленные/удалённые файлы относительно родительского снапшота.

### 11.5 Хранение снапшотов

| Аспект | Решение |
|--------|---------|
| **Где хранятся метаданные** | PostgreSQL (таблица snapshots: тип, версия, Git commit, ссылка на файлы) |
| **Где хранятся файлы** | File Storage (S3-совместимое) |
| **Дедупликация** | Неизменённые файлы между снапшотами могут ссылаться на один объект в File Storage (content-addressable) |
| **Сжатие** | tar.gz для полных снапшотов, индивидуальное сжатие для инкрементальных diff |
| **Шифрование** | Опционально: снапшоты могут шифроваться at rest |

## 12. Resync Strategy

### 12.1 Что такое Resync

Resync — это процесс полной перестройки вычисляемых данных (Graph, embedding-и, кэши) из авторитетных источников (исходный код, PostgreSQL). Resync выполняется, когда инкрементальное обновление невозможно или нецелесообразно.

### 12.2 Когда нужен Resync

| Причина | Объяснение |
|---------|------------|
| **Расхождение Graph и кода** | Инкрементальное обновление пропустило изменения (сбой, ошибка в Indexer). Консистентность не восстановима инкрементально. |
| **Обновление Indexer** | Новая версия Indexer извлекает больше информации из кода (новые типы узлов, новые отношения). Старый Graph неполон. |
| **Добавление нового языка** | Проект начал использовать новый язык программирования. Для этого языка нужен новый парсер. |
| **Смена embedding-модели** | Новая модель создаёт embedding-и другой размерности или с другой семантикой. Старые embedding-и несовместимы. |
| **Восстановление после сбоя** | Neo4j или Vector Store повреждены и не могут быть восстановлены из бэкапа (или бэкап устарел). |
| **Миграция данных** | Изменение схемы данных, не поддерживаемое инкрементально (например, переход с pgvector на Qdrant). |

### 12.3 Процесс Resync

Resync — длительная операция (минуты или часы, в зависимости от размера проекта). Она выполняется в фоне, не блокируя основные операции системы.

**Этапы Resync графа:**

```
1. Подготовка
   ├── Создать новую пустую базу данных Neo4j (или очистить существующую).
   ├── Установить статус проекта: RESYNC_IN_PROGRESS.
   └── Уведомить все модули: "Graph временно недоступен для этого проекта".

2. Сканирование файловой системы
   ├── Indexer сканирует все файлы проекта.
   ├── Для каждого файла: парсинг AST → извлечение Symbol-ов.
   └── Результат: полный набор Node и Edge для всего проекта.

3. Запись в Neo4j
   ├── Батчевая запись узлов и рёбер.
   ├── Установка graph_version для всех узлов.
   └── Валидация: проверка checksum графа.

4. Завершение
   ├── Переключение активной базы данных Neo4j (blue-green).
   ├── Установить статус проекта: ACTIVE.
   ├── Публикация события: GraphResyncCompleted.
   └── Все модули возобновляют работу с новым графом.
```

**Этапы Resync embedding-ей:**

```
1. Для каждого активного Knowledge в PostgreSQL:
   ├── Сгенерировать новый embedding через Embedding Provider.
   ├── Записать в Vector Storage.
   └── Обновить ссылку embedding_id в PostgreSQL.

2. Для каждого Code Chunk (если включена векторизация кода):
   └── Аналогичный процесс.

3. Очистить старые embedding-и.
```

**Этапы полного Resync проекта (крайний случай):**

```
1. Граф: Resync графа (см. выше).
2. Embedding-и: Resync embedding-ей (см. выше).
3. Кэши: инвалидация всех кэшей Redis для проекта.
4. Снапшоты: создать новый Stable Snapshot на основе актуального состояния.
```

### 12.4 Blue-Green Deployment для Neo4j

Для минимизации времени недоступности графа при Resync используется подход blue-green:

- **Blue:** текущая активная база данных Neo4j, обслуживающая запросы.
- **Green:** новая база данных, в которую записывается результат Resync.

После завершения Resync:
1. Green проходит валидацию.
2. Активной становится Green.
3. Blue сохраняется ещё 24 часа на случай отката.
4. Blue удаляется.

Время недоступности графа: секунды (переключение указателя на активную БД).

### 12.5 Приоритизация при Resync

Если выполняется Resync, а пользователь создаёт новую Task:

| Действие | Поведение при Resync |
|----------|----------------------|
| **Создание Task** | Разрешено. Task будет ждать завершения Resync перед стартом Run. |
| **Запуск Run** | Приостановлен до завершения Resync. |
| **Запросы к Graph** | Если Blue ещё активен — обслуживаются из Blue. Если Green переключён — из Green. |
| **Запросы к Knowledge** | Обслуживаются из PostgreSQL (всегда активно). |
| **Индексация** | Приостановлена до завершения Resync. |

## 13. Backup Strategy

### 13.1 Что бэкапируется

| Хранилище | Что бэкапируется | RPO | RTO |
|-----------|------------------|-----|-----|
| **PostgreSQL** | Полный дамп БД (pg_dump) + WAL-архивирование (Point-in-Time Recovery) | 5 минут (WAL) | 15 минут |
| **Neo4j** | Полный бэкап БД (neo4j-admin backup) | 24 часа (может быть перестроен Resync) | 4 часа (восстановление из бэкапа) или время Resync |
| **Vector Storage (pgvector)** | В составе бэкапа PostgreSQL | 5 минут (WAL) | 15 минут |
| **Vector Storage (Qdrant/external)** | Снапшот коллекции | 24 часа (может быть перестроен) | Время восстановления снапшота или время Resync |
| **Redis** | RDB-снапшот (опционально) | 1 час (если включено) | 5 минут |
| **File Storage (S3/MinIO)** | Репликация бакета (межрегиональная или cross-region) | Зависит от провайдера (обычно near-real-time) | Зависит от провайдера |
| **Git-репозиторий** | Git — сам по себе распределённая система. Клиенты (GitHub/GitLab) обеспечивают бэкап. | N/A (распределённый) | N/A |

### 13.2 Стратегия бэкапа PostgreSQL

| Аспект | Решение |
|--------|---------|
| **Тип** | Непрерывное WAL-архивирование + ежедневный полный дамп |
| **Полный дамп** | `pg_dump` ежедневно в 03:00. Хранится: 7 ежедневных + 4 еженедельных + 12 ежемесячных |
| **WAL** | Непрерывное архивирование WAL-сегментов. RPO: 5 минут |
| **Проверка** | Ежемесячное тестовое восстановление из бэкапа на staging-окружении |
| **Шифрование** | Бэкапы шифруются at rest (AES-256) |
| **Хранение** | Отдельный S3-бакет `backups-postgresql` в другом регионе (cross-region) |

### 13.3 Стратегия бэкапа Neo4j

| Аспект | Решение |
|--------|---------|
| **Тип** | `neo4j-admin backup` ежедневно |
| **Хранение** | 7 ежедневных бэкапов |
| **Почему не чаще** | Graph может быть полностью перестроен Resync из исходного кода. Бэкап — для ускорения восстановления, а не как единственный способ |
| **Проверка** | Ежемесячное тестовое восстановление |

### 13.4 Стратегия бэкапа File Storage

| Аспект | Решение |
|--------|---------|
| **Тип** | Cross-region репликация бакета (если поддерживается провайдером) |
| **Альтернатива** | Периодическая синхронизация бакета в другой регион (rclone, aws s3 sync) |
| **Частота** | Ежедневно для критических данных (snapshots, artifacts) |
| **Версионирование бакета** | Включено для защиты от случайного удаления объектов |

### 13.5 Что НЕ бэкапируется

| Данные | Почему не бэкапируются |
|--------|------------------------|
| **Redis (все данные)** | Все данные в Redis — временные или кэш. Восстанавливаются из авторитетных источников. RDB-снапшот опционален и служит только для ускорения cold start. |
| **Workspace** | Временное окружение. После завершения Run — не нужно. |
| **Context** | Одноразовый. |
| **Очереди** | После обработки — не нужны. При сбое — восстанавливаются из состояния в PostgreSQL. |

## 14. Data Retention Policy

### 14.1 Принципы retention

| Категория | Retention | Обоснование |
|-----------|-----------|-------------|
| **Проект (метаданные)** | Бессрочно (пока проект активен) + 90 дней после удаления (soft delete) | Возможность восстановления ошибочно удалённого проекта. |
| **Task и Run** | Бессрочно (для активных проектов) | История выполнения — основа для обучения Knowledge. |
| **Artifact** | Бессрочно | Неизменяемый результат работы. Аудит. |
| **Knowledge (активное)** | Бессрочно | Накопленная интеллектуальная ценность. |
| **Knowledge (Deprecated)** | Бессрочно или до ручного удаления | Исторический контекст для понимания эволюции знаний. |
| **Decision (ADR)** | Бессрочно (все статусы) | Архитектурные решения не удаляются. |
| **Conversation и Message** | Бессрочно (для активных проектов) или retention 1 год | Аудит взаимодействия с LLM. |
| **Event Log** | 90 дней (online) + 1 год (архив) | Высокий объём. Старые события агрегируются в метрики. |
| **Stable Snapshot** | 10 последних + 1 в неделю × 3 месяца + 1 в месяц × 1 год | Баланс возможности отката и объёма. |
| **Pre-Run Snapshot** | На время Run + 24 часа | Краткосрочная страховка. |
| **Incremental Snapshot** | До создания Stable Snapshot | Промежуточный, заменяется Stable. |
| **Execution Logs** | 30 дней (online) + 6 месяцев (архив) | Отладка и аудит. |
| **AI Call Logs** | 90 дней (online) + 1 год (архив) | Аудит использования LLM, биллинг. |
| **Metrics (агрегированные)** | 1 год (online) + бессрочно (архив) | Анализ трендов. |

### 14.2 Hard vs Soft Delete

| Операция | Поведение |
|----------|-----------|
| **Soft Delete** | Применяется к: Project, Task, Knowledge, Feature. Запись помечается `deleted_at`, но не удаляется физически. Через retention period — hard delete. |
| **Hard Delete** | Применяется к: Event Log (по истечении retention), Execution Logs, AI Call Logs, снапшотам (по истечении retention). Запись удаляется физически. |
| **Никогда не удаляется** | Artifact, Decision (ADR), Message. |

### 14.3 Архивация

Данные, вышедшие за retention period online-хранения, но подлежащие долгосрочному хранению, архивируются:

```
Online (PostgreSQL/Neo4j/Redis)
    │ по истечении retention
    ▼
Warm Archive (S3 Infrequent Access)
    │ по истечении extended retention
    ▼
Cold Archive (S3 Glacier)
    │ по истечении legal retention
    ▼
Permanent Delete
```

Архивированные данные доступны для чтения, но с задержкой (минуты для Warm, часы для Cold).

## 15. Общие принципы хранения данных

### Принцип 1: Single Source of Truth

Каждая единица информации имеет ровно одно авторитетное хранилище. Все остальные хранилища содержат либо копии (кэш), либо производные данные (вычисляемые).

| Данные | Авторитетный источник | Копии/Производные |
|--------|-----------------------|-------------------|
| Метаданные задач | PostgreSQL | — |
| Структура кода | Исходные файлы (Git) | Neo4j (Graph), Redis (кэш) |
| Знания (Knowledge) | PostgreSQL | Vector Storage (embedding), Neo4j (связи), Redis (кэш) |
| Артефакты | File Storage | PostgreSQL (метаданные) |

### Принцип 2: Хранилище выбирается под паттерн доступа

Не существует "универсального" хранилища. Выбор хранилища определяется тем, как данные будут запрашиваться, а не тем, как они выглядят.

| Паттерн доступа | Хранилище |
|-----------------|-----------|
| Транзакционное обновление состояния | PostgreSQL |
| Графовый траверс (зависимости, влияние) | Neo4j |
| Семантический поиск (ближайшие соседи) | Vector Storage |
| Кэш (горячие данные, sub-ms доступ) | Redis |
| Потоковая запись/чтение больших файлов | File Storage |

### Принцип 3: Вычисляемые данные всегда восстановимы

Neo4j (Graph), Vector Storage (embedding-и), Redis (кэш) не являются авторитетными источниками. Их потеря не приводит к потере информации — только к временной деградации производительности. При проектировании всегда задаётся вопрос: "Можно ли восстановить эти данные из авторитетного источника?"

### Принцип 4: ACID для критического, Eventual для производного

- **ACID (PostgreSQL):** Task, Run, Knowledge, ADR, Provider, Event Log. Данные, потеря или повреждение которых недопустимы.
- **Eventual Consistency (Neo4j, Vector Storage, Redis):** Данные, которые могут быть восстановлены или перестроены из ACID-источников.

### Принцип 5: Immutable-first

Данные, которые описывают факты (а не текущее состояние), должны быть неизменяемыми:
- Artifact — immutable.
- Decision (после Accepted) — immutable.
- Message — immutable.
- Event — immutable.

Изменяемые данные описывают текущее состояние:
- Task.status — mutable.
- Run.status — mutable.
- Knowledge.status — mutable.
- Provider.config — mutable.

### Принцип 6: Версионирование вместо перезаписи

При изменении данных, которые использовались для принятия решений, создаётся новая версия, а не перезаписывается существующая:
- Knowledge.version = N+1 (старая версия сохраняется).
- Graph.graph_version = new_git_commit (старые узлы сохраняются с пометкой `graph_version_until`).
- ADR.superseded_by = new_adr_id.

### Принцип 7: Данные живут дольше кода

Knowledge, Decision, Artifact, Event Log — это интеллектуальная ценность, накопленная системой. Они не зависят от версии кода, реализации модуля или выбора библиотеки. При миграции системы эти данные сохраняются и переносятся.

### Принцип 8: Партицирование по проекту

Данные разных проектов изолированы:
- PostgreSQL: партицирование таблиц по `project_id`.
- Neo4j: отдельная база данных на проект.
- Vector Storage: отдельная коллекция на проект.
- File Storage: отдельный префикс в бакете на проект.

Это обеспечивает:
- Безопасность: проект не может получить доступ к данным другого проекта.
- Производительность: индексы и кэши не смешиваются.
- Обслуживание: бэкап/восстановление/удаление одного проекта не затрагивает другие.

### Принцип 9: Шифрование чувствительных данных

- API-ключи Provider-ов: шифруются at rest (AES-256) в PostgreSQL.
- Персональные данные пользователей: шифруются at rest.
- Снапшоты: опциональное шифрование at rest.
- Сетевое взаимодействие: TLS для всех соединений между сервисами и хранилищами.

### Принцип 10: Минимизация хранимых данных

- Не хранить то, что можно вычислить. Graph хранится, потому что его вычисление (парсинг AST) дорого. Context не хранится, потому что его сборка дёшева относительно хранения.
- Не хранить дубликаты. Artifact-ы с одинаковым содержимым (content-addressable) хранятся один раз.
- Агрегировать старые данные. Event Log старше 90 дней агрегируется в метрики; сырые события перемещаются в Cold Archive.
