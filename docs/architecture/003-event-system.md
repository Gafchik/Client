## 1. Цель Event System

### 1.1 Почему платформа использует события

Client — это система с высокой степенью внутренней сложности. В её ядре — множество независимых модулей (Indexer, Graph, Knowledge, Research, Planner, Execution, Context Builder, Workspace), каждый из которых решает свою специализированную задачу. Однако эти модули не существуют в вакууме: результат работы одного модуля является входными данными для другого.

Пример: когда Indexer завершает парсинг изменённого файла, это должно привести к:
- обновлению графа (Graph),
- инвалидации кэша графа (Redis),
- проверке, не устарели ли Knowledge, связанные с изменёнными узлами (Knowledge),
- уведомлению UI о том, что структура проекта изменилась (Frontend).

Если связать модули напрямую (синхронные вызовы), получится жёсткая耦合 (tight coupling), при которой:
- отказ одного модуля каскадно обрушивает всю цепочку,
- добавление нового модуля требует изменения всех существующих,
- тестирование модуля невозможно без развёртывания всей системы,
- невозможно понять, кто и почему вызвал тот или иной модуль.

**Событийная архитектура** решает эти проблемы, заменяя прямые вызовы на публикацию событий. Модуль-издатель не знает, кто подписан на его события. Модуль-подписчик не знает, кто опубликовал событие. Единственная точка связи — само событие.

### 1.2 Какие проблемы решает

| Проблема | Решение через события |
|----------|----------------------|
| **Tight Coupling** | Модули знают только о событиях, но не друг о друге. Indexer не знает о существовании Context Builder. |
| **Каскадные отказы** | Если один подписчик упал, другие продолжают обработку. Событие будет доставлено упавшему подписчику позже (retry). |
| **Сложность добавления новых модулей** | Новый модуль просто подписывается на нужные события. Никакой существующий модуль не меняется. |
| **Невозможность аудита** | Каждое событие сохраняется в Event Log (PostgreSQL). Полная история: что произошло, когда, в каком контексте. |
| **Синхронные блокировки** | Издатель публикует событие и продолжает работу. Подписчики обрабатывают асинхронно. Нет блокировок. |
| **Сложность мониторинга** | События образуют естественный граф причинно-следственных связей. Метрики строятся на основе потока событий. |
| **Невоспроизводимость состояния** | Имея Event Log, можно воспроизвести состояние системы на любой момент времени (event sourcing). |

### 1.3 Границы событийной системы

Событийная система НЕ заменяет:
- **Прямые запросы на чтение.** Если модулю нужны актуальные данные, он запрашивает их напрямую из авторитетного хранилища (PostgreSQL, Neo4j). События — это уведомления о том, что данные изменились, а не сами данные.
- **Транзакционные операции.** Создание Task и связанного Run происходит в одной транзакции PostgreSQL. Событие `TaskCreated` публикуется после фиксации транзакции.
- **Критические синхронные операции.** Если операция должна быть выполнена до того, как издатель продолжит работу, используется прямой вызов. События — для асинхронных реакций.

## 2. Основные принципы

### Принцип 1: Loose Coupling (Слабая связанность)

Модуль-издатель публикует событие в общий канал. Модуль-подписчик подписывается на события определённого типа. Ни издатель, ни подписчик не имеют прямой ссылки друг на друга. Они знают только формат события.

**Что это даёт:**
- Модули можно разрабатывать, тестировать и развёртывать независимо.
- Один модуль можно заменить другим без изменения остальных — при условии, что новый модуль публикует/обрабатывает те же события.
- Подписчиков на одно событие может быть много (fan-out). Добавление нового подписчика не требует изменения издателя.

**Ограничение:** слабая связанность не означает полное отсутствие контракта. Формат события — это контракт. Изменение формата события — breaking change, которое должно быть версионировано (см. Принцип 5: Immutable Events).

### Принцип 2: Event-Driven (Событийная управляемость)

Система реагирует на события, а не на команды. Модуль-издатель сообщает: "произошло X", а не "сделай Y". Решение о том, что делать в ответ на событие, принимает подписчик.

**Пример:**
- ❌ Командный стиль: Indexer говорит Graph: "обнови узел Class:UserService" (команда).
- ✅ Событийный стиль: Indexer публикует `FileIndexed` (событие). Graph сам решает, что с этим делать (обновить узел, создать новый, удалить старый — в зависимости от изменений в файле).

**Что это даёт:**
- Логика реакции инкапсулирована в подписчике.
- Одно событие может вызывать разные реакции у разных подписчиков.
- Событие — это факт. Факты не устаревают и не отменяются (в отличие от команд).

### Принцип 3: Eventually Consistent (Согласованность в конечном счёте)

События обрабатываются асинхронно. Это означает, что после публикации события состояние системы не обновляется мгновенно во всех модулях. Существует окно несогласованности (обычно — миллисекунды или секунды), в течение которого разные модули видят разное состояние.

**Пример:**
- Indexer публикует `FileChanged` в момент T0.
- Graph обновляет узел в Neo4j в момент T0 + 50ms.
- Knowledge проверяет устаревшие Knowledge в момент T0 + 200ms.
- UI получает уведомление в момент T0 + 100ms.

В окне [T0, T0+200ms] Graph уже обновлён, а Knowledge ещё нет. Это нормально.

**Что это даёт:**
- Высокая производительность (нет синхронных блокировок).
- Устойчивость к сбоям (если Knowledge временно недоступен, событие будет обработано позже).

**Ограничение:** eventual consistency не применяется к операциям, требующим строгой согласованности. Создание Task и Run — ACID-транзакция в PostgreSQL. Событие `TaskCreated` публикуется только после фиксации.

### Принцип 4: Idempotency (Идемпотентность)

Обработчик события должен давать одинаковый результат при однократной и многократной обработке одного и того же события. Это критически важно, потому что события могут доставляться повторно (at-least-once семантика).

**Пример идемпотентной обработки:**
- Событие: `FileIndexed(file_id=X, symbols=[...])`
- Обработчик Graph: "Создать узлы для symbols, если они ещё не существуют". При повторной обработке узлы уже существуют — операция не делает ничего (no-op).

**Пример НЕ идемпотентной обработки:**
- Событие: `TaskStatusChanged(task_id=1, new_status=Executing)`
- ❌ Обработчик: `UPDATE tasks SET status = 'Executing' WHERE id = 1` — идемпотентно.
- ❌ Обработчик: `INSERT INTO status_history (task_id, status) VALUES (1, 'Executing')` — НЕ идемпотентно. При повторной обработке запись будет добавлена дважды. Решение: добавить `event_id` как ключ уникальности.

### Принцип 5: Immutable Events (Неизменяемость событий)

Событие — это фиксация факта, который уже произошёл. После создания событие не может быть изменено. Нельзя "отозвать" событие или изменить его данные. Если факт изменился (статус задачи изменился снова), публикуется новое событие.

**Что это даёт:**
- Event Log — достоверный источник истории.
- Воспроизводимость: replay событий всегда даёт один и тот же результат (при идемпотентной обработке).
- Отладка: всегда можно посмотреть, какие события привели к текущему состоянию.

**Версионирование формата событий.** Если формат события меняется, создаётся новая версия события (например, `FileIndexed.v2`). Старая версия продолжает обрабатываться существующими подписчиками. Новые подписчики могут поддерживать обе версии или только новую. Это обеспечивает обратную совместимость.

## 3. Жизненный цикл события

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Event Lifecycle                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  1. EMISSION (Излучение)                                            │
│     ├── Модуль фиксирует факт и создаёт объект события.            │
│     ├── Событие содержит: ID, тип, timestamp, emitter, payload,    │
│     │   контекст (project_id, task_id, run_id).                     │
│     └── Событие неизменяемо с момента создания.                    │
│                              │                                       │
│                              ▼                                       │
│  2. PUBLICATION (Публикация)                                        │
│     ├── Событие передаётся в диспетчер событий (Event Bus).        │
│     ├── Диспетчер сохраняет событие в Event Log (PostgreSQL) —     │
│     │   персистентная запись.                                      │
│     ├── Диспетчер публикует событие в оперативный канал            │
│     │   (Redis Pub/Sub или Stream) для подписчиков.                │
│     └── Издатель продолжает работу, не дожидаясь обработки.       │
│                              │                                       │
│                              ▼                                       │
│  3. ROUTING (Маршрутизация)                                        │
│     ├── Диспетчер определяет тип события.                          │
│     ├── Диспетчер находит всех подписчиков на этот тип.           │
│     └── Событие доставляется каждому подписчику.                   │
│                              │                                       │
│                              ▼                                       │
│  4. HANDLING (Обработка)                                           │
│     ├── Подписчик получает событие.                                │
│     ├── Подписчик проверяет идемпотентность:                       │
│     │   "Обрабатывал ли я уже событие с этим ID?"                  │
│     ├── Если да → пропускает (ACK).                                │
│     ├── Если нет → выполняет бизнес-логику.                        │
│     ├── Результат: success или error.                              │
│     └── Подписчик подтверждает обработку (ACK).                    │
│                              │                                       │
│                              ▼                                       │
│  5. COMPLETION (Завершение)                                        │
│     ├── Все подписчики подтвердили обработку (ACK).                │
│     ├── Событие помечается как COMPLETED в Event Log.              │
│     └── Событие готово к архивации по Retention Policy.            │
│                                                                      │
│  Альтернативный путь:                                               │
│                                                                      │
│  5a. FAILURE (Отказ)                                                │
│      ├── Подписчик не смог обработать событие (error).             │
│      ├── Если ошибка retryable:                                     │
│      │   └── Событие возвращается в очередь с backoff.             │
│      ├── Если ошибка non-retryable:                                 │
│      │   └── Событие помечается как DEAD_LETTER.                   │
│      └── Требуется ручное вмешательство оператора.                 │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.1 Кто создаёт событие

Событие создаёт модуль-издатель (Event Emitter). Издателем может быть:
- **Модуль ядра:** Indexer, Graph, Knowledge, Research, Planner, Execution, Context Builder.
- **Агент AI:** Developer Agent, Reviewer Agent, Researcher Agent (через Execution).
- **Системный компонент:** диспетчер очереди задач, планировщик Resync, мониторинг.
- **Пользователь:** через API или UI (действия пользователя транслируются в события через API-слой).

### 3.2 Кто публикует

Публикацию выполняет **Event Bus** — централизованный диспетчер событий, который:
- Принимает событие от издателя.
- Гарантирует персистентное сохранение в Event Log (PostgreSQL).
- Доставляет событие подписчикам через оперативный канал (Redis).
- Управляет очередями, повторными попытками, дедупликацией.

Издатель не публикует событие напрямую подписчикам. Он всегда проходит через Event Bus.

### 3.3 Кто обрабатывает

Обрабатывают событие **подписчики (Event Handlers)**. Подписчиком может быть:
- Любой модуль ядра (один модуль может быть подписчиком на множество типов событий).
- Внешний сервис (через адаптер, преобразующий внутреннее событие во внешний формат).

Один и тот же модуль может быть и издателем, и подписчиком. Например:
- Graph является подписчиком на `FileIndexed` (чтобы обновить узлы) и издателем `GraphUpdated` (чтобы уведомить остальных).

### 3.4 Когда событие считается завершённым

Событие считается завершённым, когда:
1. Все подписчики подтвердили обработку (ACK).
2. Все повторные попытки для упавших подписчиков исчерпаны (или обработка успешна).
3. Событие помечено как `COMPLETED` в Event Log.

Если один из подписчиков не может обработать событие и все попытки исчерпаны, событие помечается как `DEAD_LETTER`. Это нештатная ситуация, требующая ручного вмешательства. Само событие при этом остаётся в Event Log (как свершившийся факт), но помечается как необработанное конкретным подписчиком.

## 4. Категории событий

События разделены на категории по источнику их возникновения. Категория определяется тем, **какой модуль или процесс** является издателем события, а не тем, какие данные содержит событие.

| Категория | Издатель | Описание |
|-----------|----------|----------|
| **Project Events** | API / Project Manager | Жизненный цикл проекта: создание, открытие, закрытие, архивирование, изменение конфигурации |
| **Repository Events** | Repository Manager / Git Watcher | События, связанные с Git-репозиторием: клонирование, синхронизация, обнаружение новых коммитов |
| **Indexer Events** | Indexer | Процесс индексации: начало сканирования, завершение файла, завершение индексации, ошибки парсинга |
| **Graph Events** | Graph Manager | Изменения графа: обновление узла, добавление ребра, завершение обновления графа |
| **Knowledge Events** | Knowledge Manager | Жизненный цикл Knowledge: создание, обновление, deprecation, связывание с узлами |
| **Research Events** | Research Engine | Процесс исследования: старт, промежуточные результаты, завершение |
| **Planning Events** | Planner | Процесс планирования: формирование Execution Plan, обнаружение конфликтов |
| **Execution Events** | Execution Engine | Выполнение шагов плана: старт Run, завершение Run, создание Artifact |
| **Context Events** | Context Builder | События сборки контекста: построение завершено, инвалидация кэша контекста |
| **AI Events** | Agent / Provider System | Взаимодействие с LLM: запрос, ответ, ошибка провайдера, rate limit |
| **Workspace Events** | Workspace Manager | Жизненный цикл Workspace: создание, инициализация, слияние, удаление |
| **Task Events** | Task Manager | Жизненный цикл задачи: создание, смена статуса, завершение, отмена |
| **System Events** | System / Scheduler / Monitor | Системные события: Resync, смена конфигурации, сбои, мониторинг |

## 5. Каталог событий

Для каждого события указаны:
- **Издатель (Emitter):** модуль, который создаёт событие.
- **Подписчики (Subscribers):** модули, которые реагируют на событие.
- **Зачем существует:** какую бизнес-ценность несёт событие, почему оно нужно в системе.

### 5.1 Project Events

#### ProjectCreated

| Аспект | Значение |
|--------|----------|
| **Издатель** | API / Project Manager |
| **Подписчики** | Indexer (запускает первичную индексацию), Graph Manager (инициализирует пустую БД Neo4j), Repository Manager (клонирует Git-репозиторий), Knowledge (инициализирует пространство знаний) |
| **Зачем существует** | Запускает каскад инициализации всех подсистем для нового проекта. Без этого события проект был бы создан в PostgreSQL, но ни одна подсистема не узнала бы о его существовании. |

#### ProjectOpened

| Аспект | Значение |
|--------|----------|
| **Издатель** | API / Project Manager |
| **Подписчики** | Indexer (проверяет, не устарела ли индексация), Graph Manager (проверяет актуальность графа), Knowledge (загружает актуальные знания), Frontend (обновляет UI) |
| **Зачем существует** | Открытие проекта пользователем. Может происходить многократно. Не требует полной реиндексации — только проверки актуальности. |

#### ProjectClosed

| Аспект | Значение |
|--------|----------|
| **Издатель** | API / Project Manager |
| **Подписчики** | Indexer (приостанавливает индексацию для проекта), Graph Manager (закрывает соединения с Neo4j для проекта), Context Builder (инвалидирует все кэши контекста для проекта), Frontend (освобождает UI) |
| **Зачем существует** | Освобождение ресурсов. Индексация и мониторинг для неактивного проекта не должны потреблять CPU и память. |

#### ProjectArchived

| Аспект | Значение |
|--------|----------|

| **Подписчики** | Knowledge Manager (сохраняет Execution Report как Knowledge), Artifact Manager (архивирует артефакты), Frontend (уведомляет пользователя) |
| **Зачем существует** | Задача полностью выполнена. Все результаты сохраняются в Knowledge и архивируются. |

#### TaskCancelled

| Аспект | Значение |
|--------|----------|
| **Издатель** | Task Manager |
| **Подписчики** | Execution Engine (останавливает все Run-ы), Workspace Manager (откатывает все Workspace), Frontend (уведомляет) |
| **Зачем существует** | Пользователь отменил задачу. Все связанные процессы останавливаются, ресурсы освобождаются. |

#### TaskRejected

| Аспект | Значение |
|--------|----------|
| **Издатель** | Task Manager |
| **Подписчики** | Knowledge Manager (сохраняет причины отказа как Knowledge — negative knowledge), Frontend (уведомляет с причинами) |
| **Зачем существует** | Задача отклонена системой (невыполнима). Причины сохраняются как Knowledge — это ценный опыт. |

### 5.13 System Events

#### ResyncRequested

| Аспект | Значение |
|--------|----------|
| **Издатель** | Scheduler / Admin API |
| **Подписчики** | Indexer (запускает полную переиндексацию), Graph Manager (инициализирует чистую БД Neo4j), Knowledge (ставит на паузу проверку актуальности), Context Builder (инвалидирует все кэши) |
| **Зачем существует** | Запрос на полную перестройку графа. Все модули переводятся в режим Resync. |

#### ResyncCompleted

| Аспект | Значение |
|--------|----------|
| **Издатель** | Graph Manager (после завершения Resync графа) |
| **Подписчики** | Knowledge (возобновляет проверку актуальности), Context Builder (перестраивает кэши), Frontend (уведомляет пользователя) |
| **Зачем существует** | Resync завершён. Система возвращается в нормальный режим работы. |

#### SnapshotCreated

| Аспект | Значение |
|--------|----------|
| **Издатель** | Snapshot Manager |
| **Подписчики** | Backup Manager (реплицирует снапшот), Frontend (обновляет список доступных снапшотов) |
| **Зачем существует** | Создан новый снапшот. Он должен быть сохранён в бэкапе и отображён в UI. |

#### SystemHealthCheckFailed

| Аспект | Значение |
|--------|----------|
| **Издатель** | System Monitor |
| **Подписчики** | Admin Notification (уведомляет администратора), Task Scheduler (приостанавливает запуск новых задач) |
| **Зачем существует** | Критический сбой в одной из подсистем. Система переходит в режим деградации, чтобы не усугубить проблему. |

#### ConfigReloaded

| Аспект | Значение |
|--------|----------|
| **Издатель** | Configuration Manager |
| **Подписчики** | Все модули (перечитывают свои секции конфигурации) |
| **Зачем существует** | Глобальное изменение конфигурации системы. Модули не должны требовать перезапуска для применения новой конфигурации. |

## 6. Event Flow: ключевые сценарии

В этом разделе описаны последовательности событий для ключевых сценариев работы системы. Каждая схема показывает причинно-следственные связи: какое событие порождает следующие, какие модули участвуют.

**Обозначения в схемах:**
- `→` — синхронный вызов или прямая причинная связь.
- `E:` — событие публикуется.
- `H:` — событие обрабатывается (Handle).
- Отступ показывает, что действие происходит внутри обработчика события.

### 6.1 Открытие проекта

```
Пользователь открывает проект (UI/API)
    │
    ▼
Project Manager:
    │ Обновляет статус Project в PostgreSQL: ACTIVE
    │
    ├──→ E: ProjectOpened
    │
    ├── H(Indexer): проверяет, не устарела ли индексация
    │       │ Если индекс устарел → запускает инкрементальную индексацию
    │       │       └──→ E: IndexStarted
    │       │           └──→ H(Frontend): отображает прогресс
    │       │           └──→ E: IndexCompleted (после завершения)
    │       │               └──→ H(Graph): финализирует граф
    │       │               └──→ E: GraphUpdated
    │       │                   └──→ H(Context): инвалидирует кэш
    │       │                   └──→ H(Frontend): обновляет UI
    │       │
    │       └── Если индекс актуален:
    │               └──→ E: IndexStatusCurrent (системное)
    │
    ├── H(Graph): проверяет актуальность графа
    │       └── Если требуется Resync:
    │               └──→ E: ResyncRequested
    │                   └──→ (см. сценарий "Полный Resync")
    │
    ├── H(Knowledge): загружает актуальные Knowledge для проекта
    │       └── Knowledge активны, проект готов к работе
    │
    └── H(Frontend): обновляет UI
            └── Проект открыт, все системы актуальны
```

```

    │

    │       ├── Устанавливает статус проекта: RESYNC_IN_PROGRESS
    │       ├── Очищает очередь инкрементальной индексации
    │       ├──→ E: IndexStarted (полная индексация)
    │       │
    │       ├── Сканирует все файлы проекта
    │       │   Для каждого файла:
    │       │       ├── Парсинг AST
    │       │       ├── Извлечение Symbol-ов
    │       │       └──→ E: FileIndexed
    │       │           └── H(Graph): добавляет/обновляет узлы в Green БД Neo4j
    │       │
    │       └──→ E: IndexCompleted
    │           │
    │           ├── H(Graph):
    │           │       ├── Финализирует Green БД Neo4j
    │           │       ├── Переключает активную БД: Green → Blue
    │           │       ├── Удаляет старую Blue БД
    │           │       └──→ E: GraphUpdated
    │           │           │

    │           │           │       ├── Проверяет актуальность всех Knowledge
    │           │           │       │   для изменённых узлов
    │           │           │       ├── Помечает устаревшие как "potentially stale"
    │           │           │       └──→ E: KnowledgeStaleDetected (для каждого)
    │           │           │
    │           │           ├── H(Context):
    │           │           │       ├── Инвалидирует все кэши контекста
    │           │           │       └──→ E: ContextCacheInvalidated

    │           │
    │           └── H(Context):
    │                   └── Предсобирает новые кэши контекста (опционально, фоном)
    │
    ├── H(Knowledge):
    │       └── Приостанавливает проверку актуальности до завершения Resync
    │
    ├── H(Task Scheduler):
    │       └── Приостанавливает запуск новых Run до завершения Resync
    │           (создание Task разрешено, но Run не стартует)
    │
    └── H(Graph):
            ├── Инициализирует Green БД Neo4j
            └── Готов к приёму узлов от Indexer

... (процесс индексации и обновления графа, см. выше) ...

После завершения всех этапов:
    │
    └──→ E: ResyncCompleted
            │
            ├── H(Task Scheduler):
            │       └── Возобновляет запуск Run-ов
            │
            ├── H(Knowledge):
            │       └── Возобновляет проверку актуальности
            │
            └── H(Frontend):
                    └── Уведомляет пользователя: Resync завершён
```

## 7. Event Reliability

### 7.1 Повторная обработка (Retry)

События доставляются с гарантией **at-least-once**. Это означает, что событие будет доставлено каждому подписчику минимум один раз, но в случае сбоев может быть доставлено повторно.

**Стратегия повторных попыток:**

| Аспект | Решение |
|--------|---------|
| **Retry-ошибки** | Временные сбои: таймаут соединения с Neo4j, временная недоступность Redis, rate-limit Vector Storage. |
| **Non-retry-ошибки** | Постоянные ошибки: невалидный формат события, нарушение бизнес-правил, ошибка валидации. |
| **Backoff** | Экспоненциальный: 1с → 2с → 4с → 8с → 16с → 32с → 64с. Максимум 7 попыток. |
| **Dead Letter** | После исчерпания всех попыток событие помещается в Dead Letter Queue (DLQ). Требуется ручное вмешательство. |
| **Retry per subscriber** | Каждый подписчик имеет собственный счётчик попыток. Отказ одного подписчика не влияет на других. |

### 7.2 Идемпотентность

Идемпотентность — это свойство обработчика: многократная обработка одного и того же события даёт тот же результат, что и однократная.

**Механизм обеспечения идемпотентности:**

Каждое событие имеет уникальный идентификатор (`event_id`). Подписчик хранит список обработанных `event_id` (processed event registry) и перед обработкой проверяет:
- Если `event_id` уже в списке → пропустить (ACK без обработки).
- Если `event_id` нет в списке → обработать и добавить в список.

Где хранится список обработанных `event_id`:
- **Для краткосрочной памяти (минуты):** Redis (TTL = время жизни события в очереди + запас).
- **Для долгосрочной памяти (дни):** PostgreSQL (processed_events table, партицированная по дате, с retention).

**Пример неидемпотентной операции и её исправления:**

```
❌ НЕ идемпотентно:
Обработчик: INSERT INTO notifications (user_id, message) VALUES (...)

Исправление:
1. Добавить event_id в таблицу notifications как UNIQUE.
2. Использовать INSERT ... ON CONFLICT (event_id) DO NOTHING.
✅ Идемпотентно.
```

### 7.3 Обработка ошибок

**Классификация ошибок:**

| Тип ошибки | Пример | Действие |
|------------|--------|----------|
| **Transient (временная)** | Таймаут сети, БД перезагружается, ресурс временно недоступен | Retry с backoff |
| **Permanent (постоянная)** | Невалидный формат события, нарушение FK, ошибка бизнес-логики | Dead Letter |
| **Partial (частичная)** | Из 10 узлов графа 3 создать не удалось | Компенсирующая транзакция (откат созданных 7) или Retry идемпотентно |
| **Dependency (зависимая)** | Подписчик Б не может обработать, потому что подписчик А ещё не обработал | Retry с увеличенной задержкой (ожидание А) |

**Принцип fail-open vs fail-closed:**

- **Fail-open:** Если Knowledge не может проверить устаревшие знания при `GraphUpdated`, система продолжает работу. Knowledge могут быть потенциально устаревшими, но система не блокируется.
- **Fail-closed:** Если Workspace не может быть инициализирован, Run не стартует. Система блокирует прогресс задачи, потому что выполнение без Workspace невозможно.

### 7.4 Порядок обработки

**Гарантии порядка:**

| Сценарий | Гарантия |
|----------|----------|
| **В рамках одного подписчика** | События обрабатываются в порядке публикации (FIFO). Это обеспечивается очередью подписчика. |
| **Между разными подписчиками** | Порядок не гарантируется. Подписчик А может обработать событие раньше, чем подписчик Б. |
| **Причинно-следственная связь** | Если событие Б публикуется в ответ на обработку события А, между ними существует естественная причинная связь. Подписчики Б получат событие Б только после того, как обработчик А опубликует его. |

**Пример с порядком:**

```
E1: FileIndexed (file A)
E2: FileIndexed (file B)
E3: IndexCompleted

Гарантируется:
- Graph обработает E1 раньше E2 (очередь FIFO).
- Graph обработает E3 только после E1 и E2 (Indexer публикует E3 после обработки всех файлов).

НЕ гарантируется:
- Context обработает E1 раньше, чем Graph обработает E2.
  (разные очереди, разная скорость обработки)
```

### 7.5 Дедупликация

Дедупликация предотвращает обработку одного и того же события более одного раза. Дедупликация работает на двух уровнях:

**Уровень 1: Event Bus (централизованная).**
- Event Bus хранит словарь `event_id → статус доставки`.
- Если издатель пытается опубликовать событие с уже существующим `event_id` (retry публикации), Event Bus игнорирует дубликат.
- Словарь хранится в Redis с TTL 1 час.

**Уровень 2: Подписчик (локальная).**
- Подписчик хранит список обработанных `event_id`.
- При получении события проверяет: если `event_id` уже обработан → ACK без обработки.
- Список в Redis (краткосрочный) + PostgreSQL (долгосрочный).

## 8. Масштабирование

### 8.1 Горизонтальное масштабирование подписчиков

Событийная архитектура естественным образом поддерживает горизонтальное масштабирование. Подписчики — это независимые процессы, которые могут быть размножены для обработки большего потока событий.

```
                    Event Bus
                        │
            ┌───────────┼───────────┐
            │           │           │
       [Подписчик А-1] [Подписчик А-2] [Подписчик А-3]
            │           │           │
            └───────────┴───────────┘
                    │
            Обрабатывают события параллельно
            (разные event_id, конкурентная очередь)
```

**Механизм: конкурентная очередь (competing consumers).**

- Очередь событий для подписчика А шардируется.
- Экземпляры А-1, А-2, А-3 конкурентно забирают события из очереди.
- Каждое событие обрабатывается ровно одним экземпляром.
- Если один экземпляр падает, события перераспределяются между оставшимися.

**Что НЕЛЬЗЯ масштабировать горизонтально:**

- Подписчики, которые требуют строгого порядка обработки (FIFO) для одних и тех же данных. Например, обновление одного и того же узла графа должно происходить последовательно (FileIndexed_1 → FileIndexed_2, а не параллельно).

**Решение:** шардирование очереди по ключу (например, `project_id + file_id`). Все события для одного файла попадают в один шард и обрабатываются последовательно. События для разных файлов обрабатываются параллельно.

### 8.2 Масштабирование Event Bus

Event Bus масштабируется:
- **Redis Cluster:** шардирование каналов Pub/Sub и очередей.
- **Партицирование Event Log:** запись в PostgreSQL с партицированием по `project_id` и времени.
- **Разделение на топики:** разные категории событий могут использовать разные каналы (Project Events, Indexer Events, etc.) для независимого масштабирования.

### 8.3 Изоляция проектов

Каждый проект имеет собственные очереди и подписчиков:
- События проекта А не могут повлиять на производительность обработки событий проекта Б.
- Resync проекта А не блокирует обработку событий проекта Б.
- При росте количества проектов система масштабируется добавлением новых экземпляров подписчиков.

### 8.4 Поток событий при высокой нагрузке

При пиковой нагрузке (большой проект, много коммитов, частые Resync) возможен backlog событий. Система должна справляться с backlog:

**Backpressure:**
- Если очередь подписчика растёт быстрее, чем обрабатывается, подписчик сигнализирует Event Bus.
- Event Bus замедляет публикацию событий этого типа.
- Издатель получает сигнал "замедлись".
- Indexer снижает темп индексации.

**Приоритизация:**
- Критические события (TaskCreated, RunCompleted) имеют приоритет.
- Некритические события (IndexProgressUpdated) могут быть задержаны.
- Приоритет реализуется через отдельные очереди с разным приоритетом (priority queues).

**Batching:**
- События одного типа могут батчироваться для снижения накладных расходов.
- Например, 100 `FileIndexed` в рамках одного коммита могут быть объединены в одно `FilesBatchIndexed`.
- Батчинг снижает количество обращений к Neo4j и количество транзакций.

### 8.5 Будущая эволюция

Событийная архитектура позволяет в будущем:
- **Заменить Redis на внешний брокер** (без изменения логики модулей). Модули работают с абстрактным Event Bus, а не с конкретной технологией.
- **Добавить Event Sourcing.** Event Log уже содержит все события. Можно перейти на Event Sourcing, где состояние системы восстанавливается из Event Log, а PostgreSQL хранит только материализованные представления (projections).
- **Добавить CQRS.** Разделение команд (команды → изменения) и запросов (запросы → чтение) на основе событий.
- **Репликация событий между инстансами.** Для мульти-регионального развёртывания события могут реплицироваться между дата-центрами.

## 9. Общие правила и ограничения

### Правило 1: События уведомляют, а не командуют

Событие говорит: "произошло X", а не "сделай Y". Подписчик сам решает, что делать в ответ на событие.

**Пример:**
- ✅ Правильно: `FileIndexed(file_id=42, symbols=[...])` — "файл проиндексирован, вот его символы".
- ❌ Неправильно: `UpdateGraphNodes(file_id=42, symbols=[...])` — "обнови узлы графа". Это команда, а не событие.

**Почему:** если в будущем появится новый модуль, которому тоже нужна информация о проиндексированном файле, он просто подпишется на `FileIndexed`. Если бы это была команда `UpdateGraphNodes`, новому модулю пришлось бы создавать новое событие, а Indexer пришлось бы менять.

### Правило 2: События не изменяют состояние напрямую

Событие — это неизменяемая запись о факте. Оно не содержит логики изменения состояния. Логика изменения состояния находится в обработчике, а не в событии.

### Правило 3: Обработчики независимы

Каждый обработчик должен быть способен выполнить свою работу, даже если другие обработчики того же события упали. Обработчик не должен полагаться на то, что другой обработчик уже завершил работу.

**Исключение:** причинно-следственная связь. Если обработчик Б публикует событие в ответ на событие А, он делает это асинхронно. Подписчики на событие Б не знают и не должны знать, что событие Б было порождено событием А.

### Правило 4: Запрещены циклические зависимости

Цепочка событий не должна образовывать цикл.

**Пример цикла (ЗАПРЕЩЕНО):**
```
GraphUpdated → Knowledge проверяет актуальность → KnowledgeUpdated → 
Context инвалидирует кэш → ContextCacheInvalidated → Context перестраивает кэш
→ ContextBuilt → ... → GraphUpdated (цикл!)
```

**Как избежать:**
- Обработчики не публикуют события, которые могут вернуться к ним же.
- Context не публикует события, которые могут вызвать обновление графа.
- Knowledge не публикует события, которые могут вызвать перестроение контекста без явного запроса.

**Архитектурный надзор:** каждое новое событие должно проверяться на предмет потенциальных циклов на этапе проектирования.

### Правило 5: Событие содержит идентификатор контекста

Каждое событие содержит:
- `project_id` — в рамках какого проекта произошло событие.
- `task_id` — если событие связано с задачей.
- `run_id` — если событие связано с выполнением.
- `correlation_id` — сквозной идентификатор для трассировки цепочки событий (используется `task_id` или `run_id` как correlation).

Это обеспечивает трассировку: по `correlation_id` можно найти все события, связанные с одной задачей.

### Правило 6: События версионируются

Формат события — это контракт. Изменение формата — breaking change.

**Стратегия версионирования:**
- Каждое событие имеет поле `event_version` (семантическое версионирование: MAJOR.MINOR).
- MAJOR-изменение (добавление обязательного поля, изменение типа существующего поля): новая версия события. Старая версия продолжает поддерживаться в течение переходного периода.
- MINOR-изменение (добавление опционального поля): обратная совместимость, версия не меняется, старые подписчики игнорируют новое поле.

### Правило 7: События не должны быть слишком "тонкими" или слишком "толстыми"

**Слишком тонкое событие:**
```
FileIndexed(file_id=42)
```
Подписчику приходится идти в Indexer и спрашивать: "А какие символы ты нашёл?" Это доп. запрос и tight coupling.

**Слишком толстое событие:**
```
FileIndexed(file_id=42, symbols=[1000 элементов], file_content="...", ast_json="...", metrics={...})
```
Событие становится слишком большим, его дорого передавать и хранить.

**Правильный размер:** событие содержит достаточно данных, чтобы подписчик мог выполнить свою работу без дополнительных запросов к издателю, но не дублирует всю модель данных.

**Для `FileIndexed`:**
- ID файла
- Список изменённых Symbol-ов (имя, тип, FQN, строка, изменения)
- ID предыдущей версии файла (для diff)
- НЕ содержит: тело файла, полный AST

### Правило 8: Транзакционные границы

Событие публикуется **после** фиксации транзакции в авторитетном хранилище (PostgreSQL). Это гарантирует, что:
- Данные, на которые ссылается событие, уже сохранены.
- Если транзакция откатилась, событие не публикуется.

**Порядок:**
1. Начать транзакцию PostgreSQL.
2. Выполнить изменения (создать Task, обновить статус и т.д.).
3. Зафиксировать транзакцию (COMMIT).
4. Опубликовать событие.
5. Если публикация не удалась — данные уже в БД, событие может быть доставлено позже (retry публикации).

### Правило 9: События логируются всегда

Каждое событие записывается в Event Log (PostgreSQL) независимо от того, есть ли у него подписчики. Это обеспечивает:
- Аудит: все события сохранены, даже если никто не слушал.
- Воспроизводимость: можно "переиграть" историю событий.
- Отладку: всегда видно, какие события происходили в системе.

### Правило 10: Подписчик не блокирует издателя

Издатель публикует событие и продолжает работу немедленно. Он не ждёт, пока подписчики обработают событие. Это обеспечивает:
- Производительность: быстрые операции не ждут медленных.
- Устойчивость: если подписчик упал, издатель продолжает работать.

**Исключение:** если операция издателя критически зависит от результата обработки события, это не должен быть событийный паттерн. В этом случае используется синхронный вызов.
