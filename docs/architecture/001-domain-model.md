## 1. Цель Domain Model

Domain Model — это формализованное описание всех сущностей (entities), их атрибутов, поведения, отношений и жизненных циклов в системе Client. Она служит единым языком (Ubiquitous Language) для всех участников проекта: разработчиков, архитекторов, AI-агентов и пользователей.

### Зачем нужна единая Domain Model

1. **Единый язык.** Когда разработчик говорит "Task", а модуль Execution говорит "Job", а Knowledge хранит "WorkItem" — это три названия одной сущности, которые приводят к путанице, багам и невозможности переиспользования знаний. Domain Model фиксирует терминологию: есть ровно одна сущность **Task**, и все модули используют именно этот термин.

2. **Границы ответственности.** Domain Model определяет, где заканчивается ответственность одной сущности и начинается другой. Это предотвращает дублирование логики между модулями и гарантирует, что каждое понятие имеет ровно одного владельца.

3. **Независимость от реализации.** Domain Model описывает *что* существует в системе, а не *как* это хранится или реализовано. Сущность Knowledge существует в предметной области независимо от того, хранится ли она в PostgreSQL, Neo4j или файловой системе. Это позволяет менять реализацию без изменения модели.

4. **Основа для всех модулей.** Каждый модуль системы (Indexer, Graph, Knowledge, Planner, Execution, etc.) оперирует сущностями из Domain Model. Модуль не имеет права вводить собственные понятия, дублирующие или противоречащие модели.

5. **Прослеживаемость.** Domain Model определяет связи между сущностями. Когда Task переходит в состояние Completed, модель точно определяет: какие Artifact-ы должны быть созданы, какие Run-ы должны быть завершены, какое Knowledge должно быть обновлено. Это делает поведение системы предсказуемым и проверяемым.

### Границы Domain Model

Domain Model описывает **предметную область**, а не реализацию. В этом документе:
- **Есть:** сущности, их атрибуты, поведение, отношения, жизненные циклы, инварианты.
- **Нет:** схем баз данных, SQL-запросов, REST-эндпоинтов, форматов сообщений, структур данных в коде.

## 2. Главные сущности системы

Каждая сущность описана по единому шаблону:

- **Назначение** — для чего существует эта сущность в системе.
- **Ответственность** — за что эта сущность отвечает, какие инварианты поддерживает.
- **Жизненный цикл** — как сущность создаётся, изменяется и уничтожается.
- **Кто создаёт** — какой модуль или актор порождает экземпляр сущности.
- **Кто использует** — какие модули или сущности потребляют эту сущность.

---

### 2.1 Project

**Назначение:**  
Project — это корневая сущность, представляющая собой весь программный проект, с которым работает Client. Project объединяет все остальные сущности, относящиеся к одной кодовой базе: файлы, модули, настройки, историю, задачи.

**Ответственность:**
- Предоставление корневого контекста для всех операций Client.
- Хранение метаинформации: название, описание, репозиторий, язык(и), владельцы.
- Определение границ области действия системы в рамках данного проекта.
- Связывание всех сущностей в единое целое.

**Жизненный цикл:**
- Создаётся при инициализации нового проекта в Client (пользователем).
- Активно существует на протяжении всей работы с проектом.
- Может быть заархивирован (Archived) при завершении активной разработки.
- Может быть удалён (Deleted) при полном удалении проекта из Client.

**Кто создаёт:** Пользователь (через CLI, IDE-плагин или веб-интерфейс).

**Кто использует:**
- Все модули системы (каждая операция выполняется в контексте конкретного Project).
- Graph (строит граф для конкретного Project).
- Knowledge (хранит знания в привязке к Project).
- Indexer (индексирует файлы конкретного Project).

---

### 2.2 Workspace

**Назначение:**  
Workspace — это изолированное окружение выполнения для конкретного Task. Workspace включает в себя файловую систему (снимок состояния проекта), Git-репозиторий, Docker-контейнеры, sandbox-окружения, терминал и хранилище артефактов.

ВАЖНО: Workspace — это **окружение**, а не просто набор утилит. Это то *место*, где происходит выполнение. Каждый Run получает собственный экземпляр Workspace.

**Ответственность:**
- Изоляция выполнения: изменения в одном Workspace не влияют на другие.
- Предоставление доступа к файловой системе (чтение/запись в sandbox-режиме).
- Управление Git-операциями в рамках задачи (ветки, коммиты, diff).
- Создание и управление Docker-контейнерами для sandbox-выполнения.
- Хранение артефактов задачи (Research Report, Execution Plan, etc.).
- Создание и восстановление снимков состояния (snapshot).
- Обеспечение безопасности: весь пользовательский код выполняется в sandbox.

**Жизненный цикл:**
- Создаётся автоматически при старте Run (на основе последнего стабильного Snapshot проекта).
- Активно существует на время выполнения Run.
- Модифицируется Execution: запись файлов, коммиты, создание артефактов.
- Закрывается при завершении Run:
  - При успехе: изменения сливаются в основной проект.
  - При неудаче: Workspace уничтожается, изменения откатываются.
- Может быть сохранён как Snapshot для возможности отката.

**Кто создаёт:** Execution (автоматически при старте Run).

**Кто использует:**
- Execution (все операции с файлами и Git-ом).
- Research Engine (чтение файлов, Git blame, Git log).
- Context Builder (чтение содержимого файлов для сборки контекста).
- Impact Analysis Engine (запрос текущего состояния файлов).
- Indexer (отслеживание изменений файлов для инкрементального обновления).

---

### 2.3 Module

**Назначение:**  
Module — это логическая группировка кода в рамках Project. Module может соответствовать директории, пакету (package), неймспейсу (namespace) или иной структурной единице, принятой в языке проекта. Module является промежуточным уровнем между Project и отдельными File.

**Ответственность:**
- Группировка связанных файлов и символов.
- Определение границ ответственности внутри кодовой базы.
- Предоставление информации о зависимостях между модулями (Module A зависит от Module B).
- Выступает единицей анализа для Impact Analysis Engine.

**Жизненный цикл:**
- Создаётся Indexer-ом при обнаружении новой директории/пакета в файловой системе.
- Изменяется при добавлении/удалении файлов внутри модуля.
- Удаляется при удалении соответствующей директории/пакета.
- Может быть переименован при переименовании директории (через Git).

**Кто создаёт:** Indexer (автоматически при сканировании файловой системы).

**Кто использует:**
- Graph (как узел в графе проекта).
- Impact Analysis Engine (для определения зоны влияния на уровне модулей).
- Context Builder (для включения информации о модуле в контекст).
- Planner (для группировки шагов по модулям).

---

### 2.4 File

**Назначение:**  
File — это представление одного файла исходного кода в проекте. File является контейнером для Symbol-ов (классов, функций, методов), которые в нём определены.

**Ответственность:**
- Предоставление содержимого файла для анализа и модификации.
- Отслеживание изменений (через Git).
- Связывание с Symbol-ами, которые в нём определены.

**Жизненный цикл:**
- Обнаруживается Indexer-ом при сканировании файловой системы (или через событие FileChanged).
- Парсится: из содержимого извлекаются Symbol-ы.
- Изменяется: Execution модифицирует содержимое в рамках Run.
- Может быть переименован или перемещён (Git отслеживает эти операции).
- Может быть удалён.

**Кто создаёт:** Indexer (обнаруживает существующие файлы при сканировании) или Execution (создаёт новые файлы в рамках Run).

**Кто использует:**
- Indexer (как источник для извлечения Symbol-ов).
- Graph (как узел в графе, связанный с Symbol-ами).
- Context Builder (для включения содержимого в контекст).
- Research Engine (для анализа содержимого, Git blame, Git log).
- Execution (для модификации содержимого).

---

### 2.5 Folder

**Назначение:**  
Folder — это представление директории в файловой системе проекта. Folder обеспечивает иерархическую организацию File-ов и Module-ов.

**Ответственность:**
- Организация файлов в иерархическую структуру.
- Предоставление информации о вложенных Folder-ах и File-ах.
- Связь с Module: одна Folder может соответствовать одному Module (если Module определён на уровне директории).

**Жизненный цикл:**
- Обнаруживается Indexer-ом при сканировании.
- Может быть создана Execution в рамках Run.
- Может быть переименована или перемещена.
- Может быть удалена.

**Кто создаёт:** Indexer (обнаруживает) или Execution (создаёт новую).

**Кто использует:**
- Indexer (для навигации по файловой системе).
- Graph (как узел иерархии).
- Workspace (для организации файловой структуры).

---

### 2.6 Symbol

**Назначение:**  
Symbol — это абстрактная сущность, представляющая любой именованный элемент исходного кода. Symbol является базовой абстракцией для Class, Interface, Method, Function, Property, Constant, Enum, Type и других структурных элементов кода.

**Ответственность:**
- Предоставление общей идентичности для всех элементов кода (имя, местоположение, тип).
- Связывание с File, в котором определён Symbol.
- Предоставление информации для навигации по коду и анализа зависимостей.

**Жизненный цикл:**
- Извлекается Indexer-ом из AST при парсинге File.
- Изменяется при модификации File (инкрементально переиндексируется).
- Удаляется при удалении File или удалении определения из File.

**Кто создаёт:** Indexer (из AST).

**Кто использует:**
- Graph (как узел в графе — конкретный тип узла зависит от типа Symbol-а).
- Research Engine (для поиска релевантных элементов кода).
- Context Builder (для включения информации о Symbol-ах в контекст).
- Impact Analysis Engine (для анализа зависимостей).

**Конкретные типы Symbol:** см. отдельные сущности: [Class](#27-class), [Interface](#28-interface), [Method](#29-method), [Function](#210-function), [Route](#211-route).

---

### 2.7 Class

**Назначение:**  
Class — это конкретный тип Symbol, представляющий класс в объектно-ориентированном программировании. Class инкапсулирует состояние (Property) и поведение (Method).

**Ответственность:**
- Определение структуры и поведения объектов.
- Наследование (extends) и реализация интерфейсов (implements).
- Предоставление информации о свойствах, методах, родительском классе, реализуемых интерфейсах.

**Жизненный цикл:**
- Аналогичен Symbol: создаётся Indexer-ом при парсинге File.
- Изменяется при рефакторинге (добавление/удаление методов и свойств).
- Может быть переименован (рефакторинг).
- Может быть удалён.

**Кто создаёт:** Indexer (из AST).

**Кто использует:**
- Graph (узел типа Class).
- Impact Analysis Engine (поиск всех классов, зависящих от данного).
- Research Engine (поиск похожих классов, анализ иерархии).
- Context Builder (включение структуры класса в контекст).

---

### 2.8 Interface

**Назначение:**  
Interface — это конкретный тип Symbol, представляющий интерфейс (контракт) в объектно-ориентированном программировании. Interface определяет набор методов, которые должны быть реализованы классами.

**Ответственность:**
- Определение контракта для классов.
- Обеспечение полиморфизма.
- Предоставление информации о методах, которые должны быть реализованы.

**Жизненный цикл:**
- Аналогичен Class.

**Кто создаёт:** Indexer (из AST).

**Кто использует:**
- Graph (узел типа Interface).
- Impact Analysis Engine (поиск всех классов, реализующих данный интерфейс — изменение интерфейса затрагивает все реализации).
- Research Engine (поиск контрактов, анализ абстракций).
- Context Builder.

---

### 2.9 Method

**Назначение:**  
Method — это конкретный тип Symbol, представляющий метод класса или интерфейса. Method определяет поведение, которое может быть вызвано.

**Ответственность:**
- Реализация конкретного поведения.
- Приём параметров и возврат результата.
- Предоставление сигнатуры (имя, параметры, возвращаемый тип, видимость).

**Жизненный цикл:**
- Аналогичен Symbol.
- Может быть изменён (добавление/удаление параметров, изменение возвращаемого типа).
- Может быть перемещён в другой Class (рефакторинг).

**Кто создаёт:** Indexer (из AST).

**Кто использует:**
- Graph (узел типа Method, связанный с Class через BELONGS_TO).
- Impact Analysis Engine (поиск всех вызовов метода — CALLS).
- Research Engine (поиск методов с похожей сигнатурой).
- Context Builder (включение сигнатуры и тела метода в контекст).

---

### 2.10 Function

**Назначение:**  
Function — это конкретный тип Symbol, представляющий функцию (не метод класса). Function существует независимо от классов.

**Ответственность:**
- Реализация поведения на уровне модуля/файла.
- Приём параметров и возврат результата.
- Предоставление сигнатуры.

**Жизненный цикл:**
- Аналогичен Method.

**Кто создаёт:** Indexer (из AST).

**Кто использует:**
- Graph (узел типа Function).
- Impact Analysis Engine (поиск всех вызовов функции).
- Research Engine.
- Context Builder.

---

### 2.11 Route

**Назначение:**  
Route — это конкретный тип Symbol, представляющий HTTP-маршрут в веб-приложении. Route связывает HTTP-метод (GET, POST, PUT, DELETE) и URL-шаблон с Method (контроллером), который его обрабатывает.

**Ответственность:**
- Определение точки входа HTTP API.
- Связывание URL с конкретным Method.
- Предоставление информации о HTTP-методе, URL-шаблоне, параметрах, middleware.

**Жизненный цикл:**
- Аналогичен Symbol.
- Может быть изменён (изменение URL, метода, параметров).
- Может быть удалён при удалении контроллера или снятии аннотации маршрута.

**Кто создаёт:** Indexer (из AST — аннотации, атрибуты, конфигурации маршрутов).

**Кто использует:**
- Graph (узел типа Route, связанный с Method и Module).
- Research Engine (поиск существующих маршрутов, анализ API-поверхности).
- Impact Analysis Engine (изменение маршрута затрагивает клиентов API).
- Context Builder.

---

### 2.12 API Endpoint

**Назначение:**  
API Endpoint — это представление внешнего API, с которым взаимодействует проект. В отличие от Route (который представляет внутренний маршрут), API Endpoint представляет внешнюю точку интеграции (Stripe API, Google OAuth, AWS S3).

**Ответственность:**
- Предоставление информации о внешнем API: URL, метод, параметры, аутентификация.
- Связывание с кодом проекта, который использует этот внешний API.
- Отслеживание изменений внешнего API (версионирование).

**Жизненный цикл:**
- Может быть создан Indexer-ом (обнаружение вызовов внешних API в коде).
- Может быть создан Research Engine (при исследовании внешнего API в ходе задачи).
- Может быть создан пользователем (добавление известного внешнего API в Knowledge).
- Обновляется при изменении способа взаимодействия.
- Может быть помечен как устаревший (Deprecated) при миграции на новую версию API.

**Кто создаёт:** Indexer, Research Engine, или пользователь.

**Кто использует:**
- Graph (узел типа APIEndpoint, связанный с Method-ами, которые его вызывают).
- Knowledge (хранение документации внешних API).
- Research Engine (поиск информации о внешнем API при исследовании).
- Context Builder (включение информации о внешнем API в контекст).

### 2.13 Database Table

**Назначение:**  
Database Table — это представление таблицы базы данных, с которой работает проект. Связывает схему данных с кодом, который её использует.

**Ответственность:**
- Предоставление информации о структуре таблицы (колонки, типы, индексы, связи).
- Связывание с Migration, которая создала/изменила таблицу.
- Связывание с Method-ами и Class-ами, которые читают/пишут в таблицу.

**Жизненный цикл:**
- Может быть создана Indexer-ом (из Migration-файлов, ORM-схем, SQL-файлов).
- Может быть создана Research Engine (при исследовании существующей БД).
- Изменяется при применении Migration.
- Может быть удалена.

**Кто создаёт:** Indexer или Research Engine.

**Кто использует:**
- Graph (узел типа DatabaseTable).
- Impact Analysis Engine (изменение таблицы затрагивает все классы, работающие с ней).
- Research Engine (поиск структуры данных).
- Context Builder (включение схемы таблицы в контекст).

---

### 2.14 Migration

**Назначение:**  
Migration — это представление миграции базы данных. Migration описывает изменение схемы базы данных: создание/удаление/изменение таблиц, колонок, индексов.

**Ответственность:**
- Предоставление информации об изменении схемы БД.
- Связывание с Database Table, которые затрагивает миграция.
- Отслеживание порядка применения миграций (последовательность).
- Предоставление статуса миграции (Applied, Pending, Rolled Back).

**Жизненный цикл:**
- Обнаруживается Indexer-ом (файлы миграций в проекте).
- Может быть создана Execution в рамках Run (генерация новой миграции).
- Применяется (Applied) при выполнении миграции.
- Может быть откатана (Rolled Back).

**Кто создаёт:** Indexer (обнаруживает) или Execution (генерирует новую).

**Кто использует:**
- Graph (узел типа Migration).
- Impact Analysis Engine (изменение миграции затрагивает таблицы и код).
- Context Builder.
- Planner (определение порядка применения миграций).

---

### 2.15 Component

**Назначение:**  
Component — это представление UI-компонента в frontend-части проекта. Component инкапсулирует визуальное представление, состояние и поведение элемента пользовательского интерфейса.

**Ответственность:**
- Определение структуры UI-компонента (props, state, events, slots).
- Связывание с File, в котором определён компонент.
- Предоставление информации о зависимостях от других Component-ов.
- Предоставление информации о Route-ах, которые рендерят компонент.

**Жизненный цикл:**
- Аналогичен Symbol: создаётся Indexer-ом при парсинге файлов компонентов.
- Изменяется при рефакторинге.
- Может быть удалён.

**Кто создаёт:** Indexer (из AST frontend-файлов).

**Кто использует:**
- Graph (узел типа Component).
- Impact Analysis Engine (изменение компонента затрагивает все компоненты, которые его используют).
- Research Engine (поиск похожих компонентов).
- Context Builder (включение структуры компонента в контекст).

---

### 2.16 Knowledge

**Назначение:**  
Knowledge — это фундаментальная сущность долговременной памяти системы. Knowledge представляет любую единицу знания, накопленную в ходе работы Client: архитектурные решения, best practices, результаты исследований, уроки, извлечённые из выполненных задач, внешнюю документацию.

Knowledge существует независимо от конкретных задач и проектов и может быть переиспользовано между проектами.

**Ответственность:**
- Хранение единицы знания в структурированном виде.
- Версионирование: каждое знание имеет версию и историю изменений.
- Связывание с сущностями, к которым знание относится (через Graph).
- Предоставление семантического поиска по знаниям (через эмбеддинги).
- Поддержка актуальности: устаревшие знания помечаются как Deprecated.

**Жизненный цикл:**
- Создаётся в результате:
  - Research (Research Report сохраняется как Knowledge).
  - Execution (Execution Report и извлечённые уроки).
  - Принятия архитектурного решения (ADR как Knowledge).
  - Импорта внешней документации.
- Обновляется при получении новых данных по той же теме.
- Может быть помечено как Deprecated (устарело).
- Может быть заменено новым Knowledge (со ссылкой на предшественника).
- Может быть удалено (при полной неактуальности).

**Кто создаёт:** Research Engine, Execution, пользователь.

**Кто использует:**
- Research Engine (поиск релевантных знаний при исследовании).
- Context Builder (включение знаний в контекст).
- Planner (использование прошлых планов и уроков).
- Impact Analysis Engine (использование истории для прогноза рисков).
- Graph (Knowledge связывается с узлами графа через RELATES_TO).

---

### 2.17 Research

**Назначение:**  
Research — это сущность, представляющая процесс исследования предметной области задачи. Research управляет серией исследовательских вопросов и агрегирует ответы из различных источников (Graph, Knowledge, Git, внешние API, документация).

Research — это **процесс**, результат которого — Research Report (который сохраняется как Knowledge).

**Ответственность:**
- Формулировка исследовательских вопросов на основе задачи.
- Оркестрация сбора информации из источников.
- Анализ перекрёстных связей между собранными данными.
- Выявление информационных пробелов.
- Формирование структурированного Research Report.

**Жизненный цикл:**
- Создаётся Research Engine при получении Task.
- Проходит через серию итераций: формулировка вопросов → сбор → анализ → выявление пробелов → дополнительные вопросы.
- Завершается формированием Research Report.
- Research Report сохраняется как Knowledge.

**Кто создаёт:** Research Engine (автоматически при старте обработки Task).

**Кто использует:**
- Impact Analysis Engine (Research Report как входные данные).
- Context Builder (Research Report как источник информации для контекста).
- Planner (Research Report как обязательный входной артефакт).
- Knowledge (Research Report сохраняется для будущего переиспользования).

---

### 2.18 Context

**Назначение:**  
Context — это сущность, представляющая структурированный набор информации, подготовленный для передачи LLM-модели. Context компилируется из множества источников и оптимизируется под ограничения контекстного окна целевой модели.

Context — **одноразовый**. Каждый запуск LLM получает новый экземпляр Context, собранный специально для этого запуска.

**Ответственность:**
- Агрегация информации из Research Report, Impact Report, кода проекта, Knowledge.
- Приоритезация: что критически важно, что полезно, что опционально.
- Структурирование: system prompt, project context, task context, knowledge context.
- Проверка лимитов контекстного окна.
- Оптимизация под конкретного LLM-провайдера.

**Жизненный цикл:**
- Создаётся Context Builder-ом для каждого запроса к LLM.
- Используется один раз (передаётся LLM вместе с запросом).
- Не сохраняется (но может быть залогирован для аудита).
- Уничтожается после использования.

**Кто создаёт:** Context Builder (автоматически).

**Кто использует:**
- LLM (через Provider System) — как промпт.
- Execution (для логирования и аудита).
- Planner (контекст, использованный при планировании, сохраняется для traceability).

---

### 2.19 Task

**Назначение:**  
Task — это центральная сущность, представляющая единицу работы, поставленную пользователем. Task проходит через всю систему: от постановки до завершения, порождая Research, Run-ы, Artifact-ы и Knowledge.

Task — это то, что пользователь хочет получить: "Добавить OAuth-аутентификацию через Google", "Рефакторинг модуля UserService", "Обновить зависимости до последних версий".

**Ответственность:**
- Формализация пользовательского запроса: что нужно сделать, ограничения, приоритеты.
- Управление жизненным циклом: отслеживание текущего состояния.
- Связывание всех порождённых сущностей: Research, Impact Analysis, Runs, Artifacts.
- Предоставление единого интерфейса для мониторинга прогресса.

**Жизненный цикл:**
Подробно описан в разделе [5. Entity Lifecycle — Task](#51-task).

**Кто создаёт:** Пользователь (через интерфейс).

**Кто использует:**
- Все модули системы (Task — центральная оркестрирующая сущность).
- Research Engine (получает Task для формулировки вопросов).
- Planner (получает Task для декомпозиции).
- Execution (получает Task для выполнения).
- Graph (узел типа Task в графе).
- Knowledge (история задач сохраняется для анализа).

---

### 2.20 Run

**Назначение:**  
Run — это одна попытка выполнения Task (или одного шага Task). Каждый Run получает собственный Workspace и выполняет конкретную операцию: исследование, анализ влияния, планирование, выполнение шага, ревью.

Task может иметь множество Run-ов: исследовательский Run, Run анализа влияния, Run-ы выполнения отдельных шагов, Run финального ревью.

**Ответственность:**
- Выполнение конкретной операции в рамках Task.
- Изоляция: каждый Run имеет собственный Workspace.
- Отслеживание прогресса внутри Run (текущий шаг, статус).
- Порождение Artifact-ов (результатов выполнения).
- Обработка ошибок и повторных попыток.

**Жизненный цикл:**
Подробно описан в разделе [5. Entity Lifecycle — Run](#52-run).

**Кто создаёт:** Execution, Research Engine, Impact Analysis Engine (автоматически при старте операции).

**Кто использует:**
- Workspace (привязка окружения к Run).
- Artifact (каждый Artifact создаётся в контексте Run).
- Execution (мониторинг и управление Run).
- Planner (анализ результатов Run для корректировки плана).

---

### 2.21 Artifact

**Назначение:**  
Artifact — это результат работы Run. Artifact — это неизменяемый документ (или файл), созданный в ходе выполнения операции: Research Report, Impact Report, Execution Plan, сгенерированный код, отчёт о тестировании.

Artifact — **неизменяемый (immutable)**. После создания Artifact не изменяется. Если требуется обновление, создаётся новый Artifact, который ссылается на предыдущий.

**Ответственность:**
- Фиксация результата операции в неизменяемом виде.
- Предоставление метаинформации: какой Run создал, когда, в рамках какой Task.
- Версионирование через создание новых экземпляров.
- Связывание с другими Artifact-ами (traceability chain).

**Жизненный цикл:**
- Создаётся в результате завершения Run.
- Не изменяется никогда (immutable).
- Может быть заменён новым Artifact-ом (новой версией).
- Может быть заархивирован.
- Хранится в Knowledge как часть истории.

**Кто создаёт:** Любой модуль, выполняющий Run (Research Engine, Impact Analysis Engine, Planner, Execution).

**Кто использует:**
- Модули, потребляющие результат (например, Planner потребляет Research Report).
- Knowledge (сохраняет Artifact для будущего переиспользования).
- Graph (связывает Artifact с затронутыми узлами).

**Типы Artifact (соответствуют разделу 7 документа 000-overview):**
- **Research Report** — результат Research.
- **Impact Report** — результат Impact Analysis.
- **Context Package** — результат Context Builder.
- **Execution Plan** — результат Planner.
- **Execution Report** — результат Execution.

---

### 2.22 Decision (ADR)

**Назначение:**  
Decision (Architecture Decision Record, ADR) — это формализованное описание принятого архитектурного решения. Decision объясняет: какая проблема решалась, какие альтернативы рассматривались, какое решение было принято и почему, какие trade-off были сделаны.

Decision — **неизменяемый (immutable)**. Принятое решение не переписывается задним числом. Если решение пересматривается, создаётся новый Decision, который ссылается на заменяемый.

**Ответственность:**
- Фиксация архитектурного решения с контекстом принятия.
- Сохранение информации об альтернативах и причинах их отклонения.
- Связывание с затронутыми сущностями (через Graph).
- Обеспечение traceability: почему код выглядит именно так.

**Жизненный цикл:**
- Создаётся в результате:
  - Архитектурного обсуждения (пользователь или Planner предлагает архитектурный выбор).
  - Автоматической фиксации архитектурно значимого решения в ходе выполнения Task.
- Проходит стадии: Proposed → Accepted → Implemented.
- Может быть заменён новым Decision (Superseded).
- Может быть помечен как Deprecated (решение более не актуально, но код ещё существует).
- Никогда не удаляется (история решений сохраняется навсегда).

**Кто создаёт:** Пользователь, Planner, или Execution (при фиксации архитектурно значимого выбора).

**Кто использует:**
- Knowledge (хранение как часть Knowledge base).
- Graph (узел типа ADR в графе).
- Research Engine (поиск релевантных решений при исследовании).
- Context Builder (включение релевантных ADR в контекст).
- Planner (учёт принятых решений при планировании).

---

### 2.23 Graph

**Назначение:**  
Graph — это графовая модель кодовой базы проекта. Graph представляет код как набор Node (узлов) и Edge (рёбер), позволяя выполнять структурные запросы: поиск зависимостей, анализ влияния, поиск циклов.

Graph — это **представление** кода, а не сам код. Graph обновляется Indexer-ом при изменении файлов, но не изменяет файлы сам.

**Ответственность:**
- Хранение всех Node и Edge проекта.
- Предоставление API для графовых запросов (поиск путей, соседей, подграфов).
- Версионирование: состояние Graph привязано к состоянию кодовой базы (Git commit).
- Обеспечение консистентности с реальным состоянием кода.
- Обнаружение архитектурных нарушений (циклы, нарушение границ слоёв).

**Жизненный цикл:**
- Создаётся Indexer-ом при первичном сканировании проекта.
- Инкрементально обновляется при изменениях файлов (события FileChanged).
- Версионируется: каждая версия Graph привязана к Git commit.
- Может быть перестроен полностью при рассинхронизации (Resync).

**Кто создаёт:** Indexer (автоматически).

**Кто использует:**
- Research Engine (структурные запросы).
- Impact Analysis Engine (анализ зависимостей).
- Context Builder (информация о структуре кода для контекста).
- Knowledge (связывание знаний с узлами графа).

---

### 2.24 Node

**Назначение:**  
Node — это элемент Graph, представляющий одну сущность в графе проекта. Node может представлять: класс, метод, файл, модуль, таблицу БД, миграцию, ADR, задачу — любую сущность из Domain Model.

**Ответственность:**
- Предоставление идентичности (уникальный ID в рамках Graph).
- Хранение типа узла (Class, Method, File, Module, etc.).
- Хранение атрибутов (имя, местоположение, метрики).
- Связывание с другими Node через Edge.
- Версионирование: привязан к версии Graph.

**Жизненный цикл:**
- Создаётся Indexer-ом (для сущностей кода).
- Создаётся Knowledge (для ADR, Research Report-ов).
- Создаётся Execution (для Task, Run).
- Изменяется при изменении соответствующей сущности (инкрементальное обновление через Indexer).
- Удаляется при удалении соответствующей сущности.

**Кто создаёт:** Indexer, Knowledge, Execution (в зависимости от типа Node).

**Кто использует:**
- Graph (как элемент данных).
- Impact Analysis Engine (траверс узлов для анализа влияния).
- Research Engine (поиск связанных узлов).
- Context Builder (информация об узлах для контекста).

**Типы Node:**  
Соответствуют типам узлов, определённым в разделе [4.2.1 Graph Data Model](000-overview.md#421-graph-data-model) документа 000-overview. Все сущности из данной Domain Model, которые имеют представление в графе, существуют как Node соответствующего типа.

---

### 2.25 Edge

**Назначение:**  
Edge — это элемент Graph, представляющий направленное отношение между двумя Node. Тип Edge определяет семантику отношения: наследование, вызов, зависимость, принадлежность.

**Ответственность:**
- Предоставление направленного отношения между Node.
- Хранение типа отношения (CALLS, EXTENDS, IMPLEMENTS, DEPENDS_ON, etc.).
- Хранение атрибутов отношения (например, номер строки вызова).
- Версионирование: привязан к версии Graph.

**Жизненный цикл:**
- Создаётся Indexer-ом при обнаружении отношения в коде.
- Может быть создан другими модулями (например, RELATES_TO между Knowledge и Class).
- Изменяется при изменении кода.
- Удаляется при исчезновении соответствующего отношения в коде.

**Кто создаёт:** Indexer, Knowledge, Execution.

**Кто использует:**
- Graph (как элемент данных).
- Impact Analysis Engine (траверс рёбер для анализа влияния).
- Research Engine (поиск связанных сущностей).

**Типы Edge:**  
Соответствуют типам рёбер, определённым в разделе [4.2.1 Graph Data Model](000-overview.md#421-graph-data-model) документа 000-overview.

---

### 2.26 Provider

**Назначение:**  
Provider — это сущность, представляющая внешний сервис, используемый системой: LLM-провайдер (OpenAI, Anthropic), embedding-провайдер, поисковый провайдер, векторное хранилище.

Provider — это **абстракция над внешним сервисом**, а не сам сервис. Provider определяет контракт (интерфейс), которому должен соответствовать конкретный адаптер.

**Ответственность:**
- Определение стандартизированного интерфейса для типа сервиса.
- Маршрутизация запросов к конкретному провайдеру в зависимости от задачи.
- Управление конфигурацией: API-ключи, эндпоинты, лимиты.
- Обработка ошибок: fallback, retry, circuit breaker.
- Мониторинг: latency, стоимость, использование токенов.

**Жизненный цикл:**
- Настраивается администратором/пользователем (конфигурация).
- Активируется/деактивируется в зависимости от доступности.
- Может быть обновлён (новая версия API провайдера).
- Может быть удалён.

**Кто создаёт:** Пользователь (конфигурация) или администратор системы.

**Кто использует:**
- Все модули, взаимодействующие с внешними сервисами:
  - Research Engine (поиск во внешних источниках).
  - Execution (вызов LLM для генерации кода).
  - Context Builder (вызов LLM для планирования).
  - Knowledge (эмбеддинги и векторный поиск).

**Типы Provider:**
- **LLM Provider** — для генерации текста/кода.
- **Embedding Provider** — для создания эмбеддингов.
- **Search Provider** — для поиска во внешних источниках.
- **Vector Store Provider** — для хранения и поиска эмбеддингов.

---

### 2.27 Agent

**Назначение:**  
Agent — это сущность, представляющая специализированного AI-исполнителя. Agent имеет конкретную роль (Developer, Reviewer, Researcher, Planner) и действует в рамках этой роли, используя предоставленный ему контекст и инструменты.

Каждый Agent работает с LLM через Provider System, но имеет различные системные промпты, доступные инструменты и критерии качества.

**Ответственность:**
- Выполнение операций в соответствии со своей ролью.
- Взаимодействие с LLM через Provider System.
- Использование инструментов (Workspace, терминал, тесты).
- Соблюдение протокола выполнения (Plan → Develop → Verify → Review → Fix → Report).
- Предоставление результатов работы в виде Artifact-ов.

**Жизненный цикл:**
- Создаётся Execution при старте Run (в зависимости от требуемой роли).
- Существует в рамках Run.
- Уничтожается при завершении Run.

**Кто создаёт:** Execution.

**Кто использует:**
- Execution (управляет Agent-ами, распределяет задачи).
- Workspace (предоставляет окружение для Agent).
- Provider System (предоставляет доступ к LLM).

**Типы Agent:**
- **Developer Agent** — генерирует код, выполняет шаги плана.
- **Reviewer Agent** — выполняет ревью сгенерированного кода.
- **Researcher Agent** — формулирует исследовательские вопросы, анализирует источники.
- **Planner Agent** — выполняет стратегическую и тактическую декомпозицию (под управлением модуля Planner, который не является LLM).

---

### 2.28 Conversation

**Назначение:**  
Conversation — это сущность, представляющая диалог между пользователем и системой (или между Agent-ами) в рамках решения задачи. Conversation содержит последовательность Message (сообщений), которые формируют контекст взаимодействия.

Conversation может быть:
- Диалогом пользователя с Client (уточнение задачи, вопросы, подтверждения).
- Внутренним диалогом Agent-а с LLM (обмен промптами и ответами).
- Диалогом между Agent-ами (Developer запрашивает ревью у Reviewer).

**Ответственность:**
- Хранение истории сообщений в рамках диалога.
- Управление контекстом диалога (история сообщений, которая передаётся LLM).
- Обеспечение traceability: почему Agent принял то или иное решение (видно из истории диалога с LLM).

**Жизненный цикл:**
- Создаётся при начале взаимодействия с пользователем или при старте Run.
- Пополняется сообщениями по мере развития диалога.
- Может быть разветвлён (branch) при необходимости исследовать альтернативный путь.
- Завершается при завершении взаимодействия или Run.
- Может быть сохранён для аудита (полная история переписки с LLM).

**Кто создаёт:** Execution, API Gateway, Research Engine.

**Кто использует:**
- Agent (как контекст диалога с LLM).
- Context Builder (предыдущие сообщения добавляются в контекст).
- Пользователь (через UI).
- Knowledge (история диалогов сохраняется для анализа).

---

### 2.29 Message

**Назначение:**  
Message — это одно сообщение в рамках Conversation. Message имеет отправителя (User, Assistant, System) и содержимое (текст, код, структурированные данные).

**Ответственность:**
- Хранение содержимого одного сообщения.
- Идентификация отправителя и временной метки.
- Связывание с Conversation, в рамках которого отправлено.
- Может содержать вызовы инструментов (tool calls) и их результаты.

**Жизненный цикл:**
- Создаётся при отправке сообщения (пользователем, Agent-ом, или системой).
- Не изменяется (immutable после создания).
- Может быть скрыто из контекста (но не удалено) при превышении лимита контекстного окна.
- Хранится в Conversation на протяжении его жизни.

**Кто создаёт:** Пользователь, Agent, система (автоматические уведомления).

**Кто использует:**
- Conversation (как элемент диалога).
- Context Builder (для включения в контекст LLM).
- Knowledge (для анализа и аудита).

**Отправители Message:**
- **User** — сообщение от пользователя.
- **Assistant** — ответ от AI (LLM).
- **System** — системное сообщение (инструкции, уведомления о состоянии).
- **Tool** — результат выполнения инструмента (tool call result).

---

### 2.30 Feature

**Назначение:**  
Feature — это функциональная возможность проекта, которая объединяет несколько связанных элементов: классы, компоненты, маршруты, миграции, документацию. Feature существует на более высоком уровне абстракции, чем Module: Module — это структурная единица кода, Feature — это функциональная единица.

Примеры Feature: "Аутентификация пользователей", "Платёжная система", "Уведомления по email", "Поиск по проекту".

**Ответственность:**
- Группировка связанных элементов (классов, компонентов, маршрутов), реализующих одну функциональную возможность.
- Предоставление карты функциональности проекта.
- Связывание с Task (какие задачи относятся к данной Feature) и Decision (какие архитектурные решения приняты для данной Feature).

**Жизненный цикл:**
- Может быть создана пользователем (явное определение).
- Может быть предложена автоматически на основе анализа графа (кластер связанных сущностей).
- Может быть изменена (добавление/удаление элементов).
- Может быть помечена как завершённая (Completed).
- Может быть удалена.

**Кто создаёт:** Пользователь или автоматический анализ (Research Engine предлагает выделить Feature).

**Кто использует:**
- Graph (узел типа Feature, связанный с Module, Class, Component, Route через BELONGS_TO).
- Research Engine (поиск связанных элементов по Feature).
- Impact Analysis Engine (анализ влияния на уровне Feature).
- Context Builder (информация о Feature в контексте).

## 4. Aggregate Boundaries

Aggregate — это группа связанных сущностей, которые рассматриваются как единое целое с точки зрения изменения данных. Aggregate имеет корневую сущность (Aggregate Root), через которую выполняются все операции над элементами агрегата.

Границы агрегатов определяют:
- **Консистентность:** изменения внутри агрегата должны быть атомарными и удовлетворять всем инвариантам.
- **Изоляцию:** два агрегата не могут изменяться в одной транзакции (eventual consistency между агрегатами).
- **Доступ:** внешние сущности могут ссылаться только на Aggregate Root, но не на внутренние сущности агрегата.

### Агрегат 1: Project

**Aggregate Root:** Project

**Внутренние сущности:**
- Project (root)
- Module
- Folder
- File
- Symbol (и все его подтипы: Class, Interface, Method, Function, Route)
- Database Table (если определена через код)
- Migration
- Component
- API Endpoint

**Граница:** Структурные элементы кода. Всё, что является непосредственной частью кодовой базы, принадлежит агрегату Project.

**Правила:**
- Создание/изменение/удаление любого структурного элемента (Class, File, etc.) происходит через Workspace, который оперирует копией агрегата Project.
- Graph является представлением (projection) этого агрегата, а не частью агрегата. Graph может быть перестроен из агрегата Project.
- Module не может существовать вне Project.

---

### Агрегат 2: Task

**Aggregate Root:** Task

**Внутренние сущности:**
- Task (root)
- Research
- Run
- Artifact (все типы: Research Report, Impact Report, Execution Plan, Execution Report, Context Package)
- Workspace
- Conversation
- Message
- Agent

**Граница:** Всё, что относится к выполнению одной задачи. Task владеет всем процессом от постановки до завершения.

**Правила:**
- Run не может существовать вне Task.
- Artifact не может существовать вне Task (и Run).
- Conversation и Message существуют в контексте Task (или Run).
- Agent существует только в рамках Run.
- Workspace создаётся для Run и уничтожается вместе с ним.
- При завершении Task порождённое Knowledge сохраняется и становится независимым (не принадлежит агрегату Task).

---

### Агрегат 3: Knowledge

**Aggregate Root:** Knowledge

**Внутренние сущности:**
- Knowledge (root)
- Decision (ADR) — подтип Knowledge

**Граница:** Накопленные знания системы. Knowledge — это независимый агрегат, который может пережить Project и Task.

**Правила:**
- Knowledge не изменяет Graph (только связывается с узлами через RELATES_TO).
- Knowledge не зависит от AI-моделей или Provider-ов.
- Decision (ADR) — это специализированный тип Knowledge с дополнительными инвариантами (неизменяемость после Accepted).
- Обновление Knowledge происходит через создание новой версии, а не изменение существующей.

---

### Агрегат 4: Graph

**Aggregate Root:** Graph

**Внутренние сущности:**
- Graph (root)
- Node
- Edge

**Граница:** Графовое представление кодовой базы. Graph — это проекция агрегата Project и Knowledge.

**Правила:**
- Graph не изменяет Project (только отражает его состояние).
- Graph не зависит от AI.
- Node и Edge не могут существовать вне Graph.
- Изменения в Graph инициируются только Indexer-ом (для структурных узлов) или Knowledge (для Knowledge-узлов).
- Версионирование Graph привязано к Git-коммитам.

---

### Агрегат 5: Conversation

**Aggregate Root:** Conversation

**Внутренние сущности:**
- Conversation (root)
- Message

**Граница:** История диалога.

**Правила:**
- Message не может существовать вне Conversation.
- Message неизменяем (immutable).
- Conversation может существовать независимо от Task (например, диалог пользователя с Client до создания Task).
- Conversation может быть связан с Task, но не принадлежит агрегату Task (может пережить Task).

---

### Независимые сущности (не принадлежат агрегатам)

Некоторые сущности являются независимыми и не входят ни в один агрегат:

- **Provider** — независимая конфигурационная сущность. Настраивается на уровне системы или Project, но не принадлежит агрегату Project.
- **Feature** — независимая группирующая сущность. Связывает элементы из агрегата Project и Knowledge, но не принадлежит ни одному из них.
- **Context** — одноразовая сущность, создаваемая на стыке агрегатов. Не сохраняется после использования.

```
                     ▼
              ┌──────────────┐
              │  Planning     │──────── Задача невыполнима ──────► ┌──────────┐
              └──────┬────────┘                                     │ Rejected │
                     │ Execution Plan создан                        └──────────┘
                     ▼
              ┌──────────────┐
              │  Executing    │◄──────── Итерация (после каждого шага)
              └──────┬────────┘
                     │ Все шаги выполнены
                     ▼
              ┌──────────────┐
              │  Reviewing    │──────── Обнаружены проблемы ──────► ┌──────────┐
              └──────┬────────┘                                     │Reopened  │──► Executing
                     │ Ревью пройдено                               └──────────┘
                     ▼
              ┌──────────────┐
              │  Completed    │
              └──────┬────────┘
                     │ Знания сохранены, документация обновлена
                     ▼
              ┌──────────────┐
              │  Archived     │
              └──────────────┘
```

**Состояния:**
- **Created:** Задача создана пользователем, но ещё не обработана системой.
- **Queued:** Задача поставлена в очередь на выполнение.
- **Researching:** Research Engine выполняет исследование.
- **Impact Analyzing:** Impact Analysis Engine анализирует влияние.
- **Planning:** Planner формирует Execution Plan.
- **Executing:** Execution выполняет шаги плана. Может длиться долго (много итераций).
- **Reviewing:** Reviewer выполняет финальное ревью.
- **Reopened:** Задача возвращена на доработку (если ревью выявило проблемы).
- **Rejected:** Задача отклонена (невыполнима, противоречит архитектуре, требует уточнения).
- **Completed:** Задача успешно выполнена. Знания сохранены, документация обновлена.
- **Archived:** Задача перемещена в архив. Доступна для чтения, но не может быть изменена.

**Переходы:**
- Created → Queued: автоматически при подтверждении пользователем.
- Queued → Researching: автоматически при взятии в обработку.
- Researching → Impact Analyzing: автоматически после завершения Research.
- Impact Analyzing → Planning: автоматически после завершения Impact Analysis.
- Planning → Rejected: Planner определил задачу как невыполнимую.
- Planning → Executing: автоматически после формирования плана.
- Executing → Reviewing: автоматически после выполнения всех шагов.
- Reviewing → Completed: ревью пройдено успешно.
- Reviewing → Reopened: обнаружены проблемы, требующие доработки.
- Reopened → Executing: задача возвращается на выполнение.
- Completed → Archived: автоматически через заданный период или вручную.
- **Любое состояние → Отменена пользователем (Cancelled):** в любой момент пользователь может отменить задачу.

### 5.2 Run

```
                 ┌─────────┐
                 │ Created  │
                 └────┬─────┘
                      │ Инициализирован Workspace
                      ▼
                 ┌─────────┐
                 │ Running  │
                 └────┬─────┘
                      │
          ┌───────────┼───────────┐
          │           │           │
          ▼           ▼           ▼
    ┌──────────┐ ┌──────────┐ ┌──────────┐
    │Completed │ │  Failed   │ │Cancelled │
    └──────────┘ └────┬─────┘ └──────────┘
                      │ (если разрешены повторные попытки)
                      ▼
                 ┌─────────┐
                 │ Retrying │──────► Running
                 └─────────┘
```

**Состояния:**
- **Created:** Run создан, Workspace ещё не инициализирован.
- **Running:** Run активен. Workspace инициализирован, Agent выполняет операцию.
- **Completed:** Run успешно завершён. Artifact-ы созданы и сохранены.
- **Failed:** Run завершился с ошибкой, которая не может быть исправлена автоматически.
- **Retrying:** Run завершился с ошибкой, но разрешена повторная попытка (с новым контекстом).
- **Cancelled:** Run отменён (пользователем или системой).

**Переходы:**
- Created → Running: Workspace успешно инициализирован.
- Running → Completed: операция выполнена успешно.
- Running → Failed: неисправимая ошибка.
- Running → Cancelled: отмена пользователем или таймаут.
- Running → Retrying: ошибка, допускающая повторную попытку.
- Retrying → Running: повторный запуск.
- Retrying → Failed: исчерпаны все попытки.

### 5.3 Knowledge

```
                 ┌─────────┐
                 │ Created  │
                 └────┬─────┘
                      │
                      ▼
                 ┌─────────┐
                 │ Active   │
                 └────┬─────┘
                      │
          ┌───────────┼───────────┐
          │           │           │
          ▼           ▼           ▼
    ┌──────────┐ ┌──────────┐ ┌──────────┐
    │Updated   │ │Deprecated│ │ Archived │
    └──────────┘ └──────────┘ └──────────┘
```

**Состояния:**
- **Created:** Знание создано (из Research Report, Execution Report, ADR).
- **Active:** Знание актуально и используется системой.
- **Updated:** Создана новая версия знания. Предыдущая версия остаётся доступной для истории, но более не является актуальной.
- **Deprecated:** Знание устарело, но ещё может быть релевантно для исторического контекста.
- **Archived:** Знание перемещено в архив (более не используется, но хранится для аудита).

### 5.4 Decision (ADR)

```
                 ┌──────────┐
                 │ Proposed  │
                 └────┬─────┘
                      │ (обсуждение, анализ trade-off)
                      ▼
                 ┌──────────┐
                 │ Accepted  │──────── Отклонено ──────► ┌──────────┐
                 └────┬─────┘                            │ Rejected │
                      │                                  └──────────┘
                      ▼
                 ┌──────────────┐
                 │ Implemented   │
                 └──────┬───────┘
                        │
               ┌────────┼────────┐
               │                 │
               ▼                 ▼
        ┌───────────┐     ┌──────────┐
        │Superseded │     │Deprecated│
        └───────────┘     └──────────┘
```

**Состояния:**
- **Proposed:** Решение предложено, но ещё не утверждено.
- **Accepted:** Решение утверждено, но ещё не реализовано.
- **Rejected:** Решение отклонено (с указанием причин). Сохраняется для истории.
- **Implemented:** Решение реализовано в коде.
- **Superseded:** Решение заменено новым Decision. Ссылается на новый Decision.
- **Deprecated:** Решение более не актуально, но связанный код ещё существует.

### 5.5 Artifact

```
                 ┌─────────┐
                 │ Created  │
                 └─────────┘
                      │ Artifact неизменяем
                      │ (immutable)
                      ▼
                 ┌─────────┐
                 │ Active   │
                 └────┬─────┘
                      │
                      ▼
                 ┌──────────┐
                 │ Superseded│ (создана новая версия Artifact)
                 └──────────┘
```

**Состояния:**
- **Created:** Artifact создан в результате Run.
- **Active:** Artifact актуален и используется.
- **Superseded:** Создана новая версия Artifact. Предыдущая версия сохранена для traceability.

Artifact не имеет состояния "Deleted" — артефакты хранятся навсегда как часть audit trail.

### 5.6 Workspace

```
                 ┌──────────┐
                 │Initialized│
                 └────┬─────┘
                      │
                      ▼
                 ┌──────────┐
                 │  Active   │
                 └────┬─────┘
                      │
          ┌───────────┼───────────┐
          │           │           │
          ▼           ▼           ▼
    ┌──────────┐ ┌──────────┐ ┌──────────┐
    │Merged    │ │Discarded │ │Snapshotted│
    └──────────┘ └──────────┘ └──────────┘
```

**Состояния:**
- **Initialized:** Workspace создан на основе снимка проекта.
- **Active:** Workspace активен, операции выполняются.
- **Merged:** Изменения из Workspace слиты в основной проект (при успешном Run).
- **Discarded:** Workspace уничтожен, изменения откатаны (при неудачном Run или отмене).
- **Snapshotted:** Состояние Workspace сохранено как Snapshot для возможности отката.

## 6. Общие принципы

### Принципы целостности и независимости

1. **Knowledge не изменяет Graph.**  
   Knowledge может связываться с узлами Graph (через RELATES_TO), но не может изменять узлы или рёбра Graph. Только Indexer изменяет Graph, и только на основе реального состояния кода.

2. **Graph не зависит от AI.**  
   Graph содержит только структурную информацию о коде и связях между сущностями. Graph не содержит результатов работы AI (планов, исследований, решений). Эти данные хранятся в Knowledge.

3. **Run всегда относится к одной Task.**  
   Run не может существовать вне Task. Каждый Run выполняется в контексте конкретной Task.

4. **Context всегда одноразовый.**  
   Context создаётся для конкретного запроса к LLM, используется один раз и не сохраняется (логируется для аудита, но не переиспользуется как объект).

5. **Artifact неизменяемый (immutable).**  
   После создания Artifact не может быть изменён. Для обновления создаётся новый Artifact, который ссылается на предыдущий.

6. **Decision неизменяемый (immutable).**  
   Принятое архитектурное решение не переписывается. Если решение пересматривается, создаётся новый Decision со статусом Superseded, который ссылается на предыдущий.

7. **Message неизменяемый (immutable).**  
   Сообщение в диалоге не может быть изменено после отправки. Это обеспечивает целостность истории диалога.

8. **Workspace изолирован.**  
   Каждый Run получает собственный Workspace. Изменения в одном Workspace не влияют на другие до явного Merge.

9. **Project — корень всего.**  
   Все операции выполняются в контексте конкретного Project. Знания накапливаются в рамках Project, но могут быть переиспользованы между проектами.

10. **Feature — сквозная сущность.**  
    Feature не принадлежит ни одному агрегату. Она связывает элементы из разных агрегатов (символы из Project, знания из Knowledge) для обеспечения функциональной целостности.

### Принципы идентификации

11. **Каждая сущность имеет уникальный идентификатор (UUID).**  
    Идентификаторы генерируются при создании сущности и никогда не изменяются. Это обеспечивает traceability даже при переименовании сущности.

12. **Node в Graph имеет тот же идентификатор, что и представляемая сущность.**  
    Если Class имеет UUID `abc-123`, то соответствующий Node в Graph имеет тот же UUID. Это обеспечивает прямую связь между Domain Model и Graph без дополнительного маппинга.

### Принципы прослеживаемости

13. **Каждый Artifact ссылается на породивший его Run и Task.**  
    Цепочка прослеживаемости: Message → Conversation → Agent → Run → Artifact → Task → Project.

14. **Каждое изменение File должно иметь связанный Run.**  
    Изменения файлов, сделанные вне Client (напрямую разработчиком), отслеживаются через Git, но связываются с Run только если были сделаны в рамках Run.

15. **Каждый вызов LLM сохраняется как Message в Conversation.**  
    Полная история взаимодействия с AI доступна для аудита.

### Принципы безопасности и изоляции

16. **Provider не имеет доступа к коду проекта напрямую.**  
    Provider получает только подготовленный Context. Исходный код проекта не передаётся провайдеру без явной обработки Context Builder.

17. **Workspace всегда работает в sandbox-режиме.**  
    Код, сгенерированный AI, никогда не выполняется на хостовой системе.
