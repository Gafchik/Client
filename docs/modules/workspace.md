## 1. Назначение

### 1.1 Что такое Workspace

Workspace — это **изолированная, контролируемая, воспроизводимая среда выполнения проекта**, предоставляемая модулем Execution Engine и Agent-ам для безопасного внесения изменений в кодовую базу.

Workspace не является просто директорией с файлами. Workspace — это **инфраструктурный слой**, который обеспечивает:

- полную изоляцию одного Run от другого;
- контролируемый доступ к файловой системе проекта;
- гарантию, что ни один Agent не может повредить основной репозиторий до явного Merge;
- воспроизводимость: два Run с одинаковыми входными параметрами и одинаковым состоянием проекта дадут идентичную стартовую среду;
- полную историю изменений в рамках Run, привязанную к конкретным шагам плана;
- staging-область для артефактов выполнения (логи, сгенерированные файлы, промежуточные результаты).

### 1.2 Почему Workspace существует

В системе Client код проекта непрерывно изменяется Agent-ами. Без изолированной среды выполнения:

- параллельные задачи могли бы конфликтовать на уровне файловой системы;
- ошибка Agent-а могла бы повредить основной репозиторий без возможности отката;
- было бы невозможно определить, какие изменения принадлежат конкретному Run;
- воспроизводимость выполнения была бы недостижима;
- аудит изменений (кто, когда, почему изменил файл) был бы невозможен.

Workspace решает эти проблемы на инфраструктурном уровне, а не на уровне соглашений между модулями.

### 1.3 Какие задачи решает Workspace

| Задача | Решение |
|--------|---------|
| Изоляция параллельных Run | Каждый Run получает собственный Workspace, изменения изолированы до Merge |
| Безопасное выполнение Agent-ов | Agent не имеет прямого доступа к основному репозиторию; все операции проходят через контролируемый Workspace |
| Откат неудачного Run | Pre-Run Snapshot позволяет откатить Workspace до состояния "до начала Run" |
| Воспроизводимость | Каждый Workspace инициализируется от фиксированного состояния (Git commit + Stable Snapshot) |
| Аудит изменений | Все файловые операции в Workspace логируются с привязкой к Run, шагу плана и агенту |
| Стадийность изменений | Изменения накапливаются в Workspace, проходят валидацию (линтеры, тесты, review), и только затем сливаются в основной репозиторий |
| Промежуточное хранение артефактов | Логи, отчёты, сгенерированные файлы хранятся в Workspace до переноса в постоянное хранилище |

### 1.4 Почему Execution, Indexer и Snapshot Strategy зависят от Workspace

- **Execution Engine** выполняет шаги плана через Agent-ов. Agent-ы оперируют файлами. Без Workspace Execution Engine не имеет безопасной песочницы для выполнения операций чтения/записи. Workspace предоставляет Execution Engine гарантию, что изменения не повредят проект до завершения всех проверок. Execution Engine не управляет файловой системой напрямую — он делегирует это Workspace.

- **Indexer** индексирует файлы проекта для построения Graph. Когда Run завершается успешно и изменения сливаются в репозиторий (Merge), именно Workspace публикует события `FileChanged`, на которые реагирует Indexer. Без Workspace Indexer не имеет чёткого сигнала о том, какие файлы изменились, и был бы вынужден сканировать весь репозиторий целиком.

- **Snapshot Strategy** (описанная в Storage Architecture, `002-storage.md`, раздел 11) определяет, когда и как создаются снапшоты проекта. Workspace является **исполнителем** Snapshot Strategy: именно Workspace создаёт Pre-Run Snapshot перед началом Run, именно Workspace инициирует создание Stable Snapshot после успешного Merge. Snapshot Strategy определяет политику (когда, как часто, retention), а Workspace обеспечивает механизм.

### 1.5 Чем Workspace отличается от Repository

| Аспект | Repository (Git) | Workspace |
|--------|------------------|-----------|
| **Назначение** | Хранение и версионирование кода проекта | Изолированная среда для выполнения одного Run |
| **Время жизни** | Перманентно (весь жизненный цикл проекта) | Временно (от создания Run до Merge/Discard) |
| **Изоляция** | Один репозиторий на проект | Один Workspace на Run |
| **Изменения** | Коммиты — перманентная история | Мутабельный слой, который либо сливается в коммит, либо уничтожается |
| **Доступ** | Только через Git-операции (clone, pull, commit, push) | Прямой файловый доступ (чтение, запись, удаление) — для Agent-ов |
| **Авторитетность** | Источник истины для состояния кода | Временная копия; авторитетным становится только после успешного Merge |

Workspace **не заменяет** Repository. Workspace — это временная песочница, которая базируется на состоянии Repository и в конечном счёте либо сливается обратно в Repository, либо уничтожается.

### 1.6 Чем Workspace отличается от Filesystem

| Аспект | Filesystem | Workspace |
|--------|------------|-----------|
| **Уровень абстракции** | Низкоуровневые операции (open, read, write, unlink) | Высокоуровневые операции (checkout, merge, snapshot, discard) |
| **Изоляция** | Отсутствует (любой процесс с правами может читать/писать) | Полная (каждый Run видит только свой Workspace) |
| **Версионирование** | Отсутствует | Встроенное: Pre-Run Snapshot, инкрементальные изменения |
| **Аудит** | Отсутствует | Каждая операция логируется с привязкой к Run и шагу плана |
| **Политики безопасности** | Стандартные Unix permissions | Декларативные sandbox-политики (allowed paths, forbidden operations, tool restrictions) |

Workspace **использует** файловую систему как нижележащий механизм хранения, но добавляет поверх неё уровень изоляции, контроля, аудита и жизненного цикла.

### 1.7 Чем Workspace отличается от Project Snapshot

| Аспект | Project Snapshot | Workspace |
|--------|------------------|-----------|
| **Определение** | Неизменяемая точка сохранения состояния проекта (файлы + Graph + Knowledge) | Активная, мутабельная среда выполнения |
| **Изменяемость** | Неизменяем (immutable) | Изменяем (mutable) — Agent-ы активно меняют файлы |
| **Время жизни** | Определяется retention policy (дни, недели, месяцы) | От создания Run до его завершения |
| **Назначение** | Точка восстановления для отката | Среда для выполнения работы |
| **Создаётся кем** | Workspace (Pre-Run Snapshot) или по команде пользователя/системы | Workspace Provisioner при старте Run |

Project Snapshot — это **сохранённое состояние**, из которого создаётся Workspace. Workspace — это **живая среда**, которая может быть заснапшочена, слита или уничтожена.

## 2. Ответственность

### 2.1 Что входит в ответственность Workspace

1. **Provisioning изолированной рабочей копии проекта** из указанного Git-коммита и/или Stable Snapshot.
2. **Управление жизненным циклом Workspace:** создание, инициализация, активное использование, снапшотирование, слияние, откат, удаление.
3. **Предоставление контролируемого файлового доступа** Agent-ам: чтение, запись, создание, удаление, перемещение файлов и директорий.
4. **Создание Pre-Run Snapshot** перед началом каждого Run.
5. **Инициирование Stable Snapshot** после успешного завершения Run (через Snapshot Manager).
6. **Изоляция параллельных Run:** каждый Run работает в собственном Workspace, не видя изменений других Run.
7. **Соблюдение sandbox-политик:** ограничение операций, разрешённых путей, запрещённых паттернов файлов.
8. **Отслеживание изменений:** какие файлы изменены, созданы, удалены в рамках данного Workspace.
9. **Публикация событий** о значимых изменениях состояния Workspace (`WorkspaceInitialized`, `FileChanged`, `WorkspaceMerged`, `WorkspaceDiscarded`, `SnapshotCreated` и др.).
10. **Координация с Repository Bridge** для операций checkout, merge, commit.
11. **Предоставление staging-области для артефактов:** логи выполнения, сгенерированные файлы, промежуточные результаты работы Agent-ов.
12. **Валидация целостности Workspace:** проверка, что файловая система не повреждена, все ожидаемые файлы на месте.
13. **Обработка ошибок provisioning, merge, discard** с соответствующими событиями и fallback-стратегиями.
14. **Восстановление Workspace после сбоев:** возобновление работы с последнего согласованного состояния.
15. **Обеспечение воспроизводимости:** два Run с одинаковыми параметрами (один commit, один Snapshot) получают идентичную стартовую среду.
16. **Управление временными и системными файлами:** очистка временных артефактов после завершения Run, защита от захламления.

### 2.2 Что категорически НЕ входит в ответственность Workspace

1. **Формирование Execution Plan** — зона ответственности Planner.
2. **Принятие архитектурных решений** — зона ответственности Planner, Research Engine и пользователя.
3. **Валидация корректности кода** (линтеры, статический анализ, тесты) — зона ответственности Execution Engine (Validation Engine).
4. **Принятие решения о Merge** — Workspace предоставляет механизм Merge, но решение о том, что Merge допустим, принимает Execution Engine (Reviewer Agent) и/или пользователь.
5. **Хранение Graph** — зона ответственности модуля Graph (Neo4j). Workspace работает с файлами, а не с графовым представлением.
6. **Хранение Knowledge** — зона ответственности модуля Knowledge (PostgreSQL + Vector Store).
7. **Индексация файлов** — зона ответственности Indexer. Workspace уведомляет Indexer об изменениях через события, но не индексирует файлы сам.
8. **Управление Git-репозиторием за пределами Workspace** (clone, push, управление ветками) — зона ответственности Repository Bridge. Workspace использует Repository Bridge, но не управляет репозиторием напрямую.
9. **Разрешение конфликтов между параллельными Run** — зона ответственности Impact Analysis Engine и Planner. Workspace может обнаружить конфликт при Merge, но не разрешает его самостоятельно.
10. **Определение Snapshot Retention Policy** — зона ответственности Storage Architecture. Workspace исполняет политику, но не определяет её.
11. **Управление Docker-контейнерами** — Workspace предоставляет файловую среду; управление контейнерами для sandbox-выполнения — зона ответственности Execution Engine (Sandbox Executor).
12. **Секретность и шифрование данных** — зона ответственности Security Layer и Configuration Management. Workspace обеспечивает, что секреты не попадают в логи и артефакты, но не управляет ключами шифрования.
13. **Мониторинг и алертинг** — зона ответственности Observability Layer (OpenTelemetry, Grafana, Loki). Workspace публикует метрики и события, но не строит дашборды.
14. **Управление пользователями и правами доступа** — зона ответственности API Gateway и Auth-сервиса.

## 3. Входные данные

### 3.1 Project Files

| Аспект | Значение |
|--------|----------|
| **Описание** | Исходные файлы проекта, составляющие кодовую базу |
| **Источник** | Git Repository |
| **Authoritative Owner** | Repository (Git) |
| **Как используется** | Копируются в Workspace при provisioning как базовая файловая структура. Agent-ы читают и модифицируют их в рамках Run |
| **Mutable/Immutable** | Immutable в источнике (конкретный коммит). Становятся mutable в контексте Workspace |

### 3.2 Repository State (Branch / Commit Reference)

| Аспект | Значение |
|--------|----------|
| **Описание** | Указатель на конкретное состояние репозитория: ветка, коммит, или тег |
| **Источник** | Execution Plan (поле `baseRevision`) или Task Metadata |
| **Authoritative Owner** | Repository (Git) |
| **Как используется** | Определяет, от какой точки кодовой базы создаётся Workspace. Обеспечивает воспроизводимость: два Run от одного коммита получат идентичную стартовую среду |
| **Mutable/Immutable** | Immutable (конкретный коммит не меняется) |

### 3.3 Task / Run Metadata

| Аспект | Значение |
|--------|----------|
| **Описание** | Метаданные задачи и конкретного Run: `taskId`, `runId`, `planId` |
| **Источник** | Task Manager (через событие `RunCreated`) |
| **Authoritative Owner** | Task Manager |
| **Как используется** | Привязывает Workspace к конкретному Run. Используется для именования, логирования, аудита и трассировки |
| **Mutable/Immutable** | Immutable (метаданные фиксируются при создании Run) |

### 3.4 Execution Plan Constraints

| Аспект | Значение |
|--------|----------|
| **Описание** | Ограничения из Execution Plan, влияющие на Workspace: `allowedPaths`, `forbiddenPatterns`, `workspaceScope` |
| **Источник** | Execution Plan (создан Planner) |
| **Authoritative Owner** | Planner |
| **Как используется** | Определяет, какие части файловой системы доступны Agent-ам, какие файлы нельзя трогать, какой scope у Workspace |
| **Mutable/Immutable** | Immutable в рамках Run (план не меняется без явного Replan) |

### 3.5 Configuration

| Аспект | Значение |
|--------|----------|
| **Описание** | Конфигурация Workspace: пути хранения, политики очистки, лимиты размера, настройки sandbox |
| **Источник** | System Configuration (PostgreSQL) |
| **Authoritative Owner** | System Configuration |
| **Как используется** | Параметризует поведение Workspace: где хранить временные файлы, какой максимальный размер Workspace, как долго хранить Pre-Run Snapshot |
| **Mutable/Immutable** | Mutable (администратор может менять конфигурацию), но для активного Workspace конфигурация фиксируется на момент создания |

### 3.6 Sandbox Policies

| Аспект | Значение |
|--------|----------|
| **Описание** | Политики безопасности песочницы: разрешённые операции, запрещённые пути, ограничения сети, лимиты ресурсов |
| **Источник** | Execution Plan (поле `safetyBounds`) + System Configuration |
| **Authoritative Owner** | Execution Plan (операционные политики), System Configuration (глобальные политики) |
| **Как используется** | Применяются ко всем файловым операциям в Workspace. Операция, нарушающая политику, блокируется до выполнения |
| **Mutable/Immutable** | Immutable в рамках Run |

### 3.7 Tool Requirements

| Аспект | Значение |
|--------|----------|
| **Описание** | Требования к инструментам, которые будут выполняться в Workspace: компиляторы, линтеры, тестовые фреймворки, пакетные менеджеры |
| **Источник** | Execution Plan (поле `toolBindings`) |
| **Authoritative Owner** | Execution Plan |
| **Как используется** | Определяет, какие инструменты должны быть доступны в Workspace. Влияет на provisioning (например, `npm install` перед стартом) |
| **Mutable/Immutable** | Immutable в рамках Run |

### 3.8 Artifact Requirements

| Аспект | Значение |
|--------|----------|
| **Описание** | Требования к артефактам, которые будут созданы в Workspace: директории для логов, отчётов, сгенерированных файлов |
| **Источник** | Execution Plan |
| **Authoritative Owner** | Planner (через Execution Plan) |
| **Как используется** | Workspace резервирует staging-области под артефакты согласно этим требованиям |
| **Mutable/Immutable** | Immutable в рамках Run |

### 3.9 Snapshot Metadata

| Аспект | Значение |
|--------|----------|
| **Описание** | Метаданные снапшота, из которого создаётся Workspace: `snapshotId`, `snapshotVersion`, `graphVersion`, `knowledgeVersion` |
| **Источник** | Snapshot Manager |
| **Authoritative Owner** | Snapshot Manager |
| **Как используется** | Если Workspace создаётся из снапшота (а не из Git checkout), эта метаданные определяют, какой именно снапшот использовать |
| **Mutable/Immutable** | Immutable (ссылка на конкретный снапшот) |

## 4. Выходные данные

### 4.1 Isolated Working Copy

| Аспект | Значение |
|--------|----------|
| **Описание** | Полная изолированная копия проекта, готовая к модификации |
| **Потребитель** | Execution Engine, Agent-ы |
| **Содержание** | Все файлы проекта на момент указанного коммита/снапшота |
| **Время жизни** | От создания Workspace до его Discard или Archival |

### 4.2 Workspace State

| Аспект | Значение |
|--------|----------|
| **Описание** | Текущее состояние Workspace: статус (Initialized, Active, Snapshotted и т.д.), прогресс выполнения, активные операции |
| **Потребитель** | Execution Engine, Task Manager, UI (дашборд) |
| **Содержание** | Статус, текущий шаг, количество изменённых файлов, флаги ошибок |
| **Время жизни** | Актуально в реальном времени |

### 4.3 File Operation Results

| Аспект | Значение |
|--------|----------|
| **Описание** | Результаты файловых операций: успех/неудача, изменённые пути, метаданные |
| **Потребитель** | Execution Engine, Agent-ы |
| **Содержание** | Для каждой операции: тип (create/read/write/delete/move), путь, результат, ошибки |
| **Время жизни** | Логируются и сохраняются в рамках Workspace |

### 4.4 Workspace Snapshot

| Аспект | Значение |
|--------|----------|
| **Описание** | Полная или инкрементальная копия состояния Workspace на определённый момент (Pre-Run, Intermediate, Pre-Merge) |
| **Потребитель** | Snapshot Manager, Execution Engine (для rollback) |
| **Содержание** | Состояние всех файлов, метаданные Workspace, привязка к Run и шагу |
| **Время жизни** | Pre-Run: до завершения Run + grace period. Stable: согласно retention policy |

### 4.5 Merge Result

| Аспект | Значение |
|--------|----------|
| **Описание** | Результат слияния изменений из Workspace в основной репозиторий |
| **Потребитель** | Execution Engine, Repository Bridge, Knowledge |
| **Содержание** | Статус (success/conflict/failed), список слитых файлов, конфликты, созданный коммит |
| **Время жизни** | Сохраняется в Event Log перманентно |

### 4.6 Discard Result

| Аспект | Значение |
|--------|----------|
| **Описание** | Результат уничтожения Workspace после неудачного или отменённого Run |
| **Потребитель** | Execution Engine, Task Manager |
| **Содержание** | Статус, список уничтоженных артефактов, сохранённые логи (если применимо) |
| **Время жизни** | Фиксируется в Event Log |

### 4.7 Execution Artifacts

| Аспект | Значение |
|--------|----------|
| **Описание** | Артефакты, созданные в процессе выполнения: логи, отчёты, сгенерированные файлы |
| **Потребитель** | Knowledge, Artifact Manager, пользователь |
| **Содержание** | Файлы логов, Execution Report (промежуточный), линтер-отчёты, тестовые результаты |
| **Время жизни** | Временные артефакты — до завершения Run. Итоговые — переносятся в постоянное хранилище |

### 4.8 File Change Events

| Аспект | Значение |
|--------|----------|
| **Описание** | События об изменениях файлов: `FileChanged`, `FileCreated`, `FileDeleted` |
| **Потребитель** | Indexer, Graph, Knowledge |
| **Содержание** | Путь файла, тип изменения, предыдущий хеш (если есть), новый хеш |
| **Время жизни** | Публикуются в момент изменения, сохраняются в Event Log |

### 4.9 Workspace Diagnostics

| Аспект | Значение |
|--------|----------|
| **Описание** | Диагностическая информация о состоянии и производительности Workspace |
| **Потребитель** | Execution Engine, Observability Layer (Grafana) |
| **Содержание** | Время provisioning, размер Workspace, количество операций, ошибки, использование диска |
| **Время жизни** | Публикуется как метрики; агрегируется в Grafana |

### 4.10 Workspace Metadata

| Аспект | Значение |
|--------|----------|
| **Описание** | Метаданные Workspace, сохраняемые после завершения Run |
| **Потребитель** | Knowledge, Task Manager, Audit |
| **Содержание** | `workspaceId`, `runId`, `taskId`, `baseCommit`, `snapshotId`, список изменённых файлов, длительность |
| **Время жизни** | Сохраняется в PostgreSQL перманентно |

## 6. Workspace Model

### 6.1 Workspace Instance

**Workspace Instance** — это конкретный экземпляр изолированной рабочей среды, созданный для одного Run.

**Атрибуты:**
- `workspaceId` — уникальный идентификатор (UUID);
- `runId` — идентификатор Run, к которому привязан Workspace;
- `taskId` — идентификатор родительской Task;
- `planId` — идентификатор Execution Plan;
- `state` — текущее состояние (см. Lifecycle);
- `baseCommit` — Git commit hash, от которого создан Workspace;
- `snapshotId` — идентификатор снапшота (если создан из снапшота);
- `rootPath` — физический путь к корневой директории Workspace;
- `createdAt` — временная метка создания;
- `completedAt` — временная метка завершения (Merge/Discard);
- `metadata` — дополнительные метаданные.

### 6.2 Workspace State

Текущее состояние Workspace, определяющее допустимые операции (см. Lifecycle для переходов).

### 6.3 Workspace Identity

Каждый Workspace имеет уникальную идентичность, определяемую:
- `workspaceId` (технический идентификатор),
- `runId` (связь с бизнес-контекстом),
- `baseCommit` (версионная привязка).

Два Workspace с одинаковым `baseCommit`, `taskId` и конфигурацией должны быть **идентичны на старте** (воспроизводимость).

### 6.4 Base Revision

**Base Revision** — это неизменяемый фундамент Workspace: состояние файлов проекта на момент указанного Git-коммита. Это "нулевой слой", от которого отсчитываются все изменения.

**Характеристики:**
- Определяется `baseCommit` (Git commit hash).
- Не изменяется в течение жизни Workspace.
- Является точкой отсчёта для diff при Merge.
- Обеспечивает воспроизводимость: повторное создание Workspace от того же `baseCommit` даст идентичную стартовую среду.

### 6.5 Mutable Layer

**Mutable Layer** — это набор изменений, внесённых Agent-ами поверх Base Revision. Это единственная часть Workspace, которая изменяется в процессе Run.

**Характеристики:**
- Содержит: созданные файлы, изменённые файлы, удалённые файлы.
- Отслеживается Filesystem Manager (каждая операция логируется).
- Именно Mutable Layer анализируется при Merge.
- Именно Mutable Layer сохраняется в Intermediate Snapshot.
- При Discard Mutable Layer уничтожается безвозвратно (если не сохранены артефакты).

### 6.6 Snapshot

**Workspace Snapshot** — это сохранённое состояние Mutable Layer на определённый момент времени.

**Типы снапшотов Workspace:**
- **Pre-Run Snapshot:** Состояние Workspace до начала выполнения (равно Base Revision). Используется для rollback при сбое.
- **Intermediate Snapshot:** Состояние после важного milestone (успешное выполнение группы шагов). Опционально, определяется Execution Plan.
- **Pre-Merge Snapshot:** Состояние перед Merge. Сохраняется для аудита и возможности отката после Merge.

**Связь с Project Snapshot:**
- Pre-Run Snapshot является частным случаем Project Snapshot, ограниченным файловой системой (не включает Graph и Knowledge).
- Stable Snapshot (создаваемый после успешного Merge) является полным Project Snapshot, но инициируется Workspace после завершения Merge.

### 6.7 Merge Target

**Merge Target** — это целевой репозиторий и ветка, куда будут слиты изменения из Workspace.

**Характеристики:**
- Определяется в момент создания Workspace.
- Обычно это та же ветка, от которой был создан Workspace.
- При параллельных Run Merge Target может измениться (новые коммиты в ветке). Это обнаруживается на этапе Pre-Merge Validation.

### 6.8 Artifact Area

**Artifact Area** — выделенная область внутри Workspace для хранения артефактов выполнения.

**Структура:**
- `workspace_root/.client/artifacts/logs/` — логи выполнения;
- `workspace_root/.client/artifacts/reports/` — отчёты (линтеры, тесты);
- `workspace_root/.client/artifacts/generated/` — сгенерированные файлы;
- `workspace_root/.client/artifacts/staging/` — промежуточные результаты.

**Характеристики:**
- Изолирована от основной кодовой базы (не попадает в Merge).
- Сохраняется при Discard (артефакты могут быть ценными, даже если Run не удался).
- Очищается Cleanup Manager согласно retention policy.

### 6.9 Temporary Area

**Temporary Area** — область для временных файлов, которые не имеют ценности после завершения Run.

**Структура:**
- `workspace_root/.client/tmp/` — временные файлы.

**Характеристики:**
- Гарантированно уничтожается при Discard и Cleanup.
- Не снапшотируется (не включается в Intermediate Snapshot).
- Не попадает в Artifact Staging.

### 6.10 Run Binding

**Run Binding** — жёсткая привязка Workspace к конкретному Run.

**Характеристики:**
- Один Workspace обслуживает ровно один Run.
- Workspace не может быть переиспользован для другого Run.
- После завершения Run Workspace либо Merge-ится, либо Discard-ится.
- Run Binding обеспечивает traceability: все изменения в Workspace имеют однозначную привязку к Run.

### 6.11 Task Binding

**Task Binding** — связь Workspace с родительской Task.

**Характеристики:**
- `taskId` сохраняется в метаданных Workspace.
- Используется для группировки Workspace-ов (один Task может иметь несколько Run при retry).
- Обеспечивает контекст для Knowledge: все артефакты Workspace привязаны к Task.

### 7.2 Описание состояний

| Состояние | Описание | Допустимые операции |
|-----------|----------|---------------------|
| **Created** | Workspace зарегистрирован в системе, но физически ещё не создан. Выделен `workspaceId`, привязка к `runId`. | Нет (ожидание provisioning) |
| **Initialized** | Физическая среда создана. Base Revision скопирован. Mutable Layer пуст. Workspace готов к использованию. | Файловые операции (чтение/запись), создание снапшотов |
| **Active** | Agent-ы активно работают с Workspace. Mutable Layer изменяется. | Все файловые операции, создание Intermediate Snapshot |
| **Snapshotted** | Создан снапшот текущего состояния (Pre-Run, Intermediate или Pre-Merge). Workspace продолжает быть активным или ожидает Merge/Discard. | Файловые операции (если Active), Merge, Discard |
| **Merged** | Изменения из Workspace слиты в основной репозиторий. Workspace больше не активен. Ожидает финализации. | Чтение (для верификации), создание Stable Snapshot, Discard |
| **Failed** | В процессе работы Workspace произошла ошибка (сбой provisioning, нарушение sandbox, неконсистентность файловой системы). Требуется вмешательство. | Диагностика, Recovery, Discard |
| **Discarded** | Workspace уничтожен. Файловая система очищена. Артефакты сохранены (если были). | Нет (терминальное состояние) |
| **Recovered** | Workspace восстановлен после сбоя. Состояние восстановлено из последнего снапшота или из Base Revision. | Файловые операции, продолжение выполнения |
| **Archived** | Метаданные Workspace сохранены для долгосрочного аудита, но физическая среда уничтожена. | Нет (терминальное состояние) |

### 7.3 Переходы и инварианты

| Переход | Триггер | Инварианты |
|---------|---------|------------|
| Created → Initialized | Успешный provisioning: Base Revision скопирован, Mutable Layer пуст | `baseCommit` зафиксирован, `rootPath` существует, все файлы Base Revision на месте |
| Created → Failed | Ошибка provisioning: Git checkout не удался, недостаточно места, проблемы с правами | Ошибка зафиксирована в Event Log, `runId` и `taskId` сохранены |
| Initialized → Active | Execution Engine начинает выполнение шагов. Первый Agent получает доступ к Workspace | Pre-Run Snapshot создан (если требуется планом) |
| Active → Snapshotted | Создание Intermediate Snapshot (по требованию Execution Plan или после milestone) | Снапшот создан успешно, хеши всех файлов Mutable Layer зафиксированы |
| Active → Failed | Обнаружена неконсистентность файловой системы, нарушение sandbox, критическая ошибка | Событие `WorkspaceValidationFailed` опубликовано; Workspace заблокирован для дальнейших изменений |
| Snapshotted → Active | Продолжение выполнения после создания снапшота | Workspace разблокирован, Agent-ы могут продолжать работу |
| Active → Merged | Успешное выполнение всех шагов. Execution Engine инициирует Merge | Pre-Merge Validation пройдена, конфликтов с Merge Target нет |
| Active → Discarded | Run отменён пользователем или Execution Engine принял решение не сливать изменения | Pre-Run Snapshot сохранён (если был создан) для возможности аудита |
| Merged → Archived | Stable Snapshot создан, метаданные сохранены, Cleanup выполнен | Изменения в репозитории; метаданные в PostgreSQL |
| Merged → Discarded | Принято решение не сохранять Workspace после Merge | Файлы Workspace удалены, артефакты сохранены |
| Failed → Recovered | Recovery Manager успешно восстановил состояние из последнего снапшота | Состояние Workspace идентично состоянию на момент снапшота |
| Recovered → Active | Восстановленный Workspace готов к продолжению работы | Инварианты Active выполнены |
| Failed → Discarded | Восстановление невозможно. Принято решение уничтожить Workspace | Артефакты сохранены, событие опубликовано |
| Discarded → (terminal) | — | Workspace уничтожен безвозвратно |
| Archived → (terminal) | — | Метаданные сохранены, физическая среда уничтожена |

### 7.4 Ключевые инварианты

1. **Workspace всегда привязан к одному Run.** Нельзя изменить `runId` после создания.
2. **Base Revision никогда не изменяется.** Нельзя изменить `baseCommit` после инициализации.
3. **Merge возможен только из Active или Snapshotted.** Нельзя сливать Failed или Discarded Workspace.
4. **Discard всегда сохраняет артефакты.** Даже при аварийном Discard артефакты из Artifact Area должны быть сохранены.
5. **Каждый переход состояния публикует событие.** Система всегда знает текущее состояние каждого Workspace.

## 8. Workspace Provisioning

### 8.1 Как создаётся Workspace

Процесс provisioning — это последовательность шагов, превращающая Workspace из состояния `Created` в `Initialized`.

**Шаги provisioning:**

1. **Валидация входных параметров:**
   - Проверка, что `baseCommit` существует в репозитории.
   - Проверка, что `runId` и `taskId` валидны.
   - Проверка, что конфигурация Workspace не противоречит системным ограничениям (достаточно места, доступен Repository Bridge).

2. **Выделение物理ческого пространства:**
   - Создание `rootPath` — корневой директории Workspace.
   - Проверка доступного дискового пространства.
   - Создание структуры директорий: корень проекта, `.client/` (артефакты, временные файлы, метаданные).

3. **Получение Base Revision:**
   - **Вариант A (из Git):** Repository Bridge выполняет checkout указанного `baseCommit` в `rootPath`.
   - **Вариант B (из Snapshot):** Snapshot Manager восстанавливает состояние файлов из существующего снапшота.

4. **Верификация Base Revision:**
   - Проверка целостности: все файлы из Base Revision присутствуют.
   - Сравнение хешей файлов с эталонными (из Git tree или метаданных снапшота).

5. **Применение конфигурации Workspace:**
   - Установка sandbox-политик (разрешённые пути, запрещённые паттерны).
   - Настройка Artifact Area.
   - Инициализация системы логирования операций.

6. **Создание Pre-Run Snapshot (если требуется):**
   - Если Execution Plan требует Pre-Run Snapshot, Snapshot Manager создаёт его немедленно.
   - Pre-Run Snapshot замораживает состояние Base Revision до начала изменений.

7. **Публикация события `WorkspaceInitialized`:**
   - Содержит: `workspaceId`, `runId`, `taskId`, `baseCommit`, `rootPath`.
   - Подписчики: Execution Engine (получает подтверждение, что среда готова), Task Manager.

8. **Переход в состояние `Initialized`.**

### 8.2 Из какого source revision

Workspace всегда создаётся из конкретного, неизменяемого состояния кодовой базы:

- **Приоритет 1:** Stable Snapshot (если доступен для данного коммита). Восстановление из снапшота быстрее, чем Git checkout.
- **Приоритет 2:** Git checkout указанного `baseCommit`. Если снапшот недоступен, Workspace создаётся напрямую из Git.
- **Запрещено:** Создание из "головы ветки" (HEAD) без указания конкретного коммита. Это нарушило бы воспроизводимость.

### 8.3 Как формируется working copy

Working copy — это полная копия файлов проекта на момент `baseCommit`. Она включает:

- Все отслеживаемые Git-ом файлы (из дерева коммита).
- Игнорируемые Git-ом файлы не копируются (`.gitignore` соблюдается).
- Подмодули (submodules) инициализируются в соответствии с конфигурацией проекта.
- Файлы `.git/` не копируются в Workspace (Workspace не является Git-репозиторием — за Git-операции отвечает Repository Bridge).
- Файлы из `.gitignore` не копируются, за исключением явно указанных в конфигурации Workspace (например, локальные конфигурационные файлы).

### 8.4 Как учитывается branch/commit

- **Branch:** Определяет контекст (feature, bugfix, main), но Workspace привязывается к конкретному коммиту, а не к ветке. Ветка используется только для определения Merge Target.
- **Commit:** `baseCommit` — это жёсткая привязка. Workspace не "следит" за веткой; он зафиксирован на конкретном состоянии.
- **Отсоединённый HEAD:** Workspace всегда работает в режиме отсоединённого HEAD (detached HEAD) относительно Git. Он не привязан к ветке до момента Merge.

### 8.5 Как обеспечивается reproducibility

Воспроизводимость — это гарантия, что два Workspace, созданные с одинаковыми параметрами, будут идентичны на старте.

**Механизмы:**
- `baseCommit` однозначно определяет состояние всех файлов (Git tree hash).
- Pre-Run Snapshot фиксирует точное состояние файлов с хешами.
- Конфигурационные параметры фиксируются на момент создания Workspace и не меняются.
- Системные зависимости (версии инструментов) фиксируются в конфигурации проекта или Docker-образе.
- Случайные факторы (временные метки, порядок файлов) исключаются: файлы копируются с сохранением оригинальных временных меток из Git.

**Ограничения:**
- Внешние зависимости (npm-пакеты, composer-пакеты) могут измениться между Runs. Для полной воспроизводимости требуется lock-файл (`package-lock.json`, `composer.lock`).
- Системные инструменты (версии Node.js, PHP) должны быть зафиксированы в конфигурации проекта.

### 8.6 Как привязывается к Run

- При создании Workspace получает `runId` и `taskId`.
- Эта привязка immutable — Workspace не может быть передан другому Run.
- Run Binding используется для:
  - Логирования (все операции помечаются `runId`).
  - Трассировки событий (`correlation_id = runId`).
  - Ограничения времени жизни (после завершения Run Workspace должен быть завершён).

### 8.7 Как учитываются sandbox constraints

- Sandbox constraints из Execution Plan применяются на этапе provisioning.
- Если план ограничивает Workspace определёнными директориями (`allowedPaths`), Workspace может быть создан с виртуальным root, ограниченным этими путями.
- Если план запрещает определённые файлы (`forbiddenPatterns`), эти файлы могут быть исключены из working copy или помечены как недоступные для записи.
- Constraints валидируются: если план требует доступа к путям, которые не существуют в Base Revision, provisioning завершается с ошибкой.

## 9. Filesystem Responsibility

### 9.1 Общие принципы

Filesystem Manager — это **единственный** компонент, через который Agent-ы взаимодействуют с файловой системой Workspace. Agent-ы не имеют прямого доступа к файловой системе (ни через системные вызовы, ни через shell-команды в обход Manager).

**Все операции:**
- Проходят валидацию Sandbox Policy Manager.
- Логируются с полным контекстом (`runId`, `stepId`, `agentId`, `timestamp`).
- Проверяются на целостность после выполнения.
- При ошибке оставляют Workspace в согласованном состоянии (откат частичных изменений).

### 9.2 Чтение файлов

- **Разрешено:** чтение любых файлов в пределах `rootPath`, если они не исключены `forbiddenPatterns`.
- **Запрещено:** чтение файлов за пределами `rootPath`, чтение системных файлов хоста.
- **Формат:** возвращает содержимое файла и метаданные (размер, хеш, временная метка).
- **Частичное чтение:** поддержка чтения диапазонов (offset + limit) для больших файлов.
- **Гарантия целостности:** при чтении проверяется, что файл не был изменён в процессе чтения (сравнение хеша до и после).

### 9.3 Запись файлов

- **Полная запись (write):** перезаписывает файл целиком. Предыдущее содержимое сохраняется в истории изменений Workspace.
- **Дифференциальная запись (patch):** применяет diff к существующему файлу. Используется, когда Agent изменяет конкретные строки, а не переписывает файл целиком.
- **Атомарность:** запись либо завершается полностью, либо не применяется вовсе. Нет промежуточного состояния "файл записан наполовину".
- **Валидация после записи:** после записи проверяется, что файл корректен (не бинарный мусор, соответствует ожидаемой структуре).

### 9.4 Создание файлов

- **Новые файлы:** создаются в указанном пути в пределах `rootPath`.
- **Проверка пути:** путь должен находиться в разрешённой области (`allowedPaths`).
- **Проверка паттерна:** имя файла не должно соответствовать `forbiddenPatterns`.
- **Родительские директории:** создаются автоматически, если не существуют.
- **Конфликт:** если файл уже существует, операция может быть отклонена (если не разрешена перезапись) или выполнена как write (с сохранением предыдущей версии).

### 9.5 Удаление файлов

- **Удаление:** файл удаляется из Mutable Layer, но не из Base Revision (Base Revision неизменяем).
- **Сохранение истории:** удалённый файл сохраняется в истории изменений Workspace для возможности восстановления.
- **Защита:** файлы из `protectedPaths` (определённых в sandbox-политиках) не могут быть удалены.
- **Проверка зависимостей:** Workspace не проверяет, используется ли удаляемый файл другими файлами (это зона Impact Analysis).

### 9.6 Перемещение файлов

- **Перемещение:** атомарная операция переноса файла из одного пути в другой в пределах `rootPath`.
- **Переименование:** частный случай перемещения без изменения директории.
- **Валидация:** оба пути (источник и назначение) должны быть в разрешённой области.
- **История:** сохраняется информация о предыдущем пути.

### 9.7 Directory Operations

- **Создание директории:** рекурсивное создание с проверкой прав.
- **Удаление директории:** рекурсивное удаление с проверкой `protectedPaths`. Если директория содержит защищённые файлы, операция отклоняется.
- **Листинг директории:** возвращает список файлов и поддиректорий с метаданными.
- **Проверка существования:** проверка, что путь существует и является директорией/файлом.

### 9.8 Permissions

- Workspace не управляет Unix-правами (chmod). Права определяются процессом, под которым работает Workspace.
- Вместо Unix-прав используется **логическая модель доступа**:
  - **Read-only:** файлы Base Revision, которые не разрешено изменять.
  - **Read-write:** Mutable Layer.
  - **Forbidden:** пути, соответствующие `forbiddenPatterns`.
- Agent-ы не могут изменить модель доступа — она устанавливается при provisioning и не меняется.

### 9.9 Path Normalization

- Все пути нормализуются перед обработкой:
  - Удаление `..` (защита от directory traversal).
  - Приведение к абсолютному пути относительно `rootPath`.
  - Унификация разделителей (под конкретную ОС).
- Path traversal-атаки блокируются на уровне Filesystem Manager: попытка доступа к `../../etc/passwd` будет отклонена, даже если путь формально корректен.

### 9.10 Handling Generated, Temp, and System Files

- **Сгенерированные файлы** (результаты компиляции, сборки): хранятся в Artifact Area (`.client/artifacts/generated/`). Не смешиваются с исходным кодом.
- **Временные файлы** (промежуточные результаты): хранятся в Temporary Area (`.client/tmp/`). Автоматически очищаются Cleanup Manager.
- **Системные файлы** (`.git`, `.svn`, `.DS_Store`): исключаются из отслеживания изменений. Не видны Agent-ам и не попадают в снапшоты.
- **Lock-файлы и артефакты пакетных менеджеров** (`node_modules/`, `vendor/`): в общем случае не копируются при provisioning для экономии места и времени. Если необходимы, устанавливаются заново через Tool Invocation.

## 10. Repository Integration

### 10.1 Как Workspace связан с Git Repository

Workspace не работает с Git напрямую. Все Git-операции делегируются Repository Bridge.

**Связь:**
- **При provisioning:** Repository Bridge выполняет checkout `baseCommit` в `rootPath`.
- **При Merge:** Repository Bridge выполняет слияние изменений Workspace с Merge Target и создаёт коммит.
- **При валидации:** Repository Bridge проверяет, не изменился ли Merge Target с момента создания Workspace (новые коммиты в ветке).

Workspace хранит Git-метаданные (SHA коммита, URL репозитория, имя ветки), но не хранит `.git/` директорию в своём `rootPath`.

### 10.2 Чем отличается repository state от workspace state

| Аспект | Repository State | Workspace State |
|--------|------------------|-----------------|
| **Где хранится** | В Git (объекты, ссылки, индекс) | В файловой системе `rootPath` |
| **История** | Полная история коммитов | Изолированная история изменений в рамках Run |
| **Изменения** | Коммиты — атомарные, перманентные | Mutable Layer — непрерывные изменения |
| **Авторитетность** | Источник истины | Временная копия |
| **Доступ** | Через Git-команды | Прямой файловый доступ |
| **Ветки** | Несколько веток | Отсоединённый HEAD, привязанный к конкретному коммиту |

Repository State — это то, что было до Run и что станет после успешного Merge. Workspace State — это то, что существует только в рамках Run.

### 10.3 Как учитываются branch, commit, dirty state

- **Branch:** Используется как контекст. Workspace знает имя ветки-источника и имя ветки-назначения (Merge Target). Но Workspace всегда работает с конкретным коммитом, а не с веткой.
- **Commit:** `baseCommit` — жёсткая точка отсчёта. Все изменения Workspace вычисляются как diff относительно этого коммита.
- **Dirty state:** Workspace **никогда** не создаётся из dirty working copy репозитория. Если в репозитории есть незакоммиченные изменения, они игнорируются. Workspace создаётся только из закоммиченного состояния.

### 10.4 Как происходит sync с repository

**До начала Run:**
- Workspace Provisioner через Repository Bridge получает состояние файлов на `baseCommit`.
- Pre-Run Snapshot (если создаётся) фиксирует это состояние.

**В процессе Run:**
- Workspace не синхронизируется с репозиторием. Изменения других Run не видны до момента Merge.
- Если Merge Target изменился (новые коммиты), это обнаруживается на этапе Pre-Merge Validation.

**При Merge:**
- Repository Bridge вычисляет diff между Base Revision и текущим Mutable Layer.
- Этот diff применяется к актуальному состоянию Merge Target.
- Если есть конфликты, Merge Manager обрабатывает их согласно стратегии (см. Merge Strategy).

**После Merge:**
- Repository Bridge создаёт коммит в репозитории.
- Событие `CommitPushed` публикуется (см. раздел Events).
- Stable Snapshot создаётся.

### 10.5 Как обрабатываются rename/move/delete

- **Rename:** Отслеживается Filesystem Manager как две операции: удаление по старому пути + создание по новому пути. При Merge Repository Bridge преобразует это в Git-rename (через сравнение содержимого).
- **Move:** Аналогично rename.
- **Delete:** Файл удаляется из Mutable Layer. При Merge это преобразуется в удаление файла из репозитория (Git `rm`).

**Важно:** Workspace не пытается эвристически определить rename. Это ответственность Repository Bridge при Merge (Git сам определяет rename по similarity index).

### 10.6 Как предотвращаются unsafe mutations

**Unsafe mutations** — это изменения, которые могут повредить репозиторий или нарушить его целостность.

**Механизмы защиты:**
- Workspace не имеет доступа к `.git/` директории репозитория.
- Операции записи ограничены `allowedPaths`.
- `forbiddenPatterns` предотвращают изменение критических файлов (например, `.gitignore`, конфигурации CI/CD).
- Sandbox Policy Manager блокирует операции, классифицированные как `dangerous` и `critical`.
- Pre-Merge Validation проверяет, что изменения не конфликтуют с новыми коммитами в Merge Target.
- Workspace не может выполнять `git push` — это делается отдельно, после успешного Merge и создания Stable Snapshot.

## 11. Isolation Model

### 11.1 Почему каждый Run должен иметь изолированный Workspace

Без изоляции параллельные Run могут:
- Перезаписывать файлы друг друга, создавая гонки данных (race conditions).
- Видеть промежуточные, незавершённые изменения друг друга, что приводит к неконсистентным решениям.
- Создавать трудно обнаруживаемые конфликты, которые проявляются только при определённом порядке выполнения.
- Нарушать воспроизводимость: повторное выполнение того же Run может дать другой результат из-за вмешательства параллельного Run.

Изоляция на уровне Workspace гарантирует, что каждый Run:
- Стартует от чистого, согласованного состояния (Base Revision).
- Не видит изменений других Run.
- Не влияет на другие Run до явного Merge.
- Может быть воспроизведён независимо от других Run.

### 11.2 Как предотвращается влияние одной задачи на другую

- **Физическая изоляция:** каждый Workspace имеет свой `rootPath`. Процессы одного Workspace не могут обратиться к файлам другого Workspace.
- **Логическая изоляция:** каждый Workspace знает только свой `runId`. Информация о других Run не доступна через Workspace.
- **Изоляция событий:** события о файловых изменениях (`FileChanged`) содержат `runId` и не обрабатываются другими Run. Indexer и Graph обрабатывают их только после Merge.
- **Изоляция Git:** каждый Workspace работает со своим view репозитория (через Repository Bridge). Одновременные Merge двух Workspace сериализуются Git-ом (один коммит за другим).

### 11.3 Как работают parallel runs

При параллельных Run:
- **Разные Workspace:** Каждый параллельный Run получает свой Workspace со своим `rootPath`.
- **Одинаковый или разный baseCommit:** Если Run-ы стартуют одновременно от одного коммита, их Base Revision идентичен. Если от разных — каждый Workspace отражает своё состояние.
- **Общие ресурсы:** Workspace не разделяет файлы между параллельными Run. Единственный разделяемый ресурс — сам Git-репозиторий, но доступ к нему сериализуется Repository Bridge.
- **Параллельные шаги внутри одного Run:** Если Execution Plan допускает параллельные шаги в рамках одного Run, Execution Engine обеспечивает, что они оперируют непересекающимися областями Workspace (например, разными директориями). Workspace предоставляет механизм изоляции поддиректорий.

### 11.4 Как устроены boundaries

- **Filesystem boundary:** `rootPath` — жёсткая граница. Операции за пределами `rootPath` запрещены.
- **Scope boundary:** Дополнительное ограничение внутри `rootPath`, определяемое `allowedPaths` из Execution Plan. Agent может видеть и изменять только файлы в указанных путях.
- **Run boundary:** Workspace привязан к одному `runId`. Файлы, созданные одним Run, не видны другому.
- **Step boundary:** В рамках одного Run Execution Engine может ограничить Agent-а конкретным шагом через `workspaceScope` в плане. Workspace обеспечивает это ограничение на уровне Filesystem Manager.

### 11.5 Как решаются shared-resource risks

**Shared-resource risks** — риски, связанные с разделяемыми ресурсами, которые не могут быть полностью изолированы.

| Риск | Решение |
|------|---------|
| Одновременный Merge в одну ветку | Repository Bridge сериализует Merge-операции. Второй Merge ждёт завершения первого или завершается с конфликтом, если Merge Target изменился |
| Заполнение диска одним Workspace | Diagnostics Manager отслеживает размер Workspace. При превышении лимита публикуется warning, при критическом превышении — блокировка операций записи |
| Конкуренция за CPU/память при параллельных Run | Не зона ответственности Workspace. Решается на уровне Execution Engine (Resource Allocation) и Docker |
| Одновременное изменение одного файла в разных Run | Конфликт обнаруживается при Merge второго Run. Разрешается согласно Merge Strategy |
| Исчерпание inode в файловой системе | Cleanup Manager агрессивно очищает завершённые и discard-нутые Workspace. Diagnostics Manager отслеживает количество активных Workspace |

### 11.6 Как сохраняется reproducibility

- **Фиксация baseCommit:** Workspace не "плывёт" за веткой. Состояние на старте точно определено.
- **Изоляция от внешних изменений:** До момента Merge Workspace не видит новые коммиты в репозитории.
- **Детерминированное provisioning:** При одинаковых входных параметрах создаётся идентичная среда.
- **Детерминированные снапшоты:** Intermediate Snapshot создаётся с хешированием всех файлов, что позволяет побитово сравнить два состояния.

## 12. Sandbox and Safety

### 12.1 Sandbox boundaries

Sandbox — это логический периметр безопасности вокруг операций в Workspace.

**Границы sandbox:**
- **Filesystem boundary:** Операции ограничены `rootPath` + `allowedPaths`. Выход за пределы невозможен.
- **Network boundary:** Определяется Execution Engine (не зона Workspace). Workspace предоставляет файловую среду, но не контролирует сетевой доступ.
- **Process boundary:** Определяется Execution Engine (Docker-контейнеры). Workspace не управляет процессами.
- **Tool boundary:** Определяется Tool Registry в Execution Engine. Workspace применяет sandbox-политики к результатам выполнения инструментов.

### 12.2 Allowed operations

Операции, которые всегда разрешены (при соблюдении остальных политик):

- Чтение любых файлов в `allowedPaths`.
- Запись в файлы Mutable Layer (не защищённые `forbiddenPatterns`).
- Создание новых файлов в `allowedPaths`.
- Удаление файлов Mutable Layer (не защищённых `protectedPaths`).
- Перемещение/переименование файлов в пределах `allowedPaths`.
- Создание/удаление директорий (если они не содержат защищённых файлов).
- Листинг директорий.

### 12.3 Forbidden operations

Операции, которые **всегда** запрещены:

- Доступ к файлам за пределами `rootPath`.
- Модификация файлов, соответствующих `forbiddenPatterns`.
- Удаление файлов, соответствующих `protectedPaths`.
- Изменение структуры `.client/` (артефакты, метаданные Workspace).
- Операции, которые могут нарушить целостность Workspace (например, удаление `rootPath`).
- Выполнение произвольного кода в контексте Workspace (это зона Tool Invocation в Execution Engine, не Filesystem Manager).

### 12.4 Path restrictions

- **`allowedPaths`:** Белый список путей, к которым Agent имеет доступ. Если указан, все операции вне этих путей блокируются.
- **`forbiddenPatterns`:** Чёрный список glob-паттернов. Файлы, соответствующие этим паттернам, не могут быть изменены или удалены.
- **`protectedPaths`:** Список путей, которые нельзя удалять (но можно изменять).
- **`readOnlyPaths`:** Список путей, доступных только для чтения (даже если они в `allowedPaths`).

Пути валидируются на этапе provisioning. Если `allowedPaths` указывает на несуществующие директории, provisioning завершается с ошибкой.

### 12.5 Tool execution boundaries

Хотя Tool Execution — зона ответственности Execution Engine, Workspace применяет следующие ограничения к результатам выполнения инструментов:

- Инструмент может писать только в разрешённые пути.
- Инструмент не может изменять файлы за пределами Workspace.
- Результаты инструмента валидируются перед сохранением (проверка, что выходные данные не содержат вредоносного кода).
- Временные файлы, созданные инструментом, помещаются в Temporary Area и очищаются после завершения.

### 12.6 Destructive action control

**Destructive actions** — операции, которые могут привести к необратимой потере данных.

**Контроль:**
- **Pre-Run Snapshot:** Перед началом Run создаётся снапшот. Даже если Run разрушит Workspace, Base Revision можно восстановить.
- **Intermediate Snapshot:** Перед выполнением destructive-шагов Execution Engine может запросить Intermediate Snapshot.
- **Rollback capability:** Каждый шаг плана имеет политику отката. Workspace поддерживает откат через восстановление из снапшота.
- **Confirmation gate:** Операции классификации `dangerous` и `critical` требуют подтверждения от Execution Engine (или human approval).
- **Защита `protectedPaths`:** Критические файлы (конфигурация, `.env`, lock-файлы) защищены от удаления.

### 12.7 Secret exposure prevention

Workspace не управляет секретами, но предотвращает их случайное раскрытие:

- **Логирование:** Filesystem Manager не логирует содержимое файлов. Логируются пути, типы операций и метаданные — но не данные.
- **Артефакты:** Artifact Staging Manager проверяет артефакты на наличие паттернов секретов (API-ключи, токены) перед сохранением. При обнаружении — warning и блокировка сохранения.
- **Diff:** При публикации `FileChanged` не публикуется содержимое файла — только путь и хеш.
- **История изменений:** Не хранит полное содержимое всех версий файла — только diff.

### 12.8 Policy enforcement

**Порядок применения политик:**

1. Операция запрашивается Agent-ом.
2. Filesystem Manager проверяет путь на соответствие `allowedPaths`.
3. Проверяется на соответствие `forbiddenPatterns`.
4. Проверяется тип операции против sandbox-политик (allowed/forbidden).
5. Если операция destructive (`delete`, `write` с перезаписью), проверяется `protectedPaths`.
6. Если операция разрешена — выполняется.
7. Результат валидируется (целостность, отсутствие вредоносного содержимого).

**При нарушении политики:**
- Операция блокируется.
- Публикуется событие `WorkspacePolicyViolation` (warning/error в зависимости от серьёзности).
- Если нарушение критическое, Workspace переводится в состояние `Failed`.

### 12.9 Safety signals

**Сигналы безопасности, публикуемые Workspace:**

- **`WorkspacePolicyViolation`:** Операция была заблокирована sandbox-политикой.
- **`WorkspaceValidationFailed`:** Обнаружена неконсистентность файловой системы.
- **`WorkspaceIntegrityWarning`:** Обнаружено потенциальное нарушение целостности (например, хеш файла не совпадает с ожидаемым).
- **`WorkspaceSizeExceeded`:** Размер Workspace превысил установленный лимит.
- **`WorkspaceSecretExposureWarning`:** Обнаружен потенциальный секрет в артефактах или логах.

## 13. Snapshot Strategy

### 13.1 Что такое Workspace Snapshot

**Workspace Snapshot** — это сохранённая копия состояния файловой системы Workspace на определённый момент времени. Снапшот содержит:
- Полный список файлов с путями.
- Хеш-суммы каждого файла (для проверки целостности).
- Метаданные: `workspaceId`, `runId`, `taskId`, `baseCommit`, временная метка, причина создания.
- Тип снапшота (Pre-Run, Intermediate, Pre-Merge).

Снапшот не содержит Graph и Knowledge (это зона Project Snapshot). Workspace Snapshot — это файловый срез.

### 13.2 Когда snapshot обязателен

| Ситуация | Тип снапшота | Обязательность |
|----------|-------------|----------------|
| Перед началом выполнения Run | Pre-Run Snapshot | Обязателен, если план требует rollback capability |
| После успешного выполнения группы шагов | Intermediate Snapshot | Обязателен, если план предписывает checkpoint |
| Перед выполнением destructive-шага | Intermediate Snapshot | Обязателен, если шаг имеет `safetyBounds: destructive` |
| Перед началом Merge | Pre-Merge Snapshot | Обязателен всегда |
| После успешного Merge | Stable Snapshot | Обязателен. Инициируется Workspace, создаётся Snapshot Manager как полный Project Snapshot |
| Периодически (каждые N изменённых файлов) | Intermediate Snapshot | Опционально, согласно конфигурации |

### 13.3 Snapshot before execution

**Pre-Run Snapshot:**
- Создаётся немедленно после provisioning, до того как любой Agent получит доступ к Workspace.
- Содержит точное состояние Base Revision в `rootPath`.
- Используется как точка восстановления, если Run завершится неудачей и потребуется полный откат.
- Хранится до завершения Run + 24 часа (grace period для диагностики).

### 13.4 Snapshot after important milestones

**Intermediate Snapshot:**
- Создаётся после выполнения группы шагов (определяется Execution Plan).
- Содержит состояние Mutable Layer на момент создания.
- Используется для:
  - Ускорения rollback: можно откатиться к intermediate точке, а не к началу Run.
  - Параллельного выполнения: параллельная группа шагов может стартовать от Intermediate Snapshot.
  - Аудита: можно посмотреть состояние Workspace после каждого milestone.
- Хранится до завершения Run.

### 13.5 Stable snapshot vs transient snapshot

| Аспект | Transient Snapshot (Pre-Run, Intermediate) | Stable Snapshot |
|--------|--------------------------------------------|-----------------|
| **Когда создаётся** | В процессе Run | После успешного Merge |
| **Содержит** | Файлы Workspace | Файлы проекта + Graph + Knowledge |
| **Хранится** | До завершения Run + grace period | Согласно Retention Policy (месяцы/годы) |
| **Используется** | Rollback, отладка, аудит текущего Run | Восстановление проекта, воспроизводимость будущих Run |
| **Кто создаёт** | Workspace | Snapshot Manager (инициируется Workspace) |

### 13.6 Snapshot metadata

Каждый снапшот содержит метаданные:

- `snapshotId` — уникальный идентификатор.
- `workspaceId` — какой Workspace снапшотируется.
- `snapshotType` — `pre_run`, `intermediate`, `pre_merge`.
- `runId` — в рамках какого Run.
- `taskId` — в рамках какой Task.
- `baseCommit` — Git commit, от которого создан Workspace.
- `stepId` — для Intermediate Snapshot: после какого шага создан.
- `timestamp` — временная метка создания.
- `fileCount` — количество файлов в снапшоте.
- `totalSize` — общий размер файлов.
- `parentSnapshotId` — предыдущий снапшот (для цепочки).
- `checksum` — хеш всех хешей файлов (для проверки целостности снапшота).

### 13.7 Связь со storage architecture

Workspace Snapshot Strategy является исполнителем Snapshot Strategy, описанной в Storage Architecture (`002-storage.md`, раздел 11):

- **Хранение:** Файлы снапшотов хранятся в File Storage (специализированное файловое хранилище проекта).
- **Pre-Run Snapshot** = `Pre-Run Snapshot` из Storage Architecture (раздел 11.2).
- **Stable Snapshot** = `Stable Snapshot` из Storage Architecture. Workspace не создаёт его сам, а инициирует создание через Snapshot Manager.
- **Инкрементальность:** Snapshot Manager может хранить снапшоты инкрементально (diff относительно предыдущего), но для Workspace это прозрачно.
- **Retention:** Workspace подчиняется Retention Policy из Storage Architecture, но не определяет её.

### 13.8 Связь с project snapshot

- **Workspace Snapshot** = файловый срез Workspace. Создаётся Workspace.
- **Project Snapshot** = полный срез проекта (файлы + Graph + Knowledge). Создаётся Snapshot Manager.
- **Pre-Run Snapshot** является одновременно и Workspace Snapshot, и частью будущего Project Snapshot (файловая составляющая).
- **Stable Snapshot** (Project Snapshot) инициируется Workspace после успешного Merge, но создаётся Snapshot Manager-ом, который добавляет к файловому срезу Graph Version и Knowledge Version.

### 13.9 Reuse и cleanup snapshot-ов

- **Reuse:** Если несколько Run стартуют от одного `baseCommit`, Pre-Run Snapshot может быть переиспользован (создаётся один раз для данного коммита, используется всеми Run). Это определяется Snapshot Manager.
- **Cleanup:** Transient-снапшоты (Pre-Run, Intermediate) автоматически удаляются Cleanup Manager после завершения Run + grace period.
- **Stable Snapshot cleanup:** Согласно Retention Policy из Storage Architecture.

## 14. Merge Strategy

### 14.1 Когда Workspace может быть merged

Workspace может быть слит (merged) только при выполнении **всех** условий:

1. Все шаги Execution Plan успешно выполнены.
2. Reviewer Agent (или human reviewer) подтвердил качество изменений.
3. Все approval points, требующие human approval, пройдены.
4. Pre-Merge Validation пройдена успешно (нет конфликтов с Merge Target).
5. Workspace находится в состоянии `Active` или `Snapshotted`.
6. Workspace не имеет активных ошибок или предупреждений критического уровня.
7. Execution Engine явно инициирует Merge.

### 14.2 Что именно merge-ится

Слиянию подлежит **Mutable Layer** — изменения, внесённые Agent-ами поверх Base Revision:

- **Изменённые файлы:** Файлы, которые существовали в Base Revision и были изменены.
- **Созданные файлы:** Новые файлы, созданные в процессе Run.
- **Удалённые файлы:** Файлы, которые существовали в Base Revision и были удалены.
- **Переименованные/перемещённые файлы:** Определяются Repository Bridge при сравнении Base Revision и Mutable Layer.

**Что НЕ сливается:**
- Артефакты (`.client/artifacts/`) — они не являются частью кодовой базы.
- Временные файлы (`.client/tmp/`) — уничтожаются.
- Системные файлы (`.DS_Store`, `Thumbs.db`) — игнорируются.
- Файлы, соответствующие `forbiddenPatterns`, — даже если они были изменены, они исключаются из Merge.

### 14.3 Pre-merge validation

Перед выполнением Merge проводится серия проверок:

1. **Целостность Workspace:** Все файлы Mutable Layer доступны, не повреждены, хеши совпадают.
2. **Актуальность Merge Target:** Repository Bridge проверяет, не появились ли новые коммиты в Merge Target с момента создания Workspace.
3. **Конфликтный анализ:** Если Merge Target изменился, вычисляется пересечение изменённых файлов. Если один и тот же файл изменён и в Workspace, и в новых коммитах — это конфликт.
4. **Проверка forbiddenPatterns:** Изменённые файлы не должны соответствовать паттернам, запрещённым для Merge (например, lock-файлы без явного разрешения).
5. **Валидация коммита:** Сообщение коммита соответствует формату; author information корректна.
6. **Pre-Merge Snapshot:** Создаётся Pre-Merge Snapshot для аудита.

Если любая проверка не пройдена, Merge блокируется. В зависимости от серьёзности проблемы:
- Мелкие предупреждения: Merge продолжается с записью в лог.
- Конфликты: Merge останавливается, требуется разрешение (автоматическое или ручное).
- Критические ошибки: Workspace переходит в `Failed`.

### 14.4 Merge conflicts

**Типы конфликтов:**

| Тип | Описание | Разрешение |
|-----|----------|------------|
| **File conflict** | Один и тот же файл изменён в Workspace и в Merge Target | Git merge conflict. Если изменения в непересекающихся частях файла, Git разрешает автоматически. Если в одних и тех же строках — конфликт передаётся на разрешение |
| **Structural conflict** | Файл удалён в Workspace, но изменён в Merge Target (или наоборот) | Требуется решение: сохранить удаление или изменение |
| **Dependency conflict** | Изменения в Workspace зависят от файла, который был удалён в Merge Target | Merge блокируется. Требуется анализ зависимостей (Impact Analysis) |

**Стратегии разрешения:**
- **Force workspace:** Изменения Workspace приоритетнее. Файлы из Merge Target перезаписываются.
- **Force target:** Изменения Merge Target приоритетнее. Конфликтующие файлы из Workspace отбрасываются.
- **Manual resolution:** Конфликт передаётся человеку или Reviewer Agent-у для ручного разрешения.
- **Abort merge:** Merge отменяется. Workspace остаётся в состоянии `Active`. Execution Engine должен перепланировать.

Стратегия по умолчанию определяется Execution Plan. Для критических конфликтов всегда требуется manual resolution.

### 14.5 Safe merge boundaries

**Safe merge** — это слияние, которое не может повредить репозиторий, даже если что-то пойдёт не так.

**Границы безопасности:**
- Merge выполняется в изолированной Git-операции (временный индекс). В случае ошибки Git-индекс откатывается.
- Merge Target не изменяется до успешного завершения Merge. Если во время Merge кто-то пушит в Merge Target, операция повторяется.
- Pre-Merge Snapshot гарантирует возможность отката даже после слияния.
- Если Merge создаёт конфликт, он не разрешается автоматически (без явной стратегии).
- Коммит создаётся только после успешного завершения всех проверок.

### 14.6 Partial merge

**Partial merge** — слияние только части изменений (некоторые файлы исключаются).

**Когда применяется:**
- Execution Plan явно разрешает partial merge.
- Некоторые файлы были изменены экспериментально и не готовы к слиянию.
- Reviewer Agent принял решение, что часть изменений требует доработки.

**Ограничения:**
- Partial merge возможен только если исключаемые файлы не имеют зависимостей от сливаемых.
- Исключённые изменения сохраняются в артефактах для возможного использования в будущем.
- Partial merge фиксируется в метаданных Workspace с указанием причины и списка исключённых файлов.

### 14.7 Merge failure

Если Merge завершился ошибкой:

1. Изменения в Merge Target не применяются.
2. Workspace остаётся в состоянии `Active`.
3. Публикуется событие `WorkspaceMergeFailed` с деталями ошибки.
4. Execution Engine получает уведомление и принимает решение: retry, manual resolution, или abort.
5. Pre-Merge Snapshot сохраняется для аудита.
6. Если причина — конфликт с новыми коммитами, Execution Engine может запросить Replan у Planner.

### 14.8 Merge auditability

Каждый Merge оставляет полный audit trail:

- Pre-Merge Snapshot (состояние до Merge).
- Merge Result (diff, список файлов, статус).
- Созданный коммит (commit hash).
- События: `WorkspacePreMergeValidated`, `WorkspaceMerged` или `WorkspaceMergeFailed`.
- Записи в Event Log.

По этим записям можно восстановить: кто, когда, почему и какие именно изменения слил в репозиторий.

### 14.9 Post-merge events

После успешного Merge:

1. `WorkspaceMerged` — основное событие.
2. Snapshot Manager создаёт Stable Snapshot (событие `SnapshotCreated`).
3. Repository Bridge публикует `CommitPushed`.
4. Workspace переходит в состояние `Merged`.
5. Cleanup Manager начинает очистку (согласно grace period).
6. Indexer получает `FileChanged` для всех изменённых файлов и запускает переиндексацию.
7. Graph получает `GraphUpdated` (опосредованно через Indexer).
8. Knowledge получает сигнал о необходимости обновления.

## 15. Discard and Rollback

### 15.1 Когда Workspace discard-ится

Workspace уничтожается (discard) в следующих случаях:

- **Run failed:** Выполнение завершилось ошибкой, и восстановление невозможно/нецелесообразно.
- **Run cancelled:** Пользователь отменил задачу.
- **Merge rejected:** Reviewer или пользователь отклонил изменения.
- **Approval rejected:** Human approval не было получено на контрольной точке.
- **Sandbox violation:** Обнаружено критическое нарушение sandbox-политик.
- **Provisioning failed:** Workspace не удалось создать.
- **Manual discard:** Администратор или пользователь явно удаляет Workspace.
- **Retention expired:** Для Archived Workspace — превышен срок хранения метаданных.

### 15.2 Rollback to snapshot

**Rollback** — это восстановление Workspace до состояния, сохранённого в снапшоте.

**Процесс:**
1. Snapshot Manager загружает указанный снапшот (Pre-Run или Intermediate).
2. Mutable Layer полностью заменяется состоянием из снапшота.
3. Все изменения, сделанные после снапшота, теряются.
4. Workspace переходит в состояние `Snapshotted` (со снапшота) или `Active` (если Execution Engine продолжает выполнение).
5. Публикуется событие `WorkspaceRolledBack`.

**Что теряется при rollback:**
- Все файловые изменения после снапшота.
- Все артефакты, созданные после снапшота (если они не были сохранены отдельно).

**Что сохраняется:**
- Логи операций (для аудита).
- События, опубликованные до rollback (Event Log неизменяем).

### 15.3 Rollback after failed run

Если Run завершился неудачей:

1. Execution Engine инициирует rollback к Pre-Run Snapshot (если он был создан).
2. Или к последнему Intermediate Snapshot, если проблема возникла на позднем шаге (чтобы сохранить прогресс).
3. Если снапшотов нет, Workspace пересоздаётся заново из Base Revision (потеря всех изменений).
4. Workspace восстанавливается, Execution Engine может retry выполнение с того же или другого плана.

### 15.4 Rollback after validation failure

Если Validation Engine обнаружил, что изменения не проходят проверку (линтеры, тесты, архитектурные правила):

1. Execution Engine может инициировать rollback к Intermediate Snapshot (до шага, который внёс проблемные изменения).
2. Agent Fixer пытается исправить проблему.
3. Если исправление невозможно, изменения откатываются, шаг помечается как failed.

### 15.5 Rollback after approval rejection

Если на точке human approval пользователь отклонил изменения:

1. Execution Engine инициирует rollback к снапшоту до шагов, требующих approval.
2. Workspace восстанавливается.
3. Execution Engine может перепланировать (Planner) или отменить Run.

### 15.6 What remains persisted after discard

При Discard:

- **Сохраняется:**
  - Artifact Area (`.client/artifacts/`) — логи, отчёты, сгенерированные файлы. Они перемещаются в постоянное файловое хранилище.
  - Pre-Run Snapshot (до истечения grace period) — для аудита.
  - Метаданные Workspace в PostgreSQL (для истории).
  - Все события в Event Log (неудаляемые записи).
  - Pre-Merge Snapshot (если был создан до Discard).

- **Уничтожается:**
  - Mutable Layer (все изменённые, созданные, удалённые файлы).
  - Temporary Area (`.client/tmp/`).
  - Intermediate Snapshots (после завершения grace period).
  - Физическая директория `rootPath`.

### 15.7 What must be destroyed

При любом Discard гарантированно уничтожается:

- Физическая копия файлов в `rootPath`.
- Все временные данные в `.client/tmp/`.
- Любые данные, которые могут содержать секреты (если они не перемещены в Artifact Manager).

Уничтожение выполняется Cleanup Manager с верификацией: после удаления проверяется, что `rootPath` больше не существует или пуст.

## 16. Artifact Staging

### 16.1 Как Workspace хранит промежуточные и итоговые artifacts

Artifact Staging Manager организует хранение артефактов в структурированной форме:

```
{rootPath}/
├── .client/
│   ├── artifacts/
│   │   ├── logs/
│   │   │   ├── execution.log
│   │   │   ├── step_{stepId}.log
│   │   │   └── agent_{agentId}.log
│   │   ├── reports/
│   │   │   ├── lint-report.json
│   │   │   ├── test-report.json
│   │   │   └── review-report.md
│   │   ├── generated/
│   │   │   ├── {file1}
│   │   │   └── {file2}
│   │   └── staging/
│   │       ├── pre-merge-diff.patch
│   │       └── intermediate-results/
│   ├── tmp/
│   │   └── (временные файлы)
│   └── meta/
│       ├── workspace.json
│       └── operations.log
```

### 16.2 Где лежат execution artifacts

**Execution artifacts** — это артефакты, созданные в процессе выполнения шагов:

- **Логи выполнения:** `.client/artifacts/logs/execution.log` (общий лог) и `step_{stepId}.log` (логи конкретных шагов).
- **Отчёты валидации:** `.client/artifacts/reports/lint-report.json`, `test-report.json`.
- **Результаты AI Tasks:** `.client/artifacts/reports/ai-response-{id}.json`.
- **Сгенерированные файлы:** `.client/artifacts/generated/` — файлы, созданные Agent-ами как итоговый результат (новые исходные файлы).

### 16.3 Где лежат logs

Все логи хранятся в `.client/artifacts/logs/`:

- **Execution log:** Общий лог всего Run, агрегирующий события всех шагов.
- **Step logs:** Детальные логи каждого шага.
- **Agent logs:** Логи действий каждого Agent-а.
- **Tool invocation logs:** Результаты вызовов инструментов (stdout, stderr, exit code).

Логи хранятся в текстовом формате для удобства чтения человеком и машиной. При переносе в постоянное хранилище логи индексируются для поиска.

### 16.4 Где лежат generated outputs

**Generated outputs** — файлы, созданные в результате выполнения Run, которые должны стать частью кодовой базы:

- Новые исходные файлы (до Merge).
- Сгенерированная документация.
- Скомпилированные артефакты (если применимо).
- Миграции баз данных.

Эти файлы находятся в Mutable Layer (в корне проекта, не в `.client/`), так как они **являются** частью изменений, которые будут слиты в репозиторий.

**Важно:** Generated outputs в `.client/artifacts/generated/` — это предварительные версии, не готовые к слиянию. После валидации они перемещаются в основную файловую структуру Workspace.

### 16.5 Как артефакты связываются с Task / Run / Knowledge

- **Привязка к Task:** Все артефакты в Workspace маркируются `taskId`. При сохранении в постоянное хранилище эта привязка сохраняется.
- **Привязка к Run:** Артефакты маркируются `runId`. Если Run повторяется (retry), артефакты разных попыток различаются по `runId`.
- **Привязка к Knowledge:** Execution Report (ключевой артефакт) передаётся в Knowledge после успешного Run. Knowledge Manager извлекает из него знания (Lessons Learned, Best Practices, ADR).
- **Привязка к Graph:** Изменённые файлы (результат Run) после Merge индексируются Indexer-ом и попадают в Graph. Связь с Run сохраняется через `graphVersion` (Git commit hash, созданный при Merge).

### 16.6 Что временно, а что идёт в permanent storage

| Артефакт | Временный / Постоянный | Назначение |
|----------|------------------------|------------|
| **Execution Report** | Постоянный | Сохраняется в Knowledge, используется для будущих Run |
| **Логи выполнения** | Постоянный (с retention) | Хранятся в Loki/File Storage для аудита и отладки |
| **Отчёты валидации** | Временный (до завершения Run + grace) | Нужны для отладки, теряют актуальность после Merge |
| **Промежуточные результаты AI Tasks** | Временный | Нужны для трассировки, уничтожаются после завершения Run |
| **Pre-Run Snapshot** | Временный (до завершения Run + 24h) | Для возможности rollback |
| **Intermediate Snapshots** | Временный (до завершения Run) | Для возможности rollback |
| **Сгенерированные файлы (исходный код)** | Постоянный | Становятся частью кодовой базы после Merge |
| **Сгенерированные файлы (временные)** | Временный | Уничтожаются Cleanup Manager |
| **Метаданные Workspace** | Постоянный | Хранятся в PostgreSQL для аудита |

## 17. State and Observability

### 17.1 Workspace state tracking

State Manager отслеживает и сохраняет следующую информацию о состоянии Workspace:

- Текущее состояние жизненного цикла (Created, Initialized, Active, ...).
- Прогресс выполнения: какой шаг плана выполняется, сколько шагов завершено.
- Количество изменённых файлов (total, added, modified, deleted).
- Размер Mutable Layer.
- Количество активных операций (чтения, записи в прогрессе).
- Флаги: `hasErrors`, `hasWarnings`, `isDirty`.
- Время последней активности.
- Привязка к Agent-ам: какие Agent-ы активны в данный момент.

State Manager обновляет состояние синхронно с операциями и публикует изменения через `WorkspaceStateUpdated` (для UI-дашборда).

### 17.2 Operation logs

Каждая файловая операция логируется с полным контекстом:

```
{
  operationId: UUID,
  workspaceId: UUID,
  runId: UUID,
  taskId: UUID,
  stepId: UUID,
  agentId: UUID,
  operation: "read" | "write" | "create" | "delete" | "move",
  path: "/relative/path",
  timestamp: datetime,
  duration: ms,
  status: "success" | "failure",
  errorDetails: (если ошибка),
  metadata: {
    fileSize: bytes,
    previousHash: SHA,
    newHash: SHA
  }
}
```

Логи операций хранятся в `.client/meta/operations.log` в рамках Workspace. После завершения Run они переносятся в постоянное хранилище.

### 17.3 File change tracking

Filesystem Manager отслеживает все изменения файлов в Mutable Layer:

- **Список изменённых файлов:** Какие файлы были изменены, созданы, удалены.
- **Diff для каждого файла:** Разница между состоянием в Base Revision и текущим состоянием.
- **Агрегация по шагам:** Какие файлы были изменены на каком шаге плана.
- **Агрегация по Agent-ам:** Какой Agent изменил какие файлы.

Эта информация используется:
- Execution Engine для отображения прогресса.
- Reviewer Agent для review изменений.
- Merge Manager для определения, что именно сливать.

### 17.4 Run binding visibility

Workspace обеспечивает полную видимость своей привязки к Run:

- `workspaceId` ↔ `runId` ↔ `taskId` — цепочка доступна в любой момент.
- По `runId` можно найти все файловые изменения, логи операций и артефакты.
- По `taskId` можно найти все Workspace-ы, созданные для этой Task (включая retry).
- По `workspaceId` можно найти `baseCommit` и моментально определить состояние кодовой базы на старте.

### 17.5 Progress visibility

Для внешних потребителей (UI, Execution Engine, Task Manager) Workspace предоставляет:

- **Текущее состояние:** В каком состоянии жизненного цикла находится Workspace.
- **Прогресс Run:** Какой шаг выполняется, сколько шагов завершено, сколько осталось.
- **Статистика изменений:** Количество изменённых файлов по категориям.
- **Активные Agent-ы:** Какие Agent-ы работают в данный момент.
- **Последняя активность:** Временная метка последней операции.

Эта информация обновляется в реальном времени через события и доступна через Workspace State API (предоставляемый State Manager).

### 17.6 Diagnostics

Diagnostics and Statistics Manager собирает и публикует:

- **Метрики производительности:**
  - Время provisioning (от создания до Initialized).
  - Время выполнения отдельных операций (среднее, медиана, p95, p99).
  - Пропускная способность: операций в секунду.
  - Количество активных Workspace в системе.

- **Метрики здоровья:**
  - Количество ошибок (по типам).
  - Количество предупреждений.
  - Количество нарушений sandbox-политик.
  - Количество конфликтов при Merge.

- **Метрики использования ресурсов:**
  - Размер Workspace (текущий, средний, максимум).
  - Дисковое пространство, занятое всеми Workspace.
  - Размер Mutable Layer.
  - Количество снапшотов и их суммарный размер.

- **Аномалии:**
  - Workspace с аномально большим размером.
  - Workspace с аномально долгим временем provisioning.
  - Workspace с аномально большим количеством операций.

### 17.7 Auditability

Workspace обеспечивает полный аудит всех операций:

- **Кто:** `agentId`, `runId`, `taskId` (все операции привязаны).
- **Когда:** Временная метка каждой операции.
- **Что:** Тип операции, путь, хеши до и после.
- **Почему:** В контексте какого шага плана (`stepId`) и задачи (`taskId`).
- **Результат:** Успех/неудача, ошибки.

Аудиторский след (audit trail) хранится в:
- Операционных логах (`.client/meta/operations.log`).
- Event Log (события о значимых изменениях состояния).
- Метаданных Workspace в PostgreSQL.

### 17.8 Traceability to task/run/plan

Каждый артефакт, каждое изменение, каждое событие в Workspace имеет полную цепочку трассировки:

```
taskId → planId → runId → workspaceId → stepId → agentId → operationId
```

Это позволяет:
- Для любого изменения в коде найти: какая задача, какой план, какой Run, какой шаг, какой Agent его создал.
- Для любой задачи найти все созданные Workspace, все изменения, все артефакты.
- Восстановить полную историю: почему этот код выглядит именно так.

## 19. Interaction with Execution Engine

### 19.1 Как Execution использует Workspace

Execution Engine использует Workspace как **единственную среду выполнения** для всех Agent-ов и инструментов.

**Точки взаимодействия:**

1. **Pre-flight:** Execution Engine запрашивает создание Workspace через `WorkspaceCoordinator.Create(workspaceSpec)`. Ждёт событие `WorkspaceInitialized`.

2. **Выполнение шагов:** Для каждого шага плана:
   - Execution Engine загружает контекст шага.
   - Agent выполняет операции через Filesystem Manager (чтение/запись файлов).
   - Инструменты вызываются в sandbox, но их файловые операции также проходят через Filesystem Manager.
   - Execution Engine получает события `FileChanged` для отслеживания прогресса.

3. **Checkpoint:** Execution Engine запрашивает создание Intermediate Snapshot через `SnapshotManager.CreateSnapshot()`.

4. **Rollback:** При сбое шага Execution Engine запрашивает восстановление из снапшота.

5. **Merge:** После успешного выполнения всех шагов Execution Engine запрашивает Merge (`MergeManager.Merge()`).

6. **Discard:** При отмене или ошибке Execution Engine запрашивает Discard (`CleanupManager.Discard()`).

### 19.2 Какие гарантии Workspace даёт Execution

Workspace предоставляет Execution Engine следующие гарантии:

1. **Изоляция:** Никакой другой Run не повлияет на файлы этого Workspace.
2. **Атомарность операций:** Запись файла либо завершается полностью, либо не применяется.
3. **Восстанавливаемость:** При сбое состояние можно восстановить из снапшота.
4. **Безопасность:** Операции, нарушающие sandbox-политики, блокируются.
5. **Аудируемость:** Все операции логируются.
6. **Воспроизводимость:** При одинаковых параметрах создаётся идентичная стартовая среда.
7. **Целостность:** Повреждение файлов обнаруживается и сигнализируется.
8. **Готовность:** После `WorkspaceInitialized` среда полностью готова к работе.

### 19.3 Что Execution не должен делать в обход Workspace

**Запрещено для Execution Engine и Agent-ов:**

- Прямой доступ к файловой системе хоста (минуя Filesystem Manager).
- Прямые Git-операции (clone, commit, push) — только через Repository Bridge.
- Чтение/запись файлов за пределами `rootPath`.
- Модификация `.client/` директории.
- Обход sandbox-политик (изменение `allowedPaths` или `forbiddenPatterns` в процессе Run).
- Изменение `baseCommit` после инициализации.
- Создание файлов, соответствующих `forbiddenPatterns`.
- Выполнение destructive-операций без создания снапшота (если план этого требует).

### 19.4 Как approval points и rollback points используют Workspace

**Approval points:**
- Когда Execution Engine достигает точки human approval, он приостанавливает выполнение.
- Workspace остаётся в состоянии `Active`, но новые изменения не вносятся (Agent-ы остановлены).
- Если approval получено — выполнение продолжается.
- Если approval отклонено — Execution Engine запрашивает rollback к снапшоту до шагов, требующих approval.

**Rollback points:**
- Execution Plan определяет точки, в которых должен быть создан Intermediate Snapshot (rollback points).
- Перед выполнением рискованного шага Execution Engine запрашивает создание снапшота.
- После завершения шага и его валидации снапшот может быть удалён (если шаг успешен) или использован для отката (если шаг неудачен).

### 19.5 Как Workspace поддерживает deterministic execution

Детерминированное выполнение — это гарантия, что повторный Run с теми же входными данными даст тот же результат.

**Поддержка со стороны Workspace:**
- Фиксация `baseCommit` обеспечивает идентичное стартовое состояние.
- Pre-Run Snapshot позволяет побитово сравнить состояние "до".
- Все операции логируются с temporal ordering (временные метки), что позволяет воспроизвести последовательность.
- Изоляция предотвращает влияние внешних факторов.
- Детерминированное provisioning: одинаковые параметры → идентичная среда.

**Ограничения:**
- Внешние зависимости (пакеты) могут измениться между Run-ами. Требуется lock-файл.
- Инструменты могут иметь недерминированное поведение (например, линтер с учётом времени). Это зона ответственности Execution Engine, не Workspace.

## 20. Interaction with Indexer, Graph and Knowledge

### 20.1 Как изменения в Workspace попадают в Indexer

Indexer **не** получает уведомления о каждом `FileChanged` в процессе Run. Это предотвратило бы индексацию незавершённых, потенциально некорректных изменений.

**Поток данных после Merge:**

1. `WorkspaceMerged` публикуется с полным списком изменённых файлов (`changedFiles[]`).
2. Indexer подписан на `WorkspaceMerged`.
3. Indexer получает список файлов и запускает инкрементальную индексацию только этих файлов.
4. Indexer публикует `FileIndexed` для каждого проиндексированного файла.
5. Indexer публикует `IndexCompleted`, когда все файлы обработаны.

**Почему не во время Run:**
- Изменения в процессе Run могут быть незавершёнными (промежуточное состояние).
- Код может быть синтаксически некорректным до завершения шага.
- Индексация незавершённого состояния приведёт к "шуму" в Graph и Knowledge.

### 20.2 Как после merge запускается reindex

Reindex — это процесс актуализации Graph после изменений в кодовой базе.

**Процесс:**
1. `WorkspaceMerged` → Indexer получает `changedFiles[]`.
2. Indexer анализирует, какие символы были изменены (на основе путей файлов).
3. Indexer индексирует только изменённые файлы (инкрементально), а не весь проект.
4. Indexer публикует `FileIndexed` для каждого файла.
5. Graph получает `FileIndexed` и обновляет соответствующие узлы и рёбра.
6. Graph вычисляет новую `graphVersion` (привязанную к Git commit hash, созданному при Merge).
7. Graph публикует `GraphUpdated`.

### 20.3 Как это влияет на GraphUpdated

`GraphUpdated` — ключевое событие, сигнализирующее об изменении структуры кодовой базы.

**Цепочка:**
```
WorkspaceMerged → Indexer → FileIndexed → Graph → GraphUpdated
```

`GraphUpdated` содержит:
- `graphVersion` — новая версия графа.
- `changedNodes[]` — узлы, которые были добавлены, изменены или удалены.
- `changedEdges[]` — рёбра, которые были добавлены, изменены или удалены.

**Downstream-эффекты `GraphUpdated`:**
- Knowledge проверяет freshness/staleness знаний, связанных с изменёнными узлами.
- Impact Analysis Engine может пересчитать зоны влияния для активных задач.
- Context Builder инвалидирует закэшированные контексты, связанные с изменёнными сущностями.

### 20.4 Как это влияет на freshness/staleness в Knowledge

**Freshness** — мера актуальности Knowledge относительно текущего состояния кодовой базы (Graph).

**После `GraphUpdated`:**
1. Knowledge Manager получает `GraphUpdated` и проверяет все Knowledge, связанные с изменёнными узлами.
2. Если Knowledge ссылается на узел, который был изменён, Knowledge помечается как `potentially_stale`.
3. Если Knowledge ссылается на узел, который был удалён, Knowledge помечается как `definitely_stale`.
4. Knowledge Manager публикует `KnowledgeStalenessUpdated` для затронутых Knowledge.
5. Context Builder при построении контекста учитывает freshness-статус Knowledge.

**Связь с Workspace:**
- Именно успешный Merge в Workspace запускает цепочку, которая заканчивается обновлением freshness/staleness в Knowledge.
- Если Merge не удался, цепочка не запускается — Knowledge остаётся в актуальном состоянии.

### 20.5 Какие события связывают эти модули

```
Workspace → WorkspaceMerged ─────► Indexer (reindex)
                                       │
                                       ▼
                                  FileIndexed ─────► Graph (update)
                                                       │
                                                       ▼
                                                  GraphUpdated ──► Knowledge (staleness check)
                                                                      │
                                                                      ▼
                                                              KnowledgeStalenessUpdated ──► Context Builder
```

**Контракты:**
- `WorkspaceMerged` публикуется **только** после успешного коммита в репозиторий.
- `WorkspaceMerged` содержит полный список изменённых файлов — Indexer не должен гадать.
- `GraphUpdated` содержит список изменённых узлов — Knowledge не должен сканировать весь граф.
- Каждое событие в цепочке содержит `correlation_id = runId` для сквозной трассировки.

## 21. Performance

### 21.1 Fast provisioning

**Цели производительности:**
- Provisioning должен занимать минимальное время, чтобы не задерживать начало Run.
- Для типового проекта среднего размера (~10,000 файлов, ~200MB) provisioning должен завершаться за секунды, а не минуты.

**Механизмы:**
- Использование Stable Snapshot (если доступен) вместо Git checkout. Восстановление из снапшота значительно быстрее.
- Кэширование снапшотов: если несколько Run стартуют от одного `baseCommit`, Snapshot Manager переиспользует существующий снапшот.
- Ленивое копирование (см. copy-on-write) — не копировать все файлы физически.
- Параллельное копирование: файлы копируются в несколько потоков.
- Исключение крупных артефактов: `node_modules/`, `vendor/` не копируются при provisioning (устанавливаются отдельно).

### 21.2 Copy-on-write или эквивалентная стратегия

**Архитектурный подход: Copy-on-Write (CoW).**

Вместо полного физического копирования всех файлов Base Revision в `rootPath`, используется стратегия, при которой:

- **Base Revision** хранится как неизменяемый слой (в файловой системе с поддержкой CoW или через символические ссылки/жёсткие ссылки).
- **Mutable Layer** существует как отдельный слой поверх Base Revision.
- При чтении файла: если файл не изменён, читается из Base Revision. Если изменён — из Mutable Layer.
- При записи файла: файл копируется из Base Revision в Mutable Layer и затем изменяется.

**Преимущества:**
- Provisioning практически мгновенный: не нужно копировать тысячи файлов.
- Экономия дискового пространства: множественные Workspace от одного `baseCommit` разделяют один Base Revision.
- Workspace создаётся за O(1) вместо O(files).

**Ограничения на уровне архитектуры:**
- Конкретная реализация CoW зависит от возможностей файловой системы (APFS на macOS, Btrfs на Linux, OverlayFS в Docker).
- Архитектура не фиксирует конкретную технологию, но фиксирует **принцип**: Base Revision неизменяем и разделяем между Workspace-ами, Mutable Layer — индивидуален.

### 21.3 Snapshot reuse

- **Pre-Run Snapshot:** Если уже существует для данного `baseCommit`, не создаётся заново.
- **Intermediate Snapshot:** Создаётся инкрементально (сохраняется только diff относительно предыдущего снапшота).
- **Stable Snapshot:** Используется для provisioning будущих Workspace. Новые Run от того же коммита используют этот снапшот.

**Cache invalidation:**
- Снапшот инвалидируется, если обнаружено расхождение с `baseCommit` (например, Git garbage collection изменил историю).

### 21.4 Incremental change tracking

Вместо полного сравнения файлов при каждой операции, Workspace отслеживает изменения инкрементально:

- Каждая операция записи немедленно регистрируется в журнале изменений.
- Журнал изменений используется для:
  - Быстрого вычисления списка изменённых файлов (без сканирования всего `rootPath`).
  - Создания Intermediate Snapshot (только изменённые файлы, а не полная копия).
  - Merge (известно, какие файлы изменены — не нужно сравнивать весь проект).

### 21.5 Artifact caching

- Artifact Staging Manager проверяет, не был ли уже создан идентичный артефакт в предыдущем Run (по хешу содержимого).
- Если артефакт идентичен, он не дублируется — сохраняется ссылка на существующий.
- Это экономит место при повторных Run (retry) или параллельных Run с одинаковыми результатами.

### 21.6 Cleanup policy

Cleanup Manager работает асинхронно, не блокируя активные Workspace:

- **Transient Workspace:** Очищается немедленно после Discard. Артефакты копируются в постоянное хранилище за константное время.
- **Archived Workspace:** Физические файлы удаляются сразу. Метаданные в PostgreSQL хранятся согласно Retention Policy.
- **Снапшоты:** Удаляются по истечении retention period. Очистка запускается периодически (cron), а не при каждом Discard.
- **Орфанные (orphaned) Workspace:** Если Run был прерван аварийно и Cleanup не сработал, периодическая проверка находит и очищает орфанные Workspace.

### 21.7 Storage optimization

- **Дедупликация:** Файлы, идентичные Base Revision, не занимают дополнительного места (CoW или символические ссылки).
- **Сжатие:** Снапшоты и артефакты могут храниться в сжатом виде (если это оправдано политикой хранения).
- **Tiered storage:** Горячие данные (активные Workspace) — на быстром хранилище. Холодные данные (старые снапшоты) — на медленном, дешёвом.
- **Лимиты:** Каждый Workspace имеет лимит размера. При превышении — `WorkspaceSizeExceeded`. Это предотвращает захламление хранилища одним Run.

## 22. Failure Handling

### 22.1 Workspace не удалось создать

**Причины:**
- Git checkout не удался (репозиторий недоступен, коммит не существует).
- Недостаточно дискового пространства.
- Нарушение прав доступа (нет прав на запись в директорию Workspace).
- Конфликт с существующим Workspace (неуникальный `rootPath`).
- Некорректные sandbox-политики (указаны несуществующие `allowedPaths`).

**Обработка:**
1. Provisioning останавливается.
2. Workspace переходит в состояние `Failed`.
3. Публикуется `WorkspaceProvisioningFailed`.
4. Execution Engine получает уведомление и отменяет Run.
5. Частично созданные файлы удаляются Cleanup Manager.
6. Ошибка логируется с полным контекстом для диагностики.

### 22.2 Snapshot не создался

**Причины:**
- Недостаточно дискового пространства.
- Ошибка ввода-вывода (дисковая ошибка).
- Слишком большой Workspace (превышен лимит размера снапшота).
- Системная ошибка процесса создания снапшота.

**Обработка:**
1. Snapshot Manager сообщает об ошибке Workspace Coordinator.
2. Если это Pre-Run Snapshot — Run продолжается без снапшота, но rollback capability теряется (Execution Engine решает, допустимо ли это).
3. Если это Intermediate Snapshot — попытка создания повторяется. При повторной неудаче — Run продолжается без этой точки восстановления.
4. Если это Pre-Merge Snapshot — Merge блокируется до успешного создания снапшота.
5. Публикуется предупреждение (warning), не ошибка (error), если Run может продолжаться.

### 22.3 Merge завершился ошибкой

**Причины:**
- Конфликт с новыми коммитами в Merge Target.
- Нарушение pre-merge validation (структурный конфликт, forbidden files).
- Ошибка Git-операции (репозиторий временно недоступен).
- Неудачная попытка автоматического разрешения конфликта.

**Обработка:**
1. Merge Manager публикует `WorkspaceMergeFailed`.
2. Workspace остаётся в состоянии `Active`.
3. Execution Engine получает уведомление.
4. В зависимости от причины:
   - Конфликт: Execution Engine запрашивает Replan у Planner или manual resolution у пользователя.
   - Временная ошибка Git: автоматический retry с backoff.
   - Структурный конфликт: Merge блокируется, требуется ручное вмешательство.
5. Pre-Merge Snapshot сохраняется для возможного ручного слияния.
6. Информация о конфликте сохраняется в Knowledge (для анализа паттернов конфликтов).

### 22.4 Sandbox policy нарушена

**Причины:**
- Agent попытался записать файл в запрещённую директорию.
- Agent попытался удалить защищённый файл.
- Инструмент попытался выйти за пределы Workspace.
- Обнаружены файлы, соответствующие `forbiddenPatterns`.

**Обработка:**
1. Операция блокируется до выполнения.
2. Публикуется `WorkspacePolicyViolation`.
3. В зависимости от severity:
   - **Warning:** Операция отклонена, выполнение продолжается (Agent получает ошибку и может попробовать другой подход).
   - **Error:** Шаг помечается как failed. Execution Engine инициирует rollback к Intermediate Snapshot.
   - **Critical:** Workspace переходит в `Failed`. Run отменяется. Требуется анализ: нарушение — результат ошибки Agent-а или злонамеренной попытки.

### 22.5 Filesystem inconsistent

**Причины:**
- Хеш-суммы файлов не совпадают с ожидаемыми (повреждение данных).
- Файлы, которые должны существовать (из Base Revision), отсутствуют.
- Файловая система повреждена на уровне ОС.

**Обработка:**
1. Diagnostics Manager обнаруживает неконсистентность при очередной проверке.
2. Публикуется `WorkspaceValidationFailed`.
3. Workspace переходит в `Failed`.
4. Recovery Manager пытается восстановить:
   - Из Pre-Run Snapshot (возврат к началу).
   - Из Intermediate Snapshot (возврат к последней стабильной точке).
   - Если снапшотов нет — Workspace пересоздаётся заново из Base Revision.
5. Если восстановление невозможно — Run отменяется.

### 22.6 Repository state changed unexpectedly

**Причины:**
- В процессе Run кто-то (другой пользователь, CI/CD) изменил Merge Target (новые коммиты).
- Git-репозиторий был переписан (force push, история изменена).
- `baseCommit` больше не существует в репозитории (git gc, force push).

**Обработка:**
1. Pre-Merge Validation обнаруживает расхождение.
2. Если новые коммиты в Merge Target:
   - Merge Manager вычисляет конфликты.
   - Если конфликтов нет — Merge продолжается.
   - Если конфликты есть — `WorkspaceMergeFailed` с указанием конфликтующих файлов.
3. Если `baseCommit` не существует:
   - Критическая ошибка. Workspace не может быть слит.
   - Run отменяется. Инцидент логируется для расследования.

### 22.7 Cleanup failed

**Причины:**
- Ошибка ввода-вывода при удалении файлов.
- Файлы заблокированы другим процессом.
- Недостаточно прав для удаления.

**Обработка:**
1. Cleanup Manager повторяет попытку с backoff.
2. Если повторные попытки не удались:
   - Частично очищенный Workspace помечается для отложенной очистки.
   - Периодическая задача (cron) подбирает такие Workspace и пытается очистить снова.
   - Если и это не удаётся — административное уведомление.
3. Метаданные в PostgreSQL обновляются (состояние `CleanupFailed`).

### 22.8 Recovery partially succeeded

**Причины:**
- Только часть файлов восстановлена из снапшота.
- Некоторые артефакты потеряны.

**Обработка:**
1. Recovery Manager публикует `WorkspaceRecoveryCompleted` с флагом `partialSuccess`.
2. Execution Engine получает информацию о потерянных данных.
3. Execution Engine принимает решение:
   - Продолжить выполнение (если потерянные данные не критичны).
   - Отменить Run (если потеряны критически важные изменения).
4. Информация о частичном восстановлении сохраняется в Knowledge (для анализа надёжности системы).

## 23. Ограничения

### 23.1 Когда Workspace обязан остановить выполнение

Workspace обязан заблокировать дальнейшие операции и перейти в состояние `Failed` в следующих случаях:

| Ситуация | Причина | Действие |
|----------|---------|----------|
| Обнаружена неконсистентность файловой системы (хеши не совпадают) | Повреждение данных | Остановка, `WorkspaceValidationFailed`, попытка восстановления |
| Критическое нарушение sandbox-политики | Потенциальная угроза безопасности | Остановка, `WorkspacePolicyViolation(critical)`, Run отменяется |
| Попытка записи за пределами `rootPath` | Directory traversal атака | Остановка, инцидент безопасности |
| `baseCommit` больше не существует в репозитории | Изменение истории Git | Остановка, Run не может быть слит |
| Размер Workspace превысил жёсткий лимит (не warning-лимит, а hard limit) | Исчерпание ресурсов | Остановка операций записи, требуется вмешательство |
| Обнаружена попытка подмены Base Revision | Целостность под угрозой | Немедленная остановка, аудит |

### 23.2 Когда Workspace обязан запретить merge

Merge должен быть заблокирован в следующих случаях:

- Pre-Merge Validation не пройдена (конфликты с Merge Target).
- Workspace находится в состоянии `Failed`.
- Есть несохранённые изменения (Mutable Layer не синхронизирован с журналом изменений).
- Pre-Merge Snapshot не удалось создать.
- Обнаружены файлы, соответствующие `forbiddenPatterns`, в списке изменённых.
- Reviewer Agent не подтвердил качество изменений (если план требует review).
- Не все approval points пройдены.
- `baseCommit` не соответствует актуальному состоянию репозитория (force push).

### 23.3 Когда Workspace обязан форсировать rollback

Rollback должен быть выполнен в следующих случаях:

- Шаг Execution Plan завершился с ошибкой, и план предписывает rollback.
- Validation Engine обнаружил, что изменения не проходят критические проверки.
- Human approval отклонено на контрольной точке.
- Обнаружена неконсистентность файловой системы, и восстановление из снапшота возможно.
- Критическое нарушение sandbox-политики, и требуется откат к состоянию до нарушения.

### 23.4 Когда Workspace обязан опубликовать warning/error event

| Ситуация | Тип события | Severity |
|----------|------------|----------|
| Provisioning не удался | `WorkspaceProvisioningFailed` | Error |
| Snapshot не создался (некритично) | `SnapshotCreationWarning` | Warning |
| Обнаружено незначительное нарушение sandbox-политики | `WorkspacePolicyViolation` | Warning |
| Размер Workspace приближается к лимиту | `WorkspaceSizeExceeded` | Warning |
| Обнаружена неконсистентность (восстановимая) | `WorkspaceIntegrityWarning` | Warning |
| Обнаружена неконсистентность (невосстановимая) | `WorkspaceValidationFailed` | Error |
| Merge не удался | `WorkspaceMergeFailed` | Error |
| Cleanup не удался | `WorkspaceCleanupFailed` | Warning |
| Обнаружен потенциальный секрет в артефактах | `WorkspaceSecretExposureWarning` | Warning |
| Операция заблокирована политикой | `WorkspacePolicyViolation` | Warning/Error (зависит от severity) |

### 23.5 Когда Workspace обязан отказать в операции как unsafe

Операция считается **unsafe** и должна быть отклонена, если:

- Путь назначения выходит за пределы `rootPath`.
- Файл соответствует `forbiddenPatterns`.
- Операция удаления затрагивает `protectedPaths`.
- Операция записи затрагивает `readOnlyPaths`.
- Операция destructive (delete, перезапись) и отсутствует Pre-Run Snapshot.
- Операция требует прав, которые не предоставлены Agent-у.
- Операция нарушает safety bounds, определённые Execution Plan.

**Принцип:** Если операция не является явно разрешённой, она запрещена (default-deny). Workspace не пытается "угадать" намерения Agent-а.

## 24. Будущее развитие

### 24.1 Новые sandbox strategies

Текущая архитектура предполагает файловую изоляцию. В будущем могут быть добавлены:

- **Process-level sandbox:** Изоляция на уровне процессов (каждый Agent в отдельном контейнере). Workspace предоставляет файловый слой, Execution Engine — процессный.
- **Network sandbox:** Ограничение сетевого доступа из Workspace (разрешённые endpoints, запрет внешних соединений).
- **Resource quota sandbox:** Ограничение CPU, памяти, дискового пространства для каждого Workspace в реальном времени.
- **Time-based sandbox:** Ограничение времени жизни Workspace (TTL), после которого операции автоматически блокируются.

Все эти стратегии добавляются как слои поверх существующей модели изоляции без изменения архитектуры Workspace.

### 24.2 Remote workspaces

Сейчас Workspace создаётся на том же хосте, где работает Execution Engine. В будущем:

- **Remote Workspace:** Файловая среда размещается на удалённой машине (например, мощный build-сервер).
- **Распределённый Workspace:** Разные части проекта распределены по разным хостам (монорепозиторий).
- **Workspace as a Service:** API для создания и управления Workspace, независимое от Execution Engine.

Архитектура Workspace уже поддерживает эту возможность: Filesystem Manager абстрагирует доступ к файлам, а `rootPath` может быть сетевым путём.

### 24.3 Containerized workspaces

- **Docker-native Workspace:** Каждый Workspace — это Docker-контейнер с файловой системой, изолированной на уровне ядра.
- **OverlayFS:** Использование Docker OverlayFS для эффективного CoW между Base Revision (image layer) и Mutable Layer (container layer).
- **Podman/Kubernetes:** Поддержка альтернативных сред контейнеризации.

### 24.4 Ephemeral cloud workspaces

- **Serverless Workspace:** Среда создаётся на время Run и уничтожается немедленно после завершения (полная эфемерность).
- **Cloud-native:** Workspace размещается в облаке (AWS EBS, GCP Persistent Disk) с автоматическим масштабированием хранилища.
- **Spot-instance friendly:** Возможность быстро сохранить состояние Workspace (Snapshot) и восстановить на другой машине при прерывании spot-instance.

### 24.5 Richer artifact staging

- **Artifact pipeline:** Артефакты проходят через pipeline обработки (валидация, преобразование, индексация) перед сохранением в Knowledge.
- **Artifact versioning:** Версионирование артефактов для отслеживания изменений между Run-ами.
- **Artifact dependency graph:** Артефакты могут зависеть друг от друга. Например, Execution Report зависит от всех промежуточных отчётов.
- **Artifact promotion:** Возможность "продвинуть" артефакт из временного хранилища Workspace в постоянное Knowledge с явным указанием значимости.
- **Artifact comparison:** Сравнение артефактов между Run-ами для выявления паттернов и аномалий.

### 24.6 Policy plugins

- **Плагинная архитектура sandbox-политик:** Возможность создавать кастомные политики безопасности без изменения ядра Workspace.
- **Декларативные политики:** Политики описываются в конфигурационных файлах, а не в коде.
- **Policy as Code:** Версионирование политик вместе с кодом проекта.
- **Inheritance политик:** Политики наследуются от проекта → задачи → Run, с возможностью переопределения на каждом уровне.

### 24.7 Workspace templates

- **Pre-configured Workspace:** Шаблон с преднастроенными инструментами, зависимостями и конфигурацией для конкретного типа задач.
- **Language-specific templates:** Шаблон для PHP-проекта, Node.js-проекта, Python-проекта с соответствующими инструментами.
- **Task-type templates:** Шаблон для рефакторинга, для фикса багов, для написания тестов.
- **Project-specific templates:** Шаблон, учитывающий специфику конкретного проекта (структура директорий, используемые фреймворки).

Шаблоны ускоряют provisioning, так как не нужно каждый раз устанавливать зависимости и настраивать окружение.

## 25. Заключение

### 25.1 Workspace — это не просто техническая деталь

Workspace является **обязательным инфраструктурным слоем**, без которого невозможны ключевые свойства системы Client:

1. **Безопасность выполнения.** Без Workspace Agent-ы имели бы прямой доступ к репозиторию, и любой сбой или ошибка могли бы привести к необратимому повреждению кодовой базы. Workspace изолирует "грязную" работу от "чистого" репозитория.

2. **Воспроизводимость.** Без Workspace каждый Run мог бы давать разный результат в зависимости от состояния файловой системы в момент выполнения. Workspace фиксирует `baseCommit` и обеспечивает детерминированную стартовую среду, что является необходимым условием для воспроизводимости — одного из фундаментальных принципов Client.

3. **Параллельная работа.** Без Workspace параллельные задачи конфликтовали бы на уровне файловой системы, создавая гонки данных, трудноуловимые баги и непредсказуемое поведение. Workspace обеспечивает полную изоляцию, позволяя системе масштабироваться на множество одновременных Run.

4. **Аудит и traceability.** Без Workspace невозможно было бы однозначно связать каждое изменение в коде с конкретной задачей, планом, шагом и агентом. Workspace обеспечивает полную цепочку: `taskId → planId → runId → workspaceId → stepId → agentId → operationId → fileChange`.

5. **Контроль качества.** Без Workspace изменения попадали бы в репозиторий немедленно, без возможности проверки линтерами, тестами и reviewer-ом. Workspace реализует staged approach: изменения накапливаются изолированно, проходят многоступенчатую валидацию и только затем сливаются в основную кодовую базу.

6. **Отказоустойчивость.** Без Workspace и его снапшотов восстановление после сбоя требовало бы ручного вмешательства и анализа состояния репозитория. Workspace со Snapshot Strategy обеспечивает автоматический rollback к любому стабильному состоянию.

### 25.2 Место Workspace в архитектуре Client

Workspace — это один из трёх фундаментальных инфраструктурных слоёв, наряду с Event System и Storage Architecture. В отличие от других модулей (Indexer, Graph, Knowledge, Planner), которые реализуют бизнес-логику, Workspace обеспечивает **среду**, в которой эта бизнес-логика выполняется.

Без Workspace:
- Execution Engine не имеет среды для выполнения шагов.
- Agent-ы не имеют файловой системы для работы.
- Snapshot Strategy не имеет субъекта для снапшотирования.
- Indexer не получает сигнала об изменениях файлов.
- Knowledge не обновляет freshness/staleness.
- Вся архитектура Client теряет свою основу.

Workspace — это не "ещё один модуль". Это **канонический слой изоляции и управления средой выполнения**, который делает возможным безопасное, воспроизводимое и аудируемое автоматическое изменение кодовой базы.
